Section titled ‚ÄúDecoding Antigravity: An Agentic Workflow from Screenshot to Code Fix‚Äù

In the traditional AI-assisted programming (Copilot) model, developers often have to act as ‚Äúnavigators‚Äù: you tell the AI ‚Äúread this file,‚Äù ‚Äúexplain this code,‚Äù ‚Äúwrite a function for me.‚Äù Through conversation, you steer the AI to finish the work step by step.

Antigravity represents the next generation of Agentic AI. It is no longer a passive Q&A bot but an ‚Äúintelligent teammate‚Äù that can explore autonomously, reason proactively, and independently complete tasks in complex codebases.

This article uses a real bug-fix case to break down how Antigravity actually works.

1. Perception: More Than OCR (Perception)

Section titled ‚Äú1. Perception: More Than OCR (Perception)‚Äù

Everything starts with user feedback.

Suppose a user submits an issue (e.g., ISSUE-182):

Title: [Weice] blood glucose/uric acid combo meter (a combined glucometer and uric acid analyzer) uploads blood glucose data, but the database does not save it Description: Call POST /api/v2/users/records.json with a payload containing blood_glucose and uric_acid. Checking the database shows only uric acid is stored; blood glucose is missing.

After receiving the task, Antigravity first performs full-context perception:

  • Read the issue description: extract key information (API path, parameter structure, error symptom).
  • Analyze the tech stack: identify it as a Rails API project (backend) plus a Flutter app (frontend).
  • Locate the environment: determine that beyond code, it needs Postgres and Loki MCP servers to check logs and data.

It constructs a ‚Äútask graph‚Äù:

  • Goal: fix the missing uploaded data.
  • Entry point: backend API endpoint /api/v2/users/records.json.

2. Exploration: Reading the code like a senior engineer (Exploration)

Section titled ‚Äú2. Exploration: Reading the code like a senior engineer (Exploration)‚Äù

Antigravity does not guess; it uses its toolchain for purposeful exploration.

Step 2.1 Route locating

Section titled ‚ÄúStep 2.1 Route locating‚Äù

It starts like a human engineer, locating the code entry via the API path.

  • Tool Call: grep_search(query: "records", path: "config/routes.rb")
  • Result: finds the route mapped to Api::V2::Users::RecordsController#create.

Step 2.2 Following the trail

Section titled ‚ÄúStep 2.2 Following the trail‚Äù

Next, it digs into the controller code.

  • Tool Call: view_file(path: "app/controllers/api/v2/users/records_controller.rb")
  • Analysis: the logic shows it calls a RecordService.new(params).save service class.

3. Reasoning: Building a mental model (Reasoning)

Section titled ‚Äú3. Reasoning: Building a mental model (Reasoning)‚Äù

At this point, Antigravity is not just ‚Äúreading‚Äù code; it is running a thought simulation.

Facing the situation where blood_glucose is missing but uric_acid is stored, it quickly proposes several hypotheses:

  1. Hypothesis A: Was blood_glucose filtered out by valid_type?
  2. Hypothesis B: Did a validation error occur during DB write while the main transaction did not roll back?
  3. Hypothesis C: Is there a special filter for the [Weice] device type?

To validate the hypotheses, Antigravity proactively calls MCP tools to query real-time logs or the database:

  • Monitor: use loki_query to inspect API request logs and confirm the payload reached the server.
  • Check DB: use execute_sql to check the records table schema and confirm field definitions.

4. Execution: The precise scalpel (Action)

Section titled ‚Äú4. Execution: The precise scalpel (Action)‚Äù

Once the issue is located, Antigravity enters Execution Mode. It is meticulous:

  1. Draft a plan: Antigravity generates an implementation_plan.md listing the files and specific change points.
  2. Request approval: It shows the plan to the user: ‚ÄúI found issue X and plan to modify file Y. Do you approve?‚Äù
  3. Perform the edits: After approval, it uses the replace_file_content tool to precisely replace the problematic code block instead of rewriting the whole file.

5. Verification: Not just run, but verify (Verification)

Section titled ‚Äú5. Verification: Not just run, but verify (Verification)‚Äù

Shipping code is not the end. Antigravity knows the importance of testing the main path.

  • Automated tests: It runs rspec spec/services/record_service_spec.rb to ensure the bug is fixed without introducing regressions.
  • Generate a report: Finally, it creates a walkthrough.md with diffs and test-result screenshots to clearly show: ‚ÄúThe problem is solved, and here is the evidence.‚Äù

Conclusion: From co-pilot to co-worker

Section titled ‚ÄúConclusion: From co-pilot to co-worker‚Äù

Antigravity‚Äôs core value lies in agency.

  • It does not need you to feed it every line: give it a requirement, and it finds the relevant files itself.
  • It does not need you to act as its compiler: it runs tests, reads errors, and fixes them itself.
  • It is a closed loop: from discovering the issue to resolving it, it forms a complete workflow.

This is Antigravity - an AI partner that works alongside you, understands code and business context, and is built to solve problems.

Introduction to the Johari Window Model and Its Four Quadrants

Section titled ‚ÄúIntroduction to the Johari Window Model and Its Four Quadrants‚Äù

The Johari Window is a psychological model for improving self-awareness and interpersonal communication, proposed by psychologists Joseph Luft and Harrington Ingham in 1955. It divides personal cognition into four quadrants (or four ‚Äúwindows‚Äù):

  • Open Area (ÂºÄÊîæÂå∫): Information known to both yourself and others‚Äîpublic behaviors, attitudes, and traits. This is the ‚Äúyou know, I know‚Äù domain.

  • Blind Area (Áõ≤Âå∫): Information others can see but you overlook, such as behavioral traits apparent to others that you yourself don‚Äôt notice. This corresponds to ‚Äúyou know, I don‚Äôt‚Äù, i.e., your cognitive blind spots.

  • Hidden Area (ÈöêËóèÂå∫): Private information you know but others do not, like personal privacy or inner thoughts. This is the ‚Äúyou don‚Äôt know, I know‚Äù domain, also called the privacy zone.

  • Unknown Area (Êú™Áü•Âå∫): Potential information or traits unknown to both you and others, including untapped potential. This corresponds to the ‚Äúyou don‚Äôt know, I don‚Äôt‚Äù fully unknown domain.

The core idea is to expand the open area, reduce the blind and hidden areas, and explore the unknown area to enhance self-awareness and communication.

When mapping the Johari Window to human‚ÄìAI dialogue, we can treat ‚Äúself‚Äù as the human user and ‚Äúother‚Äù as the AI system. The axes of ‚Äúknowing/not knowing‚Äù become ‚Äúwhether the human knows‚Äù and ‚Äúwhether the AI knows.‚Äù The Johari quadrants are redefined in this context to frame human‚Äìmachine conversations. Each quadrant reflects a different human‚Äìmachine cognition setup: both know (open area), AI knows but the human doesn‚Äôt (blind area), the human knows but AI doesn‚Äôt (hidden area), and neither knows (unknown area).

Specifically, the four quadrants in human‚ÄìAI dialogue mean:

  • Open Area (human knows + AI knows): Information or context is transparent to both parties. Both share the same knowledge base and clear task premises‚Äîakin to a public knowledge domain.

  • Blind Area (human doesn‚Äôt know + AI knows): The AI holds information the user lacks. The user‚Äôs question is their blind spot, but the AI may have relevant knowledge or skills to provide an answer the user doesn‚Äôt know.

  • Unknown Area (human doesn‚Äôt know + AI doesn‚Äôt): A new problem domain where neither side has ready answers‚Äîinnovative exploration where both must discover together.

Dialogue Strategy Analysis for the Four Quadrants

Section titled ‚ÄúDialogue Strategy Analysis for the Four Quadrants‚Äù

For each quadrant, we examine suitable question types, prompting and response strategies, and the AI‚Äôs limitations. Different quadrants imply different knowledge relationships and call for adjusted dialogue tactics.

Open Area: Knowledge Is Transparent to Both Sides

Section titled ‚ÄúOpen Area: Knowledge Is Transparent to Both Sides‚Äù

The open area is ‚Äúthe human knows, the AI knows,‚Äù where both share context. Examples include having AI polish user-written text, translate a sentence, or answer a well-scoped question. Here the AI can act as a ‚Äúhigh-efficiency assistant.‚Äù

Suitable question types: Tasks with clear context and shared background‚Äîtext polishing, translation, format conversion, simple Q&A, and other routine tasks. Human and AI understand the requirements consistently, so communication costs are lowest.

Human prompting: State the task and requirements directly and clearly. Provide explicit instructions‚Äîtask, desired format, and constraints. AI response strategy: In the open area, AI is the executor: follow instructions and complete the task efficiently. With clear context, the AI can rely on its trained knowledge or skills to answer, typically with high accuracy and speed.

AI limitations: Performance is usually strong here. The risk is user over-description confusing the AI or producing verbosity (the AI tries to process redundant guidance). Conversely, if instructions are vague, even shared knowledge won‚Äôt ensure the AI grasps intent. The main challenge is clarity of requests, not knowledge gaps.

The blind area is ‚Äúthe human doesn‚Äôt know, the AI knows.‚Äù The user seeks knowledge or answers they lack‚Äîasking for concepts, data, or insight. The AI acts as an ‚Äúaccelerated learner‚Äù or knowledge provider.

Recommended question types: Knowledge and explanatory questions‚Äîdefinitions of terms, concept explainers, industry status analyses, etc. These target user blind spots, and the AI often covers them in its training data, filling knowledge gaps.

Human prompting tips: Good questioning and gradual probing are key. Since the user is unfamiliar, one query may not cover all they need. AI response strategy: Answer in detail and sequence to help the user grasp unknown content‚Äîprovide definitions, principles, examples, and extensions. Ideally, the AI responds in a teaching cadence, like a tutor guiding step by step. But it should avoid dumping too much at once and match the user‚Äôs pacing.

AI limitations: The challenge is the user ‚Äúnot knowing what they don‚Äôt know.‚Äù Vague questions can lead to unfocused or generic answers. Studies show very simple queries can make it hard for AI to infer needs, yielding broad responses‚Äîsometimes worse than traditional search. If training data are limited or biased, AI may give inaccurate answers the user can‚Äôt detect. Thus in the blind area, humans should ask from multiple angles and cross-check sources, while AI should admit uncertainty or cite sources to build trust.

Even with data, it may misread complex professional nuances or struggle to integrate new information due to model limits‚Äîmisinterpreting data meaning or missing critical context. The unknown area is ‚Äúthe human doesn‚Äôt know, the AI doesn‚Äôt know.‚Äù This is open-ended creative dialogue or frontier exploration‚Äîno standard answer, so both brainstorm together. AI and human start from the same line, with AI as a ‚Äúco-creator.‚Äù
