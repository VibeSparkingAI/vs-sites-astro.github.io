Codex 0.23.0: One‑Tap /model to OpenAI GPT‑5 High Reasoning

Vibe Sparking

With Codex CLI 0.23.0, model selection and reasoning strength become a one‑step toggle. Type /model, pick GPT‑5 + High reasoning, and complex tasks become a lot more stable.

Introduction

Section titled “Introduction”

Many developers already use Codex to write code, run tests, and edit files. But switching models or reasoning intensity used to mean changing config or restarting. Not anymore. 0.23.0 adds /model with live switching, plus High reasoning for GPT‑5 and even a new Minimal tier. It’s a tangible boost in speed and control.

What exactly changed in this release?

Section titled “What exactly changed in this release?”
  • Slash commands: Type /model in the terminal to change model/reasoning on the fly; /approvals gives you interactive approval control.
  • Reasoning control: Adjust reasoning effort at runtime; GPT‑5 also supports a minimal tier (faster, fewer reasoning tokens).
  • Login and versioning: ChatGPT Plus/Pro/Team accounts can use GPT‑5 directly in Codex; stable release is 0.23.0 (Aug 20).

Background: GPT‑5 exposes a reasoning_effort parameter (minimal | low | medium | high) to control how deeply the model “thinks.” Higher levels usually produce more reasoning tokens with higher latency; minimal trades depth for faster first tokens.

When to use High reasoning vs Minimal/Medium

Section titled “When to use High reasoning vs Minimal/Medium”

Choose High for

Choose Minimal/Medium for

  • Scaffolding, bulk lint fixes, renames: you want speed more than deep reasoning. minimal greatly reduces reasoning tokens and speeds things up.

Uzi Runs Multiple AI Agents in Parallel: Claude Code, Codex, Cursor CLI, aider, and More

Aug 23, 2025
Vibe Sparking

Uzi is a command-line tool that uses Git worktree + tmux to run multiple AI coding agents in parallel, each with its own dependencies and development port, then uses checkpoint to merge results to the main branch with one command.

Introduction

Section titled “Introduction”

One agent slowly modifying pages, another simultaneously refactoring the backend, and a third writing tests. Multi-threaded progress efficiency far exceeds “queuing for models”. This is how Uzi works: same repository, branch isolation, parallel execution, visual monitoring, one-click merging. The official README provides complete installation, configuration, and workflow examples, with v0.0.2 released on 2025-06-03.

Why Try Uzi?

Section titled “Why Try Uzi?”

High-Concurrency Experimentation: Start 3-5 agents at once, each exploring different implementations or styles, then merge the best solution. True “Isolation”: Uses Git worktree to give each agent an independent branch and directory, preventing cross-contamination. Automatic Development Environment: Each agent automatically starts a dev server with assigned ports for real-time preview. Full Management: tmux manages sessions, uzi auto automatically “confirms” tool calls, ls -w shows real-time progress and diffs. One-Click Conclusion: checkpoint rebases and commits selected agent branches back to the current branch.

Industry observation: There are also container-based solutions for isolating parallel agents; Uzi takes the worktree route, which works smoothly in monorepos too. Recent InfoQ reports have highlighted comparisons between these two approaches.

Section titled “Recommended Workflow: From Task to Merge in 5 Steps”

Scenario B | Refactoring Route Competition

Section titled “Scenario B | Refactoring Route Competition”

Should backend authentication switch libraries? Should it be layered? Run 2-3 approaches in parallel, read diffs for complexity and ports for performance perception, choose the more stable route.

Scenario C | End-to-End Coordination

Section titled “Scenario C | End-to-End Coordination”

One agent writes API, one writes e2e tests, another writes documentation and examples. Finally, only merge the one that “runs through everything,” kill and clean up other branches anytime.

Differences from Container Parallelization Cheat Sheet

Section titled “Differences from Container Parallelization Cheat Sheet”
  • Uzi (worktree): Light, fast, Git native, suitable for same-repo multi-experiments; tmux-friendly.
  • Container Solutions: Stronger isolation, heavier operations, suitable for extreme scenarios with cross-language/system dependencies.
  • Industry Perspective: InfoQ points out that Uzi’s worktree isolation can prevent agents from interfering with each other in monorepo scenarios.

Common Pitfalls and Avoidance

Section titled “Common Pitfalls and Avoidance”
  • Port Conflicts: Make portRange large enough and ensure ports aren’t occupied.
  • Duplicate Dependency Installation: Include npm/pip install in devCommand; each agent installs its own.
  • tmux Unfamiliarity: Recommend learning basic window/panel switching to improve visibility.
  • Cost and Noise: Parallel = more tokens; suggest starting with 2-3 agents for small tasks, then scaling up. (Empirical rule, combined with community discussions about “parallel mental load” reminders.)
  • Security and Keys: Don’t hardcode sensitive env into prompts; inject via environment variables.

VibeTree = Parallel Git worktree development + persistent terminals + Claude Code one-click integration, no branch switching, no lost context, efficiency soars.

Introduction

Section titled “Introduction”

Writing features, fixing production issues, then back to major refactoring… constantly git stashing, losing context back and forth is too painful. VibeTree, this open-source desktop application, combines parallel git worktrees, persistent terminals for each branch, and deep Claude CLI integration. One project opens N “clones”, each running its own tasks, each chatting with Claude.

Feature Highlights

Section titled “Feature Highlights”
  • Parallel Development: One repo, multiple worktrees, simultaneous feature/hotfix progress, goodbye stash.
  • Persistent Terminal Sessions: Each worktree has its own terminal, state never lost.
  • Claude CLI Integration: Direct claude command in each terminal, use /ide to connect VS Code/Cursor, file context aligned.
  • One-Click IDE Opening: Direct access to VS Code or Cursor from VibeTree.
  • Multi-Repo Tab Management: Multiple projects in parallel, smooth interface switching.
  • Cross-Platform: macOS / Windows / Linux, installation packages provided. Current latest version v0.0.1 (Alpha).

Section titled “Introduction”

Section titled “2) Opaque? Full-chain Observability”

4) More Secure Default Practices

Why AI Product Development Differs from Traditional Software: Continuous Calibration and Continuous Development — Uncertainty and the ‘Agency vs. Control’ Challenge

Aug 21, 2025
Vibe Sparking

Introduction

Section titled “Introduction”

The current wave of generative AI has pushed companies to ship AI‑powered features for fear of missing the moment. ByteDance launched the conversational app “Doubao,” and Tencent rolled out its in‑house chat assistant “Yuanbao.” Many product teams, under executive pressure, rapidly integrated AI: the idea looked promising, the demo dazzled, and early user feedback seemed positive. Yet once the product goes live, reality bites. The system surfaces unexpected errors; users bring a wild variety of requests; AI responses are inconsistent. Teams dig in and find the issues are intertwined and hard to attribute to a single fix, shaking confidence in the entire product approach. Worse, these hidden faults quietly erode user trust.

Why the yawning gap between a slick demo and a fragile production reality? At the root, AI products break several assumptions that traditional software depends on. If you build AI products the traditional way, you’ll hit trap after trap. Boiled down, there are two critical differences to take seriously:

  • AI products are inherently uncertain — both user inputs and model outputs are unpredictable.
  • AI products must balance agency and control — every time you grant the system more autonomous “agency,” you cede some human control. Ignore this and push autonomy too far, and you invite runaway decisions and hard‑to‑trace errors.

Many teams miss these two differences, leading to cascading problems during scale‑up and optimization: bizarre failures, bad decisions, even trust crises. To address these challenges, experienced practitioners have proposed a new product development frame that helps teams build stable, trustworthy AI features with intent. Below we unpack the two differences and introduce the Continuous Calibration / Continuous Development (CC/CD) loop that tackles them, followed by a concrete example showing how to ship AI features step by step in daily work.

The Uncertainty of AI Products

Section titled “The Uncertainty of AI Products”

Traditional software tends to be deterministic. User interactions are limited and explicit: clicking buttons, filling forms, calling fixed APIs. Developers hard‑code logic to map inputs to outputs. When something breaks, it’s usually a code bug that’s relatively easy to locate and fix.

AI products are born uncertain. They introduce a double layer of unpredictability:

  1. Input uncertainty

    • Inputs are no longer constrained to predefined operations; they may be open‑ended natural language, voice commands, images, and more.
    • Expressions are flexible, hard to exhaustively validate, and prone to ambiguity.
    • In support bots, the same question can be phrased in hundreds of ways; in content generation, prompt nuances produce drastically different results; in code assistants, vague requirements can yield entirely different implementations.

  2. Output uncertainty

    • Traditional software produces the same output for the same input; AI models generate probabilistic answers — essentially, “best guesses.”
    • The same request can vary across contexts, times, or model versions.
    • Output uncertainty makes testing harder — classic unit tests don’t fit; you need to assess quality by probabilities and trends.

Therefore, AI product development must accept: model behavior is not absolutely controllable; you must continuously observe and calibrate deviations from intent.

Balancing Agency and Control

Section titled “Balancing Agency and Control”

Another defining difference is the introduction of “agency” — the system’s ability to act and decide on the user’s behalf.

  • High‑agency examples

    • A support bot replies directly to users
    • A writing assistant auto‑publishes articles
    • A code assistant commits changes to a repo

  • Risks and issues

    • More autonomy → weaker control
    • When wrong, consequences exceed typical bugs
    • Harder to trace root causes; debugging and fixes get tougher

So autonomy must be “earned” gradually:

  • Start with assistive suggestions (low agency, high control) and human review
  • Increase permissions after proven reliability
  • Keep human takeover and rollback paths

This stepwise approach avoids large losses from AI mistakes and steadily builds user trust.

The CC/CD Framework

Section titled “The CC/CD Framework”

The CC/CD framework borrows from CI/CD but targets AI’s uncertainty and staged empowerment. It alternates Continuous Development (CD) with Continuous Calibration (CC).

Continuous Development (CD)

Section titled “Continuous Development (CD)”
  • Define capability scope; plan staged version boundaries
  • Build an app prototype; add logging and human‑fallbacks
  • Design evaluation metrics (Evals) to set baselines for calibration
  • Roll out to a small cohort and start gathering real feedback

Continuous Calibration (CC)

Section titled “Continuous Calibration (CC)”
  • Run evaluations to verify real‑world performance
  • Analyze logs to locate error patterns
  • Targeted improvements to prompts, retrieval, models, or data
  • Iterate until behavior stabilizes and becomes trustworthy

As loops progress, capabilities rise and agency expands — while safety and trust stay intact.

Case Study: Building a Support Bot in Stages

Section titled “Case Study: Building a Support Bot in Stages”

Version 1: AI‑assisted ticket routing (low agency, high control)

Section titled “Version 1: AI‑assisted ticket routing (low agency, high control)”
  • Role: classify only → route to the right team
  • Metric: routing accuracy
  • Low risk; humans can correct anytime
  • After calibration, accuracy can exceed 95%

Version 2: AI‑suggested replies (medium agency, medium control)

Section titled “Version 2: AI‑suggested replies (medium agency, medium control)”
  • Role: draft responses → humans review and send
  • Metrics: adoption rate and consistency
  • Improve retrieval to boost specificity and expertise
  • Draft adoption rises from 50% to 80%

Version 3: AI resolves issues autonomously (high agency, low control)

Section titled “Version 3: AI resolves issues autonomously (high agency, low control)”
  • Role: direct replies → restricted to high‑frequency, low‑risk scenarios
  • Metrics: auto‑resolution rate and CSAT
  • Start narrow, then expand coverage gradually
  • Keep human takeover and post‑hoc sampling in place

Through these three stages, the bot evolves from “assistive” to “self‑service,” lifting efficiency while preserving reliability.

Conclusion

Section titled “Conclusion”

Building AI products is like welcoming a brilliant but new teammate:

  • Don’t grant full autonomy on day one — let them learn the business step by step
  • Give room while keeping oversight to ensure correct direction
  • Continuous calibration with gradual empowerment is the only stable path

The core of CC/CD:

  • Facing uncertainty → observe and calibrate continuously
  • Facing autonomy → grant agency stepwise, evolve as needed

This way, AI becomes a reliable “team member,” upgrading experience and efficiency while protecting trust.

From small to massive: an old “law” breaks

Section titled “From small to massive: an old “law” breaks”

Five years ago, proposing to train a trillion‑parameter network sounded insane.

Conventional wisdom said: small models underfit; large models memorize and fail to generalize — the classic bias–variance trade‑off. In short, simple models miss patterns; overly complex ones memorize noise and bomb on new data.

Think of teaching a child addition. Two strategies: learn the carry rules, or memorize every problem and answer.

The latter yields perfect homework scores but collapses on new formats — like overfitting: stellar train accuracy, poor test performance.

So for years we obsessed over regularization and parameter parsimony. Big models were dismissed as overfit by default.

Reality surprised us. The very large models once deemed impossible now shine — from conversational AI to protein prediction — riding the wave of scale.

Double descent: a turning point

Section titled “Double descent: a turning point”

In 2019, researchers defied the warning labels and kept scaling models even after they reached 100% train accuracy.

The shock: test error went down again.

Train error fell, rose (overfit), then test error dropped sharply once more as parameters grew — a phenomenon dubbed “double descent.”

With repeated evidence across CNNs, ResNets, Transformers, “bigger is better” gained ground. Yet one question lingered:

If bigger should overfit more, why do giant models get smarter?

The secret weapon: the Lottery Ticket Hypothesis

Section titled “The secret weapon: the Lottery Ticket Hypothesis”

“Large networks contain many randomly initialized subnetworks. During training, the best ‘winning tickets’ get amplified while others fade.”

In 2018, MIT researchers found that inside a large neural net, there exist sparse subnetworks that can match the full model’s performance even after pruning ~96% of parameters.

The twist: they only work that well if you keep the original random initialization. Re‑initialize them and the magic disappears.

Hence the Lottery Ticket Hypothesis:

Large models succeed not because they learn a baroque solution, but because with enough parameters, they contain many candidate subnetworks — some happen to be highly aligned with the task. Training “draws tickets,” reinforcing the winners while suppressing the rest.

This reconciles double descent with classical theory:

Big models aren’t just memorizing; they discover simpler explanations in a vast parameter space. Occam still holds — via a subtler mechanism.

Brains and LLMs: convergent patterns

Section titled “Brains and LLMs: convergent patterns”

Human brains have ~86B neurons and quadrillions of synapses — absurdly over‑parameterized on paper — yet we generalize.

Viewed through the lottery‑ticket lens, redundancy offers a huge pool of candidates from which effective, simple patterns can be selected. Intelligence is less rote memory, more “finding structure among many possibilities.”

This also clarifies why LLMs thrive.

Buy more tickets, increase your odds. Giant language models contain countless randomly initialized subnetworks; training elevates those “born good at language.” As scale grows, more winning tickets emerge, enabling in‑context learning, reasoning, and other capabilities.

The lesson: dare to go past “common sense”

Section titled “The lesson: dare to go past “common sense””

AI researchers accidentally scratched a jackpot that shifted the field.

The bias–variance rule wasn’t overturned; it operates differently in huge parameter spaces. Large networks aren’t cheating — they follow the rules at a higher level.

Practice, then theory. Pushing beyond the comfort zone of theory yielded milestone progress. In a math‑heavy field, some breakthroughs come from embracing uncertainty.

There’s no magic behind ChatGPT’s success.

It’s a simple idea that reshapes how we think about learning.
