Bring the Agent to Your Terminal: Kimi CLI Setup & Power Tips

Oct 24, 2025
Vibe Sparking

Kimi CLI brings “terminal + coding assistant + toolchain” into a single window. It feels like a shell, but you can switch to an agent at any time. It runs commands directly, plugs into Zsh, connects to Zed/VS Code (via ACP), and supports MCP tools. It’s in tech preview, but already usable for real work. See the official repo and docs for sources.

Why this is worth trying

Section titled “Why this is worth trying”
  • Not a chat box, but a CLI that “lands” work: one key to switch into Shell mode and run native commands; press again to return to the agent to explain logs, generate scripts, and complete parameters.
  • Works with Zsh: install the zsh plugin, hit Ctrl-K to toggle agent mode in the terminal—switching back and forth is smooth.
  • Speaks ACP (Agent Client Protocol): this means it can be used as an “external agent” inside editors like Zed. One protocol connects any editor with any agent—that’s where things are heading.
  • Supports MCP tools: load third‑party tools via MCP conventions, like snapping in browser debugging and context search like Lego bricks.

By the way, Moonshot spun up the Kimi K2 / Kimi‑Dev model line this year, focused on engineering and agent capabilities, emphasizing long context and coding ability—the “engine” behind this CLI track.

Install in two steps, take off in three

Section titled “Install in two steps, take off in three”

Recommended to install with uv (clean and hassle‑free):

  • Install: uv tool install --python 3.13 kimi-cli
  • Upgrade: uv tool upgrade kimi-cli --no-cache
  • First run on macOS may pause ~10s due to security checks.
  • After install, run kimi --help to verify.

Initialization: run kimi in the directory you’ll work in, enter /setup to bind; type /help to view the command menu.

Shell/Agent mode toggle: press Ctrl-K. Shell mode runs commands directly; built‑in shell commands like cd aren’t supported yet.

Six high‑frequency use cases

Section titled “Six high‑frequency use cases”
  1. Project cold start
  • Command: /setup to establish context → have it read repo structure, generate a Makefile, and scaffolding scripts.
  • Value: more reliable than pure chat because it’s right in your working directory; scripts it generates can be run on the spot.
  1. Log firefighting
  • Run integration tests in Shell mode, pass failing stack traces to the agent for root‑cause summaries and patches.
  • Flip back to Shell and validate the fix in one keypress.
  1. Command‑line “auto‑complete++”
  • Ask it to compose grep/awk/jq pipelines based on your project context, and explain every parameter.
  • Great for large logs, CSV, and JSON batch jobs.
  1. “A terminal that can code” inside your editor
  • Use ACP with Zed: open an agent thread alongside code so it can both read files and run commands—tighten the edit‑validate loop.
  1. Plug in third‑party tools (MCP)
  • For example, attach chrome-devtools-mcp for page performance analysis, or a context service for knowledge bases.
  • Start with: kimi --mcp-config-file /path/to/mcp.json.
  1. Faster onboarding for new teammates
  • Give newcomers a one‑window “ask a command → see explanation → execute immediately” setup; no more “ask ChatGPT → copy → paste into terminal” ping‑pong.

How it differs from a typical “model chat box”

Section titled “How it differs from a typical “model chat box””
  • Acts in place: after generating a command, it runs it directly—no copy/paste dance.
  • Editor‑native collaboration: ACP lets IDEs/editors wire “runtime context, file changes, task panels” together with the agent.
  • Pluggable: MCP brings external tools in; easier to reuse than a pile of proprietary plugins.

Pitfalls and gotchas

Section titled “Pitfalls and gotchas”
  • Built‑in shell commands aren’t supported yet: e.g., cd. If you need to change directories, use pushd/popd or split steps.
  • First launch on macOS is slow: this is just security checks—wait a few seconds.
  • Choose the right installer: uv tool install creates an isolated environment; upgrades/uninstalls are clean—that’s why this article recommends it.

Who it’s for

Section titled “Who it’s for”
  • Anyone who wants to merge “write script + execute + review” into one window.
  • Zsh/Oh My Zsh power users.
  • Engineers/teams using Zed or interested in ACP.

Conclusion

Section titled “Conclusion”

If GUI Copilots are for “write and peek,” then Kimi CLI is for “write and run.” It pulls the agent out of the web page and into your terminal and editor, so each idea can quickly become an execution. It’s in tech preview now—perfect for early birds to get a head start and iterate.

Will Documentation-Driven AI Coding Become the New 'Spaghetti Mountain'? Pros and Cons of BMAD, Spec-Kit, and OpenSpec

Oct 23, 2025
Vibe Sparking

Remember the days when “spaghetti code💩” used to give everyone a headache? With AI copilots like Claude Code, Codex, and Gemini in the mix, we have stumbled into a brand-new pitfall—the “documentation spaghetti💩.” Documentation-driven AI programming has become a hot trend: write a clear specification first, then let AI generate the code. It sounds wonderful—no more guessing what the model wants, just provide a well-written brief and code appears automatically. The truth, though, is that if you take the wrong approach, the documentation can pile up into a brand-new spaghetti mountain that leaves you laughing and crying at the same time 😅.

Introduction: What on Earth Are BMAD, Spec-Kit, and OpenSpec?

Section titled “Introduction: What on Earth Are BMAD, Spec-Kit, and OpenSpec?”

Let’s start with these documentation-driven AI programming tools that are making the rounds—they each have their own personality:

BMAD Method – a “big and comprehensive” framework whose full name is Breakthrough Method for Agile AI-Driven Development. Its ambitions are huge: it tries to simulate a complete agile squad with AI. There is an analyst AI, an architect AI, a developer AI, a tester AI—each playing a role so that planning, coding, and testing all get staged like a full production. BMAD ships with a CLI and a web UI, plus special “script files” and context management so the AI team can collaborate. It usually unfolds in two phases: first the “analyst/product manager/architect” agents do the planning and design, then the “Scrum Master/developer” agents write the actual code. The upside is a thorough workflow that stays orderly—ideal for large, complex projects that demand clear documentation and division of labor. The downside is that it is cumbersome to use, has a steep learning curve, and expects the underlying model to truly understand the documentation you write.

Spec-Kit – an open-source utility on GitHub that layers a bit of “process” on top of your existing AI coding assistant. Developers use the command line to nudge AI through a “Specification → Plan → Tasks → Implementation” four-step. In short, Spec-Kit asks you to nail the requirements first, then guides AI to break work down and produce code. It turns your specification markdown into an “executable document” using commands like /specify, /plan, and /tasks. It is lightweight and flexible, does not force you to revamp your workflow, and is model-agnostic. Great for solo builders or small teams that want to add structure and improve AI-generated code quality.

OpenSpec – a minimalist newcomer that believes “simple is better.” You write a slim Markdown spec and pair it with a handful of commands to finish the job. Its flow has only three steps: submit a change proposal, let AI produce the task list, then let AI apply the spec to modify the code. The highlight is efficiency: the specs stay lean, token usage stays low, and there is barely any metadata. It works well for individuals or small teams chasing extreme speed—minor tweaks get done in minutes. The trade-off is that model support is still limited, but it is so easy to learn that there is almost no overhead.

Spot the pattern? These tools share the same philosophy: document first, then drive the AI. The goal is to reduce the uncertainty of vibe coding by giving AI a clear rulebook so code quality becomes controllable. Ideal in theory—but reality bites. If you are not careful, the documentation you create can morph into another giant spaghetti pile. Why does that happen? Read on.

Four Reasons Documentation Turns into a Spaghetti Mountain Even Faster

Section titled “Four Reasons Documentation Turns into a Spaghetti Mountain Even Faster”

1. Lack of Structure

Section titled “1. Lack of Structure”

Code is constrained by syntax—one misplaced comma and it will not even run. Documentation is much freer, and AI is not human. When a model stares at free-form, meandering text, its comprehension goes sideways. Code is deterministic; specifications are often vague and open to interpretation. If your documentation is loosely structured, the AI sees the world through fogged glass, and it is no surprise when the code it generates veers off in the wrong direction.

2. Inconsistent AI Understanding

Section titled “2. Inconsistent AI Understanding”

Even if you write a decent document, there is no guarantee the AI will interpret it the same way every time. Seasoned developers know the saying “a thousand readers, a thousand Hamlets”—and AI thinks even more differently from humans. ChatGPT might implement the feature correctly today, but a different model tomorrow could distort your intent. Documentation lacks the compiler’s safety net, so misunderstandings can hide in plain sight and quietly plant landmines.

3. AI Hallucinations Amplify Errors

Section titled “3. AI Hallucinations Amplify Errors”

We all know large language models hallucinate—confidently fabricating details. When AI uses incomplete or ambiguous documentation to write code, the stakes rise. Miss a detail in your spec and the AI might “fill in the gaps” with something that looks plausible but is wrong. Worse still, if that flawed AI-generated documentation or code gets reused later, another AI might treat it as gospel truth. Errors balloon like rumors: once they make it into formal docs, people and models alike will keep getting misled.

4. New Risks Introduced by the Tools Themselves

Section titled “4. New Risks Introduced by the Tools Themselves”

Finally, consider the complexity of the tools. Adopting Spec-Kit or BMAD might require extra software and brand-new rituals. You can end up wrestling with tooling, pouring hours into crafting specs and calibrating workflows, while actual coding time shrinks. If you do not internalize the framework, you may copy templates mechanically, letting documentation drift away from the code. Like wielding a scalpel without training—the damage can be greater than doing nothing.

How to Avoid the Pile-Up: A Tiered Playbook for Documentation Tools

Section titled “How to Avoid the Pile-Up: A Tiered Playbook for Documentation Tools”

The pitfalls sound daunting, but do not lose heart. With a tiered usage strategy, documentation-driven programming can still pave a smoother road:

Use direct prompts for tiny tweaks

Section titled “Use direct prompts for tiny tweaks”

When you are only adjusting a small logic branch or styling tweak, there is no need to assemble a grand spec. In these cases, asking Claude Code, Codex, Gemini, GitHub Copilot, or Cursor directly is fastest. Explain the request in a sentence or two and let the AI offer a diff or snippet. You save time and stay focused. As veterans like to say: small asks do not need the full ceremony; if a quick chat solves it, skip the paperwork.

Reach for Spec-Kit or OpenSpec on medium-sized changes

Section titled “Reach for Spec-Kit or OpenSpec on medium-sized changes”

If you are adding a mid-sized feature—multiple steps, not massive—consider a lightweight tool such as Spec-Kit or OpenSpec. Spend ten minutes drafting an MVP-level spec that captures key requirements and edge cases, then hand it to the AI to refine the plan and tasks. Spec-Kit decomposes work item by item; OpenSpec minimizes context switching and handles the change in a few commands. The key is to keep the spec concise, covering only what is necessary. Data shows that compact specs are easier for AI to execute faithfully and more efficient overall. Once the feature ships, remember to update canonical docs or clean up temporary specs to prevent outdated information from piling up.

Deploy BMAD for major overhauls

Section titled “Deploy BMAD for major overhauls”

For large-scale refactors or brand-new complex subsystems—the real “surgery”—you need a rigorous approach. This is where a multi-agent framework like BMAD earns its keep: it orchestrates end-to-end planning, architecture, and task allocation. Think of it as hiring a virtual AI team to keep big projects on the rails. Of course, humans must stay in the loop—never blindly accept each AI output; review and adjust. Before spinning up BMAD, weigh the costs: it takes time and demands strong model comprehension. That investment can pay off for long-lived initiatives, but the documentation it produces—specs, user stories, and more—should evolve with the code. Do not toss it aside after one run; maintain it alongside the project or you will be rebuilding from scratch next time.

No matter the scale, remember the “documentation MVP” principle: the doc should cover only what is necessary and true right now. Do not chase completeness for its own sake. Many veterans delete temporary specs once a feature is done because the code itself is already clear. Instead of hoarding outdated notes that mislead people, keep a lean, accurate reference and update it as the product evolves. Documentation earns its value by reflecting reality; once it drifts, it is just dead weight.

Conclusion

Section titled “Conclusion”

Documentation-driven programming is a double-edged sword in the AI era. Used well, it makes development more efficient and orderly; misused, it spawns fresh spaghetti mountains. Ambitious young engineers should absolutely explore new tools, but not worship them blindly. Whether we are talking code or docs, the heart of the matter is understanding the requirement and keeping information current. Whenever you decide to write a spec or pick a tool, ask yourself: “Is this making the problem clearer—or just adding complexity?”

Here’s to every developer mastering these AI tools, producing docs that stay crisp and useful, and turning documentation into an asset—not a burden. We all want higher productivity, not a literal spaghetti mountain blocking the office! 💩

Key Takeaways Cheat Sheet (with Authoritative References)

Section titled “Key Takeaways Cheat Sheet (with Authoritative References)”
  • CoreBluetooth discovery and callbacks are asynchronous. Operate only after the corresponding callbacks (such as discoverServices or discoverCharacteristics:forService:) complete; precise UUID searches are more efficient.
  • “Discover first, then act” is the conventional order for iOS BLE. Tutorials and field guides emphasize callback-driven, asynchronous thinking to avoid writing before discovery finishes.
  • You can discover services/characteristics via full-table or targeted UUIDs. Full scans are slower; targeted discovery is more stable.
  • BabyBluetooth wraps CoreBluetooth with block- and chain-style APIs, making it easier to flush deferred queues inside setBlockOnDiscoverCharacteristics.
  • Flutter Platform Channels bridge Dart and iOS/Android; the business-layer timing guarantees should live on the native side so Dart can’t dispatch writes prematurely.

Practical Checklist

Section titled “Practical Checklist”

Do these six things and iOS BLE stops tormenting you:

  • Align the sequence: lock “connect → service → characteristic → notify → write” into a state machine and reject writes until Ready.
  • Build a queue: enqueue write operations and notify requests, flush them in the callbacks.
  • Add idempotency: repeated setNotify/write calls must be re-entrant to prevent “write storms.”
  • Add retries: exponential backoff + retry limits, backed by detailed logs.
  • Cache aggressively: cache services/characteristics and maintain connection pools to lower overhead.
  • Layer responsibilities: let Flutter “look successful,” but iOS must own correctness.

The Impact of AI on Bosses, Leaders, and Engineers

Vibe Sparking

Introduction

Section titled “Introduction”

The age of artificial intelligence (AI) has arrived with a bang. From business owners to team leaders to frontline engineers, everyone’s role at work is quietly shifting. Some worry that AI will take their jobs; others are excited by the surge in efficiency. The reality is that AI is reshaping how we work at astonishing speed. How should decisions be made? How do we use the tools? How should we lead? And for introverted engineers who prefer heads‑down work in particular, is this the moment to shine—or will the social naturals race ahead? This piece looks at real cases and data to see how Bosses, Leaders, and engineers can each find new advantages in the AI tide.

Decision‑Making and Role Redesign: What Is AI Taking Over?

Section titled “Decision‑Making and Role Redesign: What Is AI Taking Over?”

AI is redrawing the workplace map. Decisions once made by people are now informed by algorithms—and in some cases automated end‑to‑end. Resume screening in HR, reconciliation in finance, and other routine workflows are increasingly handled by AI, freeing managers from administrative grind. Business owners (Bosses) can therefore focus more on strategic direction while delegating day‑to‑day operational decisions to data.

IgniteTech’s CEO Eric Vaughan took a radical step: pushing the company “All in AI,” replacing as much as 80% of staff and keeping only those willing to embrace AI. He argues that “people who don’t use AI will be replaced by those who do.” The results? Within three months of the transition, process‑optimization metrics improved by 300% across departments, and the finance team cut the quarterly close from five days to 48 hours. For many responsibilities, AI didn’t just add speed—it changed the game.

Leaders (team managers) are approving and monitoring less, but they now own the job of putting AI strategy into practice: training teams on new tools and re‑architecting processes. As Andrew Ng notes, AI gives small teams the power to rival big companies: a single engineer finishing in a day what used to take six months of prototyping is no longer a myth. That reality is loosening traditional hierarchies and pushing decision rights down to the technologists on the front lines.

On the engineering side, parts of basic coding and testing are auto‑generated, pulling engineers up into higher‑level design and optimization. Newcomers, with AI support, can deliver products in a language they don’t yet know within days. AI is redefining role boundaries: it frees people from repetition, but it also requires continuous upskilling and new responsibilities in human‑AI collaboration.

Tooling and Efficiency Leaps: How Is AI Changing the Pace of Work?

Section titled “Tooling and Efficiency Leaps: How Is AI Changing the Pace of Work?”

In the AI wave, new tools are proliferating and accelerating the pace of work. For developers, GitHub Copilot and similar code assistants feel like pair‑programming with a tireless partner; official data shows it has increased development speed by 55%. And it’s not just programmers—nearly every role now has an AI helper: marketers brainstorm with ChatGPT, designers use Midjourney to generate concept art, and product managers lean on smart analytics to interpret user data.

According to a joint Microsoft and LinkedIn study, 75% of employees already use generative AI at work, and 68% say their workload and tempo are more demanding than before—yet AI is helping them cope. Many even bring their own AI tools to work (especially in SMBs). This “AI‑partner” model dramatically boosts personal throughput—tasks that used to require a team can now be done by one person plus AI.

Leaders can therefore reallocate people from mechanical, repetitive work to creative, higher‑value problems. Microsoft’s Copilot for Office, for instance, can automatically capture meeting takeaways and generate action items, smoothing team collaboration.

Business owners are actively arming their teams, too: companies like IgniteTech reimburse every employee for ChatGPT Plus and set up dedicated “AI Days” for intensive practice. Forward‑looking Bosses have found that employees who embrace AI typically produce more—and they reward that with bonuses and promotions. Surveys show that more than two‑thirds of leaders say they won’t hire candidates who know nothing about AI; even less experienced candidates who are fluent with AI are preferred.

Engineers are also experiencing “AI hyper‑growth”: some use AI to rapidly fill skill gaps; juniors, with Copilot, produce mid‑level code; seniors become the fabled “10x engineers.” The flip side of tooling is fiercer competition—those who don’t learn AI are getting left behind. For those willing to learn, however, AI is becoming a force‑multiplier for productivity.

Leadership Style Shift: From Controller to Enabler

Section titled “Leadership Style Shift: From Controller to Enabler”

With AI and greater information transparency, leaders must evolve their role. The classic manager once acted like a commander—controlling the field and making every call. Today, AI makes analysis and options far more accessible, and frontline employees can propose data‑driven recommendations. In this environment, Leaders who cling to “I decide everything” risk slowing innovation.

As Andrew Ng has said, when “10x talent” emerges on a team, “the best person to make a decision is sometimes the engineer, not the manager.” Traditional pyramids push information up and ask leaders to decide. In the AI era, specialists armed with intelligent tools hold rich context and alternatives—and should receive greater decision autonomy.

Successful leaders are starting to play the roles of “gardener” and coach: creating fertile ground for trial‑and‑error and innovation rather than doing everything themselves. Modern organizations must distribute decision rights throughout the structure so experts at the edge can respond quickly. That requires Leaders to trust their teams, let go more, and bring technically savvy frontline colleagues into important decisions.

The same applies to the Boss level: many business owners now establish a Chief AI Officer (CAIO) role, weaving AI strategy into executive decision‑making and asking functions to report regularly to the AI team.

Beyond delegation, leadership is getting “softer.” Because AI can offer objective options, leaders increasingly need empathy and judgment to balance technology with humanity. One clear trend: managers are investing more in team learning. Surveys show that as many as 62% of employees feel they lack the skills to use AI effectively and safely, while only about one in ten believe they have what they need. In response, leaders are providing AI training resources to close the skills gap—for example, some companies host a monthly “AI Sharing Day,” with leaders participating alongside employees.

In short, AI‑era leadership is shifting from command‑and‑control to empowerment‑and‑collaboration. Not everyone who uses AI is a “junior”; the leaders who can activate these “AI power users” will be the ones who pull their teams ahead.

Opportunities for Introverted Engineers: How Does AI Amplify Quiet Strength?

Section titled “Opportunities for Introverted Engineers: How Does AI Amplify Quiet Strength?”

AI affects engineers differently, but for many who are introverted and technically strong, this may be the best moment to shine. In the past, outspoken teammates could attract attention by presenting well, while introverts—even highly capable ones—might remain unnoticed. Today, AI tools can act as an amplifier for expression and a superb partner for execution.

There’s a growing view that AI offers introverts new ways to collaborate and express ideas—letting thinking cut through social friction and reach the heart of the problem. Introverted engineers no longer need to worry as much about “performing” in front of others; they can focus on deep thought and high‑quality output. Their ability for independent inquiry is being amplified as never before.

Real‑world stories back this up. In one software team, several quiet members flourished after adopting AI. They used models to better understand customer feedback, then engaged clients with greater confidence—ultimately improving results significantly. Picture an engineer skilled at independent problem‑solving, paired with an AI copilot; together they may produce what used to take a ten‑person team. These quiet “deep divers,” once aided by AI, can deliver startling results.

Notably, the AI era is making “10x engineers” more common—they may say little, but by mastering the tools, they work several times faster than most. Companies are willing to pay for such talent; top engineers may earn three times an average engineer’s compensation.

That said, introverted engineers must beware the comfort‑zone trap. Over‑reliance on AI as a substitute for communication can weaken social skills. On one hand, AI makes it easier for introverts to stand out on the strength of their work. On the other, excellent engineers don’t avoid communication—they use AI to boost their technical articulation and still invest in the human collaboration that AI can’t replace. Play to your strengths while shoring up gaps, and the “quiet” technologist can become a surprising disruptor in the AI age.

Writing Code Is Surface—Thinking Is the Essence of Software Development

Oct 18, 2025
Vibe Sparking

“Code is just a tool; thinking is the core.”

If you’re a product manager, indie maker, or junior developer, that line may feel counterintuitive. Yet in day‑to‑day engineering, we repeatedly discover: programming is 90% thinking and 10% typing. With Claude Code, ChatGPT, and Google Gemini making code generation easier than ever, this truth is now crystal clear. If AI can write code for us—where does our value really lie?

Consider a few common scenes: to deliver a seemingly simple feature, a developer spends ages on design—then needs only half a day to write the code. Or a production bug takes two days to investigate from logs to root cause—but the fix is a single line. PMs often wonder: “It’s just a small feature—why so long to write some code?” The reality: coding has never been the hardest part. The hard part is “what exactly are we building, and how?” Software development is fundamentally about thinking, not mechanical typing.

In an AI‑accelerated era, this stands out even more. Over 81% of developers now use AI coding assistants. One bank’s tech team reported a 40% reduction in coding time and a defect‑detection accuracy up to 92% after adopting an assistant—yet the lead still emphasized: “AI‑generated code requires human review—especially around security logic and business rules. Developers must remain in charge.” In the end, concurrency control written by AI was revised by humans due to potential risks. The essence of human‑AI collaboration is clear: AI handles repetitive, standardized work; humans handle creative, complex decisions. In other words, AI can be your “helper” that outputs code—but what ultimately determines software success is still the thinking and judgment in your head.

Below, we’ll walk through real everyday scenarios to see how “thinking” dominates—and how AI coding assistants take over the busywork so you can focus on what matters most.

Scenario 1: Implementing business features — think first, then code

Section titled “Scenario 1: Implementing business features — think first, then code”

When a new requirement lands, newcomers often rush to code. Experienced engineers suppress that impulse and first invest heavily in understanding the business: what problem does the requirement solve? What rules and edge cases exist? How should data flow? Only after clarifying the logic can we write correct code. Coding is translating a mental design into a language a computer executes. The translation isn’t the hard part—having the design is.

Take a familiar task: implement “order placement deducts inventory” in e‑commerce. The time sink is clarifying the rules behind “deduct inventory”—what happens if stock is insufficient? How do we prevent overselling under concurrency? How do we roll back on failure? Each question demands careful thought and design. By comparison, the final call to the inventory service to decrement a number may be only a few lines. If the logic isn’t thought through, more code won’t save you; if it is, the code is often just a handful of lines.

With AI tools, “think before you code” becomes more efficient. Once you’ve designed the logic, ask ChatGPT or Claude to produce a first pass of the implementation—saving time and effort. But note: AI won’t think through requirements for you. It doesn’t grasp the business essence; it only generates code from your description. You must be the “brain,” conveying intent precisely. In a way, AI forces us to be more rigorous: instructions must be clear, or the output will drift. Once again: before writing business code, think through the business logic. Code quality rises and falls with the depth of your thinking.

Scenario 2: Integrating third‑party APIs — understanding beats typing

Section titled “Scenario 2: Integrating third‑party APIs — understanding beats typing”

Many of us have integrated a third‑party API. At first glance, it seems like “just a few HTTP calls”—but where’s the real time sink? Reading and understanding the documentation: which URL does what, what parameters are required, how to auth/sign, response formats, what error cases to handle. Mapping the flow and edge cases can consume hours before a single line of code is written.

For example, integrating a payment platform: you have to find the correct endpoint, understand the order schema, learn the retry strategy and error codes, then design your system’s call sequence and resilience. Once that’s clear, the coding is mostly assembling HTTP requests and parsing responses. With AI helpers, this gets easier; you can paste relevant docs or schemas and have AI draft the client code.

But API integration fundamentally tests learning and comprehension. We play the role of “translator,” turning a foreign system into local code. AI accelerates translation, but it can’t replace grasping the “original meaning.” Without context, AI might miss key parameters or misuse flows—leaving traps for humans to catch later. So invest time in understanding the API’s business meaning and planning the call strategy, not just in the mechanics of writing the code. With that mindset—and AI to produce the scaffolding—you’ll get twice the results for half the effort.

Scenario 3: Fixing bugs — analysis costs more than the edit

Section titled “Scenario 3: Fixing bugs — analysis costs more than the edit”

Debugging is everyday life. As the joke goes: “30 minutes to write, 3 hours to debug.” When a bug appears, the first step isn’t editing code—it’s investigation: reproduce, inspect logs, read surrounding code, hypothesize likely causes, and narrow the search. It’s a mystery to solve—logic, inference, and understanding the system deeply. Once you find the root, the change is often trivial.

Imagine maintaining a complex system when sporadic errors are reported. You pore over logs to triangulate the module; add debug prints and replay runs to isolate conditions; even sketch a flowchart to unravel the execution path. After this mental work, you discover a missing null check in a rare branch. Adding that guard takes five minutes—the time went into thinking and locating, not typing.

Where can AI help? Today, AI is a capable assistant, not a substitute. You can feed failing code and logs to ChatGPT to brainstorm likely causes—it often produces useful leads. But deciding the truth and validating hypotheses still falls to you. Real debugging involves the project’s context, architecture, and business logic; it’s rarely “one faulty line.” Without the full picture, AI suggestions can be off‑target. Humans carry the global model of the system, weigh signals, and decide.

Some tools are improving here. Anthropic’s Claude is strong at step‑by‑step reasoning, listing thorough investigation steps; OpenAI’s ChatGPT is quick with obvious fixes. Treat them like a conversational mirror and advisor—explaining symptoms to clarify your own thinking and catch missed details. Even if AI proposes a patch, validate it carefully. Debugging remains a human‑led exploration and inference process; AI accelerates insight.

Scenario 4: Writing tests — designing cases matters more than code

Section titled “Scenario 4: Writing tests — designing cases matters more than code”

“No tests, no quality.” Writing tests is another case where design outweighs typing: the hardest part is imagining scenarios and cases—what to test, what can go wrong. Converting that into test code is relatively mechanical.

As one senior SDET put it: designing tests is sparring with your own code. You must dream up edge cases, boundary conditions, and invalid inputs to challenge robustness. For an order system, you might design: “What happens when demand exceeds stock?”, “How should order flow behave under DB connection drops?”, “Is the result consistent under 1,000 concurrent orders?” Each scenario demands deep understanding plus a critical mindset. That mental investment is most of the work; writing the asserts and fixtures is just staging the play.

The good news: AI shines at generating test code. You can feed a function and ask for targeted unit tests. It will often produce solid coverage and even infer developer intent from comments and context. But human judgment still decides whether tests are necessary, sufficient, and reasonable. As one developer quipped: “AI is great at turning ‘write a test to verify…’ into runnable code. Your job is to invent inputs that could break the code—without slogging through the boilerplate.”

Scenario 5: Designing technical solutions — architects live in the thinking layer

Section titled “Scenario 5: Designing technical solutions — architects live in the thinking layer”

When planning a new system or technical approach, engineers switch to pure thinking mode: a new recommendation algorithm, a microservices layout, or a sharding plan. Little to no production code is written at first. We apply experience, knowledge, and creativity to sketch the blueprint on a whiteboard or in a doc.

We consider: business and non‑functional requirements (performance, scalability, security, etc.); possible options and their trade‑offs and complexity; and which path is most robust. This is mental work. We draw architecture diagrams, enumerate modules, and write design docs—visualized thinking that clarifies the path. Once the plan is set, implementation comes later.

AI can assist here too: have ChatGPT show pattern examples, industry best practices, or produce an outline for you to refine. But the architect—grounded in business context—makes the final call. If anything, AI’s rise highlights how architecture depends on human insight. General advice from corpora can’t know your team’s constraints or long‑term goals; only you can weigh them.

We’re also seeing early signs of “repository‑level” assistants that generate whole codebases from high‑level prompts. Tempting as “one‑click projects” sound, they’re not production‑ready today; outputs are often brittle. Even as they improve, they’ll still need architects to think through the design and feed it in. Research suggests these tools are most useful for freeing developers to focus on higher‑level design and deployment decisions—AI does the low‑level details, humans steer.

In short, when designing solutions, we are thinkers and decision‑makers—not just code craftsmen. A good design beats a poor implementation; a clear, reasonable design slashes build and maintenance costs. If you skip the thinking and rush into code, the hidden traps cost 10–100x later. AI doesn’t change this; it reinforces it: code can be delegated, but creative responsibility cannot.

Conclusion: Be a better thinker in the AI era

Section titled “Conclusion: Be a better thinker in the AI era”

Whether you’re a PM, indie maker, or new developer, hopefully these scenarios resonate: the core advantage in software isn’t typing code—it’s thinking deeply and solving problems. As the saying goes, “Programming is 90% thinking and 10% typing.” AI makes this truer than ever. With code generation offloaded, our value lies in asking the right questions, making sound decisions, and solving complex problems creatively—skills AI cannot yet replace.

For product managers, recognizing the “thinking load” in development helps set better timelines and expectations. When a dev estimates a week, don’t fixate on lines of code; appreciate the design tradeoffs and mental work. Try having AI produce a quick prototype; you’ll feel firsthand that moving from 0→1 is really about turning fuzzy goals into a clear plan. When the early thinking is done well, implementation flows.

For indie or full‑stack engineers, you wear many hats and must balance thinking vs. coding. With AI, offload repetitive work—scaffolds, CRUD, format conversions—and conserve your brainpower for choices only you can make: product ideas, architecture selection, and core algorithm design. Many of us are already shifting from “writing code” to “designing, decomposing, prompting, and reviewing.” The sooner you adapt, the more competitive you’ll be.

For juniors, don’t fear AI—or resign yourself to being its typist. Treat AI as mentor and tool. Ask “why,” and use it to learn higher‑level knowledge. Surveys show many developers’ top AI use case is learning—new tech, best practices, and more. First, analyze and design on your own; then ask AI for alternative implementations and compare. Learn fast by closing the gap. Avoid tossing problems to AI without understanding—otherwise you risk hollowing out your skills. AI ultimately serves thinkers. Those who don’t think will fall behind—this only becomes more true in the AI era.

Finally, AI coding assistants don’t let our brains go off duty. Quite the opposite: they push us to understand where our value lies. If typing is automatable, we should invest time in understanding the business deeply, designing carefully, enforcing quality, and innovating continuously. Rare Earths & Strategic Metals: Industry Vitamins and Defense Arteries

Section titled “Rare Earths & Strategic Metals: Industry Vitamins and Defense Arteries”

Industry context and strategic role. “Rare earths” (17 elements) are often called the vitamins of modern industry — essential for high‑performance permanent magnets (EV motors, wind turbines), avionics, guidance systems, and semiconductors. Strategic metals (e.g., antimony, cobalt, titanium, tungsten) are key to defense/high‑tech. The U.S. Department of Defense (DoD) has repeatedly warned that rare earths are indispensable to advanced weapon systems (F‑35s, submarines, missiles, UAVs). In practice, the U.S. relies heavily on China: China mines ≥60% and processes ~90% of rare‑earth products; 70% of U.S. rare‑earth compounds/metals imports (2020–2023) came from China. China is also a major supplier of antimony (armor alloys, fuzes), graphite (battery anodes), and tungsten (AP rounds, tooling). This dependency is viewed as a national‑security weak point. In 2025, China tightened controls further on rare earths and magnet exports (declaring uses for products containing heavy RE), heightening global concern. Analysts see it as signaling that Beijing may wield rare‑earth supply as leverage. The U.S. response includes reducing reliance, tariff tools, and larger resource investment.

Supply–demand and outlook. Rare‑earth demand has surged with EVs and clean energy, while supply growth remains concentrated. Export curbs by key suppliers could cause shortages and price swings. The U.S. is acting on multiple fronts: DoD funding to MP Materials for separation, and to Lynas for heavy‑RE processing in Texas; Defense Production Act (DPA) support for lithium, graphite, nickel, antimony, etc. After China restricted antimony exports in Aug‑2024, the Defense Logistics Agency (DLA) expanded procurement plans, reportedly earmarking ~$245M with U.S. Antimony (UAMY) — roughly three‑quarters of annual U.S. imports. Strategic stockpiles are set to expand: reports in Oct‑2025 suggested a ~$10B critical‑minerals reserve to reduce China reliance. Over 2025–2030, new projects in Australia/Canada could diversify supply; deep‑sea mining (e.g., The Metals Company targeting nickel/cobalt nodules) may contribute. But permitting, costs, and price cycles complicate investment — China’s position remains strong in the near term, with allied cooperation and state support needed to bridge gaps. Geopolitical risk will be persistent; decoupling is partial at best, implying a protracted, uneven supply‑chain re‑wiring.

Representative companies & investment take. The cohort includes MP Materials (MP), USA Rare Earth (USAR), U.S. Antimony (UAMY), and Ivanhoe Electric (IE). MP — owner of Mountain Pass, the only producing U.S. RE mine — is core to U.S. revival efforts, building separation and magnet capacity with government backing, and is among the more mature, revenue‑generating names. USA Rare Earth controls Texas’ Round Top deposit (heavy RE + lithium, etc.) and has been bid up post‑listing on strategic value; it remains pre‑revenue and high beta. UAMY is one of few North American antimony producers, small in scale but strategically notable given DLA contracts; fundamentals hinge more on policy than on structural growth. Overall, many names are early stage with elevated hype risk. Focus on who can deliver capacity and long‑term offtakes. Policy can whipsaw prices (bans, tariffs, contracts). Resource equities are also rate/commodity‑cycle sensitive; small caps are often first to sell off when the dollar strengthens or commodities fall.

Lithium: The “White Oil” of the EV Era

Section titled “Lithium: The “White Oil” of the EV Era”

Industry context and strategic role. Lithium — the “white oil” of the energy‑transition era — is indispensable to Li‑ion batteries. With EVs and storage booming, it has become a strategic resource. The U.S. classifies lithium as critical and sees it as central to EV‑supply‑chain autonomy. Domestic resource endowment is modest: only one producing site today (Silver Peak brine, NV), far short of needs. Much of U.S. lithium arrives via imports from South America/Australia and is then refined in China. USGS estimates U.S. dependence on lithium compounds exceeded 50% in 2022. Any geopolitical disruption or export control poses EV supply risks. Washington is backing U.S. projects (e.g., DOE loans/guarantees to Lithium Americas’ Thacker Pass; R&D into clay/geothermal‑brine extraction) and tying IRA consumer incentives to sourcing from the U.S. or FTA partners — elevating lithium’s strategic importance.

Supply–demand and outlook. Lithium saw a roller‑coaster: a 2021–2022 price spike on EV demand, followed by a 2023 correction as supply growth outpaced demand. Fastmarkets estimates 2024 lithium supply up ~22% (China, Australia, South America, Africa), while demand remained strong but temporarily lagged, creating a near‑term surplus. Medium to long term, the outlook remains positive: global lithium consumption in 2024 is ~220kt LCE (+29% y/y), with EV demand projected to grow 12% annually through 2030. At that pace, 2030 demand could be >3× today. New supply is coming: faster expansions in Australia/South America; Africa could contribute 18% of hard‑rock supply by 2030; U.S. projects (Thacker Pass; Arkansas geothermal brines) may come online 2026–2030. Risks: mining projects are multi‑year with tough permitting; U.S. timelines have slipped (e.g., Thacker Pass litigation). Refining is concentrated in China, where cash costs ($8k/t) are well below North America (>$20k/t), meaning ore may still need overseas processing. Closing the midstream gap requires heavy investment in U.S. chemical refining and community/ESG solutions. Tech shifts (solid‑state, sodium‑ion) could alter mix, but most forecasts still see Li‑ion dominance through 2030 — lithium’s centrality likely persists.

Representative companies & investment take. Upstream and midstream leaders include Albemarle (ALB), Lithium Americas (LAC), and Sigma Lithium (SGML). Albemarle, a top global producer with Chilean brines and Australian rock, is mature and cash‑generative, and plans U.S. refining capacity — a core, more defensive exposure, albeit earnings remain price‑sensitive. Lithium Americas is developing Thacker Pass, the largest U.S. clay resource, with GM investment and government support; 2026 start‑up is targeted, leaving the company in investment mode with valuation reflecting resource optionality — monitor progress and capex closely. Sigma operates high‑grade Brazilian assets and has begun trial output; an emerging cash‑flow name but subject to rumor‑driven swings. After prior exuberance, valuations have reset, offering long‑term entry points — but differentiate by stage: producers are more resilient; developers offer higher upside with higher risk. A prudent approach is a “core + satellite” mix: core allocation to leaders for baseline returns, plus smaller satellite bets on high‑beta names for potential alpha. Mind the price cycle — build on weakness and avoid chasing peaks.

Section titled “Macro Backdrop & Strategy: Investing Amid Supply‑Chain Rewiring”

These segments are shaped by macro forces. First, intensifying U.S.–China rivalry puts “de‑risking” and on‑shore manufacturing at the center of security policy, not just economics. JPMorgan CEO Jamie Dimon has said the U.S. has relied too much on unreliable supply chains for security‑critical minerals and products — a vulnerability now laid bare. Hence the government’s industrial‑policy toolkit — subsidies, tariffs, CFIUS‑style reviews, and export controls — is being deployed to reshape key sectors. Carbon‑neutral commitments also require massive clean‑energy infrastructure, turbo‑charging demand for lithium, cobalt, nickel, rare earths, etc. Meanwhile, geopolitical shocks (trade tensions, conflicts) amplify supply uncertainty; countries increasingly securitize resources (e.g., China’s repeated curbs on critical‑mineral exports; the U.S. using the NDAA and related measures to prioritize domestic supply). Second, “friend‑shoring” is real: U.S. alliances with resource‑rich partners (Australia, Canada, parts of Africa) aim to build diversified, lower‑risk networks (e.g., the Minerals Security Partnership). Expect a more distributed raw‑materials map and more cross‑border opportunities — but also local instability and competition with Chinese capital. Third, macro cycles matter: inflation can lift commodity prices and resource cash flows, but higher rates raise project financing costs and compress multiples for growth names. The recent hiking cycle pressured renewables/storage equities; over the long run, however, real‑asset scarcity can trump monetary headwinds if structural shortages emerge. Investors should weigh macro and industry cycles and time entries — adding on weakness to policy‑supported quality names can be effective.

Section titled “GPU Market Landscape: Nvidia Dominates, AMD Is Catching Up”
