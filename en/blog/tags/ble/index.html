How ChatGPT Helps Fix Bluetooth Device UI Chaos

Oct 16, 2025
Vibe Sparking

Scenario recap: You built a mobile app that scans nearby BLE devices. Your office/ward/desk is full of the same model, all with the same name. The scan results keep “jumping” as new devices arrive; just as users tap one, it gets pushed away — pure frustration for a PM.

This article doesn’t go into protocol internals. Instead, it shows how to use ChatGPT as your AI interaction partner: turn complex technical constraints into a UI your team can build, users can understand, and QA can validate.

1) Align on technical boundaries first, then pick the UI

Section titled “1) Align on technical boundaries first, then pick the UI”

For BLE scanning, a few facts directly shape your design:

  • Advertising payload: Useful info includes local name, service UUIDs, manufacturer data, Tx Power, etc. These can help you distinguish devices and filter. They’re read from the advertisement data structures.
  • iOS does not expose MAC addresses (privacy). Apps receive a CBPeripheral.identifier-style UUID, and it is not guaranteed to be permanently stable. Do not bake “display/remember MAC address” into any cross‑platform plan. Prefer short codes/manufacturer data from the advertisement as the differentiator.
  • Android can pre-filter and batch callbacks via ScanFilter and ScanSettings (e.g., filter by service UUID/manufacturer data and use setReportDelay for batched dispatch). This dramatically reduces UI jitter.

With these boundaries clear, ChatGPT can propose cross‑platform interaction patterns that are actionable, not just theory.

2) Break “I can’t tap the right one” into three solvable UX problems

Section titled “2) Break “I can’t tap the right one” into three solvable UX problems”

A. Recognition: Too many identical names — which one do I pick?

Section titled “A. Recognition: Too many identical names — which one do I pick?”

Solution: Generate a human‑recognizable short label per device.

  • Ask hardware to add a short code in the advertised name or Manufacturer Data (e.g., last 4 digits of S/N or batch + checksum). Surface it as a subtitle: Device Name · Short Code 3K7F. This is a common use of BLE advertisement.
  • iOS cannot show MAC: use the short code and/or service signature from the advertisement as the distinguishing factor. Don’t promise MAC in product specs.

B. Jitter: List jumps as it refreshes; finger misses the target

Section titled “B. Jitter: List jumps as it refreshes; finger misses the target”

Solution: Decouple “data changes” from “interface changes”.

  • Batched callbacks: On Android, enable ScanSettings.Builder#setReportDelay(xxx) so N updates arrive as a batch. Diff‑update the list once to reduce item shifts.
  • Freeze interactive zones: When the user long‑presses or enters multi‑select, freeze sorting and positions. Only update light fields inside items (e.g., RSSI value). Merge the full refresh after the action. This follows the principle “don’t let refreshes interrupt the user’s task”.

C. Decision load: Too noisy — help me narrow scope first

Section titled “C. Decision load: Too noisy — help me narrow scope first”

Solution: Put filter controls where the hand is.

  • Use Filter/Assist Chips above the list for one‑tap toggles like “Recently used only”, “Strong signal only”, “Connectable only”, “Target service only”. Chips are made for quick filtering.
  • Make filtering/sorting intent‑aligned — avoid large jumps that break focus. NN/g specifically recommends minimizing “view jumping” when applying filters.

3) A ready‑to‑ship UI package for PMs (via ChatGPT)

Section titled “3) A ready‑to‑ship UI package for PMs (via ChatGPT)”

You can paste the following to ChatGPT to polish copy, generate a prototype (describe it), or output component and acceptance lists.

(1) List item anatomy

Section titled “(1) List item anatomy”
  • Title: Device name (append batch/model if names collide)
  • Subtitle: Short code · Service overview (e.g., 3K7F · Battery, UpdateSvc)
  • Right‑side info: RSSI (supports the “tap the one you’re close to” mental model; if showing distance, prefer coarse buckets strong/medium/weak to avoid misleading precision)
  • Status badges: Recently used / New device (reduces decision cost). Use small badges or Chips (MD3).

(2) Top‑level quick filters (Chips)

Section titled “(2) Top‑level quick filters (Chips)”
  • Recently used only, Strong signal only (> −70 dBm), Connectable only, Target service only
  • Interaction policy: Keep the viewport stable on toggle. Try not to move the currently focused item.

(3) Sorting and grouping (avoid the alphabetical trap)

Section titled “(3) Sorting and grouping (avoid the alphabetical trap)”
  • Default: Sort by last interaction time descending (user mental model: the thing I just used should be first).
  • Alternative: Group into Recently used / New devices, then sort within groups by RSSI or last seen time. As NN/g notes, alphabetical sorting is rarely the best choice for decision tasks.

(4) Refresh and selection stability

Section titled “(4) Refresh and selection stability”
  • Combine Android batched callbacks with list diffing. When entering multi‑select/long‑press, freeze ordering; only “light‑update” fields inside items (e.g., RSSI).

(5) Cross‑platform identifier strategy

Section titled “(5) Cross‑platform identifier strategy”
  • Android: Use MAC (when available) or the advertised short code.
  • iOS: Use CBPeripheral.identifier plus advertised short code/service signature. Do not commit to displaying/remembering MAC.

4) Copy and naming you can drop in (EN/CN hints + field names)

Section titled “4) Copy and naming you can drop in (EN/CN hints + field names)”

Group headers (UI copy)

  • Recently Used Devices / 最近使用过的设备
  • New Devices / 新设备

Filter chips (UI copy)

  • Recently used / 只看最近使用
  • Strong signal only (> −70 dBm) / 只看强信号（> −70 dBm）
  • Connectable only / 只看可连接
  • Target service only / 只看目标服务

Interfaces/fields (snake_case)

5) Turn “plan → artifacts” with ChatGPT

Section titled “5) Turn “plan → artifacts” with ChatGPT”

Feed the “requirements + components” above to ChatGPT and ask for:

  • Prototype description for design: item hierarchy, chip interactions, empty‑state copy.
  • Engineering checklist: Android ScanFilter/ScanSettings config (with setReportDelay examples), iOS CoreBluetooth scanning and identifier usage cautions.
  • Acceptance cases: With 30+ devices, fluctuating RSSI, and stacked filters, can users still reliably select the intended device?

6) Takeaways: Know the boundaries, design the right interactions

Section titled “6) Takeaways: Know the boundaries, design the right interactions”
  • Lean on advertisement data to distinguish devices; don’t insist on MAC on iOS.
  • Use batched dispatch + frozen ordering to tame list jitter.
  • Make filtering/sorting intent‑centric with lightweight Chips; surface “Recently used/New” to lower selection cost.

With these constraints and patterns in hand, ChatGPT becomes your on‑call interaction consultant — bridging what’s buildable with what users grasp — turning the tiny pain of “I can’t tap it” into a chance for a big UX upgrade.

Triple Your HarmonyOS App Development Speed with Claude Code + Documentation-Driven Programming + BDD Business-Driven Testing

Sep 26, 2025
Vibe Sparking

This methodology specifically addresses three major pain points: “AI can write code but doesn’t understand HarmonyOS,” “ArkTS keeps throwing errors,” and “regression testing can’t keep up.” The core principle: Drive Claude Code with documentation, and secure quality through BDD automation.

Introduction: Don’t Chat Randomly with AI – Write Good Documentation First

Section titled “Introduction: Don’t Chat Randomly with AI – Write Good Documentation First”

Many developers immediately tell AI: “Help me build a HarmonyOS app for Bluetooth scanning, connecting, and time synchronization.” The result? Section titled “2. Connecting to a Device and Enabling Notifications”

Section titled “4. ArkTS Compilation Errors and Key Fixes”

Summary Document: docs/ble-arkts-fix-summary.md

  • Missing Class Properties:
