Chat UI Swift: A SwiftUI Chat Application Showcasing AnyLanguageModel and Swift-HuggingFace

Section titled “Chat UI Swift: A SwiftUI Chat Application Showcasing AnyLanguageModel and Swift-HuggingFace”

Building AI-powered chat applications on Apple platforms has traditionally required navigating multiple SDKs, APIs, and authentication flows. mattt/chat-ui-swift simplifies this landscape with a beautiful SwiftUI reference implementation that demonstrates how to integrate multiple language model providers through a unified API.

What is Chat UI Swift?

Section titled “What is Chat UI Swift?”

Chat UI Swift is a SwiftUI chat application created by Mattt Thompson, the legendary developer behind AFNetworking and NSHipster. It serves as a demonstration app for two powerful Swift libraries:

  1. AnyLanguageModel - A drop-in replacement for Apple’s Foundation Models framework with support for custom language model providers
  2. swift-huggingface - A Swift client for the Hugging Face API

GitHub Repository: mattt/chat-ui-swift Author: Mattt Thompson License: MIT

Key Features

Section titled “Key Features”

Currently Supported

Section titled “Currently Supported”
  • Apple Intelligence - Native integration with Apple Foundation Models (macOS 26+) for on-device AI
  • Hugging Face Integration - Connect to Hugging Face with OAuth 2.0 authentication
  • Streaming Responses - Real-time streaming of AI responses for a responsive user experience
  • Chat Persistence - Save and manage multiple conversations

Planned Features

Section titled “Planned Features”
  • MLX Model Support - Download and run models locally using MLX
  • CoreML Integration - Support for CoreML-optimized models
  • GGUF Format Support - Load GGUF models from Hugging Face Hub
  • Model Downloads - Browse and download models directly from Hugging Face
  • BYOK (Bring Your Own Key) - Support for OpenAI, Anthropic, and other inference providers

Understanding AnyLanguageModel

Section titled “Understanding AnyLanguageModel”

The real power behind chat-ui-swift is AnyLanguageModel, a Swift package that provides a unified API for accessing multiple language model providers.

Security Best Practices

Section titled “Security Best Practices”

The AnyLanguageModel documentation includes important guidance on API credential security:

Never Hardcode API Keys

Section titled “Never Hardcode API Keys”

Malicious actors can reverse-engineer your app binary or observe network traffic to extract embedded credentials.

Recommended Approaches

Section titled “Recommended Approaches”
  1. Bring Your Own Key (BYO) - Users provide their own keys, stored in Keychain
  2. Proxy Server - Route requests through your authenticated service

Important: Apple App Review has often rejected apps using the BYOK model. Consider using a proxy server for production apps.

Conclusion

Section titled “Conclusion”

Chat UI Swift is more than a demo app—it’s a reference implementation for building AI-powered chat applications on Apple platforms. By leveraging AnyLanguageModel and swift-huggingface, developers can:

  • Use a unified API for multiple LLM providers
  • Switch providers without changing application code
  • Integrate Apple Intelligence alongside cloud models
  • Authenticate with Hugging Face via OAuth
  • Build beautiful SwiftUI chat interfaces

The project exemplifies modern Swift development practices: protocol-oriented design, async/await concurrency, and careful attention to security.

Slidev-AI: Transform Ideas into Professional Presentations with AI

Creating presentations has always been a time-consuming task—designing layouts, formatting content, and ensuring visual consistency. Slidev-AI changes the game by leveraging LLM technology to generate elegant, developer-friendly presentations from natural language descriptions.

1. What is Slidev-AI?

Section titled “1. What is Slidev-AI?”

Slidev-AI is an open-source web application that combines the power of Large Language Models (LLMs) with Slidev—a Markdown-based presentation framework beloved by developers. It received the Best Application Award in the ModelScope MCP&Agent Competition.

Core Value Proposition

Section titled “Core Value Proposition”

“From ideas to presentations to content distribution — our mission is to shorten the distance.”

Google's 68-Page Prompt Engineering Whitepaper: 10 Best Practices That Actually Work

Dec 25, 2025
Vibe Sparking

Google recently released a 68-page whitepaper on prompt engineering authored by Lee Boonstra, and it’s already being called “the best structured, most readable, and most example-rich guide to LLM prompting as of 2025.” Here are the 10 best practices distilled from this comprehensive guide.

Why This Whitepaper Matters

Section titled “Why This Whitepaper Matters”

LLMs are prediction engines, not reasoning engines. Every word you provide guides the model to predict the next token. Understanding this fundamental principle is the key to mastering prompt engineering.

Techniques may become outdated, but the underlying principles remain constant.

1. Provide Examples, Don’t Just State Requirements

Section titled “1. Provide Examples, Don’t Just State Requirements”

Few-shot prompting consistently outperforms zero-shot approaches. When you provide 3-5 examples, the model understands your expectations far better than any amount of explanation.

Why it works: Examples let the model capture the structure, tone, and format you want. Instead of explaining what “professional” means, show a professional response.

Example:

2. Keep It Simple

Section titled “2. Keep It Simple”

Use action verbs. Avoid obscure expressions. One clear sentence beats three vague paragraphs.

LLMs don’t need pleasantries or context padding. They need direct instructions.

Preferred verbs:

  • Analyze
  • Create
  • Generate
  • Identify
  • Summarize
  • Compare
  • Extract

Before: “I was hoping you could perhaps help me write something that might be useful for understanding the topic of machine learning, if that’s not too much trouble.”

After: “Summarize machine learning in 3 sentences for a non-technical audience.”

3. Specify Output Format Explicitly

Section titled “3. Specify Output Format Explicitly”

Vague instructions produce random results. If you want JSON, say JSON. If you want 500 words, say 500 words.

The more specific your format requirements, the more predictable your outputs.

Weak: “Generate a blog post about video game consoles.”

Strong: “Generate a 3-paragraph blog post about the history of video game consoles. Use conversational style. Each paragraph should be 4-6 sentences. Include one specific statistic per paragraph.”

Pro tip: Request JSON format for structured, processable responses that naturally limit hallucinations:

4. Use Instructions, Not Constraints

Section titled “4. Use Instructions, Not Constraints”

Tell the model what TO DO, not what NOT to do.

Negation is inherently ambiguous for language models. “Don’t be verbose” is less effective than “Limit response to 100 words.”

Avoid ThisDo This Instead
”Don’t list video game names""Only discuss the console, company, year, and sales figures"
"Don’t be wordy""Limit each paragraph to 3 sentences"
"Don’t make things up""If uncertain, respond with ‘I cannot verify this’”

The psychology: Models process positive instructions more reliably because they define a clear action space rather than an infinite exclusion zone.

5. Control Token Length

Section titled “5. Control Token Length”

Cost scales linearly with tokens. Generating 1000 words costs 10x more than generating 100 words.

Always set explicit length constraints when token efficiency matters.

Techniques:

  • “Limit output to 80 characters”
  • “Explain in a tweet-length message”
  • “Respond in exactly 3 bullet points”
  • “Maximum 50 tokens”

Bonus: Token limits also force conciseness, often improving output quality.

6. Use Variables for Reusable Templates

Section titled “6. Use Variables for Reusable Templates”

Extract changeable elements into variables. One well-crafted template can be reused hundreds of times.

Benefits:

  • Consistency across multiple queries
  • Easier A/B testing
  • Programmatic prompt generation
  • Version control friendly

7. Experiment With Different Input Formats

Section titled “7. Experiment With Different Input Formats”

Questions, statements, and instructions produce different results from identical content.

Test multiple approaches:

  • Question: “What are the benefits of exercise?”
  • Statement: “Exercise has multiple benefits. Explain them.”
  • Instruction: “List 5 benefits of regular exercise with scientific backing.”
  • Role-based: “As a fitness coach, explain why clients should exercise.”

Finding: Lists, paragraphs, and dialogue formats also yield varying stability. Find what works best for your use case.

8. Mix Examples in Classification Tasks

Section titled “8. Mix Examples in Classification Tasks”

Don’t cluster all positive examples together. Randomize example order to force pattern learning, not sequence memorization.

Strong (mixed):

Why: Models learn discriminating features when forced to distinguish between adjacent examples of different classes.

9. Adapt to Model Updates

Section titled “9. Adapt to Model Updates”

GPT-4 to GPT-4.5, Gemini 1.0 to 2.0 - prompts need recalibration.

Each model generation has different strengths, quirks, and optimal configurations. What worked yesterday may underperform today.

Best practices:

  • Re-test core prompts with each model update
  • Monitor output quality metrics over time
  • Maintain a prompt changelog
  • Keep baseline examples for comparison

Reality check: Continuous prompt optimization is not optional maintenance - it’s core product work.

10. Document Every Attempt

Section titled “10. Document Every Attempt”

Use a table to track: model, temperature, prompt, output. Three days later, you won’t remember which version worked best.

Recommended tracking fields:

FieldPurpose
ModelWhich model version
TemperatureRandomness setting
PromptExact prompt used
OutputResult received
Quality1-5 rating
NotesWhat worked/didn’t

For RAG systems: Also document which factors influenced content insertion - queries, chunking settings, and chunk outputs.

Bonus: Chain of Thought (CoT) Best Practices

Section titled “Bonus: Chain of Thought (CoT) Best Practices”

For complex reasoning tasks, set temperature to 0 for rigorous, single-answer problems requiring strict logical reasoning.

Self-Consistency Prompting: Send the same Chain-of-Thought prompt multiple times at high temperature, then “vote” on the most common final answer. This makes answers more robust and reduces hallucinations.

Step-Back Prompting: Instead of solving directly, first prompt the model to consider general principles or patterns, then use that response to guide the specific task.

The Core Insight

Section titled “The Core Insight”

All 10 practices share the same underlying logic:

LLMs are prediction engines, not reasoning engines. Every word you provide guides token prediction. Specificity reduces ambiguity. Examples outperform explanations. Positive instructions beat negative constraints.

Techniques evolve. Principles persist.

Agent Skills for Context Engineering: Build Production-Grade AI Agent Systems

Section titled “Agent Skills for Context Engineering: Build Production-Grade AI Agent Systems”

In the rapidly evolving landscape of AI development, context engineering has emerged as one of the most critical disciplines for building effective agent systems. Unlike prompt engineering, which focuses on crafting effective instructions, context engineering addresses the holistic curation of all information entering a model’s limited attention window.

Agent Skills for Context Engineering by Murat Can Koylan is a comprehensive, open-source collection designed to teach the art and science of curating context to maximize agent effectiveness across any platform.

What is Context Engineering?

Section titled “What is Context Engineering?”

Context engineering is the discipline of managing the language model’s context window. The fundamental challenge is that context windows are constrained not by raw token capacity but by attention mechanics.

The Core Problem

Section titled “The Core Problem”

As context length increases, models exhibit predictable degradation patterns:

  • Lost-in-the-Middle Phenomenon - Information in the middle of long contexts gets less attention
  • U-Shaped Attention Curves - Models focus more on beginning and end of context
  • Attention Scarcity - Limited “attention budget” must be allocated wisely

The Solution

Section titled “The Solution”

Effective context engineering means finding the smallest possible set of high-signal tokens that maximize the likelihood of desired outcomes. This requires understanding what context is, how it degrades, and how to optimize it.

Key Concepts Deep Dive

Section titled “Key Concepts Deep Dive”

1. Context Fundamentals

Section titled “1. Context Fundamentals”

Understanding the anatomy of context in agent systems is essential:

Components of Context:

  • System prompts
  • Tool definitions
  • Retrieved documents
  • Message history
  • Tool outputs

Each component competes for limited attention. Effective agents prioritize high-signal information and compress or eliminate low-signal content.

2. Context Degradation Patterns

Section titled “2. Context Degradation Patterns”

Recognizing failure patterns helps prevent them:

Lost-in-Middle: Critical information placed in the middle of long contexts receives less model attention.

Context Poisoning: Bad or conflicting information corrupts the model’s understanding.

Context Distraction: Irrelevant information diverts attention from important content.

Context Clash: Contradictory instructions or data create confusion.

3. Multi-Agent Patterns

Section titled “3. Multi-Agent Patterns”

Three primary architectures for multi-agent systems:

Orchestrator Pattern: A central agent coordinates specialized sub-agents, managing task delegation and result aggregation.

Peer-to-Peer Pattern: Agents communicate directly without central coordination, suitable for collaborative tasks.

Hierarchical Pattern: Layered agent structures where higher-level agents manage groups of lower-level agents.

4. Memory Systems

Section titled “4. Memory Systems”

Effective agents require multiple memory types:

Short-term Memory:

  • Recent conversation context
  • Current task state
  • Immediate working memory

Long-term Memory:

  • Persistent knowledge stores
  • User preferences
  • Historical patterns

Graph-based Memory:

  • Relationship maps between concepts
  • Entity connections
  • Semantic networks
