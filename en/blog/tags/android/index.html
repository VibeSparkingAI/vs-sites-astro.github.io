One Prompt, Three Stacks: Shipping Session Auto‑Lock with Claude Code

Oct 15, 2025
Vibe Sparking

Write an absurdly short prompt and the agent patches Web (TypeScript), App (Flutter), and API (Rails) in one go: adds the feature, wires permissions, localizes strings, and even creates migrations and runs checks. The best part: easy to write, easy to reuse.

Sounds unbelievable? Read on.

Why this approach works now

Section titled “Why this approach works now”
  • Claude Code has plugins: slash commands, sub‑agents, MCP servers, and lifecycle hooks bundled into installable units. Teams can turn cross‑stack edits into a standard pipeline.
  • MCP (Model Context Protocol) is like giving AI a USB‑C port: plug IDEs/agents into your repos, databases, and tools with structured read/write and execution.
  • Remote MCP is supported: connect to cloud services with auth; in enterprise multi‑repo/multi‑environment setups it’s much simpler.
  • Official best practices for agentic coding emphasize structured goals and checklist‑style requirements, making outcomes more reliable.
  • New models and ecosystem momentum help. Sonnet 4.5 is stronger for complex agents and “computer use” scenarios; paired with plugins/MCP it makes the path from prompt to patch much smoother.

—

A stand‑in scenario: Session Auto‑Lock Duration

Section titled “A stand‑in scenario: Session Auto‑Lock Duration”

We’re adding a new item under Security Settings: Session Auto‑Lock (Duration).

  • Two options: Standard (2 minutes) and Strict (30 seconds)
  • Only authorized users can view/modify
  • Web Admin, App Settings, and API change together
  • i18n in English and Chinese

This feature is generic enough and avoids any sensitive business logic.

—

A structured prompt for Claude Code (drop‑in)

Section titled “A structured prompt for Claude Code (drop‑in)”

Treat the following as a “work order”: short sentences, imperative verbs, 2–4 lines per block. Fewer back‑and‑forth questions, more results.

This “clear landing points + checklisted requirements” style works best for agentic coding. Claude Code also recommends structured instructions to reduce iteration.

—

What changes look like across the three stacks

Section titled “What changes look like across the three stacks”

Web (TypeScript)

Section titled “Web (TypeScript)”
  • Add a Session Auto‑Lock dropdown under Security Settings.
  • Bind permission key session_auto_lock; hide or disable if unauthorized.
  • i18n:
    • en_US: Standard / Strict
    • zh_CN: 标准（2分钟）/ 严格（30秒）

App (Flutter)

Section titled “App (Flutter)”
  • Add the same dropdown on the Security page.
  • Read/write to the security configuration; disable or hide if unauthorized.
  • Copy matches Web for consistency.

API (Rails)

Section titled “API (Rails)”
  • Migration: session_auto_lock_duration, default standard.
  • Model constants: STANDARD / STRICT / ALL_OPTIONS.
  • Permission registration: session_auto_lock; whitelist validation on write; 403 if unauthorized.

These steps can be chained via plugins + MCP: the agent reads/writes across repos, generates migrations, runs tests, and opens PRs.

—

How teams operationalize it

Section titled “How teams operationalize it”

Preparation

  • For Web/App/API, wire up a lightweight agent or expose the sequence as a plugin command.
  • Connect local/remote MCP; bind repos and CI.

Execution

  • Trigger with the “work‑order‑style” prompt above.
  • The agent produces patches, migrations, i18n, and permission registration; then opens MR/PR.

Review

  • CI runs Lint/Tests; failures go back to the agent for fixes.
  • Humans review only key diffs and acceptance criteria.
  • Need compliance/auditing? Add operation logging and rollback scripts to the plugin.
    • Note: tooling is increasingly focused on governance and safety; agent platforms are improving policy and usage controls.

—

Pitfalls and tips

Section titled “Pitfalls and tips”
  • Don’t write scattered instructions. Short imperative sentences with concrete targets work best.
  • Hard‑code the permission keys, defaults, and whitelist options; don’t leave them to model “creativity”.
  • Provide the final copy strings up front to avoid another round trip.
  • Remote MCP is great, but enforce authentication and least privilege; separate environments and roles.
  • Save this prompt in the team’s instruction library/plugin. Next time you only swap the “feature name + enum options”.

—

Closing

Section titled “Closing”

Let standards handle the complex parts, and humans keep the simple parts. With one structured prompt + pluginized agent + MCP, you can flatten the “cross‑stack, cross‑repo, cross‑language” busywork. You focus on defining the requirement and making the final call; let the machine run the rest.

Why rename the APK file?

Section titled “Why rename the APK file?”

By default it’s app-release.apk. When sending to QA or uploading to CI artifacts, everything has the same name. It’s guesswork to identify versions — frustrating and error-prone.

With a descriptive name you immediately know:

  • Which branch produced it
  • Which version it is
  • When it was built

Productivity goes up immediately.

What you need

Section titled “What you need”

  • A standard Flutter/Android project.

  • Any Android Gradle Plugin (AGP) version works, but naming approaches differ. The robust approach is to copy/rename after assemble. That’s the path used below and is widely adopted under newer AGP APIs.

  • Flutter’s --dart-define values are passed via Gradle -Pdart-defines as Base64. We can decode and read them in build.gradle.

Copy and use: add this to android/app/build.gradle

Section titled “Copy and use: add this to android/app/build.gradle”

Place under the android { ... } section with applicationVariants (Groovy syntax). Idea: get branch name + decode --dart-define + build the new file name + copy/rename after assemble. This “copy and rename” technique is more reliable across AGP versions.

Notes:

  • Local uses git rev-parse; CI can fallback to the GIT_BRANCH environment variable.
  • Newer AGP often breaks “rename in place”; copy+rename is the portable approach.
  • See Flutter docs for build configuration locations and packaging steps.

  • Flutter output directory quirks Some builds land under flutter-apk/ or ignore outputFileName. Use the copy+rename fallback.

  • CI with detached HEAD git rev-parse can’t read the branch. Use env fallback (GIT_BRANCH or CI‑specific vars).

Android Studio Finally Supports Gemini Agent for Automated Coding

Aug 3, 2025
Vibe Sparking

In the latest Android Studio Narwhal Feature Drop, Google has fully “embedded” Gemini into the IDE—not just for chatting, but acting like a “co-pilot” that can plan tasks and modify code in batches. You can use Rules to instantly lock in your project’s coding standards, and even connect your own Gemini API Key to unlock 1 million tokens of ultra-long context. In other words, Android development now has its first truly practical Agent Mode: it can read and write multiple files, run Gradle, create branches, submit PRs, and even write unit tests before fixing its own bugs—leaving you with nothing but nodding or shaking your head.

Introduction

Section titled “Introduction”

“How do you handle implementing a feature that requires changes across dozens of files?” In the past, we could only repeatedly switch windows, copy paths, and manually search through code. Now, Gemini Agent Mode takes over these mechanical steps—you just need one high-level instruction. The AI enhancements in Narwhal Feature Drop are designed to leave “thinking” to humans and “tedious work” to machines.

Get Started with Agent Mode in 1 Minute

Section titled “Get Started with Agent Mode in 1 Minute”
  1. Click Gemini → Agent in the sidebar, enter your goal: “Add dark mode to the homepage.”
  2. The Agent will first show a breakdown plan: which files need changes, which Gradle tasks to run.
  3. For each step, it generates code, applies patches, and runs tests—you can accept/reject each one.
  4. After everything passes, submit a PR with one click.

Why is it more powerful?

  • Supports cross-file batch modifications, breaking free from the “can only return one code snippet” bottleneck.
  • Built-in IDE permissions allow direct disk read/write and script execution.
  • Transparent planning with the ability to intervene and adjust at any time.

Custom Prompt Rules

Section titled “Custom Prompt Rules”

Want AI to always output Kotlin + Jetpack Compose? Open Prompt Library → Rules, write down your project conventions, and they’ll be automatically applied to every conversation thereafter. Teams can check .idea into version control, and rules will be shared along with the code—no more “verbal reminders” needed.

Connect Gemini 2.5 Pro, Unlock 1M Context

Section titled “Connect Gemini 2.5 Pro, Unlock 1M Context”

Free-tier models only have a few thousand tokens of context, often “forgetting” context during complex refactoring. Narwhal supports “Bring Your Own Key”: enter your Gemini API Key in settings to switch to 2.5 Pro, fitting entire code chains into a single conversation. Paid GCP projects enjoy this privilege by default.

Other AI-Powered Details

Section titled “Other AI-Powered Details”
  • XR Project Wizard: While primarily focused on XR, it also leverages Gemini logic prompts to automatically populate templates and best practices.
  • MCP Extensions: The initial version supports stdio transport and can further integrate with external tools like GitHub, with full streaming protocol support coming in the future.
  • Performance Optimizations: This stable release incorporates extensive feedback from Canary users, fixing issues like context loss and plan freezing.

Conclusion

Section titled “Conclusion”

Agent Mode isn’t just an “intelligent dialog box”—it’s a full-stack teammate that can modify code, run scripts, and write tests. It transforms Android Studio into a truly AI-powered IDE.
