Triple Your HarmonyOS App Development Speed with Claude Code + Documentation-Driven Programming + BDD Business-Driven Testing

Sep 26, 2025
Vibe Sparking

This methodology specifically addresses three major pain points: “AI can write code but doesn’t understand HarmonyOS,” “ArkTS keeps throwing errors,” and “regression testing can’t keep up.” The core principle: Drive Claude Code with documentation, and secure quality through BDD automation.

Introduction: Don’t Chat Randomly with AI – Write Good Documentation First

Section titled “Introduction: Don’t Chat Randomly with AI – Write Good Documentation First”

Many developers immediately tell AI: “Help me build a HarmonyOS app for Bluetooth scanning, connecting, and time synchronization.” The result? Claude Code produces seemingly sophisticated code that turns completely red upon compilation. Are you familiar with ArkTS? Do you understand the ArkUI lifecycle? How do you handle regression testing? All uncertain.

HarmonyOS officially provides ArkTS coding standards, ArkUI declarative development paradigms, and performance practices – these are standard components that Claude Code “crashes repeatedly if not learned properly.” ArkTS is HarmonyOS’s primary language, sharing roots with TypeScript but having its own semantics and ecosystem. Writing with old JavaScript habits makes it easy to fall into traps. The good news is: Claude Code already supports “reading repositories, editing files, writing and running tests, executing terminal commands.” As long as you provide the right context, it can consistently produce engineered results.

I. Documentation-Driven Programming: Make Claude Code Follow Your Lead

Section titled “I. Documentation-Driven Programming: Make Claude Code Follow Your Lead”

Objective: Clearly explain requirements, value, implementation, and acceptance criteria on one page; let Claude Code “build according to specifications” within the HarmonyOS context.

1) Write “One-Page” Requirements (can be directly placed in repository /docs/feature-xxx.md)

Section titled “1) Write “One-Page” Requirements (can be directly placed in repository /docs/feature-xxx.md)”

  • Functionality: Bluetooth device scanning → list display → connection → time synchronization → status reading → command sending → disconnection

  • Business Value: Store associates complete device initialization in 1 minute, reducing manual errors by 80%

  • Platform: HarmonyOS NEXT, ArkTS + ArkUI; DevEco Studio build

  • Interface/Model: Device names start with XX; display battery level after connection; “injection” command returns success flag

  • Acceptance Criteria (BDD format):

    • Given Application is on the home page
    • When Click “Start Scan”, see device starting with XX in list and click Connect
    • Then Enter connected page, see device name/battery level; clicking “Sync Time,” “Refresh Status,” “Send Injection Command” all return success

Remember to include key ArkTS/ArkUI constraints and best practices in documentation: naming, state management, performance considerations, etc. (from official style guides and state management guidelines). Provide specifications directly to Claude Code – it will follow these rules during generation.

2) Initial System Prompt for Claude Code (place in .claude/primer.md)

Section titled “2) Initial System Prompt for Claude Code (place in .claude/primer.md)”
  • Project framework (ArkUI page structure, directory conventions)
  • ArkTS code style requirements (variable/component naming, null checks, error handling)
  • Component inventory & event flow (Button → scanning → list → connection)
  • Test-first approach: Generate BDD scenarios and ArkTS test stubs first, then supplement business code
  • Failure retry: compilation error → read error → modify minimal set only

Officials and community both emphasize standards-driven coding, which significantly reduces rework and performance issues. Claude Code itself also recommends customizing context and environment (e.g., ignoring irrelevant files, providing dedicated README/CONTRIBUTING), which can significantly improve hit rates.

II. BDD Business-Driven Testing: Transform “Documentation” into “Executable Specifications”

Section titled “II. BDD Business-Driven Testing: Transform “Documentation” into “Executable Specifications””

In one sentence: Write requirements as Gherkin scenarios, run tests on DevEco/arkXtest framework, test red → AI fix → test green again.

1) Describe Business with Gherkin

Section titled “1) Describe Business with Gherkin”

The advantage of Gherkin/BDD is transforming “business language” directly into executable specifications, continuously validating whether system behavior matches documentation.

2) ArkTS Testing Implementation Selection

Section titled “2) ArkTS Testing Implementation Selection”
  • DevEco/arkXtest System: Provides unified framework for unit testing + UI testing + performance testing; UI test scripts execute on top of unit tests, meeting end-to-end validation needs.
  • IDE Support: DevEco Studio integrates testing framework and code test entry points, facilitating execution on real devices/simulators.

Some documentation also mentions HarmonyOS/ArkTS testing capability upgrades in the NEXT version, supporting more modern syntax and parameterization techniques, which can be adopted in your engineering templates as needed (based on official and authoritative sources).

Section titled “3) Example: UI Scenario Test Framework (extracted from actual project structure)”

  2. Wait for device discovery

4) Make “Test Red → AI Fix → Test Green Again” a Pipeline

Section titled “4) Make “Test Red → AI Fix → Test Green Again” a Pipeline”
  • Claude Code Loop: When failures occur, paste compilation/runtime logs back into conversation for minimal modification fixes (Claude’s official “environment tuning + context minimization” recommendations are crucial here).
  • IDE/CI Integration: One-click execution in DevEco Studio locally; CI integration with real devices/simulators for regression (refer to DevEco test entry points and official testing documentation).
  • Visual Acceptance: Place the manual acceptance step diagrams above in wiki, run Gherkin + UI tests for each version, ensuring “documentation is truth.”

III. Implementation Manual: 7-Step Checklist from 0 to 1

Section titled “III. Implementation Manual: 7-Step Checklist from 0 to 1”

Step 1 | Project Initialization

  • Create ArkTS project in DevEco Studio; select ArkUI declarative paradigm; confirm target API/device.
  • Recommend pulling ArkTS style guide and performance practices into repository /guides for both human and AI reference.

Step 2 | One-Page Requirements

  • Clearly write functionality, value, interfaces, states, exceptions, acceptance scenarios (Given/When/Then).
  • Place screenshot-style manual acceptance steps in README (the diagram section above).

Step 3 | Claude Code Startup

  • Use .claude/primer.md and /docs as primary context; limit it from reading irrelevant large files.
  • Have it generate test stubs & Gherkin first, then supplement business code and ArkUI pages.

Step 4 | ArkTS/ArkUI Coding

  • Follow naming, componentization, state management constraints; pay attention to performance practices and null/exception handling.

Step 5 | Test Implementation

  • Use arkXtest / DevEco testing for unit tests + UI tests; use Gherkin as top-level use case description (can call UI test APIs in step definitions).

Step 6 | Iterative Fixes

  • Feed failure logs to Claude Code for minimal fixes; loop locally/CI until all green.

Step 7 | Knowledge Base Accumulation

  • Record pitfalls & FAQ in /kb: such as BLE permissions, lifecycle, list performance.
  • Next time, only need to modify “one-page” requirements and a few step descriptions, Claude Code can reuse context to continue generation.

IV. Results: Quality, Speed, and Understanding – Triple Success

Section titled “IV. Results: Quality, Speed, and Understanding – Triple Success”
  • Speed: With the “documentation as code” approach, Claude Code is more stable on HarmonyOS projects, reducing back-and-forth guessing.
  • Quality: arkXtest + DevEco testing aligns unit tests/UI/performance, automates regression, leaving manual work for exploratory validation only.
  • Understanding: Standards and performance practices written in documentation, humans and AI follow the same rules, new team members can master ArkTS/ArkUI basics in 1 day.

Rapidly Develop a HarmonyOS 6 Bluetooth Device Management App with Claude Code: Bluetooth Interaction, ArkTS Compilation Fixes, and TDD/BDD Practices

Sep 22, 2025
Vibe Sparking

Rapidly Develop a HarmonyOS 6 Bluetooth Device Management App with Claude Code: Bluetooth Interaction, ArkTS Compilation Fixes, and TDD/BDD Practices

Section titled “Rapidly Develop a HarmonyOS 6 Bluetooth Device Management App with Claude Code: Bluetooth Interaction, ArkTS Compilation Fixes, and TDD/BDD Practices”

This article demonstrates how to quickly develop a Bluetooth device management app for Huawei’s HarmonyOS 6 using Claude Code. It covers typical features such as receiving Bluetooth advertisements, connecting to devices, setting the time, synchronizing status, performing infusions, and retrieving logs. The article also focuses on strategies for fixing ArkTS compilation errors and implementing TDD/BDD-oriented testing.

— Suitable for teams aiming to build BLE device applications through a “write, test, iterate quickly” approach.

1. Development Goals and Device Protocol Overview

Section titled “1. Development Goals and Device Protocol Overview”
  • Target Device: A Bluetooth device.
  • Typical Commands:
    • Read status, synchronize time, perform infusion, read records.

Implementation Files and Data Models:

  • entry/src/main/ets/utils/BleService.ets: An all-in-one wrapper for scanning and discovery, connection, notifications, and command sending.
  • entry/src/main/ets/utils/BleCommand.ets: Command construction (including CRC, length, and time/dose encoding).
  • entry/src/main/ets/utils/BleResponseParser.ets: Response parsing.
  • entry/src/main/ets/utils/BleAdvertisingParser.ets: Advertising data parsing (local name, UUID, manufacturer data, etc.).

2. How Claude Code Accelerates Native ArkTS Development

Section titled “2. How Claude Code Accelerates Native ArkTS Development”
  • Code Navigation and Refactoring:
    • Quickly locate BLE-related entry points, cross-references, and dependencies using commands; maintain consistency with batch renaming and interface extraction.
  • Assisted Fixing of ArkTS Compilation Warnings/Errors:
    • Automatically points out rule violations like arkts-no-spread and unknown/any, suggesting replacement code and type completion.
  • Generating Test Skeletons and Adding Assertions:
    • Quickly create and extend unit tests and UI integration tests based on Hypium.

By combining these capabilities, a high-frequency feedback loop of “modify → run tests → analyze logs → fix/refactor” can be established, significantly shortening the cycle from protocol design to an end-to-end working application.

3. Detailed Breakdown of the Bluetooth Interaction Flow

Section titled “3. Detailed Breakdown of the Bluetooth Interaction Flow”

1. Receiving Bluetooth Advertisements (Scanning and Parsing)

Section titled “1. Receiving Bluetooth Advertisements (Scanning and Parsing)”
  • Multi-path Scanning (HarmonyOS API v15+):
    • Entry Point: entry/src/main/ets/utils/BleService.ets startScan(...)
    • Event: BLEDeviceFind is continuously reported, internally using BleAdvertisingParser to parse the local name, service UUID, manufacturer data, etc.
  • Advertising Parsing: entry/src/main/ets/utils/BleAdvertisingParser.ets
    • Covers major AD types like Flags, Complete/Incomplete UUIDs, and Manufacturer Specific Data, making it easy to quickly identify the target device.

Associated Discovery Service: entry/src/main/ets/utils/BleDiscoveryService.ets encapsulates the discovery list, sorting by RSSI, automatic stopping on timeout, and a “trusted device” caching strategy. The UI side can directly subscribe to status/result callbacks.

2. Connecting to a Device and Enabling Notifications

Section titled “2. Connecting to a Device and Enabling Notifications”
  • After connecting and discovering services, you must “subscribe, then enable notifications, then read once more” before writing:
    • Enable Notifications: entry/src/main/ets/utils/BleService.ets
      • First, subscribe to the callback: gattClient.on('BLECharacteristicChange', ...)
      • Then, enable it: setCharacteristicChangeNotification(..., true)
      • Optionally, perform a readCharacteristicValue once to “warm up” the notification channel (required by some devices).
  • Notification Callback: entry/src/main/ets/utils/BleService.ets onCharacteristicChanged(...)
    • Use lowercase UUID comparison to avoid misses due to case differences.
    • Convert characteristic.characteristicValue to an ArrayBuffer before dispatching to a pending callback or a unified protocol parser.

3. Sending and Debugging Commands

Section titled “3. Sending and Debugging Commands”
  • Sending a Write: entry/src/main/ets/utils/BleService.ets sendCommand(...)
    • ArkTS disables object spreading (no-spread), so characteristicValue must be modified directly before writing.
    • Uniformly output hexadecimal strings for easier troubleshooting: BleCommand.toHexString(...)
  • Command Construction:
    • Read Status: entry/src/main/ets/utils/BleCommand.ets
    • Synchronize Time: entry/src/main/ets/utils/BleCommand.ets
    • Infusion Command: entry/src/main/ets/utils/BleCommand.ets (dose × 1000 converted to an integer, written in little-endian).

4. Synchronizing Time and Reading Status

Section titled “4. Synchronizing Time and Reading Status”
  • Synchronize Time: entry/src/main/ets/utils/BleService.ets, writes 0xA7 containing year, month, day, week, hour, minute, and second, then verifies through the notification channel upon completion.
  • Read Status: entry/src/main/ets/utils/BleService.ets → Wait for notification → Parse: entry/src/main/ets/utils/BleResponseParser.ets
    • Parses battery status, voltage/charge, remaining dose, single/daily limits, alarm bits, today’s usage, maximum log entries, etc.

5. Infusion

Section titled “5. Infusion”
  • entry/src/main/ets/utils/BleService.ets calls buildInjectionCommand(dose).
  • entry/src/main/ets/utils/BleCommand.ets: The dose is encoded as a 4-byte little-endian integer with a precision of 1/1000, accommodating “large/high-precision dose” scenarios.
  • Recommendations:
    • Perform pre-validation against the single/daily limits from the status, and split into packets or send in multiple parts if necessary.
    • Pay attention to MTU (typically 20 bytes) limitations and command length design.

6. Retrieving Logs (Record Parsing)

Section titled “6. Retrieving Logs (Record Parsing)”
  • Notification Handling Dispatch: entry/src/main/ets/utils/BleService.ets processReceivedData(...)
    • The “ path calls record parsing: entry/src/main/ets/utils/BleResponseParser.ets parseRecord(...)
    • Supports two types of structured data: injection records and operation records.
  • If you need to “fetch historical records by index/type,” you can add a READ_RECORD_WITH_TYPE constructor in BleCommand (a template is commented out in the project), which seamlessly integrates with the parsing logic above.

4. ArkTS Compilation Errors and Key Fixes

Section titled “4. ArkTS Compilation Errors and Key Fixes”

Summary Document: docs/ble-arkts-fix-summary.md

  • Missing Class Properties:
    • For example, if connectionPromiseResolvers is not declared, it needs to be added as Map<string, () => void> for external resolution of connection Promises.
  • arkts-no-spread:
    • Object spreading is forbidden when writing characteristic values. Directly modify existing.characteristicValue = ... (see the BleService.ets section).
  • unknown/any and Loose Literals:
    • In notification callbacks, remove temporary Record<string, string>/unknown assertions and use strong typing instead.
