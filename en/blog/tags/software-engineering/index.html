Harness Engineering: Your Job Isn’t Writing Code Anymore

Section titled “Harness Engineering: Your Job Isn’t Writing Code Anymore”

Your best engineer spent the morning rewriting an error message. Not fixing a bug. Rewriting a linter output so that an AI agent could understand what went wrong and fix it on its own.

This is what engineering looks like in 2026.

What OpenAI Actually Did

Section titled “What OpenAI Actually Did”

OpenAI’s Harness team built and shipped an internal beta product over five months. Zero lines of manually-written code. Every line was generated by Codex agents. The codebase grew past one million lines. Development time: roughly one-tenth of what a human team would have needed.

They published what they learned. The title: “Harness Engineering: Leveraging Codex in an Agent-First World.” The word choice is deliberate. Not AI engineering. Not prompt engineering. Harness engineering — building the harness that keeps agents productive and on track.

Martin Fowler picked it up and wrote his own analysis. His take: the article buries the lead in its title. “Harness” appears once in the body text, but the concept is the whole point. The tooling, the practices, the constraints that make agent-generated code maintainable at scale.

Eight lessons came out of this work. All of them point in the same direction.

1. Engineers Become Environment Designers

Section titled “1. Engineers Become Environment Designers”

The job shifted. Instead of writing code, engineers design systems, specify intent, and build feedback loops. When an agent produces bad output, the response is never “try harder.” It’s “what capability or context is missing from the agent’s environment?”

Human time and attention became the only scarce resource. Everything else — code production, review capacity, pattern enforcement — can be automated or mechanized.

2. Give Agents a Map, Not an Encyclopedia

Section titled “2. Give Agents a Map, Not an Encyclopedia”

The first instinct was to give agents everything. A massive AGENTS.md packed with coding standards, architecture decisions, project context. It backfired. Too much context crowded out the actual task.

The fix: a roughly 100-line AGENTS.md that works as a table of contents, pointing to a structured docs/ directory. Progressive disclosure. The same pattern that makes good UIs work — show what’s needed at the current level, link to depth when it’s relevant.

3. If Agents Can’t See It, It Doesn’t Exist

Section titled “3. If Agents Can’t See It, It Doesn’t Exist”

Slack threads where someone explained the payment flow. A Google Doc with the deployment runbook. Tribal knowledge your senior engineer carries around. None of it exists to an agent.

The team had to encode everything into the repository as versioned, discoverable artifacts. Same practice we’ve recommended for onboarding human engineers for years — write things down, put them where people can find them. Agents forced the issue. A nice-to-have became a hard requirement.

4. Enforce Architecture Mechanically

Section titled “4. Enforce Architecture Mechanically”

Custom linters with remediation instructions baked directly into error messages. When an agent violates an architectural boundary, the error message explains what the boundary is, why it exists, and how to fix the violation.

The error message becomes the teaching moment. Enforce boundaries centrally, allow autonomy locally. The agent doesn’t need to understand the entire architecture. It needs clear feedback when it steps out of bounds.

Martin Fowler noted that OpenAI monitored architecture “not only by the LLM-based agents, but also deterministic custom linters and structural tests.” Both kinds of enforcement, working together.

5. Boring Technology Wins

Section titled “5. Boring Technology Wins”

Agents perform better with composable, stable APIs that have strong representation in training data. The latest framework with sparse documentation and novel patterns? Agents struggle with it.

OpenAI’s team sometimes found it cheaper to reimplement a subset of functionality than to fight opaque upstream behavior from a trendy library. Boring technology — the kind with years of Stack Overflow answers and well-trodden paths — gives agents the most reliable foundation.

Fowler suggested this could push organizations toward fewer tech stacks, chosen for “AI-friendliness” rather than developer preference.

6. Entropy Management as Garbage Collection

Section titled “6. Entropy Management as Garbage Collection”

Agents are pattern replicators. They find an existing pattern in the codebase and reproduce it. Useful when patterns are good. A problem when they aren’t.

The solution: recurring background agents that continuously scan the codebase for deviations from standards and auto-fix. Periodic agents also catch documentation inconsistencies and architectural violations. Code hygiene isn’t a quarterly cleanup — it runs continuously.

7. Throughput Changes Merge Philosophy

Section titled “7. Throughput Changes Merge Philosophy”

When agents produce code fast, traditional merge gates become the bottleneck. OpenAI shifted to minimal blocking merge gates and short-lived PRs. Test flakes get follow-up runs, not blocking.

The philosophy: corrections are cheap, waiting is expensive. Optimize for throughput. Fix problems as they surface. This only works with strong automated guardrails — which circles back to the linter strategy.

8. Agent-to-Agent Review

Section titled “8. Agent-to-Agent Review”

OpenAI pushed almost all code review to agent-to-agent loops. Codex reviews its own changes, requests additional agent reviews, iterates until the review passes. Humans get escalated only when genuine judgment is required — novel architectural decisions, security-sensitive changes, product direction.

The math: if agents produce ten times the code volume, human reviewers become the bottleneck. Agent-to-agent review handles mechanical checks. Human attention gets reserved for decisions that actually need it.

What’s Missing

Section titled “What’s Missing”

Fowler raised an important gap. The OpenAI write-up emphasizes internal code quality and maintainability but says little about verification of functionality and behavior. The harness constrains how code is written and organized. It doesn’t yet validate that the code does what users need.

He also flagged a real-world concern: retrofitting harnesses onto existing, non-standardized codebases may prove economically unfeasible. This could create a divide between pre-AI and post-AI applications in terms of maintenance cost.

What This Means for You

Section titled “What This Means for You”

The engineers who thrive in the agent-first era won’t be the fastest coders. They’ll be the best environment designers — the ones who build systems where agents succeed on the first try because every piece of context, every guardrail, every feedback loop is already in place.

Start with your repo:

  • Audit your AGENTS.md. Is it a map or an encyclopedia?
  • Find the tribal knowledge that lives outside your codebase. Move it in.
  • Look at your linter error messages. Could an agent understand them and self-correct?
  • Identify the “exciting” technology in your stack. Is it helping or hurting agent performance?
  • Check if your merge gates are protecting quality or just slowing throughput.

Your agents are only as good as the world you build for them. That error message your engineer rewrote this morning? That’s the new engineering.

The engineers getting the most out of AI agents aren’t writing better prompts or picking smarter models. They’re redesigning the engineering environment itself — repo structure, linter rules, merge gates, documentation layout — so agents can operate independently.

Section titled “1. Engineers Become Environment Designers”

Section titled “3. If Agents Can’t See It, It Doesn’t Exist”

Slack threads. Google Docs. Hallway conversations. Tribal knowledge passed down through pair programming sessions.

All invisible to agents.

Composable, stable APIs with strong training-set representation outperform cutting-edge libraries that agents struggle with.

Agents have been trained on mountains of React, Express, and PostgreSQL code. They’ve seen far less of your team’s custom framework or that niche library with 200 GitHub stars.

6. Entropy Management Is Garbage Collection

Section titled “6. Entropy Management Is Garbage Collection”

Agents replicate existing patterns. All of them. Including the bad ones.

That one hacky workaround from six months ago? An agent will find it, treat it as the canonical pattern, and reproduce it across 40 files before anyone notices.

Without active maintenance, code quality degrades fast. The solution: recurring background agents that scan for deviations from established patterns and auto-fix. Continuous garbage collection for your codebase.

Minimal blocking merge gates. Short-lived PRs. Test flakes addressed with follow-up runs, not by blocking merges.

The principle: corrections are cheap, waiting is expensive.

When agents can generate and test changes faster than humans can review them, the bottleneck moves from code production to code approval. Redesign the pipeline for throughput, not for gatekeeping.

8. Agent-to-Agent Code Review

Section titled “8. Agent-to-Agent Code Review”

This is where it gets interesting.

OpenAI pushed almost all code review to agent-to-agent loops. Codex reviews its own changes, requests additional agent reviews, iterates until satisfied. Humans are escalated to only when judgment is required.

Read that again. The code review loop — something most teams consider a core human activity — has been restructured. Not eliminated. The human role shifted from reviewing every change to designing review criteria and handling edge cases.

The Pattern Underneath

Section titled “The Pattern Underneath”

Across all eight principles, the same thing happened. Nothing disappeared. Everything shifted.

The engineer didn’t become irrelevant. The engineer’s job description changed. From code author to environment designer. From reviewer to review-system designer. From architect to architecture-enforcement designer.

The code still gets written. The architecture still gets designed. The reviews still happen. But the human’s primary output shifted from code artifacts to environment artifacts — documentation structure, linter rules, merge policies, feedback loops.

Where to Start

Section titled “Where to Start”

The question isn’t whether this shift will reach your team. It’s whether you’ll design for it or react to it.

Start with one audit: can a new agent, with zero context beyond what’s in the repo, find what it needs, follow the architectural rules, and ship working code?

If not, you have your backlog.

A few concrete first moves:

  • Audit your AGENTS.md (or equivalent). Is it a wall of text or a navigable map? Trim it to a table of contents pointing to structured docs.
  • Check for invisible knowledge. What do new hires learn in their first week that isn’t written down? That’s your agent knowledge gap too.
  • Add remediation hints to your linter rules. Every error message is a teaching opportunity for agents.
  • Pick one pattern violation that keeps recurring and write a background cleanup script. Automate the janitor.
  • Try agent-to-agent review on a low-risk PR. Set up a second agent session to review the first agent’s output before a human looks at it.

The Problem with Vibe Coding

Section titled “The Problem with Vibe Coding”

“Vibe coding” - writing code “by feeling” and hoping the AI guesses your intentions - seemed efficient at first. You describe what you want, the AI generates code, you copy-paste. Fast, right?

But this apparent speed hides structural problems that inevitably emerge:

  • Context rot: As conversations lengthen, the AI “forgets” previous decisions and generates code inconsistent with what was done before
  • Emergent architecture (in the worst sense): Without an overall vision, code grows chaotically and quickly becomes impossible to maintain
  • Lack of traceability: Impossible to understand why certain technical decisions were made
  • Onboarding nightmare: New team members have no references to understand the system

Enter Spec-Driven Development

Section titled “Enter Spec-Driven Development”

Spec-Driven Development (SDD) is the professional answer. A methodology where specifications become first-class executable artifacts. You write the “spec” (structured documentation) first, then the AI generates code that respects that contract.

The spec becomes the source of truth for developers and AI agents, eliminating the ambiguities of vibe coding.

According to Thoughtworks, SDD is one of the most important engineering practices in recent years.

Section titled “BMAD Method: The Enterprise Powerhouse”

The BMAD Method (Build More, Architect Dreams) is the most complete SDD framework available today. With 21 specialized AI agents, over 50 guided workflows, and intelligence adaptive to project scale, BMAD simulates an entire agile team with specialized roles.
