Will Trae, Cursor, or Copilot Use My Code to Train Their Models?

Jul 29, 2025
Vibe Sparking

Yesterday, a post on V2EX went viral: the author discovered that Trae IDE continuously uploads code data, and even complaints on Discord were “silenced.” This revelation stirred up a hornet’s nest—developers have only one burning question: Is my code actually safe?

1️⃣ Official Statements at a Glance

Section titled “1️⃣ Official Statements at a Glance”
ToolUploads to CloudLong-term StorageUsed for Model TrainingCan Be Disabled
Trae (Domestic/Global)Yes, temporary uploads for indexing/chatPlain code deleted after processing, only embeddings retainedNo, official statement: “won’t train models except for computing embeddings”No privacy toggle
CursorYes, for completion & indexingMay retain by default; zero retention in Privacy ModeUsed for model improvement by default; no training in privacy modeHas privacy mode
GitHub CopilotYes, processed via AzureNo plain text saved, only for immediate inferenceNo training by default; personal users can opt-inManual training toggle available

Bottom Line: All three require “uploads,” but Copilot and Trae officially claim they don’t train on code; Cursor requires manual privacy mode activation for “zero training.”

2️⃣ Detailed Breakdown

Section titled “2️⃣ Detailed Breakdown”

🟢 Trae: Official Promise of “Embeddings Only”

Section titled “🟢 Trae: Official Promise of “Embeddings Only””
  • Privacy policy clearly states: “We do not use Your Content as training data for model training.”
  • Still sends code snippets to the cloud for vector indexing, deletes plain text, only retains embeddings.
  • Pain Point: No “privacy mode” like Cursor—users can’t disable uploads.

🔵 Cursor: Only Truly Private with “Privacy Mode”

Section titled “🔵 Cursor: Only Truly Private with “Privacy Mode””
  • Collects telemetry by default for model improvement.
  • Enable Privacy Mode → No storage, no training, but requests still processed by third-party LLMs (OpenAI/Anthropic).
  • Privacy mode requires manual activation—new users shouldn’t forget this.

🟣 GitHub Copilot: Enterprise-Friendly, Personal Optional

Section titled “🟣 GitHub Copilot: Enterprise-Friendly, Personal Optional”
  • Copilot Business/Enterprise: Code never used for model training.
  • Personal subscriptions: Also no training by default, with optional “Improve Copilot” toggle in settings.
  • Still uploads to Azure; completely offline solutions not yet available.

3️⃣ Practical Advice for Developers

Section titled “3️⃣ Practical Advice for Developers”
ScenarioRecommendation
Highly Confidential (Finance, Core Algorithms)Go offline. Use local LLMs or private cloud; disable any auto-completion.
Pure Open Source/Public ProjectsChoose Copilot or Cursor with minimal impact. Trae acceptable but requires accepting uploads.
Middle Ground (Partial Closed Source + Need Efficiency)Enable Cursor privacy mode or use Copilot; Trae currently lacks disable option—evaluate carefully.

4️⃣ Friendly Reminders for Tool Providers

Section titled “4️⃣ Friendly Reminders for Tool Providers”
  • Make privacy mode the default, rather than making users dig through settings.
  • Publish transparency reports: What’s uploaded? How long is it stored? Is it actually deleted?
  • Granular controls: Indexing OK, training NO—let users decide.

Conclusion

Section titled “Conclusion”

No matter how sweet AI-assisted coding gets, remember:

Code is productivity, but also a risk vector.

Periodically review your tools’ privacy policies—don’t let silent updates quietly shift your security boundaries.

AI Vibe Coding: Speeding Up or Slowing Down? Insights from a 19% Slowdown

Jul 28, 2025
Vibe Sparking

Introduction

Section titled “Introduction”

This year, AI Vibe Coding programming tools have created quite a buzz in developer circles. Many developers (myself included) thought that with AI intelligent assistants, coding efficiency would dramatically improve. However, recent research threw cold water on us: using AI tools actually slowed development by nearly 19%! This study by METR conducted controlled experiments with experienced open-source developers, yielding surprising results. Participants expected AI to reduce task completion time by 24%, but actually slowed down by nearly one-fifth. More interestingly, even after experiencing the slowdown firsthand, many programmers still believed AI “made me faster” — a kind of “psychological acceleration” illusion.

As an internet e-commerce backend developer, I actually resonate with these results. Reflecting on my own experience introducing AI assistants for coding, there were indeed many pitfalls: sometimes waiting for AI code output made me scratch my head anxiously, sometimes I had to interrupt my own thinking to accommodate AI. Below, I’ll combine actual development cases to discuss why AI programming might slow down the pace. I hope these experiences can serve as a warning to fellow developers.

The Long Wait for AI Execution

Section titled “The Long Wait for AI Execution”

Once upon a time, we coded with brain-to-hand fluidity - when inspiration struck, we’d tap it out immediately. But after introducing AI assistants, waiting for them to “think” and output became a new time sink. When implementing asynchronous message processing in my project, I thought I’d have AI help generate consumer logic. After asking a question, that little circle on screen just kept spinning, with AI “pondering” in the cloud while I could only stare and wait. Those few dozen seconds might seem insignificant, but they accumulate into substantial overhead. More troublesome is when AI occasionally outputs a huge chunk of code at once, forcing me to stop and wait for it to finish scrolling, completely disrupting my rhythm.

During these waiting periods, I’d unconsciously check emails or do other things, but my mind remained occupied with AI’s response. In this state, it’s hard to focus on other tasks, essentially putting me in “suspension”. Without AI, I might have already coded up an initial draft myself. Now, precious development time slips away waiting for the machine to “finish thinking.” Clearly, while AI can automatically produce code, its speed doesn’t necessarily match human intuitive response. The process of waiting for AI is like introducing additional latency into iterations, severely fragmenting what was once a smooth development rhythm.

Disruption from Multitask Switching

Section titled “Disruption from Multitask Switching”

With AI assistants, I once tried “parallel development”: having AI write one module while I worked on something else. But reality proved that the cost of context switching back and forth was far greater than imagined. For example: when developing idempotency handling for a payment callback interface, I had AI generate code for duplicate request validation while I debugged another piece of business logic. Minutes later, AI produced results, so I switched back to read and integrate. But with the other matter still occupying my mind, I had to shift my thinking to AI’s code. Going back and forth like this, both sides slowed down.

Humans aren’t truly adept at multitasking. In work requiring full concentration like coding, frequent context switching creates serious cognitive overhead. Each jump from my own thinking to AI’s thinking requires time to understand AI’s code intent and recall the previous problem context. It’s like listening to two people talk simultaneously - you need to constantly “fill in” context between them, and efficiency suffers accordingly. Especially when AI-generated content differs from my original code style or thinking, I need extra effort to adapt. Imagine developing a complex e-commerce backend feature during peak hours while handling AI “interruptions” like customer service - this parallelism not only didn’t save time, but slowed the entire process.

The Burden of Reviewing and Correcting AI Output

Section titled “The Burden of Reviewing and Correcting AI Output”

AI-generated code isn’t ready-to-use; careful review and debugging are often unavoidable. In actual projects, I’ve repeatedly encountered AI “hallucinations” - fabricating non-existent functions or misusing interfaces. For example, what made me both cry and laugh was when I asked AI to help implement idempotency handling for order payment callbacks. It enthusiastically wrote a bunch of code, but closer inspection revealed calls to module functions that didn’t exist in our system! AI’s overconfident wild guessing forced me to re-examine every line of code. Research found similar phenomena: developers accepted less than 44% of AI suggestions, with most code requiring modification or rewriting, and 75% of cases requiring programmers to review every line of AI output. This review and modification process essentially spent back the time supposedly saved.

Even when AI code runs, it likely doesn’t meet our business expectations. I once had AI optimize Kafka message processing logic. Its solution looked reasonable, but running it revealed improper handling of edge cases like out-of-order and duplicate messages, nearly introducing new bugs. I ended up spending time simplifying the fancy parts it “smartly” added. Another time, to solve a problem, AI provided a complex multi-threading solution that was overkill and introduced thread safety hazards. AI’s excessive enthusiasm sometimes backfires, making me spend more time on rework. Ultimately, current AI code reliability is poor, forcing me to stay vigilant and scrutinize with a critical eye, reviewing line by line like code review, fearing it might bury some trap.

Time Drain from Prompt Optimization

Section titled “Time Drain from Prompt Optimization”

Many think asking AI questions is effortless, but actually getting AI to understand you and provide useful answers requires repeatedly refining prompts. In my experience, this has almost become an art of “speaking human to AI.” When I had AI help generate product inventory synchronization logic in my project, my first problem description was met with an irrelevant answer; I had to rephrase and add business context, but the second attempt was still unsatisfactory. Going back and forth in deep dialogue with AI for several rounds felt like training a new employee: gradually correcting its misunderstandings and continuously refining requirements. Time spent crafting and optimizing prompts is invisible but very real.

Sometimes, to make AI understand complex scenarios, I’d even paste relevant code snippets and error messages, which is time-consuming itself. For example, implementing asynchronous refunds in order workflows, I needed it to focus on idempotency and transactionality, and just explaining this business context took a long message. By the time AI gradually “grasped” my intent, considerable time had passed. These back-and-forth communications essentially produced no new business value, only helped AI figure out the situation. We can say prompt optimization overhead partially offset AI coding’s acceleration advantages. We think “chatting” with AI is quick, but might actually fall into prolonged debugging conversations, dragging the pace even slower.

Conclusion

Section titled “Conclusion”

The emergence of AI programming assistants undoubtedly provides developers with new productivity tools. But as revealed by the above personal experiences and research results, current-stage AI is not a silver bullet. It might improve efficiency and reduce grunt work in simple scenarios, but faces complex business logic and large codebases where AI is often willing but unable. Developers must not only spend time waiting and switching focus back and forth, but also cover for AI mistakes and painstakingly train it. These hidden costs accumulate into productivity regression. Of course, we’re not denying AI tools’ value, but reminding everyone to rationally view their applicability: when proficiency is insufficient or application scenarios don’t match, AI might actually slow you down.

From a frontline developer’s perspective, I feel “love-hate.” AI’s new approaches and paradigms have benefited me greatly, but when it gets in the way, it’s truly frustrating. Perhaps as models evolve and our usage skills improve, this 19% disadvantage will transform into real acceleration advantages. But until then, we need more sobriety: don’t let optimistic AI expectations blind us, and continuously reflect on its cost-effectiveness in actual work. As Greyhound Research analysts said, don’t mistake developer satisfaction for productivity.

What do you think? What pitfalls have you encountered using AI programming?

Cursor 1.2 Update: Cursor Agent Now Plans Before It Codes

Jul 26, 2025
Vibe Sparking

When developing complex features, Cursor Agent will “list tasks first, then execute”

👀 Have You Also Encountered This Cursor Experience?

Section titled “👀 Have You Also Encountered This Cursor Experience?”

We often use Cursor to help write code, but when it comes to multi-step, complex tasks, it’s easy to feel “lost”:

  • What is Cursor doing? Why did it suddenly touch the database?
  • What’s it going to do next? Did it miss testing?
  • What’s the overall process? Can I see the plan first?

Especially when involving multiple files, database logic, and frontend-backend coordination, without clear planning and relying only on “step-by-step responses,” it’s easy to worry—Is Cursor really solving problems “with a plan”?

🧠 Cursor 1.2 Update: Agent Planning & To-dos Officially Live!

Section titled “🧠 Cursor 1.2 Update: Agent Planning & To-dos Officially Live!”

In Cursor 1.2, when you give Cursor a complex task, such as:

“Fix the SQL error caused by soft-deleted patients and ensure logic covers all edge cases.”

Agent will automatically:

  • Analyze the problem’s context
  • Break it down into clear subtasks
  • Generate a To-do list where each task can be checked off and progress tracked
  • During execution, dynamically mark completed steps

This is no longer a “black box” Cursor, but a transparent and controllable development partner.

🪄 Example: Fixing a Soft Delete Association Query Error

Section titled “🪄 Example: Fixing a Soft Delete Association Query Error”

Let’s look at a real development problem:

This line in Rails executes LEFT OUTER JOIN but doesn’t automatically exclude soft-deleted associated records.

After Cursor identifies the problem, it breaks down the task like this:

  1. Modify query logic to exclude soft-deleted patients
  2. Add scope to Advice model: only fetch data from active patients
  3. Write tests to verify soft-deleted data doesn’t appear in results
  4. Record this fix and best practices in memory bank for future reuse

Throughout the process, you can check what it plans to do next anytime, and can skip, adjust, or retry any step.

✍️ Coding Is No Longer “Blind Trial and Error” but “Collaborative Breakdown”

Section titled “✍️ Coding Is No Longer “Blind Trial and Error” but “Collaborative Breakdown””

This feature fundamentally changes how we collaborate with Cursor:

  • Task Transparency: You know how many steps it will take total, no more passive waiting
  • Controllable Progress: Each subtask has clear status, supports interruption and rollback
  • Experience Accumulation: Can be saved to memory with one click, automatically called for similar future problems

Put simply, Cursor Agent now acts like a reliable intern who not only writes code but first lists an execution plan for your confirmation.

🚀 How to Experience?

Section titled “🚀 How to Experience?”

Upgrade Cursor to version 1.2 or above, ensure Agent mode is enabled in your project.

Input a multi-step task, such as:

“Refactor this code section, ensure logic is clear with test coverage, and update the README while you’re at it.”

You’ll see Agent automatically pull up the To-dos panel, advancing tasks step by step, transparently and efficiently.

🧩 Food for Thought

Section titled “🧩 Food for Thought”

Are you willing to delegate “task planning” to AI? Do you think Cursor’s breakdown plans align with your usual engineering thinking?

Feel free to share your thoughts in the comments!

Consumer Market: Rapid Penetration with Emerging Scale Trends

Section titled “Consumer Market: Rapid Penetration with Emerging Scale Trends”

According to a 2025 U.S. consumer survey by Menlo Ventures and Morning Consult, AI has rapidly penetrated various daily life scenarios, becoming an “invisible tool.”

Main consumer use cases include:

  • Writing Support (51%): AI is commonly used for note-taking, letter writing, generating article drafts, and other work/study content.
  • Code Generation & Programming Assistance (47%): AI helps quickly write code, modify existing structures, or explore new approaches, especially beneficial for beginners.
  • Search Research/Learning Assistance (43%): AI serves as a summarization tool for English books and basic research assistant.
  • Visual/Audio Content Creation (37%-38%): Generating PPTs, cover images, script drafts, etc.

Menlo Ventures notes: While most consumers haven’t systematically integrated AI workflows, AI tools in “non-professional scenarios” are becoming mainstream.

Meanwhile, Tencent Research Institute points out that Chinese users show similar trends, but lean more toward “practical tasks” like education, content writing, translation assistance, and design efficiency.

Section titled “4. Multi-engineering Capabilities + Continuous Learning”

AI tools are powerful, but when encountering cross-module, edge cases, or complex business logic scenarios, developers still need to “research, understand logic, and make judgments.”

Section titled “Conclusion”

AI Coding is rapidly transforming from “experimental tool” to “essential tool.” Whether in consumer writing/programming assistance or enterprise product development, design, and testing, AI has deeply integrated into development workflows.

Over the next three years, the biggest gap won’t be “knowing how to use AI,” but “who can make AI an enabler rather than an obstacle.”

I hope every reader can find their own value anchor in the AI era.

Kiro: From a Prompt to Production Code - A Guide to Spec-Driven Agentic IDE

Jul 15, 2025
Vibe Sparking

In the era of AI coding tools “racing to the sky,” Kiro from AWS brings a refreshing perspective: it’s not just “helping you write code,” but packs the complete software engineering workflow of requirements → design → implementation → automation into an IDE, truly moving from “Vibe Coding (demo-based coding)” to “Viable Code (production-ready products).” This article will quickly introduce you to Kiro’s core features, practical usage methods, and how it differs from other Agentic IDEs.

1 | What is Kiro?

Section titled “1 | What is Kiro?”
  • Positioning: An Agentic IDE based on VS Code kernel that automatically breaks down a “single prompt” into requirements specifications, system design, and task lists, driving LLM to complete implementation.
  • Background: Developed by a “very opinionated small team” within AWS, but intentionally weakly coupled with AWS cloud services—you can log in with GitHub/Google accounts. AWS Builder ID login is also supported.
  • Models Used: Preview version includes built-in Claude Sonnet 3.7 and 4.0, switchable in settings.

2 | Three Core Capabilities

Section titled “2 | Three Core Capabilities”

Based on practical usage in real work projects, I’ve analyzed and summarized Kiro’s core functions:

2.1 Spec-driven Development

Section titled “2.1 Spec-driven Development”

  1. Three Specs Auto-Generated:

    • requirements.md —— User stories and acceptance criteria written in EARS syntax
    • design.md —— Architecture, flowcharts, database design
    • tasks.md —— Executable implementation plan (including testing, accessibility, etc.)

  2. EARS Introduction: EARS (Easy Approach to Requirements Syntax) was proposed by the Rolls-Royce engineering team in 2009, structured as “While … when … the shall …”, reducing requirement ambiguity.

  3. Significance: Compared to simple Chat-pair coding, Spec-driven development ensures every line of code AI writes has traceable origins, facilitating team collaboration and later maintenance.

2.2 Agent Hooks (Event-Triggered Automation)

Section titled “2.2 Agent Hooks (Event-Triggered Automation)”

  • Working Method: Monitors events like “file save/create/delete/manual trigger” and calls backend LLM to execute preset tasks.

  • Common Scenarios:

    • Save React component → Auto-update test files
    • Modify API Endpoint → Sync README.md documentation
    • Before Git commit → Scan for sensitive information leaks

  • Benefits: Solidifies “good habits I should develop” into automated Hook execution, shared across teams, avoiding omissions.

2.3 From Vibe Coding to Viable Code

Section titled “2.3 From Vibe Coding to Viable Code”
  • Dual Modes: When starting projects, you can choose “Prompt mode” for quick experimentation or “Spec mode” for rigorous one-step implementation. (TechTarget)
  • In-IDE Review: All AI changes are presented in diff format, supporting line-by-line approval or batch acceptance. (Kiro)
  • VS Code Ecosystem Compatibility: Open VSX plugins, themes, and shortcuts migrate almost unchanged. (DEVCLASS)
