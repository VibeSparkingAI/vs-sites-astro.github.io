Claude Code Alternative: DeepSeek R1 + Kimi K2 Quick Start Guide

Jul 16, 2025
Vibe Sparking

Claude Code Router (CCR) https://github.com/musistudio/claude-code-router is a “multi-model router” tailored for Claude Code users: it listens locally to Claude Code’s standard requests, then forwards them to DeepSeek-R1, Kimi K2, and any other LLMs according to your routing strategy. It’s compatible with OpenAI-style APIs and can add “transformers” for different models to adapt parameters and responses. This article first provides an overview of CCR’s core mechanisms, then demonstrates step-by-step how to integrate DeepSeek R1 and Kimi K2, giving you everything you need for “multi-model collaborative coding.”

1 | What Can CCR Do?

Section titled “1 | What Can CCR Do?”

1.1 Core Concept

Section titled “1.1 Core Concept”
  • CCR describes Provider → Model → Transformer → Routing Rules through ~/.claude-code-router/config.json, allowing Claude Code to hot-switch backend models in VS Code/terminal as usual using /model.
  • Built-in pluggable transformers for DeepSeek, OpenRouter, Gemini, MaxToken, etc., allowing multiple transformations to be stacked on a single model.
  • Supports both CLI and GitHub Actions modes; combined with Actions, you can leverage DeepSeek’s 25% nighttime rates for “early morning batch processing.”

2 | Integrating DeepSeek R1

Section titled “2 | Integrating DeepSeek R1”

2.1 Prepare API Key

Section titled “2.1 Prepare API Key”
  1. Apply for a key in the DeepSeek Platform console and record the sk-xxx.
  2. The official API is https://api.deepseek.com/chat/completions, fully compatible with OpenAI SDK.
  3. R1 latest version number deepseek-r1-250528 (VolcEngine mirror provides 128K context).

2.2 Modify CCR Configuration

Section titled “2.2 Modify CCR Configuration”
  • The deepseek transformer automatically limits max_tokens to 8192 and is compatible with R1’s lack of tool_choice support.
  • Nighttime (00:30 – 08:30 UTC+8) R1 calls cost only 25% of daytime rates → suitable for large-scale testing/refactoring scripts.

3 | Integrating Kimi K2

Section titled “3 | Integrating Kimi K2”

3.1 Why Choose K2?

Section titled “3.1 Why Choose K2?”
  • Moonshot AI’s latest open-source MoE model, 1T total parameters / 32B activated experts, excelling at code generation and agent calls.
  • SiliconFlow provides 16384 max_tokens quota with high-concurrency, low-latency inference services, suitable for long-chain code analysis.

3.2 Get Key

Section titled “3.2 Get Key”
  • Enter siliconflow.cn console, enable moonshotai/Kimi-K2-Instruct and get the token.
  • Can also use OpenRouter’s unified compatible endpoint https://openrouter.ai/api/v1/chat/completions.

3.3 Configuration Example

Section titled “3.3 Configuration Example”
  • The maxtoken transformer prevents CCR from accidentally exceeding SiliconFlow’s 16K max_tokens limit.
  • If you want to feed K2 all 32K+ project documentation in the same session, first use /model siliconflow,moonshotai/Kimi-K2-Instruct, then paste the full text.

3.4 Call Effect

Section titled “3.4 Call Effect”

SiliconFlow’s official cURL examples only need to replace with CCR’s local port; CCR handles forwarding and field compatibility.

5 | Common Pitfalls & FAQ

Section titled “5 | Common Pitfalls & FAQ”
  1. Slow response? DeepSeek’s official endpoint inference is relatively slow. If VS Code times out, enable UltraThink budget or switch to VolcEngine mirror.
  2. Context overflow? DeepSeek API defaults to 64K, the above example uses VolcEngine to expand to 128K; Kimi K2 is 16K, needs maxtoken protection.
  3. Wrong model name? CCR’s /model format must be provider,model, don’t forget the comma.
  4. Tool calls failing? R1 currently doesn’t support tool_choice. For function calls, fall back to DeepSeek V3, or add a tooluse transformer to V3 in CCR.
