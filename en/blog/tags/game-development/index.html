BMad Quick Dev Workflow: Building a Game in 2.5 Hours

Jan 25, 2026
Vibe Sparking

Introduction

Section titled “Introduction”

Last week, I decided to test the BMad Method’s Quick Dev workflow by building something from scratch: a stickman fighting game using Phaser 3. The goal was simple—see how much AI could autonomously handle, and where human intervention was actually needed.

Two and a half hours later, I had a complete playable game with:

  • Procedurally rendered stickman characters (no sprite assets needed)
  • Player controls (A/D move, W jump, J punch, K kick)
  • An AI opponent with approach/attack/retreat behavior
  • Health bars, hit detection, and knockback physics

Here’s how it went, step by step.

What is Quick Dev?

Section titled “What is Quick Dev?”

Quick Dev is BMad’s streamlined workflow for focused implementation tasks. It’s designed for situations where you either:

  • Have a tech-spec ready to execute, or
  • Know exactly what you want to build and want to dive straight in

The workflow uses a “step-file architecture”—each phase loads fresh to combat the “lost in the middle” problem that plagues long AI sessions. State persists through variables rather than context, allowing for marathon coding sessions without degradation.

The 6-Step Process

Section titled “The 6-Step Process”

Step 1: Mode Detection

Section titled “Step 1: Mode Detection”

What happens: The workflow captures a baseline commit (for diff tracking), loads project context if available, and determines whether you’re working from a tech-spec or direct instructions.

Human intervention required:

When I ran /bmad:bmm:workflows:quick-dev with the prompt “build a stickman fighting game in Phaser 3,” the workflow analyzed the scope and presented three options:

My choice: I selected [e] - Execute directly. I wanted to see how far AI could go without formal planning.

Why this matters: This checkpoint prevents AI from committing to approaches that might be wrong. For larger projects (platforms, multi-system work), the workflow would have escalated me to full BMad Method planning.

Step 2: Context Gathering

Section titled “Step 2: Context Gathering”

What happens: The workflow analyzes the request to identify:

  • Files to modify (or create, for new projects)
  • Relevant patterns (from existing code or frameworks)
  • Dependencies (external libraries, internal modules)
  • A mental plan with tasks and acceptance criteria

What was presented:

Human intervention required: Confirmation to proceed. I could have adjusted the plan, but it looked solid.

Step 3: Execute

Section titled “Step 3: Execute”

What happens: This is where the magic happens. The workflow implements tasks continuously—stopping only for blocking issues, not for approval between tasks.

Key principle: Do NOT halt for milestones. Execute all tasks in sequence.

What I watched happen:

The AI created a complete project structure:

Then it built each component:

  1. Project scaffolding - Vite + TypeScript + Phaser 3.80.1
  2. Stickman class - Procedural rendering using Phaser Graphics API. Each body part (head, torso, arms, legs) drawn as lines with joints
  3. Animation system - Pose-based animation with smooth transitions between idle, walk, jump, punch, kick states
  4. Combat system - Hitbox detection, damage calculation, knockback physics
  5. AI opponent - Simple state machine: approach when far, attack when close, retreat when health low

Human intervention required: None during execution. The workflow found a minor issue with jump animation (leg movement too subtle) and fixed it autonomously.

Total tokens used: ~155,000 Time: About 90 minutes of continuous work

Step 4: Self-Check

Section titled “Step 4: Self-Check”

What happens: Before presenting results, the workflow self-audits:

  • Are all tasks complete?
  • Do tests pass?
  • Are acceptance criteria satisfied?
  • Were existing patterns followed?

What was presented:

Human intervention required: None—this is automatic. But if anything had failed, the workflow would have halted.

Step 5: Adversarial Review

Section titled “Step 5: Adversarial Review”

What happens: The workflow constructs a diff of all changes and invokes an adversarial code reviewer—a separate agent that has only the diff, no context. This information asymmetry forces the reviewer to find real problems, not excuse things based on “what the AI was thinking.”

The process:

  1. Git diff from baseline commit
  2. New untracked files included
  3. Separate subagent reviews only the diff
  4. Findings categorized by severity and validity

Step 6: Resolve Findings

Section titled “Step 6: Resolve Findings”

What happens: You choose how to handle the findings:

Human intervention required: Decision on resolution approach.

My choice: [2] - Auto-fix. The findings were minor and the AI understood what needed fixing.

What was fixed:

  • F1: Changed hipOffsetX: dir * 2 to include both direction and air phase
  • F2: Increased body lean from 0.05 to 0.1 + air phase adjustment
  • F3: Skipped (noise)

Where Humans Are Still Needed

Section titled “Where Humans Are Still Needed”

After this experiment, here’s where human input proved essential:

  1. Initial direction - The workflow can’t decide what to build
  2. Mode selection - Understanding whether formal planning is needed
  3. Complex decisions - Major architectural choices still benefit from human judgment
  4. Creative direction - Art style, game feel, these are human domains

Key Takeaways

Section titled “Key Takeaways”
  1. Step-file architecture works - 2.5 hours of continuous execution without context degradation
  2. Adversarial review finds real issues - The separate reviewer caught subtle bugs the main agent missed
  3. YOLO mode is viable for focused work - With proper guardrails, AI can execute autonomously
  4. Human attention is precious - The workflow preserves it for decision points, not implementation

Would I Use This Again?

Section titled “Would I Use This Again?”

Absolutely. For focused implementation tasks—features, refactorings, bug fixes—Quick Dev is significantly faster than traditional coding. The adversarial review catches issues I would have missed, and the step-file architecture enables marathon sessions.

Would I use it for a platform rewrite? No—that’s what the full BMad Method is for. But for a game, a feature, a service? Quick Dev delivers.
