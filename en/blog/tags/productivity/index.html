Agent Client: AI Coding Agents Directly in Your Obsidian Vault

Jan 4, 2026
Vibe Sparking

Introduction

Section titled “Introduction”

Agent Client is an Obsidian plugin that brings AI coding agents directly into your knowledge management workspace. Built on the Agent Client Protocol (ACP)—an open standard pioneered by the Zed editor—this plugin creates a seamless bridge between your personal knowledge base and powerful AI coding assistants like Claude Code, Codex, and Gemini CLI.

Instead of juggling between your notes and terminal, Agent Client integrates AI agents into a dedicated right-side panel within Obsidian, transforming your vault into an intelligent development companion.

Key Features and Highlights

Section titled “Key Features and Highlights”

🤖 Direct Agent Integration

Section titled “🤖 Direct Agent Integration”

Chat with AI coding agents in a dedicated right-side panel without leaving Obsidian. Your AI assistant is just a side panel away.

📝 Note Mentions

Section titled “📝 Note Mentions”

Reference your Obsidian notes directly in conversations using @notename syntax. Automatically include the active note’s content or manually reference specific notes to provide context to your AI agent.

⚡ Slash Commands

Section titled “⚡ Slash Commands”

Quickly trigger agent actions using / commands. Browse and execute actions provided by your current agent without typing full prompts.

🔄 Multi-Agent Support

Section titled “🔄 Multi-Agent Support”

Switch between Claude Code, Codex, Gemini CLI, and any ACP-compatible custom agents. Configure multiple agents and switch between them based on your task requirements.

🎛️ Mode & Model Selection

Section titled “🎛️ Mode & Model Selection”

Change AI models (e.g., Sonnet, Haiku, Opus) and agent modes (e.g., Plan Mode) directly from the chat interface using convenient dropdown menus.

💻 Terminal Integration

Section titled “💻 Terminal Integration”

Let your agent execute shell commands and return results inline in the chat. Run build scripts, git commands, or any terminal operation without leaving the conversation.

🖼️ Image Attachments

Section titled “🖼️ Image Attachments”

Paste or drag-and-drop images directly into the chat to send them with your message—perfect for visual debugging or UI-related questions.

🔐 Permission Management

Section titled “🔐 Permission Management”

Fine-grained control over agent actions. Approve or deny file reads, edits, and command executions based on your comfort level.

Recommended Scenarios and Best Practices

Section titled “Recommended Scenarios and Best Practices”

Daily Workflow Integration

Section titled “Daily Workflow Integration”

  1. Code Review with Context: Open your project documentation in Obsidian, reference it with @notename, and ask Claude Code to review your implementation against the documented requirements.

  2. Debugging Session: When stuck on a bug, take a screenshot, paste it into the Agent Client chat, and let your AI agent analyze the error while you reference relevant error logs stored in your vault.

  3. Documentation Generation: As you build features, use the chat to generate documentation while the agent reads your code files—then save the output directly to your notes.

  4. Multi-Agent Task Switching: Use Claude Code for architectural planning, switch to Codex for code generation, and use Gemini CLI for alternative perspectives—all within the same interface.

Best Practices

Section titled “Best Practices”
  • Organize Notes by Project: Structure your vault so that project-specific notes can be easily referenced with @notename
  • Use Chat Export: Save important conversations for future reference—Agent Client supports exporting chat history
  • Leverage Terminal Integration: Run tests and builds directly from the chat to get immediate feedback
  • Permission Management: Start with restrictive permissions and gradually grant access as trust builds with the agent

Precautions, FAQ, and Troubleshooting

Section titled “Precautions, FAQ, and Troubleshooting”

Precautions

Section titled “Precautions”
  • API Key Security: API keys are stored in Obsidian’s plugin settings. Enable vault encryption if storing sensitive keys
  • Terminal Commands: Agents can execute commands—review permissions carefully before approving
  • File Editing: Agents can modify files—always review proposed changes before applying

FAQ

Section titled “FAQ”

Q: Can I use multiple agents simultaneously? A: Currently, you can switch between agents, but future updates will include multi-instance support for running multiple agents simultaneously.

Q: Does this work on mobile? A: The plugin is designed for desktop Obsidian. Mobile support depends on Obsidian’s plugin capabilities on mobile platforms.

Q: Can I use custom agents? A: Yes! Any agent that implements the Agent Client Protocol (ACP) can be configured in the plugin settings.

Section titled “Troubleshooting”

AI coding agents have transformed how developers work, but getting the most out of them requires more than just prompting. Based on the Vibe Kanban project, here’s a comprehensive guide to orchestrating AI coding agents effectively.

Planning: Always Start with a Plan

Section titled “Planning: Always Start with a Plan”

Five minutes spent planning saves ten minutes in review.

Regardless of task size, always begin with a plan. Whether your agent has a built-in “plan mode” or not, you can simply ask:

“Come up with a plan, confirm with me before making changes.”

Planning can feel like overhead, but it moves unavoidable work earlier. A plan is faster to judge than a diff.

Planning Sets the Shape

Section titled “Planning Sets the Shape”

Agents optimize for getting something working with minimal edits. Once that version exists, subsequent changes patch whatever’s there already—instead of rethinking the approach. The plan is your chance to pick the right shape before the code hardens around the wrong one.

Async Workflows: Juggle Multiple Threads

Section titled “Async Workflows: Juggle Multiple Threads”

Working asynchronously with coding agents (sometimes called “backgrounding”) lets you:

  • Juggle multiple threads at once
  • Avoid drifting into procrastination
  • Keep the agent running for longer stretches while you do other things

A typical workflow: keep the agent running continuously while you handle other tasks, checking in periodically.

Use YOLO Mode for True Async

Section titled “Use YOLO Mode for True Async”

Async only works if you’re not approving every tool call. Either use your agent’s YOLO mode, or give it an explicit allowlist of commands that covers almost everything it needs.

Otherwise you’ve reinvented pair programming—except slower.

Naming varies by platform: ‘yolo’, ‘dangerously skip permissions’, or similar.

Set Your Codebase Up to Be QA’d

Section titled “Set Your Codebase Up to Be QA’d”

If the agent can’t test its own changes, it’ll ask you to do it—which isn’t a great use of your time.

Put in the work to make changes verifiable with a dev or test command. Not everything is testable by an agent (frontend still needs human eyes), but most changes should end with “tests pass,” not “can you try this?”

Solve Dev Servers

Section titled “Solve Dev Servers”

Agents are bad at starting and cleaning up dev servers, which leads to:

  • Port conflicts
  • Wasted effort troubleshooting “something’s already running on that port” errors

Solutions like Dev Manager MCP assign ports and kill stale servers automatically.

Add Dummy Data for Offline Testing

Section titled “Add Dummy Data for Offline Testing”

Make it easy to run the project offline with realistic seed data. If the agent spends half its time getting the app into a testable state, you’ll get:

  • Flaky results
  • Slower progress

Dummy data also enables parallel agents—if they share a real database, one might work on a different schema version than another.

Combating Agent Laziness

Section titled “Combating Agent Laziness”

Coding agents make some lazy decisions repeatedly. Here’s what to watch for:

No Backwards Compatibility

Section titled “No Backwards Compatibility”

Every coding agent defaults to maintaining backwards compatibility. This is part overcaution, part laziness—keeping the old interface means not refactoring all call sites.

Add this to your system prompt:

We want the simplest change possible. We don’t care about migration. Code readability matters most, and we’re happy to make bigger changes to achieve it.

Disable Disabling Lint Rules

Section titled “Disable Disabling Lint Rules”

Agents love silencing linter errors instead of fixing them. eslint-disable-next-line and similar patterns appear across TypeScript, Rust, and likely other languages.

Ban this with plugins like eslint-comments/no-restricted-disable.

Frontend Best Practices

Section titled “Frontend Best Practices”

Separate Presentation from Logic

Section titled “Separate Presentation from Logic”

Keep leaf components presentational. Business logic (fetching data, etc.) lives in parent components.

Most frontend problems with agents involve Frankenstein components: thousands of lines of state management mixed with rendering. Separating concerns makes it easy to audit a presentational component’s props and spot irregularities.

It also helps the agent—agents look for patterns to follow. Clear folder structures give them distinct examples.

This can be semi-enforced via ESLint by disabling hooks like useState or useEffect in presentational components:

Restrict Tailwind

Section titled “Restrict Tailwind”

Agents love adding custom colors, spacing, and padding everywhere. Enforce a small set of allowed utility classes via ESLint. For example, p-4 and p-8 might be banned, but p-base and p-double are allowed and defined in your Tailwind config.

  • The component has been designed in Figma and can be retrieved using the Figma MCP server
  • All spacing, colors, and sizes should match existing Tailwind variables
  • We use [icon library]
  • If you cannot match an existing variable, ask the user what to do
