Breaking the “No-Device Repro” Challenge: How Antigravity Pinpoints Complex Bugs Through Code Reasoning

Section titled “Breaking the “No-Device Repro” Challenge: How Antigravity Pinpoints Complex Bugs Through Code Reasoning”

In software development, few things frustrate engineers more than this: “the user has a problem, but I can’t reproduce it here.”

Especially when specific hardware, specific user roles, or specific network environments are involved, reproducing the issue often requires pricey devices or complex environment setup. As an intelligent coding agent, Antigravity demonstrates a new approach: without physical devices, use pure code reasoning and full-chain data validation to precisely locate and fix the bug.

This article reviews a recent real case (Issue #181: abnormal records appearing when a doctor views patient data) to walk through this “blind” problem-solving process.

Challenge: An Unseen Site

Section titled “Challenge: An Unseen Site”

Problem description: A hospital doctor unexpectedly saw other patients’ records while using a PC/tablet to view a specific patient’s data.

Difficulties:

  1. No environment: The dev environment cannot connect to the hospital intranet, and cannot log in to that doctor’s account.
  2. No device: The specific device used by the doctor is unavailable.
  3. Limited info: Only one screenshot and a short line: “During rounds, I saw someone else’s data.”

Traditional practice would be: borrow an account, find a device, connect to the intranet, and try to reproduce. If reproduction fails, the bug often gets shelved.

Antigravity’s Problem-Solving Mindset: Logic Over Visuals

Section titled “Antigravity’s Problem-Solving Mindset: Logic Over Visuals”

Antigravity cannot “see” the screen, but it can see through the symptoms to the code essence. The core is three steps: full-chain tracing -> code archaeology -> data falsification.

Step 1: Full-Chain Tracing (UI to Database)

Section titled “Step 1: Full-Chain Tracing (UI to Database)”

Since the UI cannot be operated, read how the UI is built.

  1. Frontend locating: By searching the keyword records.json and analyzing the frontend code patient.service.ts, we quickly pinpointed the API call causing the issue:

    This clarified the direction: doctors use the generic users/records endpoint instead of the doctor-specific one.

  2. Backend tracing: Following the trail into the Rails backend, from UsersController to UserService, and finally down to the core RecordService.

Step 2: Code Archaeology (Finding the Logic Flaw)

Section titled “Step 2: Code Archaeology (Finding the Logic Flaw)”

In RecordService.query_by_user_id_or_patient_id, we found an old snippet that looks “helpful” but is actually dangerous:

Logical reasoning: The comment shows that to accommodate historical dirty data (patient_id of 0), the query forcibly adds 0 to the filter. However, when a doctor (User A) views a patient (Patient B), the system queries: user_id = Doctor_A AND patient_id IN (Patient_B, 0)

This means that as long as there are records under that doctor with patient_id 0 (test data or orphan data that was not bound), they will mix into Patient B’s list.

Step 3: Data Falsification (Nailing the Bug)

Section titled “Step 3: Data Falsification (Nailing the Bug)”

To validate the reasoning, Antigravity did not guess. It wrote SQL to run “probe detection” in the SIT environment (integration test database).

Conclusion: Data leakage is real. The extra 2918 rows are exactly those patient_id=0 ghost records being wrongly pulled into Patient B’s page.

Fix and Takeaways

Section titled “Fix and Takeaways”

Once the root cause was found, the fix was obvious: when querying a specific patient (patient_id present), drop the 0 compatibility to keep isolation pure.

This case shows the unique value of an AI agent in software engineering:

  1. Go beyond the surface: Skip UI reproduction; restore business logic through code control flow.
  2. Data-driven: Use direct database access to build test cases that verify hypotheses.
  3. Precision strikes: In millions of lines of code, quickly home in on a few critical conditional checks.

For Antigravity, there is no “unreachable” site; only logic not yet parsed.

Fixing Bugs Without the User's Phone: An Antigravity Autonomous Debugging Case Study

Dec 11, 2025
Vibe Sparking

When a user reports “the lock screen can’t be tapped to enter the app when the phone font is set to maximum,” and you don’t have that phone model on hand, what do you do? This article records how Antigravity, using only the user’s screenshots and description, explored the codebase on its own, reasoned through the issue, and successfully shipped a fix.

Background

Section titled “Background”

In mobile app development, we often hit a tricky situation: a user-reported bug cannot be reproduced in the dev environment. Possible reasons include:

  • No test device of the same model
  • Unable to mirror the user’s specific system settings
  • The issue only appears under a particular combination of conditions

Traditional methods usually require remote debugging, screen-recording analysis, or outright buying/borrowing the device. Now, Antigravity’s autonomous debugging offers a new path.

Problem Description

Section titled “Problem Description”

Issue #169: A user reports that on a certain phone, when the system font is set to maximum, the lock screen cannot be tapped to enter the app.

User-provided information:

  • Device: Samsung Galaxy S20
  • Condition: System font set to maximum
  • Symptom: The “swipe up to enter home” area on the lock screen does not respond

The AI Diagnostic Process

Section titled “The AI Diagnostic Process”

Step 1: Understand the Problem Space

Section titled “Step 1: Understand the Problem Space”

Antigravity first analyzed the key phrases:

  • “font at maximum” -> involves system font scaling (textScaleFactor)
  • “cannot tap” -> likely a layout overflow making the interactive area inaccessible
  • “lock screen” -> need to locate lock screen-related code

Step 2: Code Exploration

Section titled “Step 2: Code Exploration”

The AI used multiple tools in parallel to explore the codebase:

Key findings:

Finding 1: The main app already guards against font scaling

Finding 2: The lock screen is displayed via Overlay

Step 3: Reasoning and Analysis

Section titled “Step 3: Reasoning and Analysis”

Based on the code review, the AI inferred the root cause:

Conclusion: The lock screen is inserted via Overlay.insert() and sits outside the app’s MediaQuery wrapper. Therefore:

  1. When the user sets the system font to maximum, the lock screen text scales up.
  2. The enlarged content overflows the visible area.
  3. The “enter home” swipe area is pushed off-screen, so the user cannot interact.

Step 4: Design the Fix

Section titled “Step 4: Design the Fix”

The fix is straightforward: add a dedicated MediaQuery guard to the lock screen.

Step 5: Verification

Section titled “Step 5: Verification”

Without physical hardware, the AI used static analysis to verify:

Results showed:

  • OK No syntax errors
  • OK No compilation errors
  • Warn Only expected deprecation warnings (consistent with existing code style)

Key Reasoning Insights from the AI

Section titled “Key Reasoning Insights from the AI”

Throughout the diagnosis, the AI demonstrated several critical reasoning skills:

1. Cross-File Correlation

Section titled “1. Cross-File Correlation”

Instead of looking at lock_screen.dart in isolation, the AI traced how the lock screen is displayed (in main.dart), and spotted the mismatch between Overlay and MediaQuery.

2. Framework-Aware Reasoning

Section titled “2. Framework-Aware Reasoning”

The AI understands Flutter’s widget tree: MediaQuery scope applies only to its subtree. Overlay is not part of that subtree, so it is not constrained by the app-wide setting.

3. Mapping User Scenarios to Code

Section titled “3. Mapping User Scenarios to Code”

From the user action “font at maximum,” the AI pinpointed the technical concept textScaleFactor and how it works in Flutter.

What We Learn from This Case

Section titled “What We Learn from This Case”

For Developers

Section titled “For Developers”
  1. Mind context inheritance when using Overlay: Overlay may not inherit the desired Theme/MediaQuery settings.
  2. Global configs must cover every entry point: If you want to disable font scaling, make sure all UI entry points are covered.
  3. Include accessibility in testing: Large fonts are a common accessibility requirement and should be in the test matrix.

For AI-Assisted Development

Section titled “For AI-Assisted Development”
  1. Enough context matters: User screenshots + problem description + codebase access = sufficient diagnostic info.
  2. Static analysis is an effective validation tool: Even without a runtime environment, it can verify correctness.
  3. Reasoning compensates for missing environments: Understanding framework mechanics beats simply running tests.

Conclusion

Section titled “Conclusion”

This case shows a new paradigm for Antigravity’s autonomous debugging: reasoning-driven diagnosis based on code understanding.

Traditional debugging relies on the “reproduce -> observe -> modify -> verify” loop. When reproduction fails, the loop breaks.

Antigravity offers another path: “analyze code structure -> reason about the cause -> design a fix -> validate statically.” It does not replace traditional debugging, but serves as a powerful supplement when conditions are constrained.

Of course, final confirmation still requires testing on real devices. But the AI has already turned “searching for a needle in a haystack” into “precise pinpointing,” dramatically shortening time to resolution.

This article is based on Antigravity’s real diagnosis and fix for Issue #169 on December 11, 2025.

Decoding Antigravity: An Agentic Workflow from Screenshot to Code Fix

Section titled “Decoding Antigravity: An Agentic Workflow from Screenshot to Code Fix”

In the traditional AI-assisted programming (Copilot) model, developers often have to act as “navigators”: you tell the AI “read this file,” “explain this code,” “write a function for me.” Through conversation, you steer the AI to finish the work step by step.

Antigravity represents the next generation of Agentic AI. It is no longer a passive Q&A bot but an “intelligent teammate” that can explore autonomously, reason proactively, and independently complete tasks in complex codebases.

This article uses a real bug-fix case to break down how Antigravity actually works.

1. Perception: More Than OCR (Perception)

Section titled “1. Perception: More Than OCR (Perception)”

Everything starts with user feedback.

Suppose a user submits an issue (e.g., ISSUE-182):

Title: [Weice] blood glucose/uric acid combo meter (a combined glucometer and uric acid analyzer) uploads blood glucose data, but the database does not save it Description: Call POST /api/v2/users/records.json with a payload containing blood_glucose and uric_acid. Checking the database shows only uric acid is stored; blood glucose is missing.

After receiving the task, Antigravity first performs full-context perception:

  • Read the issue description: extract key information (API path, parameter structure, error symptom).
  • Analyze the tech stack: identify it as a Rails API project (backend) plus a Flutter app (frontend).
  • Locate the environment: determine that beyond code, it needs Postgres and Loki MCP servers to check logs and data.

It constructs a “task graph”:

  • Goal: fix the missing uploaded data.
  • Entry point: backend API endpoint /api/v2/users/records.json.

2. Exploration: Reading the code like a senior engineer (Exploration)

Section titled “2. Exploration: Reading the code like a senior engineer (Exploration)”

Antigravity does not guess; it uses its toolchain for purposeful exploration.

Step 2.1 Route locating

Section titled “Step 2.1 Route locating”

It starts like a human engineer, locating the code entry via the API path.

  • Tool Call: grep_search(query: "records", path: "config/routes.rb")
  • Result: finds the route mapped to Api::V2::Users::RecordsController#create.

Step 2.2 Following the trail

Section titled “Step 2.2 Following the trail”

Next, it digs into the controller code.

  • Tool Call: view_file(path: "app/controllers/api/v2/users/records_controller.rb")
  • Analysis: the logic shows it calls a RecordService.new(params).save service class.

Step 2.3 Into the core logic

Section titled “Step 2.3 Into the core logic”

The heart of the problem usually hides in the business layer.

  • Tool Call: view_file(path: "app/services/record_service.rb")
  • Discovery: In RecordService, Antigravity might use view_file_outline to quickly map the method structure and find the data-processing logic:

3. Reasoning: Building a mental model (Reasoning)

Section titled “3. Reasoning: Building a mental model (Reasoning)”

At this point, Antigravity is not just “reading” code; it is running a thought simulation.

Facing the situation where blood_glucose is missing but uric_acid is stored, it quickly proposes several hypotheses:

  1. Hypothesis A: Was blood_glucose filtered out by valid_type?
  2. Hypothesis B: Did a validation error occur during DB write while the main transaction did not roll back?
  3. Hypothesis C: Is there a special filter for the [Weice] device type?

To validate the hypotheses, Antigravity proactively calls MCP tools to query real-time logs or the database:

  • Monitor: use loki_query to inspect API request logs and confirm the payload reached the server.
  • Check DB: use execute_sql to check the records table schema and confirm field definitions.

Combining code logic (static analysis) with runtime state (dynamic analysis), it may spot a line that looks harmless but is the real culprit:

“Aha!” - there is the root cause.

4. Execution: The precise scalpel (Action)

Section titled “4. Execution: The precise scalpel (Action)”

Once the issue is located, Antigravity enters Execution Mode. It is meticulous:

  1. Draft a plan: Antigravity generates an implementation_plan.md listing the files and specific change points.
  2. Request approval: It shows the plan to the user: “I found issue X and plan to modify file Y. Do you approve?”
  3. Perform the edits: After approval, it uses the replace_file_content tool to precisely replace the problematic code block instead of rewriting the whole file.

5. Verification: Not just run, but verify (Verification)

Section titled “5. Verification: Not just run, but verify (Verification)”

Shipping code is not the end. Antigravity knows the importance of testing the main path.

  • Automated tests: It runs rspec spec/services/record_service_spec.rb to ensure the bug is fixed without introducing regressions.
  • Generate a report: Finally, it creates a walkthrough.md with diffs and test-result screenshots to clearly show: “The problem is solved, and here is the evidence.”

Conclusion: From co-pilot to co-worker

Section titled “Conclusion: From co-pilot to co-worker”

Antigravity’s core value lies in agency.

  • It does not need you to feed it every line: give it a requirement, and it finds the relevant files itself.
  • It does not need you to act as its compiler: it runs tests, reads errors, and fixes them itself.
  • It is a closed loop: from discovering the issue to resolving it, it forms a complete workflow.

This is Antigravity - an AI partner that works alongside you, understands code and business context, and is built to solve problems.

  • Accidentally exposing .env or sensitive logs

Not directly about images, but:

If you’re letting it edit code, move files, and run scripts, double-check permission settings. The image magic is cool—just don’t “draw” your servers away.
