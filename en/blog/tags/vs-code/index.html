Gemini CLI Companion × VS Code/Cursor: Bring gemini-2.5-pro into your IDE

Aug 31, 2025
Vibe Sparking

Wire Gemini CLI (gemini-2.5-pro) into VS Code/Cursor’s integrated terminal so it can see your workspace, understand your selections, and pop native diff windows. Smarter, steadier, less hassle.

Introduction

Section titled “Introduction”

Still doing the “copy → paste → enter” shuffle? Now, the command line = your IDE’s super-brain plugin. The latest Gemini CLI update directly bridges the VS Code integrated terminal: it can read your workspace, detect selected text, show native side-by-side diffs, and let you tweak before accepting. Nice.

Highlights (worth turning on today)

Section titled “Highlights (worth turning on today)”
  • Workspace-aware context: It can “see” your open files and read your current selection, making suggestions more precise.
  • Native full-screen diff: Proposed edits open in VS Code’s side-by-side diff. Review and fine-tune before accepting — no blind overwrites.
  • One-command toggle: /ide enable on, /ide disable off, switch anytime.
  • Cursor mostly works: The extension is marked compatible with VS Code and forks (Cursor is a VS Code fork), though in some cases you may need to install it manually.

Install & Enable (3-minute setup)

Section titled “Install & Enable (3-minute setup)”

Using VS Code as the example; Cursor is similar (manual extension install may be required — see “Common Pitfalls”).

  1. Prerequisites
  • Gemini CLI version ≥ 0.1.20
  • You must run the CLI from the IDE’s integrated terminal (not an external terminal).
  1. One-time install In the VS Code terminal, run:

Install IDE integration in gemini-cli:

This will automatically install the Gemini CLI Companion extension.

  1. Turn it on

Turn it off when needed:

Note: The marketplace page states the extension supports VS Code and VS Code forks. If you use Cursor, open the marketplace and manually search and install “Gemini CLI Companion”.

Recommended Usage (treat it as “code copilot + edit bot”)

Section titled “Recommended Usage (treat it as “code copilot + edit bot”)”

1) Select → propose → accept via native diff

Section titled “1) Select → propose → accept via native diff”
  • In a file, select the code or text you want to change.
  • In the terminal, tell Gemini your intent (or use your usual prompt template).
  • A side-by-side diff opens → you can fine-tune the right-side edits → then accept.

Great for: small bug fixes, API adjustments, renames, log unification — all “fine-grained” edits.

2) “Open files + guardrails” refactors

Section titled “2) “Open files + guardrails” refactors”
  • Open 2–3 key files involved (so it can “see” the context).
  • Provide “goal + constraints” (e.g., preserve function signatures, pass existing tests).
  • Land changes in batches via diff, balancing safety and pace.

Great for: slimming controllers, extracting hooks, standardizing error handling.

3) Selection = precise context anchor

Section titled “3) Selection = precise context anchor”
  • Use selections to keep it focused.
  • In complex files, feed only the critical functions/snippets to reduce drift.

Great for: long files and legacy-heavy modules.

VS Code vs. Cursor: smooth setup

Section titled “VS Code vs. Cursor: smooth setup”

  • VS Code: /ide install auto-installs the extension → then /ide enable.

  • Cursor:

    • In the Extensions page, manually install “Gemini CLI Companion” (VS Code forks supported).
    • If /ide install shows “No installer for Cursor”, it’s a known issue — follow the hint to install manually.
    • Cursor supports most VS Code extensions out of the box (as stated by the project and third-party reviews), but your mileage may vary.

Common Pitfalls & Troubleshooting

Section titled “Common Pitfalls & Troubleshooting”

  • Extension installed, but CLI says it can’t connect to the IDE

    • Required environment variables (such as GEMINI_CLI_IDE_WORKSPACE_PATH / GEMINI_CLI_IDE_SERVER_PORT) might not be ready.

    • Fix: Ensure the extension is enabled; open a fresh IDE terminal and try again.

  • /ide install fails in Cursor

    • Message: “No installer is available for Cursor”.
    • Fix: Manually install the extension from the marketplace.

  • No native diff appears

    • Confirm you’re running the CLI in the IDE integrated terminal, not an external terminal.

Who benefits? Best-fit scenarios

Section titled “Who benefits? Best-fit scenarios”
  • Solo developers: small, steady steps with immediate visual feedback.
  • Teams: consistent style and batch micro-edits (logs, errors, naming, comments).
  • Legacy cleanup: low-risk “sliced” refactors, landing edits via diffs.

Essentially, it turns “LLM code edits” into an auditable, controllable IDE-native workflow.

A handy onboarding checklist

Section titled “A handy onboarding checklist”
  1. npm i -g @google/gemini-cli (skip if already installed)
  2. Open the project in VS Code → in the integrated terminal run: /ide install → /ide enable
  3. Open 2–3 key files and start with small, selection-driven edits
  4. When the diff opens, fine-tune then accept → commit with a clear message
  5. Add a team norm: what changes can go through Gemini, and what must go through manual review

The official blog and extension page have key details — worth bookmarking.

Closing

Section titled “Closing”

Combine the “hardcore power” of the command line with the IDE’s visualization to boost both efficiency and certainty. Connect Gemini CLI to VS Code/Cursor today to make changes more controlled, context more precise, and collaboration smoother.

Codex Releases IDE Plugin for VS Code and Cursor

Aug 30, 2025
Vibe Sparking

Want to embed the “code-writing CodeX GPT-5” into VS Code? The Codex IDE gets it right. Read, modify, and run code in one go. This article will guide you through its highlights, installation steps, and most recommended use cases in 10 minutes.

Why Codex IDE?

Section titled “Why Codex IDE?”

It’s an “agent that can run code on its own.” Not just a chat assistant. Codex can directly read files, modify code, and run commands in your working directory. It also supports offloading large tasks to the cloud to run, and then pulling the changes back to your local machine for one-click acceptance. The official extension is compatible with VS Code / Cursor / Windsurf, with stable performance on macOS and Linux. For Windows, using WSL is recommended.

Side-by-side panel for collaboration. Drag the Codex panel to the right side of your editor. Selecting a piece of code allows you to type less and provide more accurate context, resulting in faster and more relevant responses.

Three “approval modes” for a balance of safety and efficiency. The default Agent mode can automatically read, modify, and run files within the current directory; it will ask for permission if it needs to go outside the directory or access the network. You can also switch to the chat-only Chat mode, or the Agent (Full Access) mode which has unrestricted permissions (use with caution).

Cloud execution, local observation. While chatting locally, you can trigger “Run in the cloud” with a single click. For large projects, let it work on a new branch; for smaller tasks, it’s smoother to have it finish up based on your local changes. The cloud context is maintained when you pull the changes back, so you can test them before merging.

Orchestrable and extensible. The extension is built around the open-source Codex CLI and supports the MCP and AGENTS.md specifications, making it easy for teams to write down project rules, build steps, and acceptance procedures for the agent to read.

3 Steps to Get Installed and Logged In

Section titled “3 Steps to Get Installed and Logged In”

① Install the extension Search for “Codex – OpenAI’s coding agent” in the VS Code Marketplace and click install. The process is the same for Cursor and Windsurf.

② Log in It is recommended to use your ChatGPT account (Plus / Pro / Team / Edu / Enterprise are all included). You can also use an API Key, but that requires additional configuration.

③ (Optional) Install the CLI If you want to run it purely in the terminal or for CI, install the Codex CLI: npm i -g @openai/codex or brew install codex, then run codex to get started.

Windows Tip: For a more stable experience, use it with WSL.

6 Immediately Useful Scenarios

Section titled “6 Immediately Useful Scenarios”

1) Quickly Understand Legacy Projects

Section titled “1) Quickly Understand Legacy Projects”

Let Codex run tests, trace call chains, and summarize module responsibilities in the repo to give you an “out-of-the-box tour.” Ideal for new team members or when taking over an old project.

2) Implement Small Features on the Spot

Section titled “2) Implement Small Features on the Spot”

In Agent mode, you can directly ask it to create new files, modify routes, add unit tests, and run local commands. You just need to review the PR.

3) Refactoring/Migration

Section titled “3) Refactoring/Migration”

Provide the goals and constraints, and let it commit changes in batches. If the task is time-consuming or has a large impact, switch to a cloud task to run it, and review the diff as it works.

4) Performance Profiling & Bug Reproduction

Section titled “4) Performance Profiling & Bug Reproduction”

Provide logs and reproduction steps, and let it write probe scripts or local stress test commands and execute them to generate a comparison report.

5) “Internalize” Security and Standards

Section titled “5) “Internalize” Security and Standards”

Add an AGENTS.md file to the project root: specify build/test commands, code style, commit conventions, and security points. The agent will follow these instructions, reducing “verbal agreements.”

6) Improve Team Collaboration Efficiency

Section titled “6) Improve Team Collaboration Efficiency”

Standardize the workflow of “Requirement → Cloud Run → Local Acceptance → Merge” as a team SOP. With everyone following the same process, the delivery pace becomes more stable.

3 Common Workflows You’ll Use

Section titled “3 Common Workflows You’ll Use”

Workflow A: Quick Local Changes

Section titled “Workflow A: Quick Local Changes”
  1. Switch the panel to Agent.
  2. Select code → state your intent → let it modify and run.
  3. Review the diff, and “undo/redo” if you’re not satisfied.

Workflow B: Run Large Tasks in the Cloud

Section titled “Workflow B: Run Large Tasks in the Cloud”
  1. Specify the goal and constraints (branch, test coverage).
  2. Select the environment → Run in the cloud.
  3. You can continue local development during this time; once completed, pull the changes back to your local machine for testing.

Workflow C: Chat First to Finalize the Plan

Section titled “Workflow C: Chat First to Finalize the Plan”

Switch to Chat to think through the design, interfaces, and boundaries first, then switch back to Agent for implementation.

Team Adoption Cheat Sheet (Highly Recommended)

Section titled “Team Adoption Cheat Sheet (Highly Recommended)”

  • Write standards into AGENTS.md: Installation commands, test scripts, linting rules, commit templates, permission scopes. The agent will follow what you write.

  • Choose the mode based on the scenario:

    • Use Chat for design discussions.
    • Use Agent for local changes.
    • Consider Full Access only for large-scale or network-related operations.

  • Use cloud tasks for “heavy lifting”: Migrating frameworks, batch refactoring, generating large test suites.

  • Keep the VS Code panel docked on the right to reduce window switching.

Frequently Asked Questions

Section titled “Frequently Asked Questions”

  • Can’t see it after installation? It might be collapsed in the sidebar’s extension area; drag it to the right to dock it.

  • Planning to use it on Windows? Use WSL first for a smoother experience.

  • How to update? The extension updates automatically, but you can also check for updates manually on the IDE’s extension page.

TL;DR

Section titled “TL;DR”
  • Local = Quick changes/runs, Agent mode is sufficient.
  • Cloud = Heavy lifting, preserves context, pull back for testing when done.
  • Write rules in AGENTS.md, maximizing the quality of team collaboration.

3 “Out-of-the-Box” Example Prompts

Section titled “3 “Out-of-the-Box” Example Prompts”
  • “Add a new /pricing page in apps/web using Tailwind, add e2e tests, and commit after npm run test passes.”
  • “Optimize the N+1 query in UserService using includes, and generate a benchmark comparison report.”
  • “Upgrade eslint and prettier in the Monorepo to the latest stable versions, fix all CI errors, and then create a PR.”

Voice-Driven Vibe Coding on Mac with Spokenly + Claude Code

Aug 26, 2025
Vibe Sparking

You speak, it writes, things run. Hands off the keyboard, code still flies.

Introduction

Section titled “Introduction”

Over the past two months I’ve become addicted to Vibe Coding. Pair Spokenly for voice input with Claude Code as a “capable terminal copilot”. I keep three panes open: Plan, Build, Review. I grab tea; when I’m back it’s just “acceptance” and “next steps”. Before bed I toss in a big task; next morning I check results like a ruthless boss — day’s rhythm, locked.

This guide shows the complete setup for driving Claude Code by voice on macOS with Spokenly. Install and go — you can ship tonight.

TL;DR

Section titled “TL;DR”
  • Spokenly: global hotkey, dictation in any input field; supports local Whisper, offline-ready; or bring your own cloud API key; Agent Mode to control your Mac by voice.
  • Claude Code: smart pair programmer in your terminal. After npm i -g @anthropic-ai/claude-code, just run claude; supports slash commands like /model /config /ide /compact, deep IDE integration with VS Code/JetBrains.
  • Voice tips: use Option+Enter or /terminal-setup for multi-line input; Esc Esc to edit the last message; run three-pane flow for Plan → Execute → Review.

1) Tools and Installation

Section titled “1) Tools and Installation”

1) Spokenly (speech → text)

  • Website: https://spokenly.app/ — supports on-device Apple Speech Recognition, Whisper, 100+ languages, Local-only privacy mode, Agent Mode, searchable history; Mac + iPhone.
  • App Store: free download, Pro adds cloud models; recent releases add Soniox, Parakeet v3, re-transcription on history, and more.

2) Claude Code (terminal AI)

You’ll be up in 30 seconds. Supports MCP tools, running commands, editing files, creating commits, and IDE connections.

3) IDE (optional but highly recommended)

  • VS Code/Cursor/Windsurf: when launched in the integrated terminal, claude auto-installs the extension.
  • JetBrains: install the plugin and restart; from an external terminal use /ide to connect to the IDE.

Alternative: Apple’s built-in Dictation works too, but Spokenly wins on features and privacy (especially multilingual and offline Whisper).

2) Voice control in 5 minutes

Section titled “2) Voice control in 5 minutes”

Step 1: Configure Spokenly

  • Grant microphone permission on first launch.
  • Set a global Push-to-Talk hotkey (e.g., Right Command) in Preferences.
  • Choose Local-only to use Apple Speech Recognition — fully offline and private.
  • Enable Auto language to freely mix Chinese and English.
  • Mode: “Type at cursor” so dictated text flows into your terminal/editor like you’re typing.

Step 2: Start Claude Code

  • Run claude in your project directory.
  • Need multi-line input or long requirements?
    • Use Option+Enter for multi-line; or run /terminal-setup to bind Shift+Enter to newline.
  • In IDEs: start claude in an external terminal, then /ide to connect to VS Code/JetBrains to enable selection context, IDE Diffs, and more.

Step 3: Start talking

  • Focus the Claude input box → hold PTT hotkey to speak → release to send.
  • Need to tweak the last line? Press Esc Esc to edit and resend.

3) The three-pane Vibe Coding flow: Plan | Build | Review

Section titled “3) The three-pane Vibe Coding flow: Plan | Build | Review”

Pane A: Architect (Plan)

  • Say: “Add OAuth login to a Next.js project. First provide a task list and file-change plan, each step within 20 lines.”
  • Commands: use /model to pick the model; /compact to compress context and keep focus.

Pane B: Builder (Implement)

  • Say: “Follow pane A’s plan. Write code and self-test; if errors occur, fix automatically; run scripts when needed.”
  • For structure, dictate with Option+Enter: “Block 1 is the API route; Block 2 is the UI component…”

Pane C: Reviewer (Review)

  • Commands: /review, /cost for token/cost; if needed, use /config to constrain tools and scope.
  • In IDE mode, review diffs directly in the editor for a near-native pair-programming feel.

Pro tip: use separate project directories per task to start claude. Each session history is isolated per directory for clean context switching.

4) Bedtime “overnight sprint” voice template

Section titled “4) Bedtime “overnight sprint” voice template”

Break big tasks into four spoken parts so Claude Code stays on track:

  1. Goal: one-sentence objective + success criteria
  2. Scope: which directories to touch and which to avoid (allow/deny lists)
  3. Checks: self-test scripts, lint, unit/E2E tests
  4. Deliverables: PR summary, diff, changelog

Example: “Tonight migrate the Settlement module from REST to tRPC. Only modify /server/ and /app/billing/. Run pnpm test and pnpm lint — both must pass. Output steps and risks, and provide a rollback guide.”

5) Seven tips for smoother dictation

Section titled “5) Seven tips for smoother dictation”
  • Speak structure: “Heading: … New paragraph. List: one, two, three.” Helps produce cleaner docs.
  • State constraints: “Change at most three areas, each under 30 lines.” Prevents overreach.
  • Set boundaries: “Only allow Bash: pnpm*, git*.” (combine with /config to limit tools)
  • Require checks: “After changes, run npm run test; if it fails, roll back.”
  • Mix languages: APIs and identifiers in English are recognized more accurately; Spokenly auto-detects.
  • Multi-line input: Option+Enter or /terminal-setup.
  • Edit last message: Esc Esc to revise.

6) Privacy and cost

Section titled “6) Privacy and cost”
  • Local-first: Spokenly’s Local-only mode keeps audio on-device; pair with offline Whisper.
  • Bring your own key: if you need cloud transcription, plug in OpenAI/Deepgram/Groq, etc., to control cost; Pro subscription simplifies setup.
  • Claude costs: use /cost to check usage; control model and thinking depth via /model.

7) Common pitfalls and fixes

Section titled “7) Common pitfalls and fixes”
  • Terminal newline quirks → use Option+Enter or /terminal-setup to bind Shift+Enter.
  • IDE extension not installed → run claude in the IDE’s integrated terminal; restart JetBrains; or from an external terminal use /ide to connect.
  • Dictation not landing in terminal → ensure focus on Claude input; adjust Spokenly hotkey conflicts.
  • Want Apple Dictation → System Settings → Keyboard → enable Dictation, set “Press Control key twice”.

Ready-to-read dictation prompts

Section titled “Ready-to-read dictation prompts”
  • “Add an export-CSV endpoint under /api/orders. First provide a plan, then write code, then run tests, showing each step.”
  • “Move state management on the payment page from Redux to Zustand. Only touch existing logic, keep UI. List 3 risk points.”
