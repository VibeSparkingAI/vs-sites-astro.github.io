Use BMAD in Codex like Claude Code: Put a multi‑role AI squad in your repo

Sep 25, 2025
Vibe Sparking

Install Codex and BMAD, map BMAD’s “multi‑role experts” into AGENTS.md, then in the terminal say “*agent analyst”. Codex will run a BMAD‑style agile loop from requirements to delivery.

Introduction

Section titled “Introduction”

Two pain points in projects: no one writes clear requirements, and even when they do, no one lands them properly.

BMAD turns roles like Business Analysis, Architecture, Product, Development, QA, and Scrum into version‑controlled Markdown agents. Combined with Codex’s AGENTS.md, you work as if an AI squad is embedded in your repo.

This is a practical, copy‑ready path:

  1. Install BMAD
  2. Update AGENTS.md to register BMAD roles
  3. One prompt to bring in the Analyst and start

Why Codex + BMAD?

Section titled “Why Codex + BMAD?”
  • Codex understands AGENTS.md: it follows your project rules to build, check, and propose changes, in both CLI and IDE.
  • BMAD provides reusable “Agent as Code”: each role is a .md with tasks, checklists, templates, and workflows.
  • Together: “project rules + multi‑role collaboration + executable checks.” It feels like a real team that never forgets the process.

Step 2: Register BMAD roles in AGENTS.md

Section titled “Step 2: Register BMAD roles in AGENTS.md”

AGENTS.md is a “README for agents.” It lives at the repo root, as standard Markdown. Use it to document your build/test/style/PR process and any local conventions.

Codex, Cursor, Gemini, and other agents read it; agents will follow rules you define here.

Codex Gets Slash Commands: Save Prompts as /commands for One‑Tap Reuse

Sep 16, 2025
Vibe Sparking

You can now write your frequently used prompts in Markdown, drop them into ~/.codex/prompts/, and call them in Codex with /command. This speeds up workflows immediately. Community reports confirm it works well in the CLI, and setup is simple (npm i -g @openai/codex or brew install codex).

Note: As you observed, it currently doesn’t support directory‑scoped prefixes. In my testing too, it’s strictly “filename = command name,” with no namespace from subfolders.

Why use it now?

Section titled “Why use it now?”

The real time‑saver is prompt reuse. Turn high‑frequency prompts—like “code review,” “generate tests,” or “release checklist”—into /review, /tests, and /ship. Type once, reuse anywhere. Codex has been rapidly improving its CLI/IDE experience, so it’s a great time to standardize your prompts.

Even better, this pattern mirrors Claude Code’s “custom slash commands”: store prompts as Markdown, recall with a slash. The migration cost is near zero.

Three‑step quickstart

Section titled “Three‑step quickstart”

1) Create the folder
Make sure ~/.codex/prompts/ exists (create it if not).

2) Add a Markdown file
For example, create review.md with content like:

“You are a code review assistant. Provide structured feedback on: security, performance, complexity, readability, and test gaps. Output with clear bullets.”

3) Restart your Codex session
Open Codex, type /revi… and it will autocomplete to /review. Press Enter to run. This matches community reports on folder usage and behavior.

Practical tips:

  • Command name = filename (without .md).
  • Avoid collisions with built‑ins (don’t take names like /init).
  • New files are most reliably recognized after starting a new session.

Current limitations and workarounds

Section titled “Current limitations and workarounds”

  • No directory‑scoped prefixes
    You can’t map backend/review.md and frontend/review.md to different /reviews. Add suffixes like review-fe.md and review-be.md instead.

  • Parameters/placeholders are limited
    Some users report @‑style parameters are unreliable. A reliable approach: trigger the command first, then add concrete context in the next turn.

  • Short, memorable names win
    Think of the slash menu like VS Code/Cursor quick actions—short names you can remember and type fast. Follow community best practices and conventions.

5 ready‑to‑use high‑leverage commands

Section titled “5 ready‑to‑use high‑leverage commands”

  • /review: Structured code review (security/performance/complexity/readability/tests).

  • /tests: Generate table‑driven/parameterized tests and cover edges.

  • /ship: Pre‑release/release checklist (migration, rollback, monitoring, canary scope).

  • /hotfix: Fast triage and minimal‑blast‑radius fix plan.

  • /release-notes: Human‑readable notes + tweet‑length announcement.

Save each snippet above as a .md file in ~/.codex/prompts/. After restarting your session, they’ll show up via /.

Use it alongside Claude Code

Section titled “Use it alongside Claude Code”

If your team mixes Codex + Claude Code, standardize reusable flows as Markdown prompts and use each tool’s slash menu. Both sides can consume the same files, reducing cognitive load. The Claude community also encourages this prompt‑as‑Markdown approach.

Closing

Section titled “Closing”

Standardizing and modularizing prompts is a key step in turning AI coding from “inspiration” into “process.”
Starting today, put your top 3 prompts into ~/.codex/prompts/ and let your fingers remember the joy of /.

Codex 0.23.0: One‑Tap /model to OpenAI GPT‑5 High Reasoning

Aug 24, 2025
Vibe Sparking

With Codex CLI 0.23.0, model selection and reasoning strength become a one‑step toggle. Type /model, pick GPT‑5 + High reasoning, and complex tasks become a lot more stable.

Introduction

Section titled “Introduction”

Many developers already use Codex to write code, run tests, and edit files. But switching models or reasoning intensity used to mean changing config or restarting. Not anymore. 0.23.0 adds /model with live switching, plus High reasoning for GPT‑5 and even a new Minimal tier. It’s a tangible boost in speed and control.

What exactly changed in this release?

Section titled “What exactly changed in this release?”
  • Slash commands: Type /model in the terminal to change model/reasoning on the fly; /approvals gives you interactive approval control.
  • Reasoning control: Adjust reasoning effort at runtime; GPT‑5 also supports a minimal tier (faster, fewer reasoning tokens).
  • Login and versioning: ChatGPT Plus/Pro/Team accounts can use GPT‑5 directly in Codex; stable release is 0.23.0 (Aug 20).

Background: GPT‑5 exposes a reasoning_effort parameter (minimal | low | medium | high) to control how deeply the model “thinks.” Higher levels usually produce more reasoning tokens with higher latency; minimal trades depth for faster first tokens.

Section titled “1‑minute quickstart: three steps”

Install / upgrade

(Installation methods follow the official NPM/README.)

Sign in (ChatGPT plans include GPT‑5)

After sign‑in, Codex lets you use the latest model included in your plan (including GPT‑5).

Two ways to switch

  • CLI: go straight to High reasoning (override via --config):

    (--model/-m and --config follow the official docs.)

  • Interactive one‑tap /model: In the TUI, type /model, select gpt‑5, then select High (or Minimal/Low/Medium).

Want High by default? Put it in ~/.codex/config.toml, or override per run with --config.

When to use High reasoning vs Minimal/Medium

Section titled “When to use High reasoning vs Minimal/Medium”

Choose High for

  • Complex refactors / design tradeoffs: e.g., “Split an MVC app into layered architecture with dependency inversion, plus tests.” Needs multi‑step planning and global consistency.
  • Cross‑language migrations: e.g., Python → Go while preserving concurrency/error semantics. Demands careful API and model mapping.
  • Tricky bug forensics: deep stacks, race conditions, edge cases—benefit from longer chains of thought.

Choose Minimal/Medium for

  • Scaffolding, bulk lint fixes, renames: you want speed more than deep reasoning. minimal greatly reduces reasoning tokens and speeds things up.

  1. Downshift for speed
    After core logic stabilizes, type /model and switch reasoning to medium or minimal for the bulk‑edit phase.

    Break a big goal into rounds: design → interfaces → implementation → tests. The official guidance also favors “multi‑turn, one step per turn”—in practice it’s more reliable.

Tip: Higher reasoning means more reasoning tokens and potentially higher latency. If it feels sluggish, /model → minimal/low.

FAQs and small gotchas

Section titled “FAQs and small gotchas”

  • “With ChatGPT sign‑in, is reasoning stuck at Medium?”
    Early community reports said so; but 0.23.0 supports switching /model and reasoning effort at runtime. You can change it directly in the TUI.

  • Conflicts between CLI and config?
    CLI --config takes precedence for the current run; write long‑term defaults in ~/.codex/config.toml.

  • What’s the point of Minimal?
    Per the docs, it produces almost no log‑style reasoning tokens and targets scenarios that are extremely sensitive to first‑token latency.

Closing

Section titled “Closing”

Codex 0.23.0 turns “change model/tune reasoning” from a config chore into a one‑tap action. Use High for complex work; use Minimal/Medium for bulk edits. That’s the practical aim. Now go ship that weekend project.
