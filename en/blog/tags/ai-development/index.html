Vibe Sparking

Core Insight: Can Zhipu GLM-4.7 replace the expensive Opus 4.5 in the Claude Code environment? How does its 200K context window perform within the document-driven BMAD workflow? This post reveals the answers through the real-world development of the HeartPetBond app.

1. Motivation: Why This Review?

Section titled “1. Motivation: Why This Review?”

In the AI-assisted programming realm, Claude Opus 4.5 has been the undisputed king, but its high cost (approx. $15/1M input, $75/1M output) deters high-frequency usage. DeepSeek V3, while affordable, occasionally hits bottlenecks with ultra-long contexts (>128K tokens).

I aimed to verify Zhipu GLM-4.7, a strong challenger to DeepSeek V3, by developing the HeartPetBond app to test:

  • Long Context Capability: Can the nominal 200K Tokens Context support an entire project’s documentation and codebase?
  • Output Length: Can the 128K Tokens Output generate complex Flutter pages in one go?
  • Stability & Hallucination: Does logic crumble during continuous development over several days?
  • Cost-Effectiveness: How much money can it save independent developers compared to Opus 4.5?

Context: When using DeepSeek V3 with Claude Code + BMAD, I encountered errors like API Error: 400 ... maximum context length is 131072 tokens... when the context exceeded 128K. GLM-4.7’s 200K context might be the solution.

With over 10 years of Flutter experience and previous success generating the MoneyMind App using Claude Code + BMAD, I set out to see if this new model could handle the challenge.

ClaudeCode Workflow Studio: Visual Automation Editor for AI Workflows

Section titled “ClaudeCode Workflow Studio: Visual Automation Editor for AI Workflows”

Claude Code, Anthropic’s official CLI tool for AI-assisted development, has a powerful automation system through its .claude configuration files. However, creating complex workflows with multiple sub-agents, conditional branching, and user interactions requires editing markdown and YAML files manually. ClaudeCode Workflow Studio (cc-wf-studio) by breaking-brake brings a visual, drag-and-drop editor to VSCode, making workflow creation intuitive and accessible.

What is ClaudeCode Workflow Studio?

Section titled “What is ClaudeCode Workflow Studio?”

ClaudeCode Workflow Studio is a VSCode extension that provides a visual workflow editor for Claude Code. It allows you to:

  • Design workflows visually - Drag and drop nodes onto a canvas
  • Configure without code - Set up sub-agents, skills, and MCP tools through a UI
  • Export with one click - Generate .claude/agents/ and .claude/commands/ files automatically
  • Collaborate with your team - Share workflows via Slack with preview cards

GitHub Repository: breaking-brake/cc-wf-studio VSCode Marketplace: Open VSX License: AGPL-3.0-or-later Version: 3.9.1 (as of December 2025)

Key Features

Section titled “Key Features”

1. Visual Workflow Editor

Section titled “1. Visual Workflow Editor”

The core of cc-wf-studio is its drag-and-drop canvas. Instead of writing markdown and YAML, you visually connect nodes to define workflow logic:

  • Prompt Nodes - Define reusable prompt templates with {{variable}} substitution
  • Sub-Agent Nodes - Configure autonomous AI agents with custom system prompts and tool permissions
  • Skill Nodes - Integrate Claude Code Skills from personal or project directories
  • MCP Nodes - Connect to Model Context Protocol tools for external integrations
  • Control Flow Nodes - IfElse (binary branching), Switch (multi-way branching), AskUserQuestion (user decisions)

2. AI-Assisted Workflow Refinement

Section titled “2. AI-Assisted Workflow Refinement”

One of the standout features is AI-assisted refinement. Instead of generating workflows from scratch once, you can iteratively improve them through conversation:

  1. Click the “Edit with AI” button (✨ sparkle icon)
  2. Describe your changes in natural language (max 2000 characters)

BMAD Method: How to Turn LLM System Prompts into a Reproducible AI Dev Team

Dec 27, 2025
Vibe Sparking

BMAD Method is an open-source AI-driven agile development methodology and toolkit based on the BMAD Core framework. It includes 19 professional AI Agents and 50+ Workflows covering the complete lifecycle from requirements analysis, planning, and architecture design to development and testing. It has garnered 23.6k stars on GitHub (as of December 2025). It packages system prompts, role definitions, and process instructions into structured assets, allowing you to orchestrate a team of AI experts working together like a real software development team, rather than copying and pasting fragmented prompts.

Why Do You Need a Systematic System Prompt Repository?

Section titled “Why Do You Need a Systematic System Prompt Repository?”

Most AI learners and developers encounter these pain points when using large language models:

  • Disorganized system prompts: You write a good system prompt in Claude today, modify it in ChatGPT tomorrow, and end up with prompts scattered across conversations and documents, unable to find the best version.
  • Unstable AI roles: Sometimes you want it to be a product manager, sometimes an architect, sometimes QA, but you have to re-explain each time with inconsistent results and unstable output quality.
  • Lack of closed-loop from requirements to code: Even if you’ve written many prompts, they’re mostly one-off: requirements documents have one style, architecture discussions another, development and testing yet another. These prompts lack strict artifact flow relationships, making it hard to form a replicable engineering process.
  • High collaboration costs: When you want to share your collection of prompts with colleagues, they can only copy from a long document or Notion page one by one—completely non-toolified.

If you’ve experienced similar issues, what BMAD Method does can be understood as:

Engineering, versioning, and portable AI system prompts + role division + process steps, transforming them into an AI team framework that can be repeatedly reused across various IDEs and AI tools.

What is BMAD Method Essentially?

Section titled “What is BMAD Method Essentially?”

The positioning of BMAD Method can be summarized in one sentence:

Breakthrough Method for Agile AI-Driven Development — A framework that enables AI to participate in development as an agile team.

Its core has three layers:

  • BMAD Core: A foundational framework designed for human + multi-agent collaboration, containing:

    • Agents (professional role personas) written in Markdown
    • Agent Teams composed of multiple agents
    • Workflows (strategic processes) defined in YAML
    • Templates with LLM instructions (PRD, architecture documents, Story files, etc.)

  • BMAD Method: Built on top of BMAD Core, focusing on software development workflows, breaking down the entire chain from analysis → planning → solution → implementation into multiple stages and artifacts, completed by different agents in relay.

  • BMAD Builder / Creative Intelligence Suite:

    • BMAD Builder: Allows you to design your own professional agents and modules on the same framework (e.g., legal, finance, education fields).
    • Creative Intelligence Suite: Provides creative thinking and brainstorming workflows to assist product innovation and narrative design.

In other words, BMAD Method is not just a simple prompt collection. It structures system prompts by:

  • Roles (who does it)
  • Stages (when to do it)
  • Artifacts (what to produce)
  • Tools (where to use them)

All structured into an engineering methodology and code repository that can be ported to your project.

A System Prompt Closed-Loop Example from Requirements to Launch

Section titled “A System Prompt Closed-Loop Example from Requirements to Launch”

Based on the official e-commerce example, BMAD’s complete process is roughly as follows:

  1. Analyst: Project Vision and Market Analysis

    • Command example: @analyst Create project brief for modern e-commerce platform
    • Output: project-brief.md

  2. Product Manager: Generate PRD

    • Functional Requirements (FR)
    • Non-Functional Requirements (NFR)
    • Epic list and user story hierarchy
    • Use Analyst’s Brief as input
    • Output: PRD.md

  3. Architect: System Architecture Design

    • Output: architecture.md, including tech stack, system components, data flows, and API specifications.

  4. Product Owner: Document Sharding

    • Break PRD + architecture docs into focused Epic files, such as:
      • epic-1-user-auth.md
      • epic-2-products.md

  5. Scrum Master: Generate Story Files

    • Generate Story files for a specific Epic, such as:
      • 1.1.user-registration.story.md
    • Story includes context, implementation details, acceptance criteria, etc.

  6. Developer: Code Implementation

    • Implement code around Story files (Controllers, components, database migrations, etc.) and generate test cases.

  7. QA: Quality Verification and Tracking

    • Produce a series of QA documents including test strategy, tracking matrix, quality gate reports.

In this entire process, you can see:

  • Each stage corresponds to one or more agents
  • Each agent is backed by a highly structured system prompt
  • Artifacts (documents & code) form strict tracking chains through filenames and directory structures
