Will Documentation-Driven AI Coding Become the New 'Spaghetti Mountain'? Pros and Cons of BMAD, Spec-Kit, and OpenSpec

Oct 23, 2025
Vibe Sparking

Remember the days when “spaghetti code💩” used to give everyone a headache? With AI copilots like Claude Code, Codex, and Gemini in the mix, we have stumbled into a brand-new pitfall—the “documentation spaghetti💩.” Documentation-driven AI programming has become a hot trend: write a clear specification first, then let AI generate the code. It sounds wonderful—no more guessing what the model wants, just provide a well-written brief and code appears automatically. The truth, though, is that if you take the wrong approach, the documentation can pile up into a brand-new spaghetti mountain that leaves you laughing and crying at the same time 😅.

Introduction: What on Earth Are BMAD, Spec-Kit, and OpenSpec?

Section titled “Introduction: What on Earth Are BMAD, Spec-Kit, and OpenSpec?”

Let’s start with these documentation-driven AI programming tools that are making the rounds—they each have their own personality:

BMAD Method – a “big and comprehensive” framework whose full name is Breakthrough Method for Agile AI-Driven Development. Its ambitions are huge: it tries to simulate a complete agile squad with AI. There is an analyst AI, an architect AI, a developer AI, a tester AI—each playing a role so that planning, coding, and testing all get staged like a full production. BMAD ships with a CLI and a web UI, plus special “script files” and context management so the AI team can collaborate. It usually unfolds in two phases: first the “analyst/product manager/architect” agents do the planning and design, then the “Scrum Master/developer” agents write the actual code. The upside is a thorough workflow that stays orderly—ideal for large, complex projects that demand clear documentation and division of labor. The downside is that it is cumbersome to use, has a steep learning curve, and expects the underlying model to truly understand the documentation you write.

Spec-Kit – an open-source utility on GitHub that layers a bit of “process” on top of your existing AI coding assistant. Developers use the command line to nudge AI through a “Specification → Plan → Tasks → Implementation” four-step. In short, Spec-Kit asks you to nail the requirements first, then guides AI to break work down and produce code. It turns your specification markdown into an “executable document” using commands like /specify, /plan, and /tasks. It is lightweight and flexible, does not force you to revamp your workflow, and is model-agnostic. Great for solo builders or small teams that want to add structure and improve AI-generated code quality.

OpenSpec – a minimalist newcomer that believes “simple is better.” You write a slim Markdown spec and pair it with a handful of commands to finish the job. Its flow has only three steps: submit a change proposal, let AI produce the task list, then let AI apply the spec to modify the code. The highlight is efficiency: the specs stay lean, token usage stays low, and there is barely any metadata. It works well for individuals or small teams chasing extreme speed—minor tweaks get done in minutes. The trade-off is that model support is still limited, but it is so easy to learn that there is almost no overhead.

Spot the pattern? These tools share the same philosophy: document first, then drive the AI. The goal is to reduce the uncertainty of vibe coding by giving AI a clear rulebook so code quality becomes controllable. Ideal in theory—but reality bites. If you are not careful, the documentation you create can morph into another giant spaghetti pile. Why does that happen? Read on.

Four Reasons Documentation Turns into a Spaghetti Mountain Even Faster

Section titled “Four Reasons Documentation Turns into a Spaghetti Mountain Even Faster”

1. Lack of Structure

Section titled “1. Lack of Structure”

Code is constrained by syntax—one misplaced comma and it will not even run. Documentation is much freer, and AI is not human. When a model stares at free-form, meandering text, its comprehension goes sideways. Code is deterministic; specifications are often vague and open to interpretation. If your documentation is loosely structured, the AI sees the world through fogged glass, and it is no surprise when the code it generates veers off in the wrong direction.

2. Inconsistent AI Understanding

Section titled “2. Inconsistent AI Understanding”

Even if you write a decent document, there is no guarantee the AI will interpret it the same way every time. Seasoned developers know the saying “a thousand readers, a thousand Hamlets”—and AI thinks even more differently from humans. ChatGPT might implement the feature correctly today, but a different model tomorrow could distort your intent. Documentation lacks the compiler’s safety net, so misunderstandings can hide in plain sight and quietly plant landmines.

3. AI Hallucinations Amplify Errors

Section titled “3. AI Hallucinations Amplify Errors”

We all know large language models hallucinate—confidently fabricating details. When AI uses incomplete or ambiguous documentation to write code, the stakes rise. Miss a detail in your spec and the AI might “fill in the gaps” with something that looks plausible but is wrong. Worse still, if that flawed AI-generated documentation or code gets reused later, another AI might treat it as gospel truth. Errors balloon like rumors: once they make it into formal docs, people and models alike will keep getting misled.

4. New Risks Introduced by the Tools Themselves

Section titled “4. New Risks Introduced by the Tools Themselves”

Finally, consider the complexity of the tools. Adopting Spec-Kit or BMAD might require extra software and brand-new rituals. You can end up wrestling with tooling, pouring hours into crafting specs and calibrating workflows, while actual coding time shrinks. If you do not internalize the framework, you may copy templates mechanically, letting documentation drift away from the code. Like wielding a scalpel without training—the damage can be greater than doing nothing.

How to Avoid the Pile-Up: A Tiered Playbook for Documentation Tools

Section titled “How to Avoid the Pile-Up: A Tiered Playbook for Documentation Tools”

The pitfalls sound daunting, but do not lose heart. With a tiered usage strategy, documentation-driven programming can still pave a smoother road:

Use direct prompts for tiny tweaks

Section titled “Use direct prompts for tiny tweaks”

When you are only adjusting a small logic branch or styling tweak, there is no need to assemble a grand spec. In these cases, asking Claude Code, Codex, Gemini, GitHub Copilot, or Cursor directly is fastest. Explain the request in a sentence or two and let the AI offer a diff or snippet. You save time and stay focused. As veterans like to say: small asks do not need the full ceremony; if a quick chat solves it, skip the paperwork.

Reach for Spec-Kit or OpenSpec on medium-sized changes

Section titled “Reach for Spec-Kit or OpenSpec on medium-sized changes”

If you are adding a mid-sized feature—multiple steps, not massive—consider a lightweight tool such as Spec-Kit or OpenSpec. Spend ten minutes drafting an MVP-level spec that captures key requirements and edge cases, then hand it to the AI to refine the plan and tasks. Spec-Kit decomposes work item by item; OpenSpec minimizes context switching and handles the change in a few commands. The key is to keep the spec concise, covering only what is necessary. Data shows that compact specs are easier for AI to execute faithfully and more efficient overall. Once the feature ships, remember to update canonical docs or clean up temporary specs to prevent outdated information from piling up.

Deploy BMAD for major overhauls

Section titled “Deploy BMAD for major overhauls”

For large-scale refactors or brand-new complex subsystems—the real “surgery”—you need a rigorous approach. This is where a multi-agent framework like BMAD earns its keep: it orchestrates end-to-end planning, architecture, and task allocation. Think of it as hiring a virtual AI team to keep big projects on the rails. Of course, humans must stay in the loop—never blindly accept each AI output; review and adjust. Before spinning up BMAD, weigh the costs: it takes time and demands strong model comprehension. That investment can pay off for long-lived initiatives, but the documentation it produces—specs, user stories, and more—should evolve with the code. Do not toss it aside after one run; maintain it alongside the project or you will be rebuilding from scratch next time.

No matter the scale, remember the “documentation MVP” principle: the doc should cover only what is necessary and true right now. Do not chase completeness for its own sake. Many veterans delete temporary specs once a feature is done because the code itself is already clear. Instead of hoarding outdated notes that mislead people, keep a lean, accurate reference and update it as the product evolves. Documentation earns its value by reflecting reality; once it drifts, it is just dead weight.

Conclusion

Section titled “Conclusion”

Documentation-driven programming is a double-edged sword in the AI era. Used well, it makes development more efficient and orderly; misused, it spawns fresh spaghetti mountains. Ambitious young engineers should absolutely explore new tools, but not worship them blindly. Whether we are talking code or docs, the heart of the matter is understanding the requirement and keeping information current. Whenever you decide to write a spec or pick a tool, ask yourself: “Is this making the problem clearer—or just adding complexity?”

Here’s to every developer mastering these AI tools, producing docs that stay crisp and useful, and turning documentation into an asset—not a burden. We all want higher productivity, not a literal spaghetti mountain blocking the office! 💩

Build an Efficient, Lightweight, and Controlled Spec‑Driven Workflow with OpenSpec + Claude Code

Oct 17, 2025
Vibe Sparking

When teams start collaborating with AI coding assistants, clearly answering “what to build, how to build it, and how we know it’s done” becomes crucial. This guide walks you through a practical, reviewable, and traceable development workflow using OpenSpec (documentation‑driven development) together with Claude Code (an intelligent coding assistant).

Who this is for:

  • Teams that want specs before changes in multi‑person collaboration
  • People/teams that want AI assistants embedded into a standard workflow
  • Projects that require approval, archiving, or compliance control

Core Principles — Specs Are Truth

Section titled “Core Principles — Specs Are Truth”
  • Specs are truth: openspec/specs represents the currently delivered source of truth.
  • Propose before changes: any change impacting behavior/architecture/security/performance starts in changes/[id].
  • Three‑phase loop: Create change → Implement tasks → Archive and codify.
  • AI works to specs: Claude Code always uses OpenSpec as the authoritative context.

  • Generate the openspec/ directory structure
  • Establish unified collaboration instructions for Claude Code / Cursor / Codex, etc.
  • Later you can run openspec update to refresh the guidelines

Collaborating with Claude Code

Section titled “Collaborating with Claude Code”

OpenSpec provides project context and process standards; Claude Code executes the Read → Generate → Implement → Verify cycle. After initialization, start your Claude Code session with prompts like these:

  • Fill in project context:

Please read openspec/project.md. Based on the repo’s current state, help me fully document project goals, tech stack, conventions, external dependencies, constraints, and naming standards.

  • Create the first change proposal:

I want to add “[your feature]”. Please create an OpenSpec change proposal (changes/[id]/) including proposal.md, tasks.md, and the relevant specs delta under changes/[id]/specs/.

  • Explain/enforce the workflow:

Using openspec/AGENTS.md, explain the three‑phase workflow and strictly follow it in subsequent steps: propose and validate first, implement after approval, archive when complete.

Tip: After installation, Codex/Windsurf/Cursor, etc. will also pick up the same guidelines so multiple assistants collaborate consistently.

Three‑Phase Workflow (from openspec/AGENTS.md)

Section titled “Three‑Phase Workflow (from openspec/AGENTS.md)”
  1. Create Change
  • When: new capability, breaking change, architecture/security/performance strategy changes
  • Steps:
    • Pick a unique change-id (kebab‑case, verb‑prefixed: add-, update-, remove-, refactor-)
    • Generate proposal.md, tasks.md, and design.md if needed
    • Write specs delta using operation headers: ## ADDED|MODIFIED|REMOVED|RENAMED Requirements
    • Validate: openspec validate <id> --strict
    • Only implement after approval
  1. Implement Change
  • Steps:
    • Read in order: proposal.md → design.md (optional) → tasks.md
    • Implement strictly in tasks.md order; check off each item when complete
    • Update overall task status only when all are completed
  1. Archive Change
  • After deployment:
    • Move changes/[id] to changes/archive/YYYY-MM-DD-[id]/
    • If capability behavior changed, update openspec/specs/
    • Run strict validation: openspec validate --strict
    • Optional: openspec archive [change] [--yes|-y] to auto‑archive (tooling changes may use --skip-specs)

Key Spec File Format (Important)

Section titled “Key Spec File Format (Important)”
  • Each requirement uses ### Requirement: ...
  • Each requirement must have at least one scenario using a level‑4 heading:

Do not use bullets or bold text instead of a scenario heading (the parser will fail).

  • Three types of delta changes:
    • ## ADDED Requirements for new capabilities
    • ## MODIFIED Requirements for behavior changes (paste the full updated requirement block, not a diff)
    • ## REMOVED Requirements for removals (include rationale and migration)
    • ## RENAMED Requirements for renames only

From Zero to One: A Happy Path

Section titled “From Zero to One: A Happy Path”
  1. Explore current state
  1. Choose a change ID and scaffold
  1. Write the delta (example)
  1. Strict validation
  1. Approval → Implementation
  • After approval, have Claude Code implement items in tasks.md step by step
  • Check off one item when it’s done; proceed until all are - [x]
  1. Archive and codify

Best Practices with Claude Code

Section titled “Best Practices with Claude Code”
  • Always let Claude Code “read” first: openspec/project.md, relevant specs/[capability]/spec.md, and changes/[id]/proposal.md|tasks.md
  • Execute strictly against tasks.md as the single source of truth; check off as you go
  • Prioritize edits directly related to the proposed change; avoid unrelated refactors
  • When ambiguous, ask 1–2 clarifying questions before scaffolding
  • When referencing code locations, use src/module.ts:42 single‑line pointers
  • Built‑in validation: openspec validate [change] --strict; if issues occur, use --json to diagnose

CLI Cheat Sheet

  • --strict strict validation
  • --skip-specs archive only (skip specs update)
  • --yes|-y skip confirmation (good for CI)

Proposal vs. Direct Fix — When to Use Which?

Section titled “Proposal vs. Direct Fix — When to Use Which?”

Propose (recommended) when:

  • New features/capabilities
  • Breaking changes (API/Schema)
  • Architecture/pattern shifts
  • Performance/security strategy changes that affect behavior

Direct edits are fine when:

  • Bug fixes (restoring intended behavior)
  • Copy/format/comments
  • Non‑breaking dependency upgrades
  • Minor configuration tweaks
  • Tests that cover existing behavior

Common Pitfalls and Troubleshooting

Section titled “Common Pitfalls and Troubleshooting”
  • “Change must have at least one delta”
    • Check that changes/[id]/specs/ contains .md files
    • Ensure files include ## ADDED|MODIFIED|REMOVED operation headers
  • “Requirement must have at least one scenario”
    • Scenario headings must be #### Scenario: ... (level‑4 heading)
    • Do not use bullets or bold text
  • Delta parsing failures
    • Use openspec show [change] --json --deltas-only to debug
    • Compare with originals under openspec/specs/ and ensure MODIFIED is a full replacement block

Conclusion

Section titled “Conclusion”

Use OpenSpec to make Requirements → Design → Tasks → Changes explicit from day one and treat them as the project’s source of truth. Then leverage Claude Code for efficient execution and collaboration so AI delivers within clear boundaries. The benefits:

  • Specs first to reduce rework; approvals up front to lower risk
  • Task closure with traceable outputs; archivable changes for reusable knowledge
