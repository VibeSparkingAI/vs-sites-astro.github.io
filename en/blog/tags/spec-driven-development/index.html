Gemini CLI Conductor: Context-Driven Development That Plans Before It Codes

Dec 21, 2025
Vibe Sparking

Introduction

Section titled “Introduction”

Benjamin Franklin once said: “Failing to plan is planning to fail.”

This wisdom applies perfectly to AI-assisted coding. While tools like ChatGPT and Claude excel at answering questions, they often fall short when it comes to complex, multi-session projects where context gets lost between conversations.

Enter Conductor - a game-changing extension for Gemini CLI that introduces Context-Driven Development. Instead of treating AI interactions as fleeting chat logs, Conductor helps you create formal specifications and plans that live alongside your code as persistent Markdown files.

What is Conductor?

Section titled “What is Conductor?”

Conductor is a preview extension for Gemini CLI that transforms how you work with AI on code projects. Its core philosophy is simple but powerful:

“Treat context as a managed artifact alongside your code.”

Rather than depending on impermanent chat logs, Conductor helps you:

  • Create formal specifications before writing code
  • Maintain persistent context across sessions
  • Build implementation plans that guide the AI’s behavior
  • Keep your repository as the single source of truth

Key Features

Section titled “Key Features”

1. Plan Before Implementation

Section titled “1. Plan Before Implementation”

Conductor enforces a structured workflow: Context → Spec & Plan → Implement. This ensures you think through requirements before diving into code.

2. Persistent Context Files

Section titled “2. Persistent Context Files”

All project context is stored in Markdown files that:

  • Survive beyond individual chat sessions
  • Can be version-controlled with Git
  • Enable team collaboration with shared standards

3. Hierarchical Task Organization

Section titled “3. Hierarchical Task Organization”

Work is organized into a clear hierarchy:

  • Tracks - High-level work units (features, bug fixes)
  • Phases - Major stages within a track
  • Tasks - Individual implementation steps

4. Brownfield-Friendly

Section titled “4. Brownfield-Friendly”

Unlike many AI tools that struggle with existing codebases, Conductor excels at “brownfield” development - working with mature projects where architectural nuance matters.

Installation

Section titled “Installation”

Getting started with Conductor is straightforward:

The --auto-update flag ensures you always have the latest version.

Prerequisites:

  • Gemini CLI installed and configured
  • Git repository initialized
  • Node.js (for Gemini CLI)

Commands Reference

Section titled “Commands Reference”

Conductor provides five core commands:

CommandPurpose
/conductor:setupInitial project configuration (run once per project)
/conductor:newTrackBegin a new feature or bug fix
/conductor:implementExecute planned tasks
/conductor:statusView current progress
/conductor:revertUndo work logically (tracks/phases/tasks, not commits)

Pro Tip: Add a description to newTrack: /conductor:newTrack "Add user authentication"

The Context-Driven Workflow

Section titled “The Context-Driven Workflow”

Step 1: Setup Your Project

Section titled “Step 1: Setup Your Project”

Run /conductor:setup to establish your project’s foundational context:

This creates configuration files for:

  • Product Definition - Users, goals, features
  • Tech Stack - Languages, frameworks, databases
  • Workflow Preferences - TDD approach, commit strategies
  • Code Style Guides - Formatting, naming conventions

Step 2: Create a New Track

Kiro: From a Prompt to Production Code - A Guide to Spec-Driven Agentic IDE

Jul 15, 2025
Vibe Sparking

In the era of AI coding tools “racing to the sky,” Kiro from AWS brings a refreshing perspective: it’s not just “helping you write code,” but packs the complete software engineering workflow of requirements → design → implementation → automation into an IDE, truly moving from “Vibe Coding (demo-based coding)” to “Viable Code (production-ready products).” This article will quickly introduce you to Kiro’s core features, practical usage methods, and how it differs from other Agentic IDEs.

1 | What is Kiro?

Section titled “1 | What is Kiro?”
  • Positioning: An Agentic IDE based on VS Code kernel that automatically breaks down a “single prompt” into requirements specifications, system design, and task lists, driving LLM to complete implementation.
  • Background: Developed by a “very opinionated small team” within AWS, but intentionally weakly coupled with AWS cloud services—you can log in with GitHub/Google accounts. AWS Builder ID login is also supported.
  • Models Used: Preview version includes built-in Claude Sonnet 3.7 and 4.0, switchable in settings.

2 | Three Core Capabilities

Section titled “2 | Three Core Capabilities”

Based on practical usage in real work projects, I’ve analyzed and summarized Kiro’s core functions:

2.1 Spec-driven Development

Section titled “2.1 Spec-driven Development”

  1. Three Specs Auto-Generated:

    • requirements.md —— User stories and acceptance criteria written in EARS syntax
    • design.md —— Architecture, flowcharts, database design
    • tasks.md —— Executable implementation plan (including testing, accessibility, etc.)

  2. EARS Introduction: EARS (Easy Approach to Requirements Syntax) was proposed by the Rolls-Royce engineering team in 2009, structured as “While … when … the shall …”, reducing requirement ambiguity.

  3. Significance: Compared to simple Chat-pair coding, Spec-driven development ensures every line of code AI writes has traceable origins, facilitating team collaboration and later maintenance.

2.2 Agent Hooks (Event-Triggered Automation)

Section titled “2.2 Agent Hooks (Event-Triggered Automation)”

  • Working Method: Monitors events like “file save/create/delete/manual trigger” and calls backend LLM to execute preset tasks.

  • Common Scenarios:

    • Save React component → Auto-update test files
    • Modify API Endpoint → Sync README.md documentation
    • Before Git commit → Scan for sensitive information leaks

  • Benefits: Solidifies “good habits I should develop” into automated Hook execution, shared across teams, avoiding omissions.

2.3 From Vibe Coding to Viable Code

Section titled “2.3 From Vibe Coding to Viable Code”
  • Dual Modes: When starting projects, you can choose “Prompt mode” for quick experimentation or “Spec mode” for rigorous one-step implementation.
  • In-IDE Review: All AI changes are presented in diff format, supporting line-by-line approval or batch acceptance.
  • VS Code Ecosystem Compatibility: Open VSX plugins, themes, and shortcuts migrate almost unchanged.
