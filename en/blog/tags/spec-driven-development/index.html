Goodbye Vibe Coding: Spec-Driven Development Frameworks for Professional AI Software Development

Section titled “Goodbye Vibe Coding: Spec-Driven Development Frameworks for Professional AI Software Development”

The Problem with Vibe Coding

Section titled “The Problem with Vibe Coding”

“Vibe coding” - writing code “by feeling” and hoping the AI guesses your intentions - seemed efficient at first. You describe what you want, the AI generates code, you copy-paste. Fast, right?

But this apparent speed hides structural problems that inevitably emerge:

  • Context rot: As conversations lengthen, the AI “forgets” previous decisions and generates code inconsistent with what was done before
  • Emergent architecture (in the worst sense): Without an overall vision, code grows chaotically and quickly becomes impossible to maintain
  • Lack of traceability: Impossible to understand why certain technical decisions were made
  • Onboarding nightmare: New team members have no references to understand the system

Enter Spec-Driven Development

Section titled “Enter Spec-Driven Development”

Spec-Driven Development (SDD) is the professional answer. A methodology where specifications become first-class executable artifacts. You write the “spec” (structured documentation) first, then the AI generates code that respects that contract.

The spec becomes the source of truth for developers and AI agents, eliminating the ambiguities of vibe coding.

According to Thoughtworks, SDD is one of the most important engineering practices in recent years.

BMAD Method: The Enterprise Powerhouse

Section titled “BMAD Method: The Enterprise Powerhouse”

The BMAD Method (Build More, Architect Dreams) is the most complete SDD framework available today. With 21 specialized AI agents, over 50 guided workflows, and intelligence adaptive to project scale, BMAD simulates an entire agile team with specialized roles.

Key Features

Section titled “Key Features”
  • Multi-agent: Business Analyst, Product Manager, System Architect, Scrum Master, Developer, UX Designer, QA and others
  • Scale-adaptive: Automatically adapts from bug fixes to enterprise systems
  • Open source: 100% free, no paywall
  • Cross-domain: Works for software, creative writing, business strategy, wellness

GSD (Get Shit Done): The Anti-Bureaucracy Framework

Section titled “GSD (Get Shit Done): The Anti-Bureaucracy Framework”

Section titled “Philosophy: Hidden Complexity, Exposed Simplicity”

The complexity is in the system, not in your workflow.

  • Maximum 3 tasks per plan: Each plan runs in a fresh sub-agent with 200k clean tokens
  • Git-centric: Each task is a separate commit, revertable and traceable
  • Zero context rot: Combats context loss through isolated sub-agents

Ralph Loop: The Autonomous Iteration Method

Section titled “Ralph Loop: The Autonomous Iteration Method”

The Ralph Wiggum Loop (or simply Ralph Loop), created by Geoffrey Huntley, represents a paradigm shift. Instead of maintaining perfect context, it embraces fresh starts and lets git be the memory layer.

Core Concept: Git As Memory

Section titled “Core Concept: Git As Memory”

Ralph runs AI agents in automated loops until specifications are met. Each iteration spawns a new agent process with a clean context window. The agent reads specs from disk, takes a task, implements it, and exits.

Unlike exit-hook plugins that force the same session to continue (causing context overflow), this approach cleanly terminates and restarts between tasks.

Installation

Section titled “Installation”

How It Works

Section titled “How It Works”

Claude Code automatically: works on the task, attempts to exit, the stop hook blocks the exit, the stop hook reinserts the same prompt, repeats until completion.

Documented Results

Section titled “Documented Results”

DevOps Automation: A DevOps team implemented Ralph Loop in CI/CD. When tests fail, Ralph automatically receives the report, analyzes code, proposes fixes, and reruns tests.

Results after 3 months:

  • 67% of regression bugs resolved automatically
  • Average fix time reduced from 4 hours to 23 minutes
  • Zero manual interventions for 40% of failed builds

When SDD Makes Sense

Not all projects benefit from Spec-Driven Development. For a small bug fix, the workflow is “like using a sledgehammer to crack a nut.”

Use SDD When:

  • Complex features: Functionality requiring architecture
  • Distributed teams: Where documentation is critical
  • Strict compliance: Regulated sectors (finance, healthcare)

  • Initial overhead: Requires time investment for setup and training
  • Learning curve: Teams must learn new workflows

Tessl: Giving Coding Agents the Context They Need to Succeed

Introduction

Section titled “Introduction”

If you’ve used coding agents in real projects, you know the pattern. They get 80% of the way there, then stumble on API specifics—using deprecated methods, inventing parameters that don’t exist, or making assumptions that break in production. The code compiles, but it’s wrong.

Tessl is addressing this problem head-on with a deceptively simple idea: give agents structured, versioned context about the libraries they’re using. The results are compelling—up to 3.3x improvement in correct API usage across 300+ open-source packages.

The Problem: API Misuse

Section titled “The Problem: API Misuse”

Agents fail in predictable ways. When an LLM encounters a library it doesn’t know well (or knows an outdated version of), it doesn’t admit ignorance—it guesses. This creates brittle code that:

  • Uses non-existent or deprecated APIs
  • Ignores established patterns and idioms
  • Requires multiple rounds of fixes
  • Breaks when dependencies update

Tessl calls this “abstraction adherence”—whether an agent uses a library’s intended public interfaces instead of improvising. It’s a metric that matters far more than whether code simply compiles.

How Tessl Works

Section titled “How Tessl Works”

Tessl provides “tiles”—structured, versioned documentation specifically designed for AI agents to consume. Each tile covers a specific library and includes:

  • Correct API usage patterns
  • Version-specific information
  • Idiomatic examples
  • Common pitfalls

The key innovation is that tiles integrate directly into your project as dependencies. When you run tessl sync in your project, Tessl automatically:

  1. Scans your package.json or requirements.txt
  2. Matches dependencies to available tiles
  3. Ensures tiles match your installed versions
  4. Makes this context available via MCP to any agent

The Results

Section titled “The Results”

Across 300+ open-source packages on npm and PyPI, Tessl tiles demonstrated:

  • Up to 3.3x improvement in abstraction adherence for poorly-known packages
  • 1.2x improvement overall
  • 35% fewer integration errors
  • 1.4x faster completion on API-heavy tasks

The biggest gains come from libraries that agents typically struggle with—niche packages, newer versions, or specialized use cases not well-represented in training data.

Registry and Evals

Section titled “Registry and Evals”

Tessl maintains a public registry of tiles covering thousands of open-source libraries. What’s unique is that each tile includes published evals—you can see exactly how well agents perform with that specific library.

This transparency serves two purposes:

  1. Teams can track impact over time as models improve
  2. Library authors can contribute authoritative tiles for their projects

The registry is free to explore and integrates with major agent platforms via MCP.

Spec-Driven Development

Section titled “Spec-Driven Development”

Beyond library tiles, Tessl enables teams to encode their own conventions:

  • Directory layout rules
  • Naming conventions
  • Code style preferences
  • Architecture patterns

This means every AI contribution follows team standards without manual enforcement.

In the era of AI coding tools “racing to the sky,” Kiro from AWS brings a refreshing perspective: it’s not just “helping you write code,” but packs the complete software engineering workflow of requirements → design → implementation → automation into an IDE, truly moving from “Vibe Coding (demo-based coding)” to “Viable Code (production-ready products).” This article will quickly introduce you to Kiro’s core features, practical usage methods, and how it differs from other Agentic IDEs.

1 | What is Kiro?

Section titled “1 | What is Kiro?”
  • Positioning: An Agentic IDE based on VS Code kernel that automatically breaks down a “single prompt” into requirements specifications, system design, and task lists, driving LLM to complete implementation.
  • Background: Developed by a “very opinionated small team” within AWS, but intentionally weakly coupled with AWS cloud services—you can log in with GitHub/Google accounts. AWS Builder ID login is also supported.
  • Models Used: Preview version includes built-in Claude Sonnet 3.7 and 4.0, switchable in settings.

2 | Three Core Capabilities

Section titled “2 | Three Core Capabilities”

Based on practical usage in real work projects, I’ve analyzed and summarized Kiro’s core functions:

2.1 Spec-driven Development

Section titled “2.1 Spec-driven Development”

  1. Three Specs Auto-Generated:

    • requirements.md —— User stories and acceptance criteria written in EARS syntax
    • design.md —— Architecture, flowcharts, database design
    • tasks.md —— Executable implementation plan (including testing, accessibility, etc.)

  2. EARS Introduction: EARS (Easy Approach to Requirements Syntax) was proposed by the Rolls-Royce engineering team in 2009, structured as “While … when … the shall …”, reducing requirement ambiguity.

  3. Significance: Compared to simple Chat-pair coding, Spec-driven development ensures every line of code AI writes has traceable origins, facilitating team collaboration and later maintenance.

2.2 Agent Hooks (Event-Triggered Automation)

Section titled “2.2 Agent Hooks (Event-Triggered Automation)”

  • Working Method: Monitors events like “file save/create/delete/manual trigger” and calls backend LLM to execute preset tasks.

  • Common Scenarios:

    • Save React component → Auto-update test files
    • Modify API Endpoint → Sync README.md documentation
    • Before Git commit → Scan for sensitive information leaks

  • Benefits: Solidifies “good habits I should develop” into automated Hook execution, shared across teams, avoiding omissions.

2.3 From Vibe Coding to Viable Code

Section titled “2.3 From Vibe Coding to Viable Code”
  • Dual Modes: When starting projects, you can choose “Prompt mode” for quick experimentation or “Spec mode” for rigorous one-step implementation.
  • In-IDE Review: All AI changes are presented in diff format, supporting line-by-line approval or batch acceptance.
  • VS Code Ecosystem Compatibility: Open VSX plugins, themes, and shortcuts migrate almost unchanged.

3 | Quick Start Guide

Section titled “3 | Quick Start Guide”

Step 0: Installation

Section titled “Step 0: Installation”
  1. Visit kiro.dev to download the client for your platform, log in with GitHub/Google/AWS Builder ID.

Step 1: One Prompt → Generate Specs

Section titled “Step 1: One Prompt → Generate Specs”

After a few minutes, you’ll see three Specs in the .kiro/ directory; review and fine-tune before clicking “Approve”.

Step 2: Auto-Split Tasks and Execute

Section titled “Step 2: Auto-Split Tasks and Execute”
  • Task panel shows dependencies and progress bars, with options for Autopilot Mode (fully automatic) or Supervised Mode (manual approval).

Step 3: Configure Agent Hooks

Section titled “Step 3: Configure Agent Hooks”

Save to activate—Kiro will supplement tests or throw warnings in the background.

Step 4: Review & Merge

Section titled “Step 4: Review & Merge”
  • Use Diff view to check generated code; after approval, merge to local repository with one click, then push to remote.
