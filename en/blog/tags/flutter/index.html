Practical Review: Building HeartPetBond App with Claude Code + BMAD + GLM-4.7

Dec 31, 2025
Vibe Sparking

Core Insight: Can Zhipu GLM-4.7 replace the expensive Opus 4.5 in the Claude Code environment? How does its 200K context window perform within the document-driven BMAD workflow? This post reveals the answers through the real-world development of the HeartPetBond app.

1. Motivation: Why This Review?

Section titled “1. Motivation: Why This Review?”

In the AI-assisted programming realm, Claude Opus 4.5 has been the undisputed king, but its high cost (approx. $15/1M input, $75/1M output) deters high-frequency usage. DeepSeek V3, while affordable, occasionally hits bottlenecks with ultra-long contexts (>128K tokens).

I aimed to verify Zhipu GLM-4.7, a strong challenger to DeepSeek V3, by developing the HeartPetBond app to test:

  • Long Context Capability: Can the nominal 200K Tokens Context support an entire project’s documentation and codebase?
  • Output Length: Can the 128K Tokens Output generate complex Flutter pages in one go?
  • Stability & Hallucination: Does logic crumble during continuous development over several days?
  • Cost-Effectiveness: How much money can it save independent developers compared to Opus 4.5?

Context: When using DeepSeek V3 with Claude Code + BMAD, I encountered errors like API Error: 400 ... maximum context length is 131072 tokens... when the context exceeded 128K. GLM-4.7’s 200K context might be the solution.

With over 10 years of Flutter experience and previous success generating the MoneyMind App using Claude Code + BMAD, I set out to see if this new model could handle the challenge.

4. PoC Verification & Performance

Section titled “4. PoC Verification & Performance”

After switching to GLM-4.7, I was most surprised by its “throughput speed” and “depth of understanding.”

1. Development Speed

Section titled “1. Development Speed”

In just 2-3 days, GLM-4.7 helped me complete 5 core pages for HeartPetBond (PoC phase):

  • Welcome Page: User onboarding.
  • Ready State: State checks.
  • Home Page: Core recording flow.
  • Settings: Complex configuration management.

The code pass rate was extremely high, rarely missing brackets or breaking logic even in deep Flutter Widget trees.

2. Token Usage

Section titled “2. Token Usage”

The cost breakdown is impressive:

  • PoC Phase Total: Approx. 300 Million (300M) tokens.
  • Estimated MVP: For 15 pages and App Store submission, estimated 3 Billion (3B) tokens.

Despite this massive volume, GLM-4.7’s cost is only about 1/7th of Opus 4.5. A Pro subscription (~¥200/month) essentially covers the MVP development needs.

5. Wrap Up

Section titled “5. Wrap Up”

The verdict from this field test is clear: Building mobile apps with GLM-4.7 + Claude Code + BMAD is entirely feasible and highly cost-effective.

✅ Pros

Section titled “✅ Pros”
  1. High Cost-Performance: GLM-4.7 is a perfect “affordable alternative” to Opus 4.5. It significantly outperforms DeepSeek V3 in daily coding tasks while remaining wallet-friendly.
  2. Long Window Stability: The 200K Context Window is real. It maintained clarity without “catastrophic forgetting” even after loading mass amounts of BMAD documentation.
  3. Engineering Ready: Works seamlessly with Claude Code’s toolchain to handle complex refactoring and debugging.

⚠️ Cons

Section titled “⚠️ Cons”
  1. Reasoning Minor Flaws: In very complex business flow designs, GLM-4.7’s reasoning chain occasionally misses a step compared to the meticulous Opus 4.5. This is manageable via human Code Review and clearer prompts.
  2. Multimodal Gaps: Current API support for multimodal inputs (images/files) is limited, requiring an additional zai-mcp-server configuration.

🔮 Final Verdict

Section titled “🔮 Final Verdict”

If you are an independent developer looking for a smart AI pair programmer that won’t break the bank, Claude Code + GLM-4.7 + BMAD is the combination to beat right now.

iOS App Store Submission: Complete Guide from Development to Review

Section titled “iOS App Store Submission: Complete Guide from Development to Review”

Submitting an iOS app to the App Store can feel overwhelming for first-time developers. This guide walks you through the entire process, using our MoneyMind app as a real-world example. Whether you’re building with Flutter, SwiftUI, or UIKit, these steps apply to all iOS app submissions.

Overview

Section titled “Overview”

The App Store submission process involves several key stages:

  1. App Preparation - Icon, assets, and build configuration
  2. Certificates & Provisioning - Apple Distribution certificate setup
  3. App Store Connect Setup - App information, pricing, and screenshots
  4. Build Upload - Archive and upload via Xcode
  5. Review Submission - Final checklist and submission

Let’s dive into each step.

Step 1: App Icon Creation

Section titled “Step 1: App Icon Creation”

Your app icon is the first impression users get. Apple has strict requirements:

Icon Specifications

Section titled “Icon Specifications”
  • Size: 1024 × 1024 pixels
  • Format: PNG without transparency

Why this approach works now

Section titled “Why this approach works now”
  • Claude Code has plugins: slash commands, sub‑agents, MCP servers, and lifecycle hooks bundled into installable units. Teams can turn cross‑stack edits into a standard pipeline.
  • MCP (Model Context Protocol) is like giving AI a USB‑C port: plug IDEs/agents into your repos, databases, and tools with structured read/write and execution.
  • Remote MCP is supported: connect to cloud services with auth; in enterprise multi‑repo/multi‑environment setups it’s much simpler.

  • Web Admin, App Settings, and API change together
