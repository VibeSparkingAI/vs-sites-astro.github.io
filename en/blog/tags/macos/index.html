Introduction

Section titled “Introduction”

AI coding assistants have transformed how developers work, but they’ve also introduced a new category of risk: what happens when an AI agent executes a terminal command that affects your system beyond the project directory?

Google Antigravity’s new Terminal Sandboxing feature addresses this head-on with kernel-level isolation. When enabled, all terminal commands executed by the AI agent are restricted to a designated workspace, preventing accidental modifications to system files, user directories, or other sensitive locations.

This is a significant step forward for AI agent safety, particularly as these tools gain more autonomy and the ability to run complex multi-step operations.

Key Highlights

Section titled “Key Highlights”
  • Kernel-level isolation via macOS Seatbelt (sandbox-exec) for robust security boundaries
  • File system restrictions limiting writes to workspace directory and essential system locations only
  • Independent network control — allow or deny network access while maintaining file system restrictions
  • Per-command bypass option in “Request Review” mode for flexibility when needed
  • Automatic activation in Secure Mode with network access denied by default
  • Linux support coming soon (currently macOS-only)

Requirements

Enabling Sandboxing

Section titled “Enabling Sandboxing”

Sandboxing is disabled by default but can be toggled in Antigravity User Settings:

  1. Open Antigravity Settings
  2. Find the Terminal section
  3. Toggle “Enable Terminal Sandboxing” to turn sandboxing on or off
  4. Optionally toggle “Sandbox Allow Network” to control network access independently

When Secure Mode is enabled, sandboxing activates automatically with network access denied for maximum protection.

How Sandboxing Works

Section titled “How Sandboxing Works”

Under the Hood: Seatbelt

Section titled “Under the Hood: Seatbelt”

Google Antigravity leverages Apple’s Seatbelt sandboxing mechanism (sandbox-exec), which provides kernel-level isolation for processes. This is the same technology macOS uses to sandbox apps from the App Store.

When sandboxing is enabled:

  1. File System: The agent can only write to the designated workspace directory and read from essential system locations
  2. Network Access: Controlled independently via the “Sandbox Allow Network” toggle
  3. Violations: Any operation outside permitted boundaries is immediately blocked with a clear error message

Common Issues

Section titled “Common Issues”

Issue: Commands that previously worked now fail

Solution: Check if the command tries to access files outside the workspace. Common culprits:

  • Package managers installing to global directories
  • Build tools that cache outside the project
  • Configuration files in ~/.config or similar

Issue: Agent terminates unexpectedly

Solution: Some workflows may break when sandboxed. Review the agent’s error logs for sandbox-related messages and adjust the workspace scope or use per-command bypass.

A: No, sandboxing is currently opt-in, but this may change in future releases. Google Antigravity suggests that defaults may evolve as the feature matures.

A: Currently macOS-only due to the use of Apple’s Seatbelt mechanism. Linux support is explicitly listed as “coming soon.” Windows support details are not yet available.

Q: How does this compare to Docker or VMs?

A: Sandboxing is lighter-weight than full containerization. It’s a security boundary for terminal commands specifically, not a full environment isolation solution like Docker.
