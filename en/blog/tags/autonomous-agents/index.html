The Self-Driving Codebase: Background Agents and the Next Era of Software Delivery

Mar 2, 2026
Vibe Sparking

While most people slept, Cursor’s AI system generated roughly 8,000 commits. That was in a single eight-hour stretch. Over seven days, the system accumulated 10 million tool calls, running several hundred agents concurrently, with peak throughput of approximately 1,000 commits per hour. The humans in the room weren’t writing code. They were watching dashboards and calibrating direction.

That’s the self-driving codebase. It’s already here.

The Thing You’ve Been Calling a Bot Is Different Now

Section titled “The Thing You’ve Been Calling a Bot Is Different Now”

We’ve had automated code for years. Build servers. Linting pipelines. Dependency bots that open a PR when a library releases a new version. These are scripted systems. They execute predetermined steps.

Background agents are different in one meaningful way: they reason.

When a background agent gets triggered by a security disclosure, it doesn’t follow a flowchart. It reads the CVE, examines the affected code, generates a fix, runs tests against the fix, and creates a pull request explaining what it changed and why. The agent made decisions at each step. The humans review the output.

That gap — between executing a script and making decisions — is the entire story.

What Cursor Proved

Section titled “What Cursor Proved”

Cursor published their self-driving codebase research in early 2026. The numbers are worth sitting with.

A multi-agent system ran for seven days without stopping. Peak throughput: roughly 1,000 commits per hour. Total tool calls: 10 million. The project was a web browser — not a toy demo. Hundreds of agents operated concurrently on real engineering work.

The architecture that made this work wasn’t obvious. Earlier attempts failed.

A single powerful model (Opus 4.5) lost track of what it was doing. It “frequently stopped to proclaim success despite being far from it.” Self-coordination with equal-role agents sharing state files reduced 20 effective agents down to 1-3 due to lock contention. Agents avoided complex tasks to minimize conflicts. Neither approach scaled.

What worked was a hierarchy:

  • A root planner that owned the entire scope but never wrote code
  • Recursive subplanners that subdivided work while maintaining ownership
  • Independent workers that operated on isolated repository copies and delivered handoffs
  • Information flowing upward without global synchronization

Individual failures didn’t cascade. The system was anti-fragile.

The system also accepted a small constant error rate. Fixing errors quickly across distributed agents proved more efficient than preventing all errors upfront. When Agent A broke the build, Agent B noticed and fixed it. Ultimate CI/CD — where the CI can fix the bugs it finds.

Three prompting insights that shaped the architecture:

  1. Constraints outperform instructions. “No TODOs, no partial implementations” worked better than prescriptive guidance.
  2. Avoid checkbox mentality. High-level intent with agent autonomy outperformed granular task lists.
  3. Quantify scope. “Generate 20-100 tasks” conveyed intent better than “generate many tasks.”

The researchers noted: “Agents have immense engineering skill but will follow instructions to the end, good or bad.”

The Infrastructure Story

Section titled “The Infrastructure Story”

Running a few agents on a laptop is not background agents. Enterprise-grade background agents need three things.

Isolated compute. Each agent gets its own sandboxed environment that spins up on demand and tears down when the task is done. No shared state that can corrupt between jobs. No resource contention between parallel workstreams.

Event routing. Something has to decide when to trigger agents. A new pull request. A security advisory. A failing test in CI. A scheduled dependency audit. The event routing layer watches for these signals and dispatches agents with appropriate context and scope.

Governance. Permissions and blast radius controls — what can an agent touch, what can it not touch. Audit trails for every action an agent takes. Human review gates via pull requests before anything merges. Failure isolation so one agent’s broken state doesn’t propagate to others.

The key safety insight: agents are as safe as their surrounding infrastructure. A well-governed agent with limited blast radius is safer than an engineer with unconstrained production access. The pull request remains the critical human review point. That doesn’t change.

The Role That’s Changing

Section titled “The Role That’s Changing”

The phrase from background-agents.com captures it well: developers move “on the loop” rather than “in the loop.”

In the traditional model, a developer is in the loop — writing every line, making every decision keystroke by keystroke. In the background agent model, developers are on the loop — they set direction, review outputs, calibrate agent behavior, and handle judgment calls that agents can’t resolve.

The skills that shift in importance:

Prompt engineering and constraint design. What constraints produce reliable agent output? “No partial implementations” is more effective than “finish the task fully.” Learning what to say — and what to leave unsaid — is a skill.

System architecture for agent work. How do you structure a codebase so agents can operate independently? Small, self-contained modules. Clear interface contracts. Minimal cross-cutting dependencies. These aren’t new principles. They become more important when agents are the primary workers.

Review at the right depth. An agent’s technically-correct solution can introduce the subtle wrong — an approach that compiles, passes tests, and still misses the intent. Developers who catch this are valuable. Developers who rubber-stamp agent output are not.

Trust calibration. Knowing when to let the agent run and when to intervene. Reading the signals that indicate an agent is off-track. Setting the right amount of autonomy for the task at hand.

The coding itself becomes one skill among many, not the defining skill.

What This Looks Like in Practice

Section titled “What This Looks Like in Practice”

Background agent use cases that are deployed today:

Dependency updates. An agent audits all dependencies on a schedule, generates PRs for safe minor updates, reads changelogs for major versions and flags breaking changes for human review. Not a bot opening issues — an agent that understands the upgrade.

CVE remediation. Triggered by a security advisory, the agent locates affected code, generates a fix, tests it, and creates a PR with an explanation of the vulnerability and the fix. Time from disclosure to reviewed patch PR: minutes, not sprint cycles.

CI/CD migration. A legacy build pipeline needs to move to a new platform. An agent handles the translation file by file, testing each step. The engineer reviews the plan and approves the output. No one has to manually rewrite 200 pipeline steps.

Test coverage expansion. An agent analyzes coverage gaps, writes tests for uncovered paths, runs them to verify correctness, and submits for review. Targeted coverage, not random test generation.

Linting and standards enforcement. New coding standards rolled out across multiple repositories. An agent applies them consistently, without the coordination cost of distributing the work across a team.

None of these require the agent to make product decisions. They’re all well-scoped, reviewable, and bounded. That’s the pattern: background agents handle the work that’s clear enough to specify and important enough to do consistently.

The Uncomfortable Implication

Section titled “The Uncomfortable Implication”

The agents will follow instructions to the end, good or bad.

Cursor documented this directly. Vague directives led agents to pursue obscure features. Explicit specifications improved outcomes. When a single executor received too many simultaneous objectives — plan, explore, research, spawn tasks, review code, merge outputs — it exhibited dysfunction: sleeping randomly, refusing to delegate, claiming premature completion.

This means the human’s job is to be a good architect of agent work. Structure problems so agents can solve them. Provide constraints that produce good outcomes. Set clear scope boundaries. Review with enough domain knowledge to catch the wrong that passes all tests.

The developer who does this well is more valuable than the developer who types faster.

Getting Started

Section titled “Getting Started”

You don’t need a full background agent platform to start developing these skills.

Use Claude Code, Cursor, or GitHub Copilot Agent mode to run longer autonomous tasks and practice reviewing the outputs. Notice when the agent produces technically correct but contextually wrong results — that’s the review skill you’re developing.

Structure your next project with cleaner module boundaries and clearer interfaces. Not because agents are reading your code yet, but because it’s practice for the architecture principles that make agent work reliable.

When you write prompts for AI coding tools, focus on constraints over instructions. “Never add new dependencies without explicit approval” is a constraint. “Think carefully before adding dependencies” is not.

Pay attention to the background agent infrastructure landscape. The three pillars — isolated compute, event routing, governance — are where the enterprise tooling is consolidating.

The Point

Section titled “The Point”

The self-driving codebase isn’t a future state. For specific workloads — dependency maintenance, security patches, standards enforcement, test coverage — it’s operational today.

The infrastructure question is: isolated compute, event routing, governance. Without all three, you have a powerful laptop assistant, not background agents.

The role question is: on the loop, not in the loop. Review, calibrate, direct. Architect agent work rather than execute it.

The positioning question is: which skills matter in this structure? Constraint design. System architecture. Deep review. Trust calibration.

The developer who learns to work this way has a larger effective output than the developer who optimizes for typing speed. The transition is already underway.

  • Maintain ability to debug without AI

Vague Requirements:

  • AI needs clarity to be effective

Context Loss:

  • AI forgets details over long sessions
  • Maintain persistent context files
  • Re-establish context for new sessions

  • AI can write working but suboptimal code
  • Establish testing standards
  • Use code reviews even with AI
