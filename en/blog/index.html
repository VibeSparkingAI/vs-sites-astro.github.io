Section titled “/simplify and /batch: Two Claude Code Commands You’re Probably Not Using”

You just finished a feature. Forty-seven files touched, tests pass, CI is green. You open a PR and your teammate leaves 23 comments — duplicated logic in three places, a helper function that already exists in utils, and an N+1 query hiding inside a loop.

Claude Code ships with two commands that prevent this. Most people don’t know they exist.

/simplify: Your Pre-PR Quality Gate

Section titled “/simplify: Your Pre-PR Quality Gate”

/simplify runs three specialized review agents in parallel on your recently modified code:

AgentWhat It Catches
Code ReuseDuplicated logic, redundant patterns, existing helpers you missed
Code QualityReadability issues, naming problems, structural concerns
EfficiencyPerformance bottlenecks, unnecessary computation, wasted work

After analysis, it aggregates findings and applies fixes automatically.

How to Use It

Section titled “How to Use It”

Run it after finishing a feature, after fixing a bug, before opening a PR. It catches structural issues that linters miss — linters check style, /simplify checks architecture.

What Happens Under the Hood

Section titled “What Happens Under the Hood”

Three agents launch simultaneously. The Code Reuse agent scans for patterns that appear in multiple places and checks whether utility functions already handle that logic. The Code Quality agent reads your code the way a reviewer would — variable names, function decomposition, control flow clarity. The Efficiency agent profiles the logic for unnecessary allocations, redundant loops, and operations that could be batched.

They finish. Findings get merged. Fixes get applied. You review the diff.

That’s the whole workflow.

/batch: Parallel Codebase Migrations

Section titled “/batch: Parallel Codebase Migrations”

Now for the bigger problem. You need to migrate 40 files from one API format to another. Or replace every use of lodash with native equivalents. Or add type annotations to all untyped function parameters.

File by file, this takes days. /batch does it in parallel.

Three Phases

Section titled “Three Phases”

Phase 1 — Research and Plan. /batch explores the affected files, decomposes the work into independent units (typically 5 to 30 tasks), and determines how to verify each one.

Phase 2 — Spawn Workers. One background agent per unit, each running in an isolated git worktree. Every agent implements its task, runs /simplify on its changes, executes tests, commits, and opens a pull request.

Phase 3 — Track Progress. A status table updates as workers complete, showing which PRs are ready for review.

Usage

Section titled “Usage”

Each worker operates independently. If one fails, the others keep going. Every PR is independently mergeable.

When /batch Works Well

Section titled “When /batch Works Well”
  • Framework migrations
  • API contract updates across callers
  • Convention enforcement across a codebase
  • Dependency replacements
  • Test generation for untested modules

When It Doesn’t Fit

Section titled “When It Doesn’t Fit”
  • Tightly coupled changes that require coordination between units
  • Exploratory refactoring where you don’t yet know the target
  • Single-file modifications (just use /simplify)

Here’s the part that ties them together: every /batch worker automatically runs /simplify on its changes before committing. So each generated PR has already been through the three-agent review. Your human reviewers get cleaner PRs with fewer issues to flag.

Your Code Review Has a Blind Spot

Section titled “Your Code Review Has a Blind Spot”

Adversarial code review is good at catching design flaws, naming issues, and logic errors. But it has a gap. Branching paths with nested conditions - the ones where input values sit right at a boundary - tend to slip through. You know the kind: what happens when the array is empty? When the counter hits exactly zero? When the user passes null where a string was expected?

BMAD v6.0.4 and GDS v0.1.10 shipped together. Here’s what changed and why it matters.

Edge Case Hunter

Section titled “Edge Case Hunter”

The new Edge Case Hunter is a method-driven review tool. It traces every branching path in your code and checks boundary conditions systematically. It reports only unhandled gaps - no noise, no generic suggestions.

It works alongside adversarial review, not instead of it. Adversarial review catches structural and design problems. Edge Case Hunter catches execution path problems.

Here’s what happened during development: the first version forced the tool to always report findings. If it found nothing, it would halt and retry. The result? The AI fabricated issues. When you pressure an LLM to always produce output, it will produce output - even if there’s nothing to report.

The fix was to remove the zero-findings halt condition (#1797). Let the tool say “I traced all paths, nothing is missing.” That single change made the tool trustworthy. False positives destroy confidence faster than missed bugs.

Section titled “Test Architect Enhancements”

Test Architect got four upgrades:

Agent Teams - Coordinate multiple testing agents working in parallel. One agent handles unit tests, another runs integration tests, a third checks API contracts. They share context and avoid duplicating work.

Sub-agents - Complex test scenarios that would overwhelm a single agent can be decomposed. A parent agent defines the test plan, sub-agents execute individual test suites.

Configurable tool modes - Set the testing approach at input. Choose between strict TDD, exploratory testing, or regression-focused modes before the first test runs.

Pact.io contract testing - Service-to-service contracts built into the BMAD workflow. Define what your API promises, verify that consumers and providers agree, catch breaking changes before they reach production.

Understanding Statistical Process Control

Section titled “Understanding Statistical Process Control”

Historical Context

Section titled “Historical Context”

Statistical Process Control was pioneered by Walter A. Shewhart at Bell Laboratories in the 1920s. Facing the challenge of maintaining consistent product quality in mass manufacturing, Shewhart developed the control chart in 1924.

The fundamental insight: not all variation is created equal.

After World War II, W. Edwards Deming introduced SPC to Japanese industry, where it became a cornerstone of the manufacturing quality revolution. This eventually evolved into Six Sigma methodology, which has saved billions in manufacturing costs.

Core Principles

Section titled “Core Principles”

SPC distinguishes between two types of variation:

Common Cause Variation:

  • Inherent to the process
  • Results from multiple random factors (materials, environment, minor equipment fluctuations)
  • Predictable within statistical bounds
  • Cannot be eliminated without changing the process itself

Special Cause Variation:

  • Results from specific, identifiable sources (equipment failure, operator error, defective materials)
  • Unpredictable and sporadic
  • Should be investigated and corrected immediately

Control Charts: The Primary Tool

Section titled “Control Charts: The Primary Tool”

A control chart visualizes process behavior over time with three key elements:

  1. Center Line: Process average or target value
  2. Upper Control Limit (UCL): Typically mean + 3 standard deviations
  3. Lower Control Limit (LCL): Typically mean - 3 standard deviations

When data points fall within control limits and display random variation, the process is considered “in control.” Specific patterns signal trouble:

  • Points outside control limits: Immediate investigation required
  • Seven consecutive points above/below center line: Trend detected
  • Seven consecutive increasing/decreasing points: Systematic shift
  • Two out of three points in outer third zone: Early warning

Why SPC Works

Section titled “Why SPC Works”

SPC provides several advantages:

  1. Proactive vs. Reactive: Detects problems before they produce defects
  2. Data-Driven: Replaces subjective judgment with objective metrics
  3. Continuous Improvement: Systematic approach to reducing variation
  4. Cost Reduction: Prevention is cheaper than correction

“When a measure becomes a target, it ceases to be a good measure.”

  • Pass tests while missing business logic requirements
  • Optimize for measurable aspects while ignoring unmeasurable quality factors

Mitigation:

  • Include human judgment in quality assessment
  • Rotate and update metrics periodically
  • Focus on process improvement, not individual performance evaluation

Approaches:

  • Segmented Control Charts: Separate charts for different model versions

Section titled “2. Skill Hot-Reload: Update Without Restart”

Agents accumulate context over long sessions. Killing them to deploy a fix means losing all that context.

3. Session Teleportation: Move Your Agent

Section titled “3. Session Teleportation: Move Your Agent”

Sometimes you need to move a running session to another machine. Maybe you’re switching laptops. Maybe the session started on a server and you want it local.

/teleport session-id serializes the full session state—conversation history, tool states, everything—and reconstructs it elsewhere. The agent picks up exactly where it left off.

This works across machines. I’ve teleported sessions from my MacBook to a Linux server mid-conversation.

4. Dynamic MCP Loading: Solve Context Pollution

Section titled “4. Dynamic MCP Loading: Solve Context Pollution”

Here’s the problem: loading all 50 MCP servers at startup dumps thousands of tool descriptions into your context window. Most calls only use 2-3 tools. You’re paying tokens for tools you never touch.

The fix: ENABLE_TOOL_SEARCH=true

Instead of loading everything upfront, the agent searches for relevant tools on demand. Nick measured an 85% reduction in token overhead with this approach.

The tradeoff? Slightly higher latency on first tool use while it searches. Worth it for most workloads.

You get specialization without manual routing.
