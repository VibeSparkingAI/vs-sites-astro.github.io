Harness Engineering: Your Job Isn’t Writing Code Anymore

Section titled “Harness Engineering: Your Job Isn’t Writing Code Anymore”

Your best engineer spent the morning rewriting an error message. Not fixing a bug. Rewriting a linter output so that an AI agent could understand what went wrong and fix it on its own.

This is what engineering looks like in 2026.

What OpenAI Actually Did

Section titled “What OpenAI Actually Did”

OpenAI’s Harness team built and shipped an internal beta product over five months. Zero lines of manually-written code. Every line was generated by Codex agents. The codebase grew past one million lines. Development time: roughly one-tenth of what a human team would have needed.

They published what they learned. The title: “Harness Engineering: Leveraging Codex in an Agent-First World.” The word choice is deliberate. Not AI engineering. Not prompt engineering. Harness engineering — building the harness that keeps agents productive and on track.

Martin Fowler picked it up and wrote his own analysis. His take: the article buries the lead in its title. “Harness” appears once in the body text, but the concept is the whole point. The tooling, the practices, the constraints that make agent-generated code maintainable at scale.

Eight lessons came out of this work. All of them point in the same direction.

1. Engineers Become Environment Designers

Section titled “1. Engineers Become Environment Designers”

The job shifted. Instead of writing code, engineers design systems, specify intent, and build feedback loops. When an agent produces bad output, the response is never “try harder.” It’s “what capability or context is missing from the agent’s environment?”

Human time and attention became the only scarce resource. Everything else — code production, review capacity, pattern enforcement — can be automated or mechanized.

2. Give Agents a Map, Not an Encyclopedia

Section titled “2. Give Agents a Map, Not an Encyclopedia”

The first instinct was to give agents everything. A massive AGENTS.md packed with coding standards, architecture decisions, project context. It backfired. Too much context crowded out the actual task.

The fix: a roughly 100-line AGENTS.md that works as a table of contents, pointing to a structured docs/ directory. Progressive disclosure. The same pattern that makes good UIs work — show what’s needed at the current level, link to depth when it’s relevant.

3. If Agents Can’t See It, It Doesn’t Exist

Section titled “3. If Agents Can’t See It, It Doesn’t Exist”

Slack threads where someone explained the payment flow. A Google Doc with the deployment runbook. Tribal knowledge your senior engineer carries around. None of it exists to an agent.

The team had to encode everything into the repository as versioned, discoverable artifacts. Same practice we’ve recommended for onboarding human engineers for years — write things down, put them where people can find them. Agents forced the issue. A nice-to-have became a hard requirement.

4. Enforce Architecture Mechanically

Section titled “4. Enforce Architecture Mechanically”

Custom linters with remediation instructions baked directly into error messages. When an agent violates an architectural boundary, the error message explains what the boundary is, why it exists, and how to fix the violation.

The error message becomes the teaching moment. Enforce boundaries centrally, allow autonomy locally. The agent doesn’t need to understand the entire architecture. It needs clear feedback when it steps out of bounds.

Martin Fowler noted that OpenAI monitored architecture “not only by the LLM-based agents, but also deterministic custom linters and structural tests.” Both kinds of enforcement, working together.

5. Boring Technology Wins

Section titled “5. Boring Technology Wins”

Agents perform better with composable, stable APIs that have strong representation in training data. The latest framework with sparse documentation and novel patterns? Agents struggle with it.

OpenAI’s team sometimes found it cheaper to reimplement a subset of functionality than to fight opaque upstream behavior from a trendy library. Boring technology — the kind with years of Stack Overflow answers and well-trodden paths — gives agents the most reliable foundation.

Fowler suggested this could push organizations toward fewer tech stacks, chosen for “AI-friendliness” rather than developer preference.

6. Entropy Management as Garbage Collection

Section titled “6. Entropy Management as Garbage Collection”

Agents are pattern replicators. They find an existing pattern in the codebase and reproduce it. Useful when patterns are good. A problem when they aren’t.

The solution: recurring background agents that continuously scan the codebase for deviations from standards and auto-fix. Periodic agents also catch documentation inconsistencies and architectural violations. Code hygiene isn’t a quarterly cleanup — it runs continuously.

7. Throughput Changes Merge Philosophy

Section titled “7. Throughput Changes Merge Philosophy”

When agents produce code fast, traditional merge gates become the bottleneck. OpenAI shifted to minimal blocking merge gates and short-lived PRs. Test flakes get follow-up runs, not blocking.

The philosophy: corrections are cheap, waiting is expensive. Optimize for throughput. Fix problems as they surface. This only works with strong automated guardrails — which circles back to the linter strategy.

8. Agent-to-Agent Review

Section titled “8. Agent-to-Agent Review”

OpenAI pushed almost all code review to agent-to-agent loops. Codex reviews its own changes, requests additional agent reviews, iterates until the review passes. Humans get escalated only when genuine judgment is required — novel architectural decisions, security-sensitive changes, product direction.

The math: if agents produce ten times the code volume, human reviewers become the bottleneck. Agent-to-agent review handles mechanical checks. Human attention gets reserved for decisions that actually need it.

What’s Missing

Section titled “What’s Missing”

Fowler raised an important gap. The OpenAI write-up emphasizes internal code quality and maintainability but says little about verification of functionality and behavior. The harness constrains how code is written and organized. It doesn’t yet validate that the code does what users need.

He also flagged a real-world concern: retrofitting harnesses onto existing, non-standardized codebases may prove economically unfeasible. This could create a divide between pre-AI and post-AI applications in terms of maintenance cost.

What This Means for You

Section titled “What This Means for You”

The engineers who thrive in the agent-first era won’t be the fastest coders. They’ll be the best environment designers — the ones who build systems where agents succeed on the first try because every piece of context, every guardrail, every feedback loop is already in place.

Start with your repo:

  • Audit your AGENTS.md. Is it a map or an encyclopedia?
  • Find the tribal knowledge that lives outside your codebase. Move it in.
  • Look at your linter error messages. Could an agent understand them and self-correct?
  • Identify the “exciting” technology in your stack. Is it helping or hurting agent performance?
  • Check if your merge gates are protecting quality or just slowing throughput.

Your agents are only as good as the world you build for them. That error message your engineer rewrote this morning? That’s the new engineering.

Somewhere along the way, you became an unpaid QA engineer for a machine. Nobody told you that was the job description.

The contract nobody signed

Section titled “The contract nobody signed”

Every AI coding tool on the market follows the same contract: you describe what you want, the AI generates a first draft, and you take it from there.

“Take it from there” means:

  • Reviewing for correctness
  • Checking for security holes
  • Re-prompting when the output misses the mark
  • Re-teaching conventions it forgot overnight
  • Repeating the cycle 3-5 times per task

This is the workflow that billions of dollars in AI investment has produced: the developer as babysitter.

The missing piece isn’t intelligence

Section titled “The missing piece isn’t intelligence”

The problem isn’t that the models are stupid. GPT-4, Claude, Gemini — they can all write serviceable code. The problem is structural. These tools lack executive function.

They generate and stop. They don’t plan before acting. They don’t evaluate their own output. They don’t iterate when something falls short. They don’t remember what they learned yesterday.

You fill that gap. Every review, every re-prompt, every “no, that’s not what I meant” — that’s executive function you’re donating for free.

OpenKoi: Executive Function as a Service

Section titled “OpenKoi: Executive Function as a Service”

OpenKoi is an open-source project that names this problem and builds a solution around it. It calls the concept “Executive Function as a Service” (EFaaS).

The project is a self-iterating AI agent system written in Rust. Ships as a single static binary, zero runtime dependencies, works with any model provider. But the engineering isn’t the interesting part. The cognitive architecture is.

How it works

Section titled “How it works”

  1. Sovereign — A directive shapes the approach (“Direct, test-driven, security-conscious”)
  2. Parliament — Five internal agencies deliberate: Guardian (safety), Economist (cost), Empath (user impact), Scholar (correctness), Strategist (long-term)

The verb matters. Other tools say “run.” OpenKoi says “think.”

Every re-prompt is a plan the tool didn’t make. Every manual review is an evaluation it skipped. Every “I already told you this” is a learning it lost. The aggregate cost is real — not in dollars, but in the cognitive overhead of babysitting a system that’s smart enough to babysit itself.

OpenKoi’s answer is architectural: give the agent a parliament, a world model, a memory, and quality rubrics, then let it iterate. You review the output that matters. The agent handles the mechanical loops that used to eat your afternoons.

Stop babysitting. Start directing.

Scrapling: The Web Scraping Framework That Survives Website Redesigns

Section titled “Scrapling: The Web Scraping Framework That Survives Website Redesigns”

You build a scraper. You tune the CSS selectors. Data flows. Two weeks later, the target site tweaks its HTML. Your selectors break. Your pipeline goes silent. You rewrite. Again.

If you’ve scraped the web for any length of time, you know this loop. Scrapling was built to end it.

What makes Scrapling different

Section titled “What makes Scrapling different”

Scrapling is a Python web scraping framework by Karim Shoair. It covers the full spectrum — HTTP requests, headless browsers, stealth fetching, spider-based crawling, proxy rotation, a CLI, even an MCP server for AI integration. But the feature that sets it apart is adaptive element tracking.

Here’s the idea: when you scrape a page, Scrapling can store a structural fingerprint of the elements you select. Later, if the website changes its class names, moves elements around, or swaps out its CSS framework entirely, you pass adaptive=True and the parser relocates those elements by matching against the stored fingerprint. Not by guessing. By comparing structural similarity.

This is the shift. Traditional scraping treats website changes as your problem. Scrapling treats them as a solvable engineering challenge.

The Sameness Problem

Section titled “The Sameness Problem”

Every major LLM trained on the same web data. The same Bootstrap layouts, the same Tailwind templates, the same UI kit screenshots. Without specific guidance, they converge on the statistical average of “website.”

Section titled “The Vocabulary Gap”

Harness Engineering: OpenAI’s 8 Rules for an Agent-First World

Section titled “Harness Engineering: OpenAI’s 8 Rules for an Agent-First World”

The 800-Line File That Had to Die

Section titled “The 800-Line File That Had to Die”

A team at OpenAI killed their 800-line AGENTS.md file. Not because it was wrong — because it was too right. The agents couldn’t find what they needed in a wall of context.

They replaced it with a 100-line table of contents pointing to structured files in a docs/ directory. That single decision captures what’s happening to software engineering right now.

The engineers getting the most out of AI agents aren’t writing better prompts or picking smarter models. They’re redesigning the engineering environment itself — repo structure, linter rules, merge gates, documentation layout — so agents can operate independently.

OpenAI calls this “harness engineering.” Muratcan Koylan distilled the key patterns from OpenAI’s published practices, and what emerges looks nothing like how most teams work today.

The 8 Principles

Section titled “The 8 Principles”

1. Engineers Become Environment Designers

Section titled “1. Engineers Become Environment Designers”

The job shifted. It’s no longer about writing code — it’s about designing systems, specifying intent, and building feedback loops.

When something fails, the question isn’t “try harder.” It’s “what capability is missing?”

Human time and attention are the only scarce resources left. Everything else can be delegated — if the environment is right.

2. Give Agents a Map, Not an Encyclopedia

Section titled “2. Give Agents a Map, Not an Encyclopedia”

Progressive disclosure beats comprehensive documentation. Every time.

A ~100-line AGENTS.md as a table of contents, pointing to a structured docs/ directory, outperforms a massive all-in-one reference. Too much context crowds out the actual task.

The same pattern that works for onboarding human engineers works for agents. Maybe even better, because agents follow what they can see and ignore what they can’t.

3. If Agents Can’t See It, It Doesn’t Exist

Section titled “3. If Agents Can’t See It, It Doesn’t Exist”

Slack threads. Google Docs. Hallway conversations. Tribal knowledge passed down through pair programming sessions.

All invisible to agents.

If knowledge isn’t encoded as versioned, discoverable artifacts in the repo, it effectively doesn’t exist for your fastest workers. Think of it as onboarding practices — but for non-human engineers who will never ask a clarifying question over coffee.

4. Enforce Architecture Mechanically, Not Through Instructions

Section titled “4. Enforce Architecture Mechanically, Not Through Instructions”

This one is sharp.

Custom linters with remediation instructions baked into error messages. The error output itself becomes agent context. An agent hits a boundary violation, reads the error message, and knows exactly how to fix it — without consulting any documentation.

Strict layered architecture with validated dependency directions. The principle: enforce boundaries centrally, allow autonomy locally.

Telling agents “don’t do X” in a markdown file is a suggestion. Making X trigger a build failure is a rule.

5. Boring Technology Wins

Section titled “5. Boring Technology Wins”

Composable, stable APIs with strong training-set representation outperform cutting-edge libraries that agents struggle with.

Agents have been trained on mountains of React, Express, and PostgreSQL code. They’ve seen far less of your team’s custom framework or that niche library with 200 GitHub stars.

Sometimes it’s cheaper to reimplement a subset of functionality than fight opaque upstream behavior. Pick what agents already know.

6. Entropy Management Is Garbage Collection

Section titled “6. Entropy Management Is Garbage Collection”

Agents replicate existing patterns. All of them. Including the bad ones.

That one hacky workaround from six months ago? An agent will find it, treat it as the canonical pattern, and reproduce it across 40 files before anyone notices.

Without active maintenance, code quality degrades fast. The solution: recurring background agents that scan for deviations from established patterns and auto-fix. Continuous garbage collection for your codebase.

7. Throughput Changes Merge Philosophy

Section titled “7. Throughput Changes Merge Philosophy”

Minimal blocking merge gates. Short-lived PRs. Test flakes addressed with follow-up runs, not by blocking merges.

The principle: corrections are cheap, waiting is expensive.

When agents can generate and test changes faster than humans can review them, the bottleneck moves from code production to code approval. Redesign the pipeline for throughput, not for gatekeeping.

8. Agent-to-Agent Code Review

Section titled “8. Agent-to-Agent Code Review”

This is where it gets interesting.

OpenAI pushed almost all code review to agent-to-agent loops. Codex reviews its own changes, requests additional agent reviews, iterates until satisfied. Humans are escalated to only when judgment is required.

Read that again. The code review loop — something most teams consider a core human activity — has been restructured. Not eliminated. The human role shifted from reviewing every change to designing review criteria and handling edge cases.

The Pattern Underneath

Section titled “The Pattern Underneath”

Across all eight principles, the same thing happened. Nothing disappeared. Everything shifted.

The engineer didn’t become irrelevant. The engineer’s job description changed. From code author to environment designer. From reviewer to review-system designer. From architect to architecture-enforcement designer.

The code still gets written. The architecture still gets designed. The reviews still happen. But the human’s primary output shifted from code artifacts to environment artifacts — documentation structure, linter rules, merge policies, feedback loops.

Where to Start

Section titled “Where to Start”

The question isn’t whether this shift will reach your team. It’s whether you’ll design for it or react to it.

Start with one audit: can a new agent, with zero context beyond what’s in the repo, find what it needs, follow the architectural rules, and ship working code?

If not, you have your backlog.

A few concrete first moves:

  • Audit your AGENTS.md (or equivalent). Is it a wall of text or a navigable map? Trim it to a table of contents pointing to structured docs.
  • Check for invisible knowledge. What do new hires learn in their first week that isn’t written down? That’s your agent knowledge gap too.
  • Add remediation hints to your linter rules. Every error message is a teaching opportunity for agents.
  • Pick one pattern violation that keeps recurring and write a background cleanup script. Automate the janitor.
  • Try agent-to-agent review on a low-risk PR. Set up a second agent session to review the first agent’s output before a human looks at it.

Section titled “The Real Bottleneck”

Most developers interact with AI coding agents the same way: type a rough description of what they want, hit enter, and hope for the best. The Two Roles

Section titled “The Two Roles”

The workflow separates responsibilities cleanly:

You (requirement owner): Provide verifiable requirements. Make key decisions. Final approval on merge and release.

Step 0: Structured Requirement Input

Section titled “Step 0: Structured Requirement Input”

  • One-liner: “I want [X] so that [Y]”

  • Constraints: Tech stack, APIs, things that must not change, permissions, launch window
  • Minimum Viable Acceptance (MVA): Checkboxes written as testable statements
    • “When [input], the system should [behavior]”
    • “When error/unauthorized/not-found, system should [fallback]”
    • “Must not affect [list 3-5 regression points]”
  • Out of scope: What you are explicitly not building

This takes five minutes. It saves hours.

Step 1: AI Produces the Spec

Section titled “Step 1: AI Produces the Spec”

The AI generates a one-page executable Spec covering:

  1. Goals and non-goals
  2. User/caller behavior (main flow + key boundaries)
  3. Interface/data contracts (params, responses, error codes, idempotency, compatibility)
  4. Change scope (modules, files, blast radius)
  5. Risks and dependencies (external services, migrations, permissions, security, performance)
  6. Acceptance criteria (P0 must-haves + optional items)
  7. Rollback/degradation strategy (at minimum one actionable sentence)
  8. Deliverables checklist (code, tests, docs, PR description)

This is the only “strong confirmation point” in the entire workflow.

Step 2: You Confirm the Spec

Section titled “Step 2: You Confirm the Spec”

Your response is binary:

  • “Spec OK, go.” or
  • “Spec needs changes: A, B, C”

Once you approve the Spec, you don’t need to confirm every step afterward. The AI executes autonomously unless it hits an exception gate (more on that below).

This single gate eliminates most rework. The AI and you are now aligned on what “done” means before a single line of code is written.

Step 3: AI Breaks Down Tasks

Section titled “Step 3: AI Breaks Down Tasks”

The AI produces 3-8 tasks, each with:

  • Task name (starts with a verb)
  • Inputs and dependencies
  • Files/modules to change
  • Deliverables (code, config, migrations, docs)
  • Self-test method (command, test case, expected result)
  • Definition of Done (lint, test, type-check requirements)
  • Rollback point (feature flag, config, compatibility path)

Step 4: Incremental Execution with Self-Checks

Section titled “Step 4: Incremental Execution with Self-Checks”

No “big bang” implementations. For each task, the AI outputs:

  1. What changed (summary + key diff explanation)
  2. Self-check results (lint, unit tests, logs, key test cases)
  3. Risk flags (newly discovered boundaries or dependencies)
  4. Next step

This rhythm catches problems early and produces reviewable evidence at every step.

Step 5: Exception Gates

Section titled “Step 5: Exception Gates”

During execution, the AI must stop and ask for your decision if any of these occur:

  • Requirement conflict or ambiguous interpretation
  • New dependency, permission, or data migration needed
  • External contract change (API, fields, behavior)
  • Significant risk escalation (security, performance, rollback infeasibility)
  • Scope creep beyond out-of-scope boundaries

After you decide, the AI updates only the affected Spec sections and continues.

These gates prevent the AI from silently going off-track. They’re the safety net that makes autonomous execution trustworthy.

Step 6: Layered Acceptance

Section titled “Step 6: Layered Acceptance”

The AI generates and executes a verification checklist with evidence.

P0 (always do):

  • Functional: main path + key boundaries (null, invalid, unauthorized, not-found)
  • Regression: the 3-5 points you specified in requirements
  • Automation: format/lint + critical unit tests or contract tests
  • Error handling: no swallowed errors, traceable logs, no sensitive data leaks
  • Compatibility: old behavior preserved (or explicit breaking change with migration guide)
  • Rollback: one-click revert available (flag, config, rollback steps)

P1 (recommended based on importance):

  • Performance: critical path latency, query count, large dataset behavior
  • Security: input validation, injection risks, permission boundary audit
  • Observability: metrics and instrumentation for production debugging

P2 (optional):

  • Extended test matrix (multi-platform, load testing, chaos)
  • Structural refactoring (only if obvious debt)

Step 7: Testing

Section titled “Step 7: Testing”

Layered testing plan:

  • Local/unit: always
  • Integration: when interfaces, databases, or external dependencies are involved
  • Staging/canary: when real traffic or high-risk changes are involved

Step 8: Release with Guardrails

Section titled “Step 8: Release with Guardrails”
  • Feature branch development, never push directly to main
  • Small commits (1-2 per task)
  • PR includes: change summary, test evidence, risk points, monitoring notes, rollback steps, migration scripts if applicable
  • Merge only when CI green and P0 passes
  • Canary deployment for risky changes: flag on, small traffic, then full rollout

Step 9: Post-Launch Verification

Section titled “Step 9: Post-Launch Verification”

After deployment, verify against the Spec’s acceptance criteria:

  • Smoke test critical paths
  • Monitor logs and metrics for anomalies (error rates, latency, key indicators)
  • If anomalies: execute rollback plan, document root cause

Step 10: Definition of Done

Section titled “Step 10: Definition of Done”

A requirement is “done” when all of these are true:

  • All P0 acceptance criteria pass
  • PR merged and deployed
  • Rollback plan verified (exercised or at minimum executable)
  • Post-launch verification complete
  • Records linked: PR, test report, acceptance report

Why This Works

Section titled “Why This Works”

The workflow succeeds because it solves the actual problem: not AI capability, but the absence of structure around AI execution.

One strong gate (Spec) aligns intent. Small steps catch drift. Exception gates prevent silent failures. Layered acceptance makes verification practical rather than exhaustive. Rollback readiness makes shipping safe.

The AI becomes a disciplined executor rather than a chaotic code generator. You stay in the driver’s seat without micromanaging every line.

Your AI Agent Skills Are Broken. Here’s the Fix.

Section titled “Your AI Agent Skills Are Broken. Here’s the Fix.”

You’ve been writing skills wrong.

Not a little wrong. Fundamentally wrong. Most developers approach AI agent skills like documentation for a junior developer — dump some context, add headers, throw in examples, hope it works. The result: bloated context windows, misfiring triggers, inconsistent output.

Minko Gechev’s skills-best-practices repo treats this problem with the rigor it deserves. The core insight is simple and sharp: skills aren’t documentation. They’re programs. The runtime is an LLM.

The anatomy of a skill

Section titled “The anatomy of a skill”

Every skill follows one structure:

No README. No changelog. No library code. If a file doesn’t serve the agent at runtime, it’s waste.

SKILL.md is the brain — high-level procedures and navigation. References hold API docs and cheatsheets. Scripts handle fragile, repetitive operations where variation is a bug. Assets store templates the agent copies. That’s the entire surface area.

Discovery is where skills die

Section titled “Discovery is where skills die”

The agent sees exactly two things before deciding to load your skill: name and description. That’s 1,024 characters to determine whether your skill exists or doesn’t.

Bad description: “React skills.”

Good description: “Creates and builds React components using Tailwind CSS. Use when the user wants to update component styles or UI logic. Don’t use it for Vue, Svelte, or vanilla CSS projects.”

Notice the negative triggers. You have to tell the agent when NOT to fire. Without them, your Angular migration skill will activate on React projects. Your Python testing skill will trigger on Go code.

The name field has strict rules too: 1-64 characters, lowercase letters, numbers, hyphens only, no consecutive hyphens, and it must match the parent directory name. angular-testing lives in angular-testing/SKILL.md. No exceptions.

Context window is memory. Manage it.

Section titled “Context window is memory. Manage it.”

The guide calls it “Just-in-Time Loading.” Don’t pull references into context until the agent needs them. Keep SKILL.md lean: procedures and navigation only. Move schemas, API docs, and cheatsheets to references/. Move templates to assets/. Tell the agent explicitly when to read each file: “See references/auth-flow.md for specific error codes.”

Flat subdirectories only. One level deep. references/schema.md works. references/db/v1/schema.md doesn’t. This keeps the mental model simple for both you and the agent.

This is memory management for a runtime that charges per token.

Write for machines, not humans

Section titled “Write for machines, not humans”

Three rules that change how you author skill instructions:

Step-by-step numbering. Define workflows as strict chronological sequences. Map decision trees explicitly: “Step 2: If you need source maps run ng build --source-map. Otherwise, skip to Step 3.” No ambiguity. No room for creative interpretation.

Concrete templates over prose. Agents pattern-match well. A JSON template in assets/ beats paragraphs describing what the output should look like. Don’t describe — show.

Third-person imperative. “Extract the text…” not “You should extract…” or “I will extract…” This framing produces more consistent agent behavior.

Pick one term per concept and stick with it. Use domain-native vocabulary. In Angular, say “template” — not “html”, “markup”, or “view.” Consistency reduces the chance of the agent conflating similar concepts.

Offload fragile work to scripts

Section titled “Offload fragile work to scripts”

Don’t ask an LLM to write complex parsing logic from scratch every time a skill runs. Give it a tested Python or Bash script in the scripts/ directory. Design scripts as tiny CLIs.

The critical detail: make scripts return descriptive, human-readable error messages on stdout/stderr. When a script fails with “Error: Unable to resolve CommonJS dependency ‘lodash’ — try adding it to the Vite optimizeDeps.include array,” the agent can self-correct. When it fails with exit code 1 and no message, the agent is stuck.

Don’t create files that waste tokens

Section titled “Don’t create files that waste tokens”

Skills are for agents, not humans. The guide is explicit about what NOT to create:

  • Documentation files: No README.md, CHANGELOG.md, or INSTALLATION_GUIDE.md
  • Redundant logic: If the agent already handles a task reliably, delete the instruction
  • Library code: Skills contain tiny, single-purpose scripts. Long-lived libraries belong elsewhere

Every file in a skill directory consumes context. Be ruthless about what earns its place.

Test skills with LLMs

Section titled “Test skills with LLMs”

This is where the guide separates itself from typical “best practices” lists.

Discovery validation. Paste your YAML frontmatter into a fresh LLM chat. Ask it to generate three prompts that should trigger the skill and three that shouldn’t. If the description can’t discriminate between them, rewrite it. Then go further — prompt agents with real assignments and inspect which skills they chose and why.

Logic validation. Feed the LLM your entire SKILL.md and directory tree. Have it simulate execution step-by-step against a concrete task. For each step, it writes out what it’s doing, which file it’s reading or running, and — critically — flags every point where your instructions force it to guess. These are “execution blockers”: ambiguities that will produce inconsistent results in production.

Adversarial testing. Switch roles. Make the LLM attack your skill as a QA tester. “What if the esbuild script fails on a CommonJS dependency? What if the user’s angular.json has custom Webpack builders that Vite doesn’t support? What assumptions am I making about the Node environment?” Don’t fix anything yet — just collect the questions.

Architecture refinement. After answering the adversarial questions, have the LLM restructure your skill. Push dense content out of SKILL.md into references/ and assets/. Add error handling sections. Enforce progressive disclosure so the context window stays lean.

The guide also points to SkillsBench — a benchmark for evaluating skills — as inspiration for building your own evals.

The endgame

Section titled “The endgame”

Gechev’s repo includes a meta-skill: a skill that creates other skills. That’s the direction this is heading. Agents writing instructions for agents, validated by agents.

We’re not there yet. But treating skills as first-class software artifacts — with architecture, testing, and deployment discipline — is the difference between an agent that occasionally helps and one that reliably ships.

AI-Native R&D Team Blueprint: Stop Bolting AI Onto Old Processes

Section titled “AI-Native R&D Team Blueprint: Stop Bolting AI Onto Old Processes”

You installed Copilot. You bought Claude Code subscriptions. Someone on the team uses ChatGPT to write unit tests. You tell yourself, “We’re already using AI.”

But nothing has changed. Same processes. Same meetings. Manual testing. Last-minute all-nighters before release. AI became a faster pen, but you’re still writing the same way.

This post breaks down a 15-slide blueprint document — the “AI-Native R&D Team Blueprint.” It answers one question: what’s the actual gap between “bolting an AI plugin onto old processes” and “rebuilding your entire R&D pipeline around AI capabilities”?

The Gap Between “Add a Tool” and “Change How You Work”

Section titled “The Gap Between “Add a Tool” and “Change How You Work””

The blueprint opens with two columns. Left: “AI-Assisted.” Right: “AI-Native.”

In AI-Assisted mode, AI is a personal productivity tool. Humans write code, AI autocompletes. Existing processes stay the same. Quality depends on manual effort. Scale by hiring more people.

In AI-Native mode, AI is the team’s core production capacity. Humans become definers and verifiers — the entire middle layer of execution goes to AI. Processes are redesigned around what AI can do. Quality relies on automated testing. Scale by adding tokens (trade compute for headcount).

These aren’t different points on the same spectrum. They’re two entirely different production models.

Linear Division of Labor Is Collapsing

Section titled “Linear Division of Labor Is Collapsing”

R&D teams used to have a clear chain: Product → Development → Testing. Everyone guarded their segment.

AI broke that chain. Pure execution roles — people who only write code, only build pages, only run manual tests — are being replaced at scale. Not tomorrow. Now.

Two paths remain for the people who stay:

Move up. Architecture, product direction. System-level understanding, tech decisions, designing collaboration strategies between humans and AI.

Move deep. Quality definition, test strategy. From manual executor to the person who defines the quality baseline.

No third path. The middle ground is vanishing.

Product Managers: Writing PRDs Isn’t Valuable Anymore

Section titled “Product Managers: Writing PRDs Isn’t Valuable Anymore”

This might sting.

Drawing prototypes, writing PRDs, generating solution documents — those used to be the core output of a PM. AI leveled execution ability. A junior PM can now produce a PRD with AI that looks structurally identical to what a five-year veteran writes.

So what’s still valuable? Judgment.

Deciding what to build and what not to. Checking whether the underlying logic holds. Anticipating technical debt. Having the discipline to say “we’re not doing this.” AI can’t do any of that.

The blueprint puts it bluntly: the most important skill is no longer getting things done — it’s deciding what things should be done.

The new PM method is called “define downward, judge upward.” Deliverables are no longer feature lists. They’re layered product definitions — Role, Journey, Story — plus value judgment at each layer. The clearer the bottom-level definition, the more solid the ground AI works on.

Developers: You Can’t Just Be a Coder Anymore

Section titled “Developers: You Can’t Just Be a Coder Anymore”

The blueprint draws the developer’s new work cycle as a loop: define clearly → drive AI generation → verify accuracy → submit PR/MR → continuous Code Review → back to clear definition.

Notice: “writing code” is no longer the core action. Defining and verifying are.

Three evolution paths:

Architecture evolution. System-level design, tech decisions, AI collaboration strategies.

Product integration. Deep business understanding. Independent product definition. Driving AI from definition to end-to-end delivery.

Quality depth. Quality engineering, test strategy definition, maintaining the quality floor for AI-generated code.

The execution layer uses tokens as leverage. But to use leverage well, you need stronger architectural comprehension — the ability to judge whether AI-generated code meets standards and maintains backward compatibility.

QA’s Transformation: From Click-Testing to Defining Quality Baselines

Section titled “QA’s Transformation: From Click-Testing to Defining Quality Baselines”

The blueprint draws a radar chart comparing traditional QA versus AI-native QA across five dimensions: development background, manual testing experience, business comprehension, domain language depth, and quality definition ability.

AI writes code fast, but there’s a trap: it produces the illusion of “passing tests just to pass.” Surface-level green. Underneath, logic gaps.

QA can no longer be manual executors. QA must become the people who define quality baselines. Having a development background — understanding code and frameworks — becomes a hard requirement.

Not about testing more. About testing precisely.

CTO vs. Architect: Clear the Division

Section titled “CTO vs. Architect: Clear the Division”

The blueprint splits technical leadership into two roles.

CTO / Tech Lead handles organization and strategy: setting the pace and priorities of AI transformation, adjusting team structure ratios, managing global risk, driving organizational change.

Architect handles system and collaboration design: designing the human-AI collaboration model, building scaffolding to prevent token waste, defining Code Review and backward compatibility strategies, maintaining standard systems.

Architects used to produce solution documents. Now they should produce “AI-executable project standard systems.” From documents to instruction sets — that’s the real upgrade.

The New Pipeline: Define-Generate-Verify Sandwich

Section titled “The New Pipeline: Define-Generate-Verify Sandwich”

Traditional pipelines schedule by feature modules. AI-native pipelines deliver in closed loops per user journey.

Three layers:

Definition layer (human-led). Product definition (user journeys), technical definition (architecture specs/contracts), quality definition (test strategies).

Generation layer (AI-led). Code generation, test case generation, technical doc generation, automated Code Review, refactoring execution.

Verification layer (human-AI collaborative). Automated regression verification, human PR/MR approval, product experience acceptance.

No more scheduling by feature modules. Instead, deliver complete closed loops organized around user journeys. That’s the biggest pipeline-level shift.

Infrastructure Sets the Ceiling

Section titled “Infrastructure Sets the Ceiling”

AI can be as capable as it wants. Without infrastructure, none of it works. The blueprint positions DevOps and the automated test network as the foundation of AI productivity.

Three infrastructure layers:

  1. Automated test foundation. API interface testing → E2E core journey validation → pipeline integration. This is the bedrock.
  2. CI/CD pipeline. High-frequency parallel support, rapid feedback loops.
  3. Code hosting and AI workbench. GitLab/GitHub permission management, AI load balancing.

Without automated testing running, the closed loop breaks. No matter how fast AI writes code, if bugs have no safety net, the speed advantage drops to zero.

DevOps shifts identity — from operations provider to AI productivity infrastructure provider.

Standards Are AI’s Instruction Set

Section titled “Standards Are AI’s Instruction Set”

This is the most insightful slide in the entire deck, in my opinion.

The toolchain has four blocks: Claude Code / AI IDEs for programming execution. GitLab / GitHub for code hosting. Swagger / OpenAPI for interface contracts. But the critical piece is the fourth — the standards repository.

Development standards, architecture docs, interface contracts. These documents have a new meaning now: they’re the work manual AI must read before every task.

Without standards, AI is an uncontrollable black box. You give it a prompt, it might produce running code, but with inconsistent style, incompatible interfaces, and conflicts with existing systems. With standards, AI output gains consistency and predictability.

Standards are commands. That sentence deserves a spot on every engineering team’s wall.

Talent Structure Has to Follow

Section titled “Talent Structure Has to Follow”

The blueprint draws three overlapping circles: Product Manager (strategy and journey definition), Developer (architecture and contract definition), Quality Engineer (test strategy and baseline definition). The intersection at the center is the AI execution layer.

Hiring strategy changes are explicit:

Significantly reduce hiring for pure coding roles. Increase QA talent with solid development backgrounds. Stop evaluating PMs on prototype tool proficiency — prioritize business insight and judgment instead.

All three roles are converging toward a “product + technology” hybrid. The space for people who only do one thing is shrinking.

How to Land It: Three-Phase Progression

Section titled “How to Land It: Three-Phase Progression”

The blueprint’s transformation principle is eight words: don’t stop the business, don’t mass-replace.

Phase 1: Pilot validation (4-6 weeks). Pick projects with clear boundaries that don’t block core business. Get API testing running, establish initial standards. Run one complete AI-assisted workflow end to end. The pilot is augmentation, not replacement.

Phase 2: Experience rollout (4-8 weeks). Systematize standards across the full team. Enter the first legacy project. Follow the path: supplement tests → AI fixes bugs → refactoring.

Phase 3: Normalized operations (ongoing). New projects default to AI-native process. PR/MR + automated testing + human review becomes muscle memory. Architecture standards get maintained and iterated by AI autonomously.

What About Legacy Projects?

Section titled “What About Legacy Projects?”

Most teams aren’t starting from scratch. There are old projects, tech debt, legacy code everywhere. The blueprint offers a four-gear progressive refactoring model:

  1. Add tests. Define clearly, fix code boundaries.
  2. Automated regression foundation. Build an impenetrable safety net.
  3. AI participates in iteration. Let AI handle bug fixes and non-core requirements first.
  4. Architecture cleanup and refactoring. Use AI’s multi-threading capability to clean up tech debt in parallel.

Cleaning tech debt used to be expensive and blocked business progress. The logic has changed: as long as safety boundaries are clearly defined, AI can handle large-scale restructuring work automatically, across multiple threads.

Seven Laws for the Wall

Section titled “Seven Laws for the Wall”

The blueprint’s final page lists seven core laws. The essentials:

  1. Judgment is scarcer than generation. Deciding what to do matters far more than getting it done.
  2. Definition-driven, not execution-driven. Humans define boundaries and rules. AI executes within the framework.
  3. Think in layers. From strategy down to execution. Reject unstructured guidance.
  4. Quality is the absolute foundation. Without automated test networks, AI-produced code is massive technical debt.
  5. Progressive transformation. Advance through pilots step by step. Never stop core business for transformation.
  6. Standards are commands. Your development docs and interface contracts are literally AI’s system manual.
  7. Products no longer exist by default. Before building anything, prove why a native AI conversation can’t solve the problem.

Number seven is particularly interesting. It says: not every requirement needs to become a product. If a conversation can handle it, why spend team resources building something? This is AI-native thinking pushed to its logical extreme on the question of “what should we build?”

Final Thoughts

Section titled “Final Thoughts”

After reading this blueprint, the dominant feeling isn’t anxiety. It’s clarity.

Anxiety comes from the vague fear of “AI is going to replace me.” The blueprint makes the specifics concrete: pure execution gets replaced, definition and judgment don’t. Click-testing gets replaced, quality strategy doesn’t. Writing PRDs gets replaced, deciding whether something should exist doesn’t.

Once you have clarity, the next step is action. You don’t need to get it perfect on day one. Find a project with clear boundaries. Add automated tests. Let AI run one complete workflow. See what flows and what sticks.

Transformation isn’t revolution. It’s swapping out old gears for new ones, step by step, until the whole machine runs faster.

Act as an insider trading analyst. Review recent insider trading activity in a selected industry. Return the top 5 companies and explain: who is buying and how much, historical impact of similar insider activity, what this reflects about management confidence, any potential red flags.

Act as a contrarian stock analyst. Identify stocks that have experienced significant sell-offs or negative sentiment but maintain strong fundamentals.

Usage tip: The value of this prompt is forcing a contrarian perspective. But distinguish between “wrongly punished” and “genuinely broken” — AI provides the analytical framework, but the final call is yours.

Usage tip: Best for long-horizon investors. The key metrics are payout ratio and free cash flow coverage — if a company borrows to fund dividends, even a great growth track record isn’t reliable.

Context Development Lifecycle: Your AI Agent Isn’t Broken, Your Context Is

Section titled “Context Development Lifecycle: Your AI Agent Isn’t Broken, Your Context Is”

You upgraded to the latest model. You wrote a better prompt. You even put together a CLAUDE.md file. And your AI coding agent still builds the authentication module wrong — for the third time this week.

The model isn’t the problem. Your context is.

The Assumption That Just Broke

Section titled “The Assumption That Just Broke”

The software industry spent decades optimizing how fast humans write code. CI/CD pipelines, agile sprints, pair programming, code review automation. All of it built on one assumption: developer velocity is the bottleneck.

AI coding agents broke that assumption. They scaffold services, fix bugs, ship features. Code production speed is no longer the constraint.

But something else is.

Every time you start a new session with an AI agent, you’re onboarding a new hire who knows everything about programming and nothing about your project. Your architecture decisions, your naming conventions, why that authentication flow works the way it does, which database columns are actually deprecated — none of that comes with the model.

The bottleneck moved. Most teams haven’t noticed.

Where Context Lives Today

Section titled “Where Context Lives Today”

Take an honest look at where your team’s context actually lives:

  • A .cursorrules file someone wrote three months ago. Nobody has updated it since the API redesign.
  • A CLAUDE.md copy-pasted from a blog post. It mentions “React best practices” but your team migrated to Svelte in January.
  • A Slack thread where your senior engineer explained the payment flow. Half the team was in a different timezone and missed it.
  • An architecture decision record that lives in a Confluence page nobody bookmarks.

This is the state of context management in 2026. Fragmented, unversioned, untested, rotting in silence.

The DevOps Parallel

Section titled “The DevOps Parallel”

Patrick Debois coined the term “DevOps” in 2009. His insight was simple: development and operations are inseparable. Treating them as separate teams with separate goals created friction that slowed everything down.

He sees the same pattern now.

Human intent and agent execution are inseparable. You can’t improve agent output without improving the instructions agents receive. And those instructions — the context — need the same rigor we eventually brought to code delivery.

DevOps gave us a lifecycle for delivery. Context needs one too.

The Context Development Lifecycle

Section titled “The Context Development Lifecycle”

Debois proposes four stages. Together they form the Context Development Lifecycle — CDLC.

1. Generate: Make the Implicit Explicit

Section titled “1. Generate: Make the Implicit Explicit”

Most of what makes your team effective is implicit. Your senior engineer knows not to use that ORM method because it causes N+1 queries. Your tech lead knows the billing service has a race condition under high load. Your designer knows the brand guidelines say “never blue on blue.”

None of that is written down in a form an agent can use.

Generation means capturing three layers:

  • Technical context. Coding standards, library usage patterns, architectural decisions, known footguns.
  • Project context. Current priorities, scope boundaries, what’s in flight, what’s blocked.
  • Business context. Why the system exists, what customers expect, compliance constraints, what success looks like.

The agent can draft these specifications. The human decides what’s true. This isn’t documentation for documentation’s sake — it’s the instruction set your agents will execute against.

2. Evaluate: Test-Driven Development for Context

Section titled “2. Evaluate: Test-Driven Development for Context”

You wouldn’t ship code without tests. Why ship context without them?

Evaluation means defining expected agent behaviors through scenarios. “Given this context, the agent should use repository pattern for data access.” “Given this context, the agent should never call the payment API directly.” “Given this context, error messages should include correlation IDs.”

When the agent gets it wrong, resist the impulse to blame the model. That failure is a specification you didn’t write.

Results are statistical, not deterministic. LLMs won’t produce identical output every time. Evaluate across multiple runs. Look for patterns.

3. Distribute: Context as Infrastructure

Section titled “3. Distribute: Context as Infrastructure”

Context becomes powerful when it’s shared. But sharing requires infrastructure.

Think about how code dependencies work. npm, pip, cargo — they give you versioning, integrity checks, update mechanisms. Context needs the same treatment.

Without a registry, without versioning, without a way to push updates, context rots. That skill someone wrote six months ago, before the breaking API change? It’s teaching agents the wrong pattern right now. Nobody knows.

The incentive structure here is better than documentation ever had. Documentation is a chore with delayed, diffuse benefits. Context distribution directly improves the output every developer gets from their agents. The feedback loop is tight.

4. Observe: Close the Loop

Section titled “4. Observe: Close the Loop”

Deploy agents. Watch what they do.

When an agent asks a clarifying question, that’s a gap in your context. When it makes an unexpected architectural choice, that’s an ambiguity you didn’t resolve. When it produces working code that misses the point, that’s an unstated assumption.

These signals are gold. They tell you exactly where your context is insufficient, contradictory, or stale. Feed them back into generation. Add evaluations. Redistribute.

The loop closes: Generate → Evaluate → Distribute → Observe → Generate.

Why Infinite Context Windows Don’t Fix This

Section titled “Why Infinite Context Windows Don’t Fix This”

“But we have million-token context windows now. Just load everything.”

Bigger windows ease a capacity constraint. They don’t solve a quality problem.

Loading your outdated .cursorrules, your current architecture docs, your old onboarding guide, and three conflicting Slack threads into the same context window doesn’t make the agent smarter. It gives the agent more material to be confused by.

The problem shifts from curation (what do I include?) to governance (is what I included consistent, current, and correct?). Infinite windows make governance harder, not easier.

The Compounding Advantage

Section titled “The Compounding Advantage”

Here’s why this matters strategically.

Teams that invest in context quality create a flywheel. Better context produces better agent output. Better output generates clearer signals about what’s working. Clearer signals produce better context.

This compounds in four directions:

  1. Agent quality. Every coding agent, regardless of who runs it, handles the domain correctly.
  2. Team knowledge. The process of formalizing context forces engineers to clarify their own understanding.
  3. Onboarding speed. New team members inherit structured context from day one — and so do their agents.
  4. Organizational alignment. Distributed context keeps teams consistent without centralized enforcement.

This advantage is hard to replicate. It’s built from your organization’s own institutional knowledge. A competitor can use the same model. They can’t copy your context.

What to Do Monday Morning

Section titled “What to Do Monday Morning”

If you’re reading this and wondering where to start:

  1. Audit your current context. Find every .cursorrules, CLAUDE.md, AGENTS.md, and spec file your team uses. Check when they were last updated. Check if they contradict each other.
  2. Pick one pain point. The module your agent consistently gets wrong? Write better context for that specific area. Be concrete.
  3. Write an eval. Define what “correct agent behavior” looks like for that module. Run it. See what fails.
  4. Version it. Put your context in source control if it isn’t already. Treat changes to context with the same review process as changes to code.
  5. Watch the output. After improving context for one area, measure whether agent output quality improves. If it does, you have your proof of concept.

The Question

Section titled “The Question”

The teams that win the AI-assisted development era won’t be distinguished by which model they use. Every team will have access to frontier models.

They’ll be distinguished by the quality of context they feed those models. Generated rigorously. Evaluated systematically. Distributed reliably. Improved continuously.

The question isn’t whether your team is using AI coding agents. It’s what you’ve invested in giving those agents the context to use well.

The Coding Agent Harness: Why Context Engineering Beats Model Shopping

The Coding Agent Harness: Why Context Engineering Beats Model Shopping

Section titled “The Coding Agent Harness: Why Context Engineering Beats Model Shopping”

Your team adopted an AI coding agent last month. The first week felt electric. By week three, the pattern became clear: sometimes the agent nails it, sometimes it produces code that looks plausible but crumbles under review. Duplicate logic in three places. A helper function recreated from scratch when one already exists in utils. An N+1 query hiding inside a loop.

You start wondering if you picked the wrong model.

You didn’t. The model was never the problem.

The Insight Most Teams Miss

Section titled “The Insight Most Teams Miss”

Julian de Angelis at MercadoLibre has been deploying AI coding agents across nearly 20,000 developers and thousands of repositories. His conclusion after operating at that scale:

Your AI coding agent is a brilliant new hire on day one. Fast, eager, and with zero context about your codebase, your conventions, or your architecture.

You wouldn’t hand a new engineer a Jira ticket that says “fix this bug” and walk away. You’d onboard them: docs, tech stack overview, style guides, release workflows, access to internal tools, a mentor to review their work.

That onboarding system is what de Angelis calls the agent harness - the structured set of controls that shape what the agent knows, what it can do, and how its work gets validated. The underlying discipline is context engineering: designing and controlling everything an LLM sees before it generates a single token.

The Constraint Nobody Talks About

Section titled “The Constraint Nobody Talks About”

Every popular AI coding agent - Claude Code, Cursor, Codex - runs the same fundamental loop:

Read -> Plan -> Code -> Validate -> Iterate

The agent reads your files, plans an approach, writes code, runs tests, and loops back if something fails. At every iteration, the context window grows. System prompts, tool definitions, conversation history, file contents, terminal output, the agent’s own generated code - all of it accumulates in a finite workspace.

Here’s the part that surprises people: past roughly 60% utilization of the context window, more context makes the agent actively worse. This phenomenon is called context rot. Even with windows reaching 1M tokens, the fundamental challenge remains. Every token spent on a verbose tool result or an overly long rules file is a token not available for the actual task.

The best agent harnesses are designed with this reality at the center: inject the right context at the right moment, and avoid polluting the window with anything else.

Four Levers That Actually Matter

Section titled “Four Levers That Actually Matter”

1. Custom Rules

Section titled “1. Custom Rules”

Files like CLAUDE.md, cursor/rules, or AGENTS.md get injected into the agent’s context at the start of every interaction. This is the most accessible lever.

What goes in:

  • Tech stack and architecture patterns
  • Naming conventions and code style
  • Testing philosophy (“always write unit tests with table-driven inputs”)
  • Common pitfalls specific to your codebase
  • Anti-patterns you’ve seen the agent produce repeatedly

What doesn’t belong:

  • Entire API documentation (too long, wastes context)
  • Obvious instructions (“write clean code”)
  • Contradictory rules

Practical tips:

  • Keep them under 500 lines. Precision matters.
  • Split rules into focused files by concern: architecture, testing, code style, security. The agent loads only what’s relevant.
  • Include few-shot examples. Models learn from examples better than abstract instructions.
  • Make them living documents. Every time the agent makes a mistake, update the rules. Over weeks, this file becomes remarkably effective.

2. MCP Servers

Section titled “2. MCP Servers”

MCP (Model Context Protocol) servers are plugins that give the agent tools beyond reading and writing files. Without them, the agent is limited to what’s in the repo. With them, it can:

  • Query your database to understand schema and data
  • Search internal documentation or wikis
  • Look up API contracts
  • Interact with CI/CD pipelines
  • Access design specs from Figma
  • Validate its own output against real systems

MCP servers bridge the gap between the agent and your organization’s knowledge. They give the agent access to the same resources a human developer would use.

3. Skills

Section titled “3. Skills”

Skills give the agent new capabilities and domain expertise. A skill is a directory with an entrypoint file that can bundle templates, example outputs, reference docs, and executable scripts.

The design is clever: only a short description stays in context permanently, so the agent knows what’s available. The full content loads only when the skill is invoked - by the user or automatically when the agent detects relevance. This on-demand loading means skills can be as detailed as needed without permanently consuming context budget.

They come in two flavors:

  • Reference skills inject knowledge: conventions, patterns, domain context
  • Task skills provide step-by-step instructions for specific actions

The real power is that skills can bundle and execute scripts, run in isolated subagents with their own clean context window, and compose together. Programmable, context-aware, composable units of agent behavior.

4. Spec-Driven Development

Section titled “4. Spec-Driven Development”

This one reframes how you interact with agents entirely.

Consider this prompt: “Make a new feature to add new items from the backoffice.”

Seems simple. But it says nothing about: which tech stack, where the backoffice is, API contracts, storage layer, idempotency handling, authorization rules.

The agent will read the codebase, plan something reasonable, and produce a working MVP. You add an item and it works. You click “add” again and it inserts the item twice. It was “obvious” the agent should have managed idempotency by ID. In your business, it’s “obvious” that only one role can insert items.

The problem isn’t the agent. It’s the bare initial input. The bottleneck sits between the chair and the screen.

Spec-Driven Development (SDD) solves this by writing detailed specifications before the agent writes code. What the feature does, how it integrates, edge cases, acceptance criteria, test plans. The spec becomes the harness - it consolidates architecture decisions, guidance, and review criteria into a single artifact that engineers the agent’s entire context window in one shot.

And you can use the agent to write the specs too.

The Feedback Loop

Section titled “The Feedback Loop”

The four levers define what the agent knows and can do. The piece that makes the whole system converge is the feedback loop: tests, linters, type checkers, build scripts. Every tool that produces a pass/fail signal becomes a self-correction mechanism.

Agent hooks make this enforceable. They are user-defined commands that execute automatically at specific points in the agent’s lifecycle. Wire your validation into a Stop hook, and the agent literally cannot finish until the checks pass. Not a suggestion it might ignore. An enforced gate.

The more structured feedback available, the less human intervention is needed.

MercadoLibre’s Playbook at 20,000 Developers

Section titled “MercadoLibre’s Playbook at 20,000 Developers”

Theory is fine. Here’s what works at scale with 9+ technology stacks, thousands of legacy repositories, and multiple knowledge domains:

Standardized rules per technology. A curated set of custom rules at the org level, tuned to each tech stack, internal libraries, and code review standards. Teams inherit the base and layer on repo-specific rules. Getting the specificity right is critical - too specific breaks legacy projects, too loose lets unapproved libraries through (security risk).

Internal MCP servers. Built-in access to internal cloud platform capabilities, internal SDKs, business context via RAG, and updated application documentation.

Custom code review agents. Standalone review agents running in CI that analyze every PR - human-written or AI-generated - before a human reviewer sees it. They return prioritized findings so engineers see critical issues first. Human reviewers focus on design decisions instead of catching style violations. Faster review cycles, more consistent quality, and a safety net that scales with PR volume, not reviewer headcount.

Start Here

Section titled “Start Here”

If you haven’t built an agent harness yet, start with the simplest lever: custom rules. Create a CLAUDE.md or equivalent file with your coding conventions, common pitfalls, and a couple of concrete examples of good and bad patterns in your codebase.

Run it for a week. Every time the agent produces something wrong, add a rule. By the end of the month, that file will have quietly transformed your results more than any model upgrade could.

The teams that win with AI coding agents won’t be the ones using the fanciest model. They’ll be the ones with the best-engineered harness.

What is WebReel?

Section titled “What is WebReel?”

WebReel is an open-source CLI tool that records browser demos from JSON configuration files. You define what should happen — clicks, keystrokes, scrolling, typing, dragging — and WebReel handles the rest.

Under the hood: WebReel launches a headless Chrome instance, navigates to your URL, executes each step in sequence, captures screenshots at roughly 60 frames per second, composites cursor animations and keystroke overlays on each frame, and encodes the final output with ffmpeg. Both Chrome and ffmpeg are downloaded automatically on first use.
