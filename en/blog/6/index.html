terminalcp lets Claude Code manage Terminal sessions like Playwright

Aug 13, 2025
Vibe Sparking

In one sentence: terminalcp = Playwright for the terminal.
It lets an AI agent start, operate, and read any command‑line tool (LLDB/GDB, Python REPL, database shells, even other AI CLIs) like a human—and lets you and me take over the same session at any time.

Introduction

Section titled “Introduction”

You may already use Playwright/Browser for web automation—so what about an “automation console” for the command line?
terminalcp abstracts “what a human does in the terminal” into an MCP tool: start a process → send keystrokes → read output → manage multiple sessions, and also supports socket attach so you can watch and take over the AI’s actions in real time.

Why add it to your toolchain?

Section titled “Why add it to your toolchain?”

1) Truly interactive control
Not just “run a command and exit,” but persistent sessions: single‑step in LLDB/GDB, try code in a REPL, or view the “full screen” in TUI apps like vim/htop.

2) Two output modes, pick as needed

  • Terminal mode (stdout): captures the rendered screen buffer, perfect for debuggers and TUIs.
  • Stream mode: captures the raw text stream (ANSI stripped by default), and supports incremental reads with since_last—more efficient for builds/logs.

3) High‑fidelity PTY
Processes run in a pseudo‑terminal, preserving colors/cursor moves/special key sequences—just like human input; plus it manages multiple processes in the background so your agent doesn’t block.

4) Socket attach: the key to human–AI collaboration
Each process automatically exposes a Unix domain socket. You can attach to an already‑running AI session to co‑view and co‑operate, jumping in to help anytime.

5) More “AI‑friendly” than screen/tmux

  • Structured responses (JSON) instead of a wall of text.
  • Clean output (ANSI can be stripped), easier to feed into an LLM.
  • Incremental reads and lifecycle management (exit codes, etc.) built in.

How it differs from screen/tmux (tested: better for AI)

Section titled “How it differs from screen/tmux (tested: better for AI)”
  • Structured responses: JSON replies that are easy for agents to parse.
  • Reliable output capture: stdout gives the rendered screen; stream supports incremental “since_last” reads.
  • Less noise: ANSI control codes stripped by default so escape sequences don’t swamp your context.
  • State‑aware: detects process exit and error codes.
  • No guessing timing: avoids brittle sleep‑style “maybe it’s done” waits.
    You can “simulate” some of this with tmux/screen, but it’s tedious and fragile.

Best‑practice checklist

Section titled “Best‑practice checklist”
  • Use absolute paths for CLIs (e.g., Claude CLI); shell aliases don’t apply.
  • For interactive commands, use {"submit":true}—or send text and \r as two steps.
  • Use stream + since_last for long tasks to avoid re‑pulling full logs.
  • Name sessions + attach via sockets for quick collaboration and fire‑fighting.
  • Explicitly stop when finished; on MCP client exit, leftover processes are auto‑cleaned.
  • Licensed MIT—safe to integrate into internal toolchains.

Conclusion

Section titled “Conclusion”

If you’re building AI‑assisted development / internal DevTools / AIOps, terminalcp turns “everything in the terminal” into an orchestrable API: agent control, human–agent co‑operation, automation with visibility. Starting today, let your AI not just “run commands,” but actually “use the terminal.”

What Is a Slash Command? How Do I Use It?

Section titled “What Is a Slash Command? How Do I Use It?”

It’s essentially a set of “common directives in Markdown,” stored at the project or user level and invoked anytime with /name arguments.

  • Storage locations: project-level .claude/commands/; user-level ~/.claude/commands/. The filename is the command name.
  • Optional Frontmatter: description, model, argument-hint, allowed-tools, etc.
  • Supports injecting Bash output into context: prefix with ! to execute, e.g., ! git status. You must declare allowed Bash commands in the frontmatter.
  • Supports $ARGUMENTS placeholders, @file references, and namespaces (subdirectories enable names like /frontend:build).

Use case: let external CLIs handle “assembling, pulling code, running tests, generating patches,” while Claude coordinates and summarizes. The official docs explicitly support this workflow.
