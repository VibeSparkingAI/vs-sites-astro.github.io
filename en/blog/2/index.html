Codex Gets Slash Commands: Save Prompts as /commands for One‑Tap Reuse

Sep 16, 2025
Vibe Sparking

You can now write your frequently used prompts in Markdown, drop them into ~/.codex/prompts/, and call them in Codex with /command. This speeds up workflows immediately. Community reports confirm it works well in the CLI, and setup is simple (npm i -g @openai/codex or brew install codex).

Note: As you observed, it currently doesn’t support directory‑scoped prefixes. In my testing too, it’s strictly “filename = command name,” with no namespace from subfolders.

Why use it now?

Section titled “Why use it now?”

The real time‑saver is prompt reuse. Turn high‑frequency prompts—like “code review,” “generate tests,” or “release checklist”—into /review, /tests, and /ship. Type once, reuse anywhere. Codex has been rapidly improving its CLI/IDE experience, so it’s a great time to standardize your prompts.

Even better, this pattern mirrors Claude Code’s “custom slash commands”: store prompts as Markdown, recall with a slash. The migration cost is near zero.

Three‑step quickstart

Section titled “Three‑step quickstart”

1) Create the folder
Make sure ~/.codex/prompts/ exists (create it if not).

2) Add a Markdown file
For example, create review.md with content like:

“You are a code review assistant. Provide structured feedback on: security, performance, complexity, readability, and test gaps. Output with clear bullets.”

3) Restart your Codex session
Open Codex, type /revi… and it will autocomplete to /review. Press Enter to run. This matches community reports on folder usage and behavior.

Practical tips:

  • Command name = filename (without .md).
  • Avoid collisions with built‑ins (don’t take names like /init).
  • New files are most reliably recognized after starting a new session.

Current limitations and workarounds

Section titled “Current limitations and workarounds”

  • No directory‑scoped prefixes
    You can’t map backend/review.md and frontend/review.md to different /reviews. Add suffixes like review-fe.md and review-be.md instead.

  • Parameters/placeholders are limited
    Some users report @‑style parameters are unreliable. A reliable approach: trigger the command first, then add concrete context in the next turn.

  • Short, memorable names win
    Think of the slash menu like VS Code/Cursor quick actions—short names you can remember and type fast. Follow community best practices and conventions.

5 ready‑to‑use high‑leverage commands

Section titled “5 ready‑to‑use high‑leverage commands”

  • /review: Structured code review (security/performance/complexity/readability/tests).

  • /tests: Generate table‑driven/parameterized tests and cover edges.

  • /ship: Pre‑release/release checklist (migration, rollback, monitoring, canary scope).

  • /hotfix: Fast triage and minimal‑blast‑radius fix plan.

  • /release-notes: Human‑readable notes + tweet‑length announcement.

Save each snippet above as a .md file in ~/.codex/prompts/. After restarting your session, they’ll show up via /.

Use it alongside Claude Code

Section titled “Use it alongside Claude Code”

If your team mixes Codex + Claude Code, standardize reusable flows as Markdown prompts and use each tool’s slash menu. Both sides can consume the same files, reducing cognitive load. The Claude community also encourages this prompt‑as‑Markdown approach.

Closing

Section titled “Closing”

Standardizing and modularizing prompts is a key step in turning AI coding from “inspiration” into “process.”
Starting today, put your top 3 prompts into ~/.codex/prompts/ and let your fingers remember the joy of /.

Vibe Coding: The Biggest Perk for 35+ Architects in the AI Era

Sep 12, 2025
Vibe Sparking

Geniuses like Linus or DHH may shrug at Vibe Coding. For working architects, this is a once‑in‑a‑decade opportunity. For people with zero engineering foundation, it may actually be “pulling up seedlings to help them grow.”

—

Introduction

Section titled “Introduction”

You can feel the wind shifting. AI is no longer just “completing a few lines of code.” It has started to string together the entire engineering loop: requirements, design, implementation, testing, deployment, and operations/retro. That’s what I call Vibe Coding — an engineering approach that is anchored by architecture, operated by agents, and measured by closed loops.

The data backs this up: the share of developers using or planning to use AI tools has risen from 76% in 2024 to 84% in 2025, yet trust has declined. “Using” and “using well” are not the same.

Bottom line first: Vibe Coding is an “era bonus” for 35+ architects. You carry systems thinking, hard‑won engineering experience, and business context. AI can outsource the repetitive, mechanical, error‑prone parts. It can 10x your “abstraction and constraint” leverage — not exaggerating; mainstream studies suggest very real productivity gains from tens of percent to multiples.

What Is Vibe Coding?

Section titled “What Is Vibe Coding?”

In one sentence: Turn AI into your “engineering squad.” You are the chief designer; it handles the grunt work and the wiring.

A practical six‑step closed loop:

  1. Requirements → Scenario use cases: translate business goals into verifiable stories and constraints.
  2. Architecture → Constraints and interfaces: lock boundaries with ADRs and interface contracts.
  3. Implementation → Agent division of labor: codegen, library retrieval, dependency install, environment setup.
  4. Verification → Test‑first: regression suites and coverage thresholds.
  5. Integration → CI/CD: PR bots, automated review, staged rollout.
  6. Observability → Feedback loops: logs, metrics, rollback, data feedback.

This isn’t hand‑waving. End‑to‑end benchmarks like SWE‑bench have brought “can AI actually fix real bugs and pass tests” into daylight. Systems such as OpenHands and Refact have placed high on leaderboards, showing closed‑loop AI engineering is rapidly approaching the practical line.

Why Is This the Biggest Perk for 35+ Architects?

Section titled “Why Is This the Biggest Perk for 35+ Architects?”

Because you hold three rare assets:

  • System‑level abstraction: you know what to do and what not to do. AI needs guardrails.
  • Engineering cadence: you’ve lived through versioning, dependencies, environments, and regressions. AI fears “no constraints.”
  • Domain context: you understand data definitions, compliance, SLAs. AI doesn’t read the “unwritten rules,” you do.

Research backs the reality: when enterprises land generative AI in software development, overall speed improves materially, and with proper governance “the same team can ship more.” But governance and method are key — otherwise you end up “fast and wrong.”

Your moat is exactly this: channel AI’s “speed” into a predictable engineering process. That’s where Vibe Coding sits.

The Truth: “Geniuses Don’t Need It; Ordinary Folks Can Trip Over It”

Section titled “The Truth: “Geniuses Don’t Need It; Ordinary Folks Can Trip Over It””

Linus summed up AI sharply: “90% marketing, 10% reality.” He has also said he expects practical tooling like bug finding — the “pride and pragmatism” of masters.

DHH’s stance is similar: AI is fine for drafting and API lookup; the real joy is in typing the code yourself. Hacker geniuses don’t need “automated closed loops” to ship.

So why do ordinary people often face‑plant?

The core issue isn’t “can a model write a few lines of code,” but can you run requirements → design → implementation → verification → deployment → observability as a repeatable loop. Without that loop, you might “use AI to write code” every day yet only accumulate half‑finished parts and a bug spiral.

The data already signals it: in 2025, developer sentiment toward AI has dropped while distrust has risen — treating AI outputs as “answers,” instead of “artifacts to be verified,” sends you off course quickly.

Typical failure patterns (common among outsiders / low‑experience programmers):

  1. Requirements drift: no clear acceptance criteria → prompts keep getting longer → results drift farther.
  2. Structural entropy: no governance for directories, module boundaries, env vars, or dependency versions → unmaintainable in weeks.
  3. Blind bug‑fixing: no test baseline → manual clicking → fix one, break three.
  4. Broken data loop: no events/metrics captured → no activation, retention, or value evidence.
  5. Invisible cost/risk: model calls without quotas, caching, or auditing → cost and compliance spiral together.
  6. Always “fast to generate,” never “strict on regression”: no minimum regression set → delivery quality is luck.
  7. Only “prompt tuning,” never “interface + constraint design”: agent artifacts lack consistency → reuse fails.

The real gap: closed‑loop thinking relies on long‑term accumulation of abstraction and constraint skills, usually built over 5–10 years of:

  • CS fundamentals (compilers, OS, database normal forms, algorithms and data structures)
  • Engineering practice (version control, dependency governance, performance and incident triage, CI/CD)
  • Architectural tradeoffs (consistency vs. throughput, latency vs. cost, failure domains vs. ownership)

The Vibe Coding Playbook (Minimal, Practical)

Section titled “The Vibe Coding Playbook (Minimal, Practical)”

Principles

Section titled “Principles”
  1. Constraints before generation: define contracts, interfaces, schemas, and acceptance criteria before asking AI to generate.
  2. Test first, always: even a tiny regression pack is better than nothing.
  3. Traceability: version your prompts, inputs, and outputs; attach artifacts to PRs.
  4. Guardrails: regex/structured checks, secondary critic models, cost and timeout budgets.
  5. Human in the loop → remove over time: automate the critical path as confidence grows.

A Minimal Six‑Artifact Kit

Section titled “A Minimal Six‑Artifact Kit”
  • ADR template + interface contracts (OpenAPI/JSON Schema)
  • Minimal regression tests (happy path + 1–2 edge cases)
  • PR bot checklist (schema changes, migration, test impact)
  • Observability baseline (structured logs + 5 key metrics + trace on critical path)
  • Cost/prod budget dashboard (model $/user/week, error rate, timeout)
  • Rollback and canary switch

Agent Roles (Example)

Section titled “Agent Roles (Example)”
  • Generator: code, docs, summaries, drafts
  • Checker: compliance, duplicates, risk flags
  • Orchestrator: multi‑step flows (ingest → clean → structure → persist)

A 90‑Day Closed‑Loop Plan (Solo/Small Team)

Section titled “A 90‑Day Closed‑Loop Plan (Solo/Small Team)”

Phase A (Weeks 1–4): Problem Framing and Data

Section titled “Phase A (Weeks 1–4): Problem Framing and Data”
  • Pick a scenario: high frequency + clearly measurable savings + easy data access.
  • Five lines: problem → user → value → data → willingness to pay.
  • Baseline data/ground truth: sample 50–200 items; define acceptance criteria.
  • Minimal product spec: 1 core path, 2 safeguards, 1 success KPI.
  • Aligned metrics: activation (first core action < 60s), retention (7‑day revisit rate), unit value (time saved or cost reduced).
  • Pricing hypothesis: model (seat / usage / tiered subscription) + price anchor research.

Phase B (Weeks 5–8): Core Loop and AI Enablement

Section titled “Phase B (Weeks 5–8): Core Loop and AI Enablement”
  • Core loop: acquisition → activation → value realization → feedback into models/rules/recommendation.
  • Agent division of labor:
    • Generation: docs/summaries/strategy drafts
    • Checking: compliance/duplication/risk flags
    • Orchestration: multi‑step data flow (scrape → clean → structure → store)
  • Sustainable data: unified schema (events + business + model prompt context); build a minimal, high‑quality label set.
  • Iteration cadence: ship weekly (changelog + metric comparison) to 5–10 gray users.
  • Trust guardrails: prompt versioning, output validation (regex/rules/critic model), cost monitoring ($/user/week).

Phase C (Weeks 9–12): Monetization and Steady State

Section titled “Phase C (Weeks 9–12): Monetization and Steady State”
  • Payments and access: Stripe/Lemon Squeezy/Paddle; control rate/features by plan.
  • Observability: logs (structured), metrics (core funnels + cost), traces (critical path), and alerting.
  • Ops automation: trial expiry nudges, low‑usage wakeups, weekly usage insights (AI‑generated).
  • Risk/compliance: data masking, model call allowlist, SBOM and version baselines.
  • Growth experiments: A/B onboarding variants (prompts/sample data) → activation impact.
  • Expansion: when weekly new paid growth slows and retention is healthy → evaluate a second curve or vertical packs.

Closed‑Loop Readiness (≥80% means “loop is running”)

Section titled “Closed‑Loop Readiness (≥80% means “loop is running”)”
  1. ≥10 real external users use it for 4 consecutive weeks
  2. A single value KPI (time saved / automation success / detection recall) trends up stably

  3. 10% trial‑to‑paid validation samples

  4. Critical path fully automated (no manual intervention) + safe rollback
  5. ≥70% of generated content passes automatic quality checks; no high‑risk misuse cases

Landing Tips

Section titled “Landing Tips”
  • Don’t over‑engineer multitenancy early; start single‑tenant with permission isolation
  • Start with rules + validation, then introduce free‑form model generation
  • Cache model calls (prompt hash + semantic similarity)
  • Weekly changelogs are for users, not yourself
  • Remove one low‑usage feature every week to keep the core crisp
  • Small, high‑quality labeling loops beat blind scaling

Risks and “Fool‑Proofing”

Section titled “Risks and “Fool‑Proofing””
  • Accuracy/trust: usage and hype rise while trust drops — counter with tests and review.
  • IP/compliance: clarify model/data boundaries and terms.
  • Quality and security: adopt security baselines, dependency allowlists, SBOM, signing, and supply‑chain scanning.
  • Over‑reliance: keep manual takeover and rollback for critical paths.

Section titled “The “H200 Era” Engineering Dividend”

Closing

Section titled “Closing”

Section titled “Stage 2: Redesign and Engineer Seriously”
  • Trigger: Requirements are mostly finalized, and the scope is stable.
  • Deliverables: Database schema, API contracts, state management, module boundaries, testing strategy, etc.

Security/Non-Functional Advice: Treat security, performance, and maintainability as “Technical Enablers” and progressively incorporate them into your Definition of Done (DoD) during the second stage. Conduct Spikes for validation when necessary.

When Can You Skip the Prototype?

Section titled “When Can You Skip the Prototype?”
  • The requirements have already been validated (e.g., redoing an old feature, a clear competitor to model after).

CC‑Switch is an ultra‑light desktop app built on Tauri 2.0 (installer ~5 MB). Introduction: Why do you need a “one‑click provider switcher”?

Section titled “Introduction: Why do you need a “one‑click provider switcher”?”

As people who write code every day, we’re often more like dispatchers of cost and speed. Today you use Qwen Coder for speed; tomorrow you switch to DeepSeek v3.1 for complex reasoning; when compatibility issues show up, you switch back to Claude official sign‑in. If you still have to manually edit configs under ~/.claude and ~/.codex every time, it’s annoying and error‑prone.

CC‑Switch turns this into a single toggle: Click once, and the provider switches. Config copies are auto‑managed, rollback is safe, and you can return to official sign‑in at any time.

Highlights at a glance

Section titled “Highlights at a glance”

  • Truly lightweight: Built on Tauri 2.0, launches instantly. The official guidance emphasizes Tauri’s small‑footprint approach (more restrained than Electron). In v3.0.0, CC‑Switch migrated from Electron to Tauri 2.0. In author tests the bundle size dropped from ~80 MB to ~12 MB and startup sped up 10×.

  • Supports both Claude Code and Codex:

    • Claude Code primary config: ~/.claude/settings.json (or legacy claude.json) The official docs confirm ~/.claude/settings.json is the standard entry point.
    • Codex primary config: ~/.codex/auth.json (required) + ~/.codex/config.toml (may be empty) The Codex README also states ~/.codex/config.toml is the config center, supporting MCP, etc.

  • One‑click switch & automatic backups: Switching overlays the selected provider copy onto the primary config. If there’s a provider currently “in use”, CC‑Switch first writes the primary config back into that provider’s copy to facilitate rollback. Internal config v1→v2 migrations are timestamp‑backed up under ~/.cc-switch/.

  • Presets ready to use: DeepSeek v3.1, Qwen Coder, GLM, Kimi K2, packycode… just paste a key to configure with one click.

  • Switch back to official sign‑in anytime: Prefer Claude or Codex’s official sign‑in? Add the Official Sign‑In preset. After switching, restart your terminal to use /login (Claude Code) or “Sign in with ChatGPT” (Codex side).

  • Privacy‑friendly: Everything is stored locally only in ~/.cc-switch/config.json.

Section titled “Installation and system requirements”

  • Windows: Download CC-Switch-Setup.msi or Windows-Portable.zip from Releases.
