Claude Code 2.1.69：二十刀磨出手感

Section titled “Claude Code 2.1.69：二十刀磨出手感”

二十个功能。单独拿出来，没一个能上头条。这正是这个版本有意思的地方。

2 月 5 日到 2 月 28 日，Claude Code 连发八个版本，把自己从终端编码助手变成了多智能体开发平台——Opus 4.6、Agent Teams、Worktree 隔离、Remote Control、持久记忆、HTTP Hooks。十五天干完的活，够猛，也够粗糙。

2.1.69（3 月 4 日）干的是更难的事：让这个平台用起来舒服。技能知道自己装在哪。钩子知道是哪个 Agent 触发的。终端上能直接看到当前 effort 等级。插件改完不用重启。这些都是手感问题——手感决定了一个强大的工具到底会不会被人用下去。

技能生态

Section titled “技能生态”

两个改动，实际影响比看上去大。

/claude-api 技能 — 内置的 Claude API 和 Anthropic SDK 开发技能。要写 API 集成的时候，不用再切出去翻文档或者凭记忆猜 SDK 用法，直接在 Claude Code 里问就行。

${CLAUDE_SKILL_DIR} 变量 — 技能现在可以在 SKILL.md 里引用自己所在的目录。之前要加载同目录下的模板或数据文件，只能写绝对路径或者脆弱的相对路径。现在：

这对技能的可移植性很关键。一个作为插件发布的技能，不管装到哪个目录都能正常工作。

插件基础设施

Section titled “插件基础设施”

四个改动，插件生态变得更适合企业场景。

/reload-plugins — 不重启 Claude Code 就能激活插件改动。开发插件的时候，反馈循环从”重启、丢上下文、重新开始”变成了”reload，继续干”。

pluginTrustMessage — 企业管理员可以在插件信任弹窗里附加自定义说明。比如”只安装 plugins.corp.internal 上的内部插件”——直接写在安装确认对话框里。

pathPattern — strictKnownMarketplaces 新增正则匹配，可以按文件和目录路径来限制插件来源，比原来只能匹配域名更细致。

git-subdir 插件源 — 指向 git 仓库里的一个子目录。一个 monorepo 里有五个插件，不用再拆成五个仓库。

钩子进化

Section titled “钩子进化”

三个功能让钩子更了解平台内部状态。

InstructionsLoaded 钩子事件 — CLAUDE.md 或 .claude/rules/*.md 被加载到上下文时触发。安全团队可以审计当前哪些指令文件在生效。开发流程可以在项目规则变化时自动响应。

agent_id 和 agent_type 进入钩子事件 — 每个钩子事件都带上了触发它的 Agent 身份。子智能体有 agent_id，--agent 会话和子智能体都有 agent_type。PreToolUse 钩子现在可以对不同 Agent 执行不同策略——code-writer 可以自由用 Bash，code-reviewer 只能用 Read。

worktree 字段加入状态行钩子 — 在 --worktree 会话中，钩子能拿到 name、path、branch 和原始仓库目录。CI 集成可以追踪哪个 worktree 在跑什么任务。

交互打磨

Section titled “交互打磨”

日常操作面上的小改动。

Ctrl+U 退出 bash 模式 — 在空的 bash 提示符上按 Ctrl+U 就退出，和 Escape、Backspace 行为一致。多一条符合肌肉记忆的路径。

数字小键盘支持 — Claude 弹出的编号选项列表，以前只认 QWERTY 上面那排数字键。现在小键盘也能用了。如果你一直在戳小键盘纳闷为啥没反应，修好了。

effort 等级显示 — spinner 和 logo 现在直接显示当前 effort 设置（比如”with low effort”）。之前得打开 /config 才知道自己跑的是什么等级，现在 Claude 在思考的时候就能看到。

Agent 名称写入终端标题 — 用 claude --agent 的时候，终端标题显示 Agent 名称。开了好几个终端 tab 跑不同 Agent，一眼就能分清。

Claude Code Desktop 提示 — macOS 和 Windows 上首次启动时建议安装桌面版。最多弹三次，之后不再出现。低干扰。

配置项

Section titled “配置项”

两个针对特定环境的设置。

sandbox.enableWeakerNetworkIsolation（仅 macOS）— 让 gh、gcloud、terraform 这类 Go 程序在使用 MITM 代理时能正常验证 TLS 证书。公司网络有 TLS 审查代理的需要这个。

includeGitInstructions — 从 Claude 的系统提示里移除内置的 commit 和 PR 工作流指令。也可以用 CLAUDE_CODE_DISABLE_GIT_INSTRUCTIONS 环境变量。如果你在 CLAUDE.md 里有自定义的提交流程，不希望 Claude 的默认行为干扰，用这个。

Remote Control 增强

Section titled “Remote Control 增强”

可选的 name 参数 — /remote-control My Project 或 --name "My Project" 设置一个自定义会话标题，在 claude.ai/code 里可见。手机上同时盯着好几个远程会话的时候，名字比自动生成的 ID 好认多了。

背后的逻辑

Section titled “背后的逻辑”

单个看这些功能，就是个维护版本。放在一起看，有一条线：每个改动都对准了你和平台之间的交互面。

层面发了什么解决什么
技能/claude-api、${CLAUDE_SKILL_DIR}技能变得可移植、自感知
插件/reload-plugins、信任提示定制、git-subdir插件开发和企业部署变得实用

交互Ctrl+U、小键盘、effort 显示、Agent 名称显示日常摩擦从交互面上消失
配置TLS 代理支持、git 指令开关边缘环境不用再绕弯子

2.1.32 到 2.1.63 搭的多智能体平台很猛。2.1.69 把它磨出了手感。

试试看

Section titled “试试看”
  1. /reload-plugins — 开发或测试插件的话，这个省掉大量重启
  2. effort 等级显示 — 用 /effort 调节一下，看 spinner 上的变化
  3. ${CLAUDE_SKILL_DIR} — 把某个技能重构成引用自己目录下的模板文件
  4. InstructionsLoaded 钩子 — 写一个钩子，在项目指令文件变化时记日志
  5. Remote Control 命名 — 下次远程会话试试：/remote-control "Feature Auth"

PM Skills Marketplace: 65 个技能让 Claude Code 变成产品经理的操作系统

Section titled “PM Skills Marketplace: 65 个技能让 Claude Code 变成产品经理的操作系统”

让 Claude 写个 PRD 试试。两页文字，措辞专业，格式漂亮。但读过 Inspired 的人一眼就能看出问题：功能列表后面没有机会解决方案树，优先级排序没有假设映射支撑，用户故事没用 JTBD 框架。

你不缺知识。Teresa Torres 的持续发现、Marty Cagan 的赋能产品团队、Alberto Savoia 的预型测试、Dan Olsen 的精益产品——这些书你都读过。问题在于：这些框架躺在书架上，而 Claude 正以光速生成没有骨架的文字。

Paweł Huryn 做了一件事：把这些框架变成代码。他是 The Product Compass 的作者，PM Skills Marketplace 就是他的作品。

这是什么

Section titled “这是什么”

65 个技能，36 个链式工作流，8 个插件。全部开源，MIT 协议。

每个技能把一个经过验证的 PM 框架编码进 Claude 的领域知识——不是提示词模板，是让 Claude 在对话中自动调用的结构化认知。聊产品发现时，Claude 会加载 Teresa Torres 的机会解决方案树方法论。排优先级时，它用影响力×风险矩阵并给出实验建议。测试假设时，它用 Alberto Savoia 在 The Right It 里提出的预型测试方法。

GitHub: phuryn/pm-skills 作者: Paweł Huryn (The Product Compass) 协议: MIT 兼容工具: Claude Code, Cowork, Gemini CLI, Cursor, Codex CLI, Kiro

三层架构

Section titled “三层架构”

设计很干净。技能是积木，命令把积木串成工作流，插件按领域分组。

技能 (Skills)

Section titled “技能 (Skills)”

每个技能给 Claude 一项特定 PM 任务的领域知识。有些是独立参考——prioritization-frameworks 涵盖 9 种排序框架（Opportunity Score、ICE、RICE、MoSCoW、Kano 等），Claude 在你讨论优先级时自动调用。有些是工作流组件，多个命令共享。

技能在对话中自动加载，不用手动调用。想强制加载某个技能，用 /plugin-name:skill-name。

命令 (Commands)

Section titled “命令 (Commands)”

命令是亮点。拿 /discover 来说，它串联四个技能：

  1. brainstorm-ideas — 从 PM、设计师、工程师三个视角做头脑风暴
  2. identify-assumptions — 在价值、可用性、可行性、可实现性四个维度识别风险假设
  3. prioritize-assumptions — 用影响力×风险矩阵排序，附带实验建议
  4. brainstorm-experiments — 设计精益实验来验证高优先级假设

一个命令，完整的发现周期。做完之后，它会建议下一步该用哪个命令。

和写提示词有什么区别

Section titled “和写提示词有什么区别”

三点。

结构而非模板。 技能给 Claude 领域知识，不是填空式模板。输出会根据你的具体情况调整，同时保持框架的严谨性。

链式工作流。 命令把技能串联成序列，和 PM 实际的工作流程一致。发现阶段的输出直接喂给战略阶段，战略再喂给执行。你不用手动编排。

自动加载。 技能在对话中按需激活。聊定价时，Claude 会自动调用定价策略、支付意愿分析、竞争定价模型——不用你点名。

Vibe Sparking

你的 AI 智能体技能写错了，这里是修法

Section titled “你的 AI 智能体技能写错了，这里是修法”

你写技能的方式一直有问题。

不是小问题，是根子上就不对。多数开发者写 AI 智能体技能的思路跟写给初级开发者的文档一样——堆上下文、加标题、塞几个例子，然后祈祷智能体能搞定。结果呢？上下文窗口撑爆、触发器乱飞、输出忽好忽坏。

Minko Gechev 最近发了个叫 skills-best-practices 的仓库，认真对待了这个问题。他的核心观点很直接：技能不是文档，是程序。运行时是 LLM。

技能的结构

Section titled “技能的结构”

每个技能只有一种结构：

不要 README。不要 changelog。不要库代码。运行时用不到的文件就是浪费。

SKILL.md 是大脑——放高层流程和导航。references 放 API 文档和速查表。scripts 处理容易出错、重复性高的操作，这类任务里变异就是 bug。assets 存模板让智能体复制。就这些。

发现机制是技能成败的关键

Section titled “发现机制是技能成败的关键”

智能体决定是否加载你的技能之前，只看两样东西：name 和 description。总共 1024 个字符决定你的技能存在还是不存在。

烂描述：“React 技能。”

好描述：“使用 Tailwind CSS 创建和构建 React 组件。当用户想更新组件样式或 UI 逻辑时使用。不适用于 Vue、Svelte 或原生 CSS 项目。”

注意负触发。你得告诉智能体什么时候不该启动。不写的话，你的 Angular 迁移技能会在 React 项目上跳出来。

name 字段也有规矩：1-64 个字符，只允许小写字母、数字、连字符，不能连续用连字符，而且名字必须和父目录名一致。angular-testing 就得放在 angular-testing/SKILL.md 里。

上下文窗口就是内存，得管理

Section titled “上下文窗口就是内存，得管理”

指南管这叫”按需加载”（Just-in-Time Loading）。参考资料在智能体用到的时候再拉进来。SKILL.md 保持精简：只放流程和导航。把 schema、API 文档、速查表搬到 references/，模板搬到 assets/。在 SKILL.md 里明确写出什么时候读哪个文件：“具体错误码见 references/auth-flow.md。”

子目录只放一层。references/schema.md 可以。references/db/v1/schema.md 不行。

这就是给按 token 计费的运行时做内存管理。

给机器写，不是给人写

Section titled “给机器写，不是给人写”

三条规则会改变你写技能指令的方式：

编号步骤。 工作流定义成严格的顺序序列。决策分支要画清楚：“第 2 步：如果需要 source map 就运行 ng build --source-map。否则跳到第 3 步。“不留模糊空间。

用具体模板代替描述。 智能体擅长模式匹配。assets/ 里的一个 JSON 模板比几段文字描述管用得多。别描述——直接展示。

第三人称祈使句。 “提取文本……”而不是”你应该提取……”也不是”我将提取……”这种表述能让智能体行为更稳定。

概念命名要统一。用领域原生词汇。Angular 里就说”template”，别说”html""markup""view”。一致性减少智能体混淆。

把脆弱的工作交给脚本

Section titled “把脆弱的工作交给脚本”

不要让 LLM 每次运行技能都从头写复杂的解析逻辑。在 scripts/ 目录里放一个测试过的 Python 或 Bash 脚本，设计成小型 CLI。

关键细节：脚本的错误信息要写得具体、人能读懂。脚本失败时输出”Error: Unable to resolve CommonJS dependency ‘lodash’ — try adding it to the Vite optimizeDeps.include array”，智能体就能自己纠正。只返回退出码 1 没有消息，智能体就卡住了。

别创建浪费 token 的文件

Section titled “别创建浪费 token 的文件”

技能是给智能体用的，不是给人看的。指南明确列出不该创建的东西：

  • 文档文件：README.md、CHANGELOG.md、INSTALLATION_GUIDE.md 都不要
  • 冗余逻辑：智能体已经能可靠处理的任务，把指令删掉
  • 库代码：技能里只放小型单用途脚本，长期维护的库放别的地方

技能目录里的每个文件都消耗上下文。对谁能进来要狠心。

用 LLM 测试技能

Section titled “用 LLM 测试技能”

这部分让这份指南跟一般的”最佳实践”清单拉开了距离。

发现验证。 把你的 YAML 前置信息粘贴到一个新的 LLM 对话里。让它生成三个应该触发你技能的提示词和三个不应该触发的。如果描述分不清，就重写。再进一步——用真实任务去提示智能体，检查它选了哪个技能、为什么选。

逻辑验证。 把整个 SKILL.md 和目录结构喂给 LLM。让它针对一个具体任务一步步模拟执行。每一步写出它在做什么、读哪个文件或运行哪个脚本——最关键的是标记出每个它被迫猜测的地方。这些”执行阻塞点”就是指令中的歧义，会导致生产环境里的不一致输出。

对抗测试。 角色互换。让 LLM 当 QA 测试员来攻击你的技能。“如果 esbuild 脚本在 CommonJS 依赖上失败了怎么办？如果用户的 angular.json 有自定义 Webpack builder、Vite 不支持呢？我对 Node 环境做了什么隐含假设？“先别改——只收集问题。

架构精炼。 回答完对抗性问题后，让 LLM 重构你的技能。把密集内容从 SKILL.md 推到 references/ 和 assets/ 里。加错误处理部分。执行渐进式披露，保持上下文窗口精简。

指南还提到了 SkillsBench——一个评估技能的基准——可以用来构建自己的评估体系。

终局

Section titled “终局”

Gechev 的仓库里有一个元技能：用来创建其他技能的技能。这是方向——智能体给智能体写指令，由智能体验证。

现在还没到那一步。但是把技能当成一等软件制品来对待——有架构、有测试、有部署纪律——这就是”偶尔帮上忙”和”稳定交付”之间的分水岭。

速查表

Section titled “速查表”
原则规则
结构SKILL.md + scripts/ + references/ + assets/
SKILL.md 大小不超过 500 行
子目录平铺，只一层

加载按需加载，不是预加载

验证LLM 驱动：发现、逻辑、对抗、精炼
浪费不要 README、changelog、库代码

别让 AI 乱写一气：需求快速落地的 11 步闭环工作流

2026年3月5日
Vibe Sparking

别让 AI 乱写一气：需求快速落地的 11 步闭环工作流

Section titled “别让 AI 乱写一气：需求快速落地的 11 步闭环工作流”

你让 AI 加一个支付重试机制。它给你搭了一套新的支付服务。你让它修个日期格式化的 bug。它把整个日期工具库重构了。你说加一个 API 端点。它起了一个带独立数据库的微服务。

AI 写了两分钟。你修了两小时。

听着耳熟？不是 AI 不行，是你没给它流程。

真正的瓶颈在哪

Section titled “真正的瓶颈在哪”

大多数开发者跟 AI 编码助手的交互模式都一样：打一段大概描述想要什么，回车，然后祈祷。输出不对，就追加各种修正提示。“不是这样的。""我说的是另一个表。""你为什么改了那个文件？”

看起来像对话，实际是混战。

AI 手里没有结构化需求。没有验收标准。不知道”做完了”是什么意思。不知道哪些东西不能碰。你招一个人类开发者不会这样 onboarding。但换成 AI，流程就全扔了。

@123olp 在 X 上发了一条帖子（2026-03-03），把这个问题讲透了，同时给出了完整方案：一套用 AI 快速落地需求的闭环工作流。不是那种虚头巴脑的”提示词工程”指南，是正经的工程流程，有闸门、有检查点、有回滚策略。

拆开来讲。

两个角色

Section titled “两个角色”

分工很清楚：

你（需求 owner）： 提供可验收的需求。做关键决策。最后拍板合并和发布。

AI（执行者）： 产出 Spec、任务拆分、代码实现、自测、验收报告、发布材料。

核心原则：一次强确认（Spec）+ 小步提交 + 分层验收（P0/P1/P2）+ 随时可回滚。

第 0 步：结构化需求输入

Section titled “第 0 步：结构化需求输入”

多数人在这里就翻车了。写一句”加个支付重试”就指望 AI 读心术。

换个方式：

  • 一句话： 我想要 X，以便 Y
  • 对象/场景： 谁用、在哪用、主路径是什么
  • 约束： 技术栈、接口、不可改的点、权限、上线窗口
  • 最小可验收版本（MVA）： 用可测试的语句写
    • 当 [输入] 时，系统应 [行为]
    • 异常/无权限/不存在时应 [兜底]
    • 不应影响 [列 3-5 个回归点]
  • 不做什么： 明确 out-of-scope

花五分钟写这些，能省掉后面好几个小时。

第 1 步：AI 产出 Spec

Section titled “第 1 步：AI 产出 Spec”

AI 生成一份不超过一页的可执行 Spec，固定包含：

  1. 目标与非目标
  2. 用户/调用方行为（主流程 + 关键边界）
  3. 接口/数据契约（入参、出参、错误码、幂等、兼容策略）
  4. 改动范围（模块、文件、影响面）
  5. 风险与依赖（外部依赖、迁移、权限、安全、性能）
  6. 验收标准（P0 必须项 + 可选项）
  7. 回滚/降级策略（最少一句话可操作）
  8. 交付物清单（代码、测试、文档、PR 描述）

整个工作流里唯一的”强确认点”。

第 2 步：你确认 Spec

Section titled “第 2 步：你确认 Spec”

你的回复只有两种：

  • “Spec ok，开干”
  • “Spec 需改：A、B、C”

Spec 确认后，后续不用再反复确认。AI 自主执行，除非触发异常闸门（下面讲）。

就这一道闸门，干掉了大部分返工。AI 和你在写第一行代码之前就对齐了”做完是什么样”。

第 3 步：AI 拆任务

Section titled “第 3 步：AI 拆任务”

AI 产出 3-8 个任务，每个包含：

  • 任务名（动词开头）
  • 输入和依赖
  • 要改的文件和模块
  • 交付物（代码、配置、迁移脚本、文档）
  • 自测方法（命令、用例、预期结果）
  • 完成定义 DoD（lint、test、类型检查是否必须）
  • 回滚点（开关、配置、兼容路径）

第 4 步：小步执行 + 即时自检

Section titled “第 4 步：小步执行 + 即时自检”

不搞一把梭。每个任务，AI 输出：

  1. 做了什么改动（摘要 + 关键 diff）
  2. 自检结果（lint、单测、日志、关键用例）
  3. 风险提示（发现了新的边界或依赖）
  4. 下一步

这个节奏能在早期发现问题，每一步都有可审阅的证据。

第 5 步：异常闸门

Section titled “第 5 步：异常闸门”

执行过程中，出现以下任一情况，AI 必须停下来问你：

  • 需求冲突或理解无法确定
  • 需要新增依赖、权限、数据迁移
  • 要改对外契约（接口、字段、行为）
  • 风险明显上升（安全、性能、回滚不可行）
  • 超出 out-of-scope

你拍板后，AI 只更新 Spec 里受影响的段落，继续执行。

这些闸门防止 AI 悄悄跑偏。有了这道安全网，自主执行才值得信任。

第 6 步：分层验收

Section titled “第 6 步：分层验收”

AI 生成验收清单并执行，附带证据。

P0（必做）：

  • 功能：主路径 + 关键边界（空值、非法、无权限、不存在）
  • 回归：你在需求里列的那 3-5 个点
  • 自动化：format/lint + 关键单测或契约测试
  • 错误处理：不吞错、可定位日志、敏感信息不泄露
  • 兼容：旧行为不被破坏（或明确 breaking + 迁移指引）
  • 回滚：能一键回到原行为

P1（推荐）：

  • 性能：关键路径耗时、查询次数、大数据量行为
  • 安全：输入校验、注入风险、权限边界复核
  • 可观测性：必要的指标和埋点

P2（可选）：

  • 更完整测试矩阵（多端、压测、混沌测试）
  • 结构性重构（明显有债务才做）

第 7 步：分层测试

Section titled “第 7 步：分层测试”
  • 本地/单元：必做
  • 集成：涉及接口、数据库、外部依赖时必做
  • 预发/灰度：涉及真实流量或高风险时建议做

第 8 步：带护栏的发布

Section titled “第 8 步：带护栏的发布”
  • feature 分支开发，不直推主分支
  • 小步 commit（每个 task 1-2 个）
  • PR 包含：变更摘要、测试证据、风险点、监控点、回滚步骤、迁移脚本（如有）
  • CI 绿灯 + P0 通过才允许合并
  • 高风险变更：先开开关、小流量，再全量

第 9 步：上线后验证

Section titled “第 9 步：上线后验证”

部署后，按 Spec 的验收标准做一次线上验证：

  • 关键路径冒烟
  • 监控日志和指标有无异常（错误率、延迟、关键指标）
  • 有异常就按回滚方案执行，记录原因

第 10 步：完成定义

Section titled “第 10 步：完成定义”

满足这些条件才算”需求完成”：

  • Spec 对应的 P0 全通过
  • PR 合并、部署完成
  • 回滚方案可用且已验证
  • 上线后验证完成
  • 记录沉淀：链接到 PR、测试报告、验收报告

为什么有效

Section titled “为什么有效”

这套工作流解决的是实际问题：不是 AI 能力不够，而是 AI 执行缺乏结构。

一道强闸门（Spec）对齐意图。小步执行捕捉偏差。异常闸门防止失控。分层验收让验证可操作而不是穷举。回滚准备让发布安全。

AI 从一个混乱的代码生成器变成一个有纪律的执行者。你掌握方向盘，但不用微操每一行。

明天就试

Section titled “明天就试”

挑一个小功能。写结构化需求。让 AI 出 Spec。确认它。看看区别。

这套工作流来自 @123olp 在 X 上的帖子（2026-03-03）。实用、经过验证，用什么 AI 编码助手都行——Claude Code、Cursor、GPT，或者下一个出来的。

把输入结构化。确认一次。让它跑。

你装了 Copilot，买了 Claude Code 的订阅，团队里有人用 ChatGPT 写单元测试。你觉得”我们已经在用 AI 了”。

但你心里清楚，什么都没变。流程还是那个流程，开会还是那些会，测试还是手动点，上线还是最后一天通宵。AI 只是变成了一支更快的笔，写字的人还是在用老办法写。

这篇文章拆解一份 15 页的蓝图文档——《AI 原生研发团队蓝图》。它回答一个问题：从”给旧流程加个 AI 插件”到”围绕 AI 能力重建整条研发生产线”，中间到底差了什么？

“加个工具”和”换个活法”之间差了什么

Section titled ““加个工具”和”换个活法”之间差了什么”

蓝图一上来就画了两列对比。左边是”AI 辅助”，右边是”AI 原生”。

AI 辅助模式下，AI 是个人效率工具。人写代码、AI 帮忙补全。原有流程不变，质量靠人工保证，想扩产能就加人。

AI 原生模式下，AI 是团队的核心生产力。人的角色变成了定义者和验证者——中间那一大段执行交给 AI。流程围绕 AI 能力重新设计，质量靠自动化测试兜底，扩产能靠加 Token（用算力换人力）。

这两者的差别不是程度上的差异。它们是两套完全不同的生产关系。

线性分工正在瓦解

Section titled “线性分工正在瓦解”

以前的研发团队有一条清晰的链：产品 → 开发 → 测试。每个人守自己的一段。

AI 打断了这条链。纯执行型角色——只写代码、只做页面、只跑手动测试——正在被大规模替代。不是明天，是现在。

留下来的人只有两条路可走：

往上走。 去做架构、做产品方向。理解系统全局，做技术决策，设计 AI 和人的协作策略。

往深走。 去做质量定义、做测试策略。从手工执行者变成质量底线的制定者。

没有第三条路。中间地带正在消失。

产品经理：写 PRD 的能力不值钱了

Section titled “产品经理：写 PRD 的能力不值钱了”

这一点可能会让不少产品经理不舒服。

过去，画原型、写 PRD、生成方案——这些就是产品经理的核心产出。但 AI 把执行能力拉平了。一个初级产品经理用 AI 写出来的 PRD，和一个有五年经验的人写的，格式上看不出太大差别。

那什么还值钱？判断力。

判断什么该做、什么不该做。判断底层逻辑对不对。预判技术债务。决定”克制不做”。这些没法交给 AI。

蓝图里有一句话说得很直接：最重要的能力不再是把事情做成，而是决定什么事情该被做成。

新一代产品工作法叫”往下定义，往上判断”。交付物不再是功能清单，而是分层的产品定义——角色(Role)、旅程(Journey)、故事(Story)——加上每一层的价值判断。底层定义越清楚，AI 干活的地基越扎实。

开发者：你不能只是个码农了

Section titled “开发者：你不能只是个码农了”

蓝图把开发者的新工作循环画成了一个圈：明确定义 → 驱动 AI 生成 → 验证准确性 → 提交 PR/MR → 持续 Code Review → 回到明确定义。

注意，这个循环里”写代码”已经不是核心动作了。核心动作是定义和验证。

开发者有三条进化方向：

架构演进。 做系统整体设计、技术决策，设计 AI 协作策略。

产品融合。 深入业务，能独立做产品定义，能驱动 AI 从定义到端到端交付。

质量纵深。 深入质量工程，定义测试策略，保障 AI 开发的质量底线。

执行层用 Token 当杠杆。但要用好杠杆，你需要更强的架构理解力——能判断 AI 生成的代码是否符合规范，能保证向后兼容。

QA 的蜕变：从点点点到定义质量底线

Section titled “QA 的蜕变：从点点点到定义质量底线”

蓝图给 QA 的变化画了一张雷达图。传统 QA 和 AI 原生 QA 相比，五个维度的差距非常明显：开发背景、业务理解力、特定语言深度、质量定义能力。

AI 写代码非常快，但有个坑：它会产生”为了通过测试而通过”的幻觉。表面上测试全绿，底下可能藏着逻辑漏洞。

这意味着 QA 不能再做手工执行者。QA 必须变成质量底线的定义者，而开发背景（懂代码、懂框架）成了硬性门槛。

一句话总结：不在于测试多，在于测试精。

CTO 和架构师：分工要清楚

Section titled “CTO 和架构师：分工要清楚”

蓝图把技术领导层分成两个角色。

CTO / 技术负责人负责组织和战略：决定 AI 转型的节奏和优先级，调整团队结构比例，把控全局风险，关注组织变革。

架构师负责系统和协作设计：设计人与 AI 的协作体系，构建防 Token 浪费的项目脚手架，定义 Code Review 和向后兼容策略，关注规范体系。

过去，架构师产出的是方案文档。现在，架构师产出的应该是 “AI 可执行的项目规范体系”。从文档到指令集，这才是真正的升级。

新流水线：“定义-生成-验证”三明治

Section titled “新流水线：“定义-生成-验证”三明治”

传统流水线按功能模块排期。AI 原生流水线按用户旅程闭环交付。

三层结构：

定义层（人类主导）。 产品定义（用户旅程）、技术定义（架构规范/契约）、质量定义（测试策略）。

生成层（AI 主导）。 代码生成、测试用例生成、技术文档生成、自动 Code Review、重构执行。

验证层（人机协同）。 自动化回归验证、人工 PR/MR 审批、产品体验验收。

不再按功能模块排期，而是以用户旅程为单位进行整条链路的闭环交付。这是流水线层面最大的一个变化。

基础设施决定天花板

Section titled “基础设施决定天花板”

AI 再能干，没有基础设施也白搭。蓝图把 DevOps 和自动化测试网定义为 AI 生产力的基石。

三层基础设施：

  1. 自动化测试兜底。 API 接口测试 → E2E 核心旅程验证 → 流水线集成。这是最底层的基石。
  2. CI/CD 流水线。 高频并发支持，快速反馈。
  3. 代码托管与 AI 工作台。 GitLab/GitHub 权限管理，AI 负载均衡。

没有自动化测试跑通，闭环就会断裂。AI 写代码再快，出了 bug 没人兜底，快速迭代的优势直接归零。

DevOps 的身份也变了——从运维保障者变成 AI 生产力基础设施提供者。

规范就是 AI 的指令集

Section titled “规范就是 AI 的指令集”

这是蓝图里我觉得最有洞察力的一页。

工具链分四块：Claude Code / AI IDE 做编程执行，GitLab / GitHub 做代码托管，Swagger / OpenAPI 做接口契约。但最关键的是第四块——规范沉淀体系。

开发规范、架构说明、接口契约，这些文档现在有了新含义：它们是 AI 每次开工前必须读取的工作手册。

没有规范，AI 就是不可控的盲盒。你给它一个 prompt，它可能写出能跑的代码，但风格不统一、接口不兼容、和现有系统冲突。有了规范，AI 的产出才有一致性和可预测性。

规范即指令。这句话值得贴在每个技术团队的墙上。

人才结构也要跟着变

Section titled “人才结构也要跟着变”

蓝图画了一个三圈交叉的图：产品经理（战略与旅程定义）、开发工程师（架构与契约定义）、质量工程师（测试策略与底线定义）。三者中间的交集是 AI 执行层。

招聘策略的变化很明确：

大幅减少纯编码型开发岗位的招聘。增加具备扎实开发背景的 QA 人才。产品经理招人不再考核原型工具熟练度，优先考察业务洞察力和判断力。

简单说，三个角色都在向”产品+技术”复合体靠拢。纯粹只做一件事的岗位空间在收缩。

怎么落地：三阶段渐进式

Section titled “怎么落地：三阶段渐进式”

蓝图的变革要诀是八个字：不停业务，不大换血。

阶段一：试点验证（4-6 周）。 挑边界清晰的项目，不阻断核心业务。把 API 测试跑通，沉淀初始规范。跑完一套完整的 AI 辅助编写流程。试点是增量，不是对原有团队的直接替代。

阶段二：经验推广（4-8 周）。 把规范体系化并赋能全团队。切入第一个存量项目。按”补测试 → AI 修 Bug → 重构”的路径落地。

阶段三：常态化运作（持续）。 新项目默认走 AI 原生流程。PR/MR + 自动测试 + 人工审批成为肌肉记忆。架构规范由 AI 自主维护并持续迭代。

存量项目怎么办

Section titled “存量项目怎么办”

大部分团队不是从零开始的。手上一堆老项目、技术债、遗留代码。蓝图给出了四步渐进式重构的齿轮模型：

  1. 补测试。 先明确定义，固定代码边界。
  2. 自动化回归兜底。 建立安全防护网。
  3. AI 参与迭代。 让 AI 先处理 bug 修复和非核心需求。
  4. 架构梳理与重构。 借 AI 多线程能力并行清理技术债。

过去清理技术债既贵又阻碍业务。现在的逻辑变了：只要安全边界定义清楚，AI 可以自动化地、多线程地承担大规模重构工作。

七条法则，贴在墙上

Section titled “七条法则，贴在墙上”

蓝图最后一页是七条核心法则。我不逐条翻译了，直接说重点：

  1. 判断比生成更稀缺。 决定做什么，远比把东西做出来重要。
  2. 定义驱动，而非执行驱动。 人定边界与准则，AI 在框架内执行。
  3. 层次化思考。 从战略层推导到执行层，拒绝无结构的指令。
  4. 质量是绝对基石。 没有自动化测试网，AI 产出的代码就是巨大的负债。
  5. 渐进式转型。 通过试点稳步推进，绝不因为转型而停核心业务。
  6. 规范即指令。 你的开发文档、接口契约，就是 AI 的系统说明书。
  7. 产品不再默认存在。 在动手建构之前，先证明为什么不能用原生 AI 对话解决问题。

第七条特别有意思。它在说：不是所有需求都需要变成一个产品。如果一个对话就能搞定的事，为什么要花团队资源去开发？这是 AI 原生思维对”什么该做”这个问题的极端追问。

最后的话

Section titled “最后的话”

这份蓝图读下来，最大的感受不是焦虑，而是清晰。

焦虑来自”AI 要取代我了”这种模糊的恐惧。但蓝图把”到底什么会被取代、什么不会”说得很具体：纯执行被取代，定义和判断不会。手动点测试被取代，质量策略制定不会。写 PRD 被取代，决定该不该做不会。

清晰之后，下一步就是行动。不用一步到位。找一个边界清楚的项目，补上自动化测试，让 AI 跑一轮完整流程。看看哪里顺，哪里卡。

转型不是革命。是一步一步地，把旧齿轮换成新齿轮，让整台机器跑得更快。

AI 编程智能体的驾驭术：为什么上下文工程比换模型管用

Section titled “AI 编程智能体的驾驭术：为什么上下文工程比换模型管用”

你的团队上个月引入了一个 AI 编程智能体。第一周感觉很兴奋。到第三周，规律变得清晰：有时候智能体写出的代码拿来就能用，有时候代码看着合理但经不起审查。三个地方出现重复逻辑。utils 里明明有现成的工具函数，它从头写了一个新的。循环里藏着一个 N+1 查询。

你开始怀疑是不是选错了模型。

不是。模型从来不是问题。

大多数团队忽略的关键

Section titled “大多数团队忽略的关键”

Julian de Angelis 在 MercadoLibre（拉丁美洲最大的科技公司）负责 AI 编程智能体的推广，覆盖了近 20000 名开发者和上千个代码仓库。他在这个规模上得出的结论：

你的 AI 编程智能体就像入职第一天的新人。聪明、积极、手速快，但对你的代码库、编码规范和系统架构一无所知。

你不会给一个新工程师扔一张 Jira 卡片写着”修这个 bug”然后就走了。你会做入职培训：给文档，介绍技术栈，讲清楚代码风格，说明发布流程，开通工具权限，安排人做代码审查。

这个入职体系就是 de Angelis 所说的”智能体驾驭系统”（agent harness）——围绕智能体搭建的一整套结构化控制，决定它知道什么、能做什么、输出怎么验证。背后的底层学科叫上下文工程（context engineering）：设计和控制 LLM 在生成任何内容之前所能看到的一切。

没人说的那个约束条件

Section titled “没人说的那个约束条件”

所有主流 AI 编程智能体——Claude Code、Cursor、Codex——都在跑同一个基本循环：

读取 -> 规划 -> 编码 -> 验证 -> 迭代

智能体读你的文件，规划方案，写代码，跑测试，失败了就回头重来。每轮迭代，上下文窗口都在增长。系统提示词、工具定义、对话历史、文件内容、终端输出、智能体自己生成的代码——全部堆积在一个有限的工作空间里。

让人意外的部分来了：上下文窗口用到大约 60% 的时候，继续塞入更多上下文反而会让智能体表现更差。这种现象叫上下文腐化（context rot）。即使窗口扩到 100 万 token，这个问题依然存在。每一个花在冗长工具输出或过长规则文件上的 token，都是从实际任务的可用空间里偷走的。

好的驾驭系统围绕这个现实来设计：在正确的时机注入正确的上下文，其余一概不放。

四个真正管用的杠杆

Section titled “四个真正管用的杠杆”

1. 自定义规则

Section titled “1. 自定义规则”

CLAUDE.md、cursor/rules、AGENTS.md 这类文件会在每次交互开始时注入智能体的上下文。这是最容易上手的杠杆。

该写什么：

  • 技术栈和架构模式
  • 命名规范和代码风格
  • 测试要求（“所有单元测试必须使用表驱动输入”）
  • 代码库里常见的坑
  • 你已经看到智能体反复犯的错误

不该写什么：

  • 整份 API 文档（太长，浪费上下文）
  • 正确的废话（“写干净的代码”）
  • 自相矛盾的规则

实操建议：

  • 控制在 500 行以内。精准比全面更有用。
  • 按关注点拆分成多个文件：架构、测试、代码风格、安全。智能体只加载需要的部分。
  • 放几个具体示例。模型从例子中学到的东西比抽象指令多得多。
  • 把它当成活文档。智能体每次犯错，就更新规则。几周之后这个文件会变得非常有效。

2. MCP 服务

Section titled “2. MCP 服务”

MCP（Model Context Protocol）服务是给智能体扩展文件读写之外能力的插件。没有它们，智能体只能看到仓库里有什么。有了它们，智能体能够：

  • 查询数据库，理解表结构和数据
  • 搜索内部文档和知识库
  • 查找 API 接口约定
  • 和 CI/CD 流水线交互
  • 访问 Figma 上的设计稿
  • 用真实系统验证自己的输出

MCP 服务弥合了智能体和组织知识之间的鸿沟。让它获取人类开发者日常使用的同一套资源。

3. 技能（Skills）

Section titled “3. 技能（Skills）”

技能给智能体注入新的能力和领域知识。一个技能是一个目录，包含入口文件，可以捆绑模板、示例输出、参考文档和可执行脚本。

设计上的巧妙之处：只有一段简短描述常驻上下文，让智能体知道有这个技能可用。完整内容仅在技能被调用时加载——用户手动触发或智能体检测到相关性后自动加载。按需加载意味着技能可以写得足够详细，而不会长期占用上下文空间。

技能分两种：

  • 知识型技能注入上下文：编码规范、设计模式、业务领域知识
  • 任务型技能提供具体操作步骤

真正的威力在于技能能捆绑并执行脚本，在独立的子智能体（拥有干净上下文窗口）中运行，还能互相组合。可编程、上下文感知、可组合的智能体行为单元。

4. 规格驱动开发（Spec-Driven Development）

Section titled “4. 规格驱动开发（Spec-Driven Development）”

这个杠杆会改变你和智能体交互的整个方式。

看看这个提示词：“做一个新功能，从后台管理添加新商品。”

看起来很简单。但什么都没说清楚：用什么技术栈，后台在哪里，API 接口怎么定，数据存哪里，幂等性怎么处理，权限控制怎么做。

智能体会读代码库，规划一个合理的方案，产出一个能跑的 MVP。你添加一个商品，成功了。你再点一次”添加”，同一个商品插入了两次。用 ID 做幂等”应该是”常识。在你们的业务里，只有特定角色能添加商品”应该是”常识。

问题不在智能体，在于输入太粗糙。瓶颈卡在椅子和屏幕之间。

规格驱动开发的解法是：在智能体写代码之前，先写详细的规格文档。功能做什么，怎么和现有代码集成，边界情况有哪些，验收标准是什么，测试计划怎么安排。规格文档本身就是驾驭系统——它把架构决策、执行指南和审查标准浓缩成一份文档，一次性工程化了智能体的整个上下文窗口。

而且，你可以用智能体来写规格文档。

反馈闭环

Section titled “反馈闭环”

四个杠杆定义了智能体知道什么、能做什么。让整个系统收敛的关键是反馈闭环：测试、代码检查、类型检查、构建脚本。每一个产出”通过/失败”信号的工具都是自我纠正机制。

智能体钩子（agent hooks）让这件事变得强制可执行。钩子是在智能体生命周期特定节点自动执行的用户定义命令。把验证逻辑挂到 Stop 钩子上，智能体在检查通过之前无法结束。不是它可能忽略的建议，而是一道硬性关卡。

可用的结构化反馈越多，需要的人工干预就越少。

MercadoLibre 的 20000 人实战手册

Section titled “MercadoLibre 的 20000 人实战手册”

9 种以上技术栈，上千个遗留仓库，多个业务领域。以下是他们在实际规模上验证有效的做法：

按技术栈标准化规则。 在组织层面维护一套经过精心调校的自定义规则，覆盖每个技术栈、内部类库和代码审查标准。各团队继承基础规则后叠加仓库级定制。精确度的把控很关键——规则太细会把遗留项目搞坏，太松会放过未经批准的第三方库（安全隐患）。

内部 MCP 服务。 接入内部云平台能力、内部 SDK、通过 RAG 提供的业务上下文，以及持续更新的应用文档。

自定义代码审查智能体。 在 CI 流水线中运行的独立审查智能体，分析每一个 PR——不管是人写的还是 AI 生成的。输出按优先级排序，工程师先看到最关键的问题。人类审查者把精力放在架构设计上，不用再花时间抓代码风格违规。审查周期缩短，代码质量更一致，安全网的扩展跟着 PR 数量走，不跟着审查人数走。

从这里开始

Section titled “从这里开始”

如果你还没搭建智能体驾驭系统，从最简单的杠杆开始：自定义规则。创建一个 CLAUDE.md 或等效文件，写入你的编码规范、常见陷阱，加两个代码库里好模式和坏模式的具体例子。

在 AI 编程智能体这场竞赛里，赢的不是用最新模型的团队，而是有最好驾驭系统的团队。

一个 AI 模型，能替代彭博终端、注册分析师和一整个周末的 Excel 折腾吗？Claude 4.5 已经非常接近这个目标了。下面这 8 条提示词，是把它变成你的私人股票研究分析师的具体方法。

为什么现在可以用 AI 做选股研究？

Section titled “为什么现在可以用 AI 做选股研究？”

过去做个像样的个股分析，你得：翻年报、跑估值模型、查内部人交易记录、看行业对比数据。要么花钱买 Wind/Bloomberg，要么花时间手动扒。

Claude 4.5 带来了两个变化：

  1. 推理深度足够处理多步骤的财务分析
  2. 上下文窗口足够长，能在一次对话里完成完整的筛选流程

不是说 AI 能替你做投资决策。但它可以把”信息收集 + 初步筛选 + 框架搭建”这部分工作压缩到几分钟。

8 条提示词，覆盖选股研究的核心维度

Section titled “8 条提示词，覆盖选股研究的核心维度”

以下提示词来自 @0xluffy_eth 的分享，我做了整理和补充说明。

1. 低估值选股器

Section titled “1. 低估值选股器”

用途： 从大盘里快速筛出基本面好但被低估的公司。

提示词：

作为专业股票研究分析师。扫描当前股市找出10家基本面强劲的低估值公司。应用这些筛选条件：P/E低于行业平均、过去3-5年收入和盈利稳定增长、债股比低于行业中位数、正向且增长的自由现金流、ROIC高于行业平均、分析师共识上升至少30%。

为每家公司提供：业务概述、为何低估、关键风险、内在价值范围估算。

使用建议： 可以在提示词后面加上具体行业或市场范围（比如”A股医药行业”或”美股科技板块”），让结果更聚焦。

2. 内部交易模式分析

Section titled “2. 内部交易模式分析”

用途： 追踪公司管理层的买卖行为，判断他们对自家公司的真实态度。

提示词：

作为内部交易分析师。审查选定行业最近的内部交易活动。识别以下公司：多位内部人员购买股份、公开市场购买、过去90天内购买、购买规模相对于内部人员薪酬具有意义。

返回前5家公司并解释：谁在购买和购买多少、历史影响、这反映的管理信心、任何潜在警示。

使用建议： 内部人买入比卖出更值得关注。卖出可能是流动性需求，但买入几乎只有一个原因：他们觉得便宜。

3. 市场情绪与现实差距

Section titled “3. 市场情绪与现实差距”

用途： 找出被市场情绪过度打压、但基本面其实没那么差的股票。

提示词：

作为反向股票分析师。识别遭大幅抛售或负面评价但基本面强劲的股票。

对每只股票分析：当前负面叙述、实际财务表现、问题是否临时性、相对历史平均的估值。

返回5只市场情绪与基本面最不一致的股票。

使用建议： 这条提示词的价值在于强制你从”反共识”角度思考。但要注意区分”被错杀”和”真出问题”——AI 给的是分析框架，最终判断还得自己来。

4. 股息贵族 ROI 计算器

Section titled “4. 股息贵族 ROI 计算器”

用途： 评估长期股息投资的真实回报，不光看收益率，还看可持续性。

提示词：

作为股息投资组合经理。分析连续25年增长股息的股息贵族。

对每家公司计算：假设股息再投资的过去10年总回报、当前股息收益率、股息增长率、派息比可持续性、自由现金流覆盖。

按收入可靠性和长期总回报排名。

使用建议： 适合长线投资者。关键指标是派息比和自由现金流覆盖——如果公司借钱发股息，增长记录再漂亮也不靠谱。

5. 科技股炒作 vs 基本面

Section titled “5. 科技股炒作 vs 基本面”

用途： 在科技板块里分辨谁是真增长、谁是纯估值泡沫。

提示词：

作为估值专家技术分析师。比较主要上市科技公司。对每家公司评估：收入增长vs估值倍数、利润率、自由现金流、资本效率。

识别：3只定价过度乐观增长的股票、3只相对基本面被低估的股票。解释市场错误定价。

使用建议： 在 AI 概念股泛滥的今天，这条提示词特别有用。很多公司把”AI”写进财报但收入没跟上，这个提示词能帮你分辨。

6. 行业轮动信号检测器

Section titled “6. 行业轮动信号检测器”

用途： 从宏观经济数据推导出哪些行业接下来可能走强。

提示词：

作为宏观投资策略师。分析当前宏观经济指标：利率趋势、通胀数据、GDP增长、就业条件。

基于此分析：识别未来6-12个月可能跑赢的行业、可能跑输的行业、每个观点的经济逻辑、可能失效的风险。

使用建议： 行业轮动是自上而下的分析方式。配合第1条（自下而上选股），可以先确定行业方向，再在目标行业里找个股。

7. 小盘股增长识别器

Section titled “7. 小盘股增长识别器”

用途： 在被大机构忽视的小盘股里找到高增长潜力标的。

提示词：

作为小盘股增长投资者。找出市值低于20亿美元的上市公司，满足条件：年收入增长超20%、营业利润率扩大、强劲资产负债表、内部人员持股、机构持股低。

包含：业务模式、增长驱动、执行风险、被忽视原因。

使用建议： 小盘股信息不对称更严重，正好是 AI 能帮你补信息差的领域。但流动性风险也更大，仓位控制要做好。

8. 风险调整回报优化器

Section titled “8. 风险调整回报优化器”

用途： 给定预算和风险偏好，让 AI 帮你设计投资组合。

提示词：

作为投资组合构建专家。投资组合规模：5万美元，风险偏好：保守/温和/激进，时间：指定年限。

构建多元化投资组合以最大化风险调整回报。包含：按行业资产配置、持仓规模理由、预期回报、再平衡规则、下行保护。

使用建议： 把金额、风险偏好和投资期限改成你的实际情况。这条提示词的价值不在于让你照搬结果，而在于看 AI 的配置逻辑，跟你自己的想法做对比。

使用注意事项

Section titled “使用注意事项”

这不是投资建议。 以下几点需要记住：

  • Claude 的训练数据有截止日期，不包含实时行情。它给的”筛选结果”是基于训练数据里的信息，不是当前市场快照
  • 所有财务数字都需要你去二次验证。AI 可能会编造看上去合理但实际不存在的数据
  • 这些提示词的最大价值是给你一个分析框架，而不是直接告诉你买什么
  • 在做真金白银的决策之前，至少要交叉验证两个独立数据源

Section titled “进阶用法”

没人在填的那个空白

Section titled “没人在填的那个空白”

他们不知道的是：Claude Projects、Claude Memory、Claude Skills、MCP 集成。这些工具组合起来，能让 Claude 从一个聊天机器人变成接近业务操作系统的东西。

这个信息差，就是机会。

Section titled “生意模式：Claude 迁移服务”

录下他们的工作流程。 用 Wispr Flow 这类工具转录日常操作，旁听他们开会，把那些靠肌肉记忆在跑、从来没写成文档的 SOP 全部记录下来。

上下文开发生命周期：你的 AI 编码智能体没坏，是你的上下文烂了

2026年3月4日
Vibe Sparking

上下文开发生命周期：你的 AI 编码智能体没坏，是你的上下文烂了

Section titled “上下文开发生命周期：你的 AI 编码智能体没坏，是你的上下文烂了”

你升了最新的模型。你写了更好的提示词。你甚至认认真真搞了个 CLAUDE.md。结果你的 AI 编码智能体还是把认证模块写错了——这周第三次。

不是模型的问题。是你给它的上下文不行。

一个刚被打破的假设

Section titled “一个刚被打破的假设”

软件行业花了几十年优化人写代码的速度。CI/CD 流水线、敏捷迭代、结对编程、代码审查自动化。所有这些都建立在一个前提上：开发者的产出速度是瓶颈。

AI 编码智能体打破了这个前提。它们能搭服务、修 bug、交付功能。代码生产速度不再是制约因素。

但别的东西成了。

每次你开一个新的 AI 智能体会话，你都在给一个”什么编程知识都会、但对你项目一无所知”的新人做入职培训。你的架构决策、命名规范、那个认证流程为什么要那样写、哪些数据库字段其实已经废弃了——这些东西模型自己不会带。

瓶颈转移了。大部分团队还没意识到。

上下文现在住在哪

Section titled “上下文现在住在哪”

老实看看你团队的上下文都散落在什么地方：

  • 一个 .cursorrules 文件，三个月前有人写的。API 重构之后没人动过。
  • 一份 CLAUDE.md，从某篇博客复制过来的。里面写着”React 最佳实践”，但你们一月份就迁到 Svelte 了。
  • Slack 里一条消息，你们的高级工程师解释了支付流程。一半人在另一个时区，根本没看到。
  • 一份架构决策记录，躺在没人收藏的 Confluence 页面上。

这就是 2026 年上下文管理的现状。碎片化、没版本号、没测试、在角落里悄悄腐烂。

DevOps 的既视感

Section titled “DevOps 的既视感”

Patrick Debois 在 2009 年发明了”DevOps”这个词。他当时的洞察很简单：开发和运维不能分开。把它们当成两个有不同目标的团队来对待，只会制造摩擦，拖慢所有人。

他现在看到了同样的模式。

人的意图和智能体的执行不能分开。你没法在不改善指令质量的前提下改善智能体的输出。而这些指令——也就是上下文——需要和代码交付同等程度的工程化管理。

DevOps 给了交付一个生命周期。上下文也需要一个。

上下文开发生命周期（CDLC）

Section titled “上下文开发生命周期（CDLC）”

Debois 提出四个阶段。合在一起就是上下文开发生命周期——CDLC。

1. 生成（Generate）：把脑子里的东西写出来

Section titled “1. 生成（Generate）：把脑子里的东西写出来”

让你团队厉害的大部分知识是隐性的。高级工程师知道不要用那个 ORM 方法，因为会触发 N+1 查询。技术负责人知道计费服务在高负载下有竞争条件。设计师知道品牌规范里写了”蓝色底上绝对不能再放蓝色”。

这些东西没有一条是用智能体能理解的方式写下来的。

生成意味着捕获三个层面：

  • 技术上下文。 编码规范、类库使用模式、架构决策、已知的坑。
  • 项目上下文。 当前优先级、范围边界、正在进行什么、什么被阻塞了。
  • 业务上下文。 系统为什么存在、客户期望什么、合规约束、成功长什么样。

智能体可以帮你起草这些规范。但什么是对的，人来定。这不是为了写文档而写文档——这是你的智能体要执行的指令集。

2. 评估（Evaluate）：给上下文写测试

Section titled “2. 评估（Evaluate）：给上下文写测试”

你不会不写测试就发布代码。那为什么不测就发布上下文？

评估意味着通过场景来定义期望的智能体行为。“在这个上下文下，智能体应该用 Repository 模式做数据访问。""在这个上下文下，智能体不应该直接调用支付 API。""在这个上下文下，错误信息应该带 correlation ID。”

智能体做错的时候，先别急着怪模型。那个错误是你没写出来的规范。

结果是统计性的，不是确定性的。大语言模型不会每次都产出一样的东西。多跑几次。看规律。

3. 分发（Distribute）：上下文是基础设施

Section titled “3. 分发（Distribute）：上下文是基础设施”

上下文被共享的时候才有力量。但共享需要基础设施。

想想代码依赖是怎么运作的。npm、pip、cargo——它们给你版本管理、完整性校验、更新机制。上下文需要同样的待遇。

没有注册中心、没有版本号、没有推送更新的机制，上下文就会腐烂。六个月前有人写的一个 skill，写在那个 API 破坏性变更之前——它正在教智能体错误的做法。没人知道。

这里的激励结构比写文档好得多。文档是苦差事，好处延迟而且分散。上下文分发直接改善每个开发者从智能体那里得到的结果。反馈回路很紧。

4. 观察（Observe）：闭合循环

Section titled “4. 观察（Observe）：闭合循环”

部署智能体。观察它们的行为。

智能体问澄清问题的时候，那是你上下文里的空白。它做出意外的架构选择的时候，那是你没解决的歧义。它写出了能跑的代码但偏离了意图的时候，那是你没说出来的假设。

这些信号是金子。它们精确地告诉你上下文在哪里不够、矛盾或过时。把它们反馈到生成阶段。补充评估。重新分发。

循环闭合：生成 → 评估 → 分发 → 观察 → 生成。

无限上下文窗口救不了你

Section titled “无限上下文窗口救不了你”

“我们现在有百万 token 的上下文窗口了。全塞进去就行。”

更大的窗口缓解了容量限制。它不解决质量问题。

把你过时的 .cursorrules、当前的架构文档、老的入职指南、三条互相矛盾的 Slack 消息全塞进同一个上下文窗口，不会让智能体更聪明。它只是有了更多可以被搞糊涂的素材。

问题从筛选（放什么进去？）变成了治理（放进去的东西一致吗？是最新的吗？是对的吗？）。无限窗口让治理更难，不是更容易。

复利效应

Section titled “复利效应”

这就是为什么这个事在战略上很重要。

在上下文质量上投入的团队会形成飞轮。更好的上下文产出更好的智能体输出。更好的输出产生更清晰的信号——什么有效、什么没效。更清晰的信号产出更好的上下文。

这在四个方向上产生复利：

  1. 智能体质量。 不管谁来跑，每个编码智能体都能正确处理业务领域的东西。
  2. 团队知识。 形式化上下文的过程迫使工程师理清自己的理解。
  3. 入职速度。 新人从第一天就继承结构化的上下文——他们的智能体也一样。
  4. 组织一致性。 分发的上下文让团队保持一致，不需要中央强制。

这种优势很难复制。它建立在你组织自己的知识积累之上。竞争对手可以用同样的模型。但他们复制不了你的上下文。

周一早上该做什么

Section titled “周一早上该做什么”

如果你看到这里在想从哪下手：

  1. 盘点现有上下文。 找出团队用的每一个 .cursorrules、CLAUDE.md、AGENTS.md 和规范文件。查看上次更新是什么时候。查看它们之间是否矛盾。
  2. 挑一个痛点。 智能体老是写错的那个模块？针对那个具体领域写更好的上下文。写具体的。
  3. 写一个评估。 定义那个模块的”正确智能体行为”是什么样的。跑一下。看什么过不了。
  4. 版本化。 如果上下文还没在版本控制里，放进去。对上下文的改动和对代码的改动用同样的审查流程。
  5. 看结果。 改善了某个领域的上下文之后，衡量智能体输出质量是否提高了。如果提高了，你的概念验证就有了。

那个问题

Section titled “那个问题”

赢得 AI 辅助开发时代的团队，区别不在于用了哪个模型。每个团队都能用到前沿模型。

区别在于它们给模型提供的上下文质量。严格地生成。系统地评估。可靠地分发。持续地改进。

问题不是你的团队在不在用 AI 编码智能体。问题是，你在让智能体把活干好这件事上，投入了多少。

VSDD：AI 写的代码，谁来查作业？

2026年3月3日
Vibe Sparking

没人说的问题

Section titled “没人说的问题”

AI 写了一个函数。生成了测试。测试通过了。代码审查也没问题。

发版？

我反复看到一个现象：半年后用户撞上了一个边界条件。函数处理 null 没问题，但并发访问时挂了。测试覆盖了正常路径和两三个异常路径，偏偏漏掉了输入合法但病态巨大的情况。

AI 没有写错代码。它写了”差不多对”的代码。“差不多对”比”明显错”更难抓，因为它能通过所有为了抓”明显错”而设计的检查。

行业默认的舒适区

Section titled “行业默认的舒适区”

大多数团队用 AI 写代码的流程差不多：

  1. 描述需求
  2. AI 写代码（可能顺便写测试）
  3. 人看一眼，改一改，发版

有些团队加了 TDD。有些让另一个 AI 或者人做 code review。但底层假设没变：测试通过了、reviewer 觉得还行，就够了。

这个假设有盲区。

盲区在哪：AI 太客气了

Section titled “盲区在哪：AI 太客气了”

标准的 LLM 交互天生合作导向。AI 想帮你。想和你达成一致。它审查代码的时候——哪怕审查自己写的代码——起手就带着善意。

“整体看起来不错，以下是一些小建议……”

这不叫审查。这叫夸你之后顺带提两句。

一次长对话里，AI 记得这代码是自己写的，记得你们讨论过的设计决策。它和这个代码库有了”关系”。有关系就有盲区。

什么是 VSDD

Section titled “什么是 VSDD”

VSDD（Verified Spec-Driven Development）来自开源社区，三位贡献者是 dollspace-gay、gitcnd 和 jtremback。他们把三套大多数团队当成互斥选项的方法论叠成了流水线上的三道闸门：

规格驱动开发 (SDD)： 先写契约再写代码。不光定义代码做什么，还要定义什么东西必须是可证明的。

测试驱动开发 (TDD)： 先写测试再写代码。红灯、绿灯、重构。没有失败的测试，就没有新代码。

验证驱动开发 (VDD)： 把活下来的代码扔给一个零容忍的对手反复审查，直到对手被迫编造不存在的缺陷。

三道闸门。规格定义”做什么”。测试执行”怎么做”。对抗性验证确保”没漏掉什么”。

四个角色

Section titled “四个角色”

架构师（人类开发者） — 战略视野、领域专业知识、最终拍板权。你签字确认规格，仲裁 Builder 和 Adversary 的争议，做 AI 做不了的判断。

建造者（Claude 或同类） — 写规格、生成测试、实现代码、做重构。严格 TDD 约束。不许”我先把整个东西写出来再补测试”。

追踪者（Chainlink） — 分层 issue 管理。每个规格条目、测试用例、代码实现都映射到一个可追踪的工作单元。

对手（Sarcasmotron） — 一个不同的 AI 模型，配置为零容忍。审查规格、测试、代码实现。每次审查都用全新的上下文窗口。没有关系漂移，没有累积的人情。

对手是整个方法论里最关键的创新。用不同的模型家族（比如 Builder 用 Claude，Adversary 用 Gemini），引入真正的认知多样性。对手不关心你的感受，只关心你的不变量。

六个阶段

Section titled “六个阶段”

阶段一：把一切写进规格

Section titled “阶段一：把一切写进规格”

写代码之前，Builder 输出两样东西：行为规格和验证架构。

行为规格包含前置条件、后置条件、不变量。边界条件目录列举 null、空、最大值、负数、Unicode、并发等所有边界情况。

验证架构里最关键的一项是”纯度边界图”——把确定性的纯核心和有副作用的外壳（I/O、数据库、网络）明确分开。

为什么这很重要？一个从数据库读数据、做计算、写日志全放在一个函数里的设计，形式化验证工具拿它没办法。但一个接收数据、返回结果、让调用者处理持久化的纯函数——model checker 能推理它。

这个边界必须在规格阶段就画好。因为它决定了模块划分、依赖方向和后面的整个测试策略。

Adversary 在任何测试写出来之前就审查规格。它找歧义、找缺失的边界条件、找没说出来的隐含假设、找自相矛盾的地方。

阶段二：红灯、绿灯、重构

Section titled “阶段二：红灯、绿灯、重构”

规格锁定之后，Builder 写测试。只写测试，不写实现。每个后置条件变成一个断言，每个边界条件变成一个测试。

所有测试必须先失败。如果某个测试没写实现就通过了，这个测试有问题。

然后：拿下一个失败的测试，写刚好让它通过的最少代码，跑全套测试，重复。经典 TDD，由 AI 执行。

阶段三：过火焰山

Section titled “阶段三：过火焰山”

通过测试的代码库连同规格和测试一起交给 Adversary。全新的上下文窗口。

Adversary 查什么：

  • 实现真的满足规格了吗？还是测试把一个误解编码进去了？
  • 有没有同义反复的测试——实现稍微改一下它也照样通过？
  • 隐藏耦合、竞态条件、漏掉的资源清理
  • 输入校验缺口、注入向量
  • 代码里有的行为但规格里没提到的

不许写”整体不错”。每条反馈必须是一个具体缺陷，带具体位置。

阶段四：反馈循环

Section titled “阶段四：反馈循环”

Adversary 的意见按类型回流：

  • 规格级缺陷 → 回到阶段一，改规格，重新审查
  • 测试级缺陷 → 修测试，确认它会失败，再修实现
  • 新发现的边界条件 → 加进规格，写失败测试，写实现

循环到收敛为止。

阶段五：形式化验证

Section titled “阶段五：形式化验证”

因为架构从阶段一开始就是按可验证性设计的：

  • 形式化证明（Kani、Dafny、TLA+）
  • 结构化模糊测试（AFL++、cargo-fuzz）
  • 变异测试（mutmut、Stryker）
  • 纯度边界审计

阶段六：收敛

Section titled “阶段六：收敛”

四个维度同时收敛才算完：

维度收敛信号
规格Adversary 挑的毛病只剩措辞问题
测试Adversary 找不出有意义的未测试场景
实现Adversary 被迫捏造代码中不存在的问题
验证所有形式化证明通过，fuzzer 啥也没找到

这叫”幻觉终止”——对手必须靠幻觉来凑缺陷，因为真的缺陷已经找不到了。达到这个状态的软件被称为 “Zero-Slop”。

对 AI 辅助开发意味着什么

Section titled “对 AI 辅助开发意味着什么”

核心洞察：AI 的礼貌是系统性风险。

用合作型 AI 去审查合作型 AI 的产出，就像让朋友给你批改作业。能找到一些错，但需要对抗性思维才能发现的错——找不到。

VSDD 让 AI 对抗 AI。不同模型，新鲜上下文，零容忍。这不是 code review，这是压力测试。

而人在这个过程中不是被替代了。你是架构师——审批规格、仲裁争议、做 AI 无法替代的判断。你的角色是被抬高了。

什么时候用，什么时候不用

Section titled “什么时候用，什么时候不用”

VSDD 仪式感很重。值得投入的场景：

  • 正确性不能妥协的系统（金融、医疗、基础设施）
  • 需要长期维护的代码库
  • 安全是核心关注点
  • 团队有多个可用的 AI 模型

快速原型或者一次性脚本不需要这套。但即使只加阶段三——拿另一个 AI 做一轮零容忍审查——也能抓到不少隐藏债务。

怎么开始

Section titled “怎么开始”

不用明天就搞全六个阶段。先加一个：

做一次对抗性审查。 下次完成一个功能后，开一个新对话，用不同的 AI 模型。给它零容忍指令。把规格、测试、代码全贴进去。看它找出什么。

大多数试过的开发者会对结果感到不舒服。这种不舒服就对了——说明现有流程有盲区，而你现在知道盲区在哪了。

完整可追溯性

Section titled “完整可追溯性”

VSDD 最强的属性之一：每个产物都能追溯。

问”这行代码为什么存在？“能追溯到一条具体的规格需求。问”这个模块为什么是纯函数？“答案追溯到阶段一的纯度边界图。

没有无缘无故的代码。没有缺少证据的理由。

执行摘要

Section titled “执行摘要”

英伟达Q4 FY2026交出了又一份爆表财报：收入$681亿（YoY +73%），数据中心收入$623亿（YoY +75%），净利润$430亿（YoY +94%）。全年FY2026收入$2,159亿，自由现金流$966亿，堪称人类商业史上最惊人的增长曲线之一。Q1 FY2027指引$780亿更是大幅超出华尔街$728亿的预期。

然而，这份看似完美的财报背后，市场在盘后下跌约5.5%，$2,600亿市值蒸发。这说明：当前股价已经定价了”完美执行”的预期，容错空间极小。

TL;DR:

  • 推荐动作: HOLD / 逢回调加仓——当前价位（~$177）对应Forward PE约22-30x，不算离谱但也谈不上便宜。信心水平：中高
  • 最核心的Key Force: AI推理（Inference）需求的指数级增长正在从训练（Training）接棒，成为英伟达收入的主引擎（>70%数据中心收入来自推理）
  • 最大风险 / Kill Condition: 自研ASIC芯片（Google TPU、Amazon Trainium、Broadcom定制芯片）在推理市场的份额蚕食加速。若2026年ASIC推理份额突破30%，英伟达的定价权将面临实质性压力
  • 估值 vs 当前价格: 基准情景合理价值$190-$220，隐含IRR 15-25%；熊市情景$120-$140

Key Forces（决定性力量）

Section titled “Key Forces（决定性力量）”

Key Force #1: 推理需求指数级增长——从”训练军备竞赛”到”推理即收入”

Section titled “Key Force #1: 推理需求指数级增长——从”训练军备竞赛”到”推理即收入””

Jensen Huang在财报电话会上的核心论断：“在AI的新世界里，算力等于收入（Compute equals revenues）。没有算力，就没有token生成；没有token，就没有收入增长。”

这不是空洞的口号。英伟达数据中心收入中，超过70%来自推理工作负载，而非训练。Agentic AI（智能体AI）的兴起意味着每一次用户交互都需要大量推理算力——与传统SaaS”一次训练、反复使用”的模式截然不同。

Grace Blackwell + NVLink架构在推理端实现了”每token成本降低一个数量级”，这是真正的产品力护城河。Vera Rubin平台（预计2026下半年量产）将进一步用四分之一的GPU数量训练MoE模型，推理token成本再降10x。

量化证据: CFO Colette Kress透露，从2025年初到2026年底，公司已有$5,000亿的Blackwell和Rubin订单能见度（visibility）。

这个Key Force为什么最重要: 如果推理需求确实是指数增长的，英伟达的收入天花板远高于市场当前定价。$780亿的Q1指引意味着年化>$3,100亿，而这可能只是推理Super Cycle的开端。

Key Force #2: ASIC替代——推理市场的”特洛伊木马”

Section titled “Key Force #2: ASIC替代——推理市场的”特洛伊木马””

英伟达在训练市场的统治地位（>95%份额）短期内难以撼动。但推理市场是另一个战场：

  • Google TPU第7代Ironwood已经在技术指标上接近甚至部分超越Blackwell
  • Broadcom与Anthropic达成$210亿的TPU芯片供应协议
  • 自研ASIC出货量预计2026年增长44.6%，远超GPU的16.1%增速
  • 推理对成本更敏感，定制ASIC可以针对特定模型架构做深度优化

Anthropic宣布将在100万颗TPU上训练Claude——这是一个标志性事件，说明顶级AI实验室正在认真评估非英伟达方案。

反论点: 英伟达的CUDA生态系统仍是最大护城河。但如果PyTorch/JAX等框架的硬件抽象层持续改善，CUDA的锁定效应将逐步减弱。预计到2028年，英伟达在推理市场的份额可能从当前的70%降至50-60%。

Section titled “Key Force #3: 中国出口管制——$2,000亿TAM的”半开半关””

阶段二：启动工作智能体。 每个单元分配一个后台智能体，各自在隔离的 git worktree 中运行。每个智能体完成自己的任务后，对改动运行 /simplify，跑测试，提交代码，开 PR。

阶段三：进度跟踪。 一张状态表实时更新，展示每个工作单元的完成情况和 PR 链接。

Section titled “用法”

别再吵了：AMDD 和「刚刚好」的建模艺术

2026年3月2日
Vibe Sparking

我见过的大多数敏捷团队，要么花三个月画 UML，画完没人看；要么把”可工作的软件高于详尽文档”理解成”直接写代码”，然后认证模块重构了四遍还没稳定。

两拨人都觉得对面是问题所在。两拨人都搞错了。

一个持续了几十年的伪命题

Section titled “一个持续了几十年的伪命题”

模型驱动开发（MDD）的逻辑是：先把所有事情想清楚，画完图再动手。完整的模型、穷举的规格说明、面面俱到的架构图。然后才写代码。

敏捷阵营的反驳：这不现实。需求在变，市场在变，用户自己都不知道想要什么。先做出来再说。

两边都有道理。两边走到极端都会出事。

MDD 团队花几个月交付的规格说明，等写完了产品方向已经变了。不做设计的敏捷团队，技术债越堆越高，最后系统扛不住自身的重量。两边互相指着说：“看，不听我的就是这个下场。“

跑得快的团队到底在干什么

Section titled “跑得快的团队到底在干什么”

去观察一个真正高效的敏捷团队——不是那种把瀑布流程改个名字的。你会看到一个有意思的场景：他们没有跳过建模。但也没在建模上花好几周。

典型画面：两个开发者站在白板前。一个人画了新服务和现有组件的交互草图。另一个人质疑了一个假设。他们争论了大约十分钟，擦掉一半，达成一致，回到工位写代码。一个 Sprint 里这种事可能发生三四次。

这就是敏捷模型驱动开发，AMDD。Scott Ambler 给它起了这个名字，但好的团队在有名字之前就已经这么干了。

核心概念：刚刚好就够了

Section titled “核心概念：刚刚好就够了”

AMDD 的关键词就一个：just barely good enough，刚刚好就够了。

不是”不画图”。不是”画完美的图”。是画到能往前走为止，多一张图都算浪费。

整个生命周期分四块：

1. 初始构想（Envisioning）

Section titled “1. 初始构想（Envisioning）”

项目开头花几天时间，梳理高层需求和架构方向。回答两个问题：“我们要建什么？“和”大致怎么建？”

白板上画个架构草图。用户故事写在卡片上。画个粗略的领域模型。够了。这些信息足够你做初步成本估算和团队组织。

不管项目多大，别在这上面花超过两周。时间拉得越长，你在没有实际编码反馈的情况下积累的风险就越大。

2. 迭代建模（Iteration Modeling）

Section titled “2. 迭代建模（Iteration Modeling）”

每个 Sprint 开始时，建模到够估算和计划的程度。这是本次 Sprint 需求的分析和设计，发生在计划会里。不是单独的阶段。

3. Model Storming

Section titled “3. Model Storming”

大部分实际的建模发生在这里。两三个人，五到十分钟，很少超过半小时。有人碰到一个自己解决不了的问题，拉上同事，在白板上画几下，搞清楚了，回去写代码。

XP 实践者管这叫”站立式设计会议”。叫什么不重要。做法才重要：卡住了，找人，画，解决，继续。

4. TDD 即规格说明

Section titled “4. TDD 即规格说明”

测试驱动开发不只是在测试。你的测试就是详细规格说明。验收测试记录需求。开发者测试记录设计决策。一份能执行、能自我验证的活文档。不需要另外维护一份打印出来就过时的设计文档。

为什么时机比数量更重要

Section titled “为什么时机比数量更重要”

AMDD 管用，是因为建模发生在正确的时间点。

项目第一天，你对业务领域的了解最浅。你的问题很表面。利益相关者用抽象概念回答你，因为他们也没见过任何具体的东西。所有假设都未经验证。

到了第四个迭代，你已经交付了三次可工作的软件。利益相关者用过了。他们知道哪些地方对、哪些地方需要改。你的领域知识够深了，能问出真正有用的问题。利益相关者给出的回答也更具体，因为他们在对真实的软件做反应，不是在评价理论上的规格说明。

JIT 建模把精力集中在你实际要建的东西上。传统的前期建模往往在指定没人会用的功能。那不叫规划，那叫浪费。

一个让人不太舒服的推论

Section titled “一个让人不太舒服的推论”

AMDD 改变了你需要什么样的人。

如果建模会议只有十分钟，编码却持续几个小时，那一个全职建模师在两次会议之间干什么？这个角色说不通。

AMDD 需要的是”通才型专家”（generalizing specialist）——在一两个领域有深度专长，同时在整个开发生命周期中都有基本能力。能建模，能写代码，能测试。根据工作需要在模式间切换。

这不是说每样都半吊子。是说在你的专长领域做到出色，同时在其他领域也能干活。

AMDD 和其他方法的比较

Section titled “AMDD 和其他方法的比较”

和传统 MDD： MDD 在编码前建完整模型。AMDD 在整个过程中建最小模型。MDD 倾向串行开发；AMDD 天然是迭代的。

和特征驱动开发（FDD）： FDD 坚持用特征（feature）作为主要需求载体。AMDD 不规定用什么载体——用适合当前情况的就行。

和用例驱动开发： 同样的灵活性。用用例也行，用用户故事也行。AMDD 的意思是：写代码之前先想想，但别想太久。

怎么开始

Section titled “怎么开始”

不需要推翻现有流程：

  1. 写一个功能之前，花五分钟在纸上或白板上画个草图。写完代码就把草图扔了。
  2. 两个组件不知道该怎么交互的时候，拉上同事做一次 Model Storming。定个 15 分钟的闹钟。
  3. 先写测试。让测试充当你的规格说明。别再维护没人更新的设计文档了。
  4. 每个项目开头花几天画高层架构和关键需求。够估算就行。别追求”完整”。

70-80% 的团队用白板和便利贴就够了。工具不重要，做法才重要。

结论

Section titled “结论”

“做完整设计”还是”不做设计”，是个伪命题。答案是：在合适的时间做够用的设计，然后回去写代码。

别再争论要不要建模了。在对的时间建模就行。

Vibe Sparking

大多数人睡觉的时候，Cursor 的 AI 系统生成了大约 8000 个提交。那是 8 小时内的数字。连续 7 天，这套系统累计了 1000 万次工具调用，同时运行着几百个智能体，峰值速度大约是每小时 1000 个提交。现场的工程师没有在写代码。他们盯着监控面板，调整方向。

这就是自动驾驶代码库。它已经存在了。

你以前叫它「Bot」，但它变了

Section titled “你以前叫它「Bot」，但它变了”

自动化代码我们早就有了。构建服务器、代码规范流水线、依赖更新机器人。这些是脚本系统，执行预定义好的步骤。

后台智能体不同的地方只有一个：它们会推理。

一个后台智能体被安全漏洞通告触发时，不走流程图。它读 CVE 描述，找到受影响的代码，生成修复方案，对着修复方案跑测试，然后创建一个 PR 解释它改了什么、为什么这么改。每一步都有决策。工程师审查输出结果。

「执行脚本」和「做决策」之间的这道缝隙，就是整件事情的核心。

Cursor 验证了什么

Section titled “Cursor 验证了什么”

Cursor 在 2026 年初发布了自动驾驶代码库的研究。数字值得认真看一遍。

一套多智能体系统不间断运行了 7 天。峰值：每小时约 1000 个提交。总计：1000 万次工具调用。这个项目是一个浏览器，不是 demo。几百个智能体同时在做真实的工程工作。

让这套系统跑起来的架构并不是一开始就想到的。早期的方案都失败了。

单个强模型（Opus 4.5）会跑偏，「经常在距离完成还很远的时候宣布成功」。等权限的多智能体共享状态文件的方案，由于锁竞争把 20 个有效智能体压缩成了 1-3 个。智能体为了减少冲突开始回避复杂任务。两种方案都无法扩展。

最终跑通的是一个层级结构：

  • 根规划者掌控整体范围，不写代码
  • 递归子规划者细分工作，同时保持所有权
  • 独立工作者在代码库的独立副本上运行，完成后交付
  • 信息向上流动，不需要全局同步

单个智能体失败不会连锁崩溃。这套系统具有反脆弱性。

系统还接受了一个小的常量错误率。让分布式智能体快速修复错误，比完全预防错误更有效率。智能体 A 搞崩了构建，智能体 B 发现了并修好。终极 CI/CD，CI 本身能修它找到的 bug。

三个塑造架构的提示词洞察：

  1. 约束比指令有效。「不允许有 TODO，不允许有不完整的实现」比描述性指导更有效。
  2. **别用清单思维。**给出高层意图并赋予自主权，比给出详细任务列表效果更好。
  3. 量化范围。「生成 20-100 个任务」比「生成很多任务」传递的意图更清晰。

研究者的原话是：「智能体有巨大的工程能力，但会把指令执行到底，不管指令好不好。」

基础设施的问题

Section titled “基础设施的问题”

在笔记本上跑几个智能体不是后台智能体。企业级后台智能体需要三件事。

隔离计算。 每个智能体得到自己的沙盒环境，按需启动，任务结束后销毁。没有可以在任务之间污染的共享状态，没有并行工作流之间的资源争抢。

事件路由。 必须有什么东西决定什么时候触发哪个智能体。新的 PR、安全通告、CI 失败、计划好的依赖审计。事件路由层监听这些信号，带着合适的上下文和范围分发智能体。

治理。 权限和爆炸半径控制，智能体能碰什么、不能碰什么。每个操作的审计日志。PR 是合并前必经的人工审查关卡。失败隔离，一个智能体的状态不传播到其他智能体。

关于安全的核心认识：智能体的安全性取决于它周围的基础设施。一个治理良好、爆炸半径有限的智能体，比一个拥有无限制生产权限的工程师更安全。PR 仍然是人工审查的关键节点，这一点不变。

角色在变

Section titled “角色在变”

background-agents.com 的说法很准确：开发者从「在循环中」变成「在循环上」。

传统模式里，开发者在循环中，逐字逐行写代码，每个决策都亲手做。后台智能体模式里，开发者在循环上，设定方向、审查输出、校准智能体行为、处理智能体解决不了的判断题。

变得更重要的技能：

提示词设计和约束设计。 什么约束能让智能体稳定产出？「不允许不完整的实现」比「把任务做完」更有效。学会该说什么、不该说什么是一门手艺。

为智能体工作设计系统架构。 怎么组织代码库让智能体能独立工作？小的、自包含的模块。清晰的接口契约。最少的横切依赖。这些不是新原则，但在智能体是主要工作者时变得更关键。

在正确深度上做审查。 智能体的「技术正确」方案可以引入「语境错误」，能编译、能通过测试，但偏离了意图。能抓住这个的开发者有价值。只是橡皮图章盖章的开发者没有价值。

信任标定。 知道什么时候让智能体跑、什么时候介入。读懂智能体跑偏的信号。根据任务设定合适的自主度。

写代码本身变成很多技能之一，而不是定义性技能。

现在长什么样

Section titled “现在长什么样”

今天已经落地的后台智能体用例：

依赖更新。 智能体按计划审计所有依赖，为安全的次版本更新生成 PR，为主版本更新读 changelog 并标记破坏性变更请人工审核。不是开 issue 的机器人，是理解这次升级的智能体。

CVE 修复。 安全通告触发，智能体定位受影响代码，生成修复，测试，创建 PR 并附上漏洞和修复说明。从披露到审查就绪的 PR：分钟级，不是一个迭代周期。

CI/CD 迁移。 老的构建流水线要迁移到新平台。智能体逐文件处理转换，每步都测试。工程师审查方案，批准输出。没有人需要手动重写 200 条流水线步骤。

测试覆盖率扩展。 智能体分析覆盖率缺口，为未覆盖路径写测试，验证正确性后提交审查。有针对性的覆盖，不是随机生成测试。

代码规范落地。 新的代码规范要在多个代码库推行。智能体一致地应用它们，没有把工作分发给一整个团队的协调成本。

这些都不需要智能体做产品决策。范围清晰、输出可审查、边界明确，这是模式：后台智能体处理那些「清晰到可以规范」且「重要到值得始终如一地做」的工作。

不那么舒服的那部分

Section titled “不那么舒服的那部分”

智能体会把指令执行到底，不管指令好不好。

Cursor 直接记录了这个。模糊的指令让智能体追求偏门特性。明确的规范改善了结果。当单个执行者同时收到太多目标，它会出现功能障碍：随机停下来、拒绝委派、提前宣布完成。

这意味着人的工作变成：成为智能体工作的好架构师。把问题组织得让智能体能解决。提供产生好结果的约束。设定清晰的范围边界。用足够深的领域知识审查，能抓到那个通过了所有测试但仍然是错的东西。

能做好这件事的开发者比打字快的开发者更有价值。

从哪里开始

Section titled “从哪里开始”

不需要完整的后台智能体平台才能开始培养这些技能。

用 Claude Code、Cursor 或 GitHub Copilot Agent 模式跑更长的自主任务，练习审查输出。注意智能体什么时候产出「技术正确但语境错误」的结果，那是你在培养审查技能。

把下个项目设计成更干净的模块边界和更清晰的接口。不是因为智能体现在就在读你的代码，而是练习让智能体工作可靠的架构原则。

给 AI 编程工具写提示词时，关注约束而不是指令。「不得在未经明确批准的情况下添加新依赖」是约束。「添加依赖前仔细想想」不是。

关注后台智能体基础设施的进展。隔离计算、事件路由、治理，企业工具正在这三个方向聚集。

结语

Section titled “结语”

自动驾驶代码库不是未来状态。对特定工作负载，依赖维护、安全补丁、规范落地、测试覆盖率，它今天就在运行。

基础设施问题：隔离计算、事件路由、治理。三者缺一，你有的是个强力的笔记本助手，不是后台智能体。

角色问题：在循环上，不是在循环中。审查、校准、指挥。架构智能体工作，而不是执行它。

定位问题：这套结构里哪些技能重要？约束设计、系统架构、深度审查、信任标定。

学会这样工作的开发者，有效产出比优化打字速度的开发者大得多。这个转变正在进行。
