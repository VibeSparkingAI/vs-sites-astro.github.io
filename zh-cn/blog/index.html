AI时代的工程师成长心法

2025年9月16日
Vibe Sparking

本文总结四个常被忽视却极其关键的成长心法：少争对错多争共赢、先做出来再优化、不怕撞题坚持差异化、别浪费时间证明自己。

引言

Section titled “引言”

在 AI 技术飞速发展的今天，工程师面临的不仅是技能的升级，更是人生观与职场心态的考验。你是否曾因据理力争而得罪同事？是否为追求完美错过了窗口？这些经历很常见，也正是本文想探讨的。多年踩坑后，我总结出四条在实践中屡试不爽的成长心法，愿你读后有所共鸣。

1. 利益思考：沟通不争对错，只争共赢

Section titled “1. 利益思考：沟通不争对错，只争共赢”

初入职场时，我也热衷证明“我是对的”。后来逐渐发现：在组织环境里，“对”往往不是最重要的，“对项目/团队有利”才是。执着于抽象“优雅”与“纯粹逻辑”，却忽视交付、风险、资源与收益的平衡，容易被视为“抬杠”。

关键转变：

  • 从“我要赢”换成“团队是否整体更好”
  • 从“理论完美”换成“本期目标价值最大化”
  • 从“辩论立场”换成“协作博弈→共赢方案”

一个实用小技巧：发言或反驳前，默数 5 秒并自问：

  1. 这句话是否能推动决策或结果？
  2. 是否对对方及团队公平且有益？
  3. 是否有更低冲突成本的表达方式？

少争对错，多争共赢，你的合作者标签、影响力与信任度都会自然提升。

2. 不过早优化：先做出来，验证了再说

Section titled “2. 不过早优化：先做出来，验证了再说”

完美主义常让第一版迟迟无法落地。用户未验证、价值未闭环，却在性能、抽象、体系化设计上过度投入，是典型的“过早优化”。
第一版（V0 / MVP）的使命只有两个：

  • 验证有没有真实用户要用
  • 验证路径是否值得继续押注

实践建议：

  • 设定“本轮最小成功标准”（MSL：Minimal Success Level）
  • 控制首次交付投入上限（人日 / 复杂度 / 依赖）
  • 留“可重构缓冲区”而非一次性定死架构
  • 记录假设→上线→反馈→迭代节奏（加速循环而非堆功能）

当出现：稳定核心使用频次 + 明确 100+ 真实使用者 + 真金白银或强留存信号，再系统化重构/性能优化才划算。

口号不是“将就”，而是“快速求证 + 有的放矢”。

3. 不要被逻辑吓怕：有人做过也没事，做出你的不同

Section titled “3. 不要被逻辑吓怕：有人做过也没事，做出你的不同”

“别人已经做过了”不是否决理由，而是市场存在与可学习样本的信号。差异化来源不只在技术本身，还可能在：

  • 视角：同一问题的新表述、切口、受众聚焦
  • 体验：流程压缩、摩擦点消除、反馈即时性
  • 价值交换方式：开源/闭源、订阅/一次性、社群 + 工具复合形态
  • 叙事与人格：信任与文化调性构成隐形壁垒

执行步骤示例：

  1. 列出现有解的优缺点矩阵
  2. 标记未被充分服务的用户场景 / 情绪点
  3. 以“减法”定义你的核心差异最小集合（Differentiated Core）
  4. 先做出可感知差异的 Demo，收集真实反馈再扩展

不要高估“重复”的风险，低估“持续 + 差异 + 聚焦”的复利。

4. 不浪费时间证明自己：把精力用在创造和快乐上

Section titled “4. 不浪费时间证明自己：把精力用在创造和快乐上”

技术人最稀缺的不是工具，而是持续深度投入的注意力。与其在无意义的争论、情绪性拉扯中内耗，不如将胜负欲转化为构建力。

常见内耗陷阱：

  • “我一定要让他承认我更懂”
  • “他们不认可=我不行”
  • “必须回击每一句挑衅”

替换策略：

  • 以成果 / 代码 / 指标 / Demo 说话
  • 识别“无效争论”：不可证伪 + 无决策权 + 情绪主导
  • 设定个人“注意力黑名单”：不触发、不追击、不复读
  • 建立正向回路：输出（文章/开源/原型） → 反馈 → 改进

你的作品会替你建立身份与声誉，而不是一时的键盘胜负。

写在最后

Section titled “写在最后”

四条心法回顾：

  1. 少争对错，多谋共赢
  2. 先验证价值，再精雕优化
  3. 不怕撞题，坚持差异化
  4. 不耗于证明，专注持续创造

AI 时代工具红利在加速“能力折旧”：仅靠会用某技术栈已不足以形成长期护城河。真正形成稳固复利的是：

  • 心态弹性
  • 策略性执行
  • 差异化表达
  • 专注与持续复盘

成长道路难免曲折，只要方向对了，就不怕路远。愿我们既拥抱技术浪潮，也能沉淀内在的判断力与创造力。

Claude Code + ArkTS 开发鸿蒙 App：几分钟做出 BLE 低功耗蓝牙管理器

2025年9月16日
Vibe Sparking

用 Claude Code 做“懂鸿蒙 ArkTS 的结对程序员”，在 ArkTS 里写出能扫描 / 连接 / 读写 / 订阅的 BLE 管理器，真机可跑、日志可查、问题可排。

引言｜为啥用 Claude Code 搭 ArkTS？

Section titled “引言｜为啥用 Claude Code 搭 ArkTS？”

做 BLE 低功耗蓝牙管理器的都懂：协议细碎、厂商千差、版本在飞。HarmonyOS/ArkTS 的文档又多又散，新人上手容易迷路。 我的做法很接地气：把官方文档“喂”给 Claude Code，让它在代码区当“看着手册写代码”的搭档；我们专注业务和调试节奏。

  • ArkTS 是鸿蒙原生首选语言，基于 TS 强化了静态检查与性能（官方定义与入门见这里 https://developer.huawei.com/consumer/cn/arkts/）。
  • BLE 在鸿蒙上走 @ohos.bluetooth.ble 模块，核心能力：扫描、广播、GATT 客户端/服务端。
  • 真正的数据交互走 GATT：读写特征、订阅通知。

这套组合拳的好处：减少瞎猜，直接照文档写；Claude Code 负责搜、比对、补全；我们控制架构、状态机与日志。

快速打样｜ArkTS + BLE 的基本动作

Section titled “快速打样｜ArkTS + BLE 的基本动作”

权限与能力 在 entry/src/main/module.json5 里声明所需权限（示例，按你的 API/版本微调）：

  • ohos.permission.USE_BLUETOOTH（启用蓝牙）
  • ohos.permission.DISCOVER_BLUETOOTH（扫描 / 配置）
  • ohos.permission.LOCATION（扫描关联位置申请，按 UX 规范前台说明）

模块入口 ArkTS BLE 走 @ohos.bluetooth.ble，扫描用事件 ble.on('BLEDeviceFind', cb) 获取结果；发现设备后创建 GattClientDevice 进行连接与 GATT 读写/订阅。

1）让它看文档再回答：在问题里贴上 @harmony-os-docs/... 路径，要求它回答带出处。 2）先要“设计稿”：让 Claude 先给「目录结构 + 状态机 + 事件流 + 关键 API 对应表」。 3）针对版本差异：要求输出“API 版本矩阵”，比如 startBLEScan vs startScan，哪里不确定标注“需在本机 SDK 验证”。 4）自动生成单测/桩：让它产出“模拟回调/虚拟通知”的快速用例，覆盖断线重连/权限拒绝等场景。 5）日志协议：约定 [BLE] 前缀 + 关键字段，一键搜集；结合“合理蓝牙使用”最佳实践设置扫描窗口与频率。

常见坑位清单（避免踩雷）

Section titled “常见坑位清单（避免踩雷）”
  • 权限：蓝牙 & 发现 & 位置三件套，按 UX 规范做前台授权提示。
  • 扫描拿不到设备：确认外设在广播、App 权限 OK；有版本用 startBLEScan，有用 startScan(filters)，事件是 BLEDeviceFind。
  • 读写失败：多半是 UUID 不对、属性权限不匹配、或没建立通知/指示。复核 GATT 设计。
  • 耗电/后台：扫描要控时长与频率，遵循“合理使用蓝牙”最佳实践。

验收标准（Acceptance Criteria 对照）

Section titled “验收标准（Acceptance Criteria 对照）”

Section titled “什么是 Vibe Coding？”

而普通人为什么容易“翻车”？

核心不在“会不会让模型写几行代码”，而在能不能把需求→设计→实现→验证→部署→观测跑成一个可重复的闭环。缺这个闭环，就会出现：看上去每天都在“用 AI 写代码”，但实际产出只是堆叠的半成品与 Bug 循环。

当前数据已经给出信号：2025 年开发者对 AI 的好感度下降、对其结果的“不信任”上升——说明把 AI 输出当“答案”而非“待验证工件”，极容易被带进歧途。

典型失败模式（外行 / 低经验程序员常见）：

  1. 需求漂移：没有明确验收条件，提示词越写越长，结果越偏。
  2. 结构失控：目录、模块边界、环境变量、依赖版本无治理，几周后不可维护。
  3. 盲修 Bug：缺测试基线 → 只能肉眼点点点 → 改一处炸三处。
  4. 数据链断裂：不采集事件 / 指标，谈不上激活、留存、价值证据。
  5. 成本与风险不可见：模型调用无配额、无缓存、无审计，成本曲线和合规风险一起失控。
  6. 永远“快生成” → 从不“严回归”：没有最小回归集，交付质量全靠运气。
  7. 只会“提示词调优” → 不会“接口与约束”设计：Agent 产物缺一致性，复用失败。

根本差距在于：闭环思维依赖长期积累的抽象与约束能力，而这通常来自 5–10 年的：

  • 计算机基础（编译原理、操作系统、数据库范式、算法与数据结构）
  • 工程化实战（版本管理、依赖治理、性能与故障定位、CI/CD）
  • 架构取舍（一致性 vs. 吞吐、延迟 vs. 成本、通用性 vs. 简洁）
  • 工具肌肉记忆（IDE 深度使用、调试器 / Profiling、脚本化自动化）
  • 领域上下文（数据口径、合规、SLA、业务节奏）

当这些底座缺位时，“AI Coding / Vibe Coding”只会退化成：

拼提示词 + 复制片段 + 临时修补

也就很难跨过三个门槛：

  1. 从“功能凑齐”到“质量可回归”
  2. 从“有人试用”到“指标可验证”
  3. 从“技术 Demo”到“可收费的最小产品”

短期（未来 1–2 年）内，如果不了解上述基础与工程套路，AI Coding 更像玩具与幻觉放大器；对具备架构与闭环经验的人，它则是“抽象力 ×10 的杠杆”。想缩小差距，不是再多抄几个提示词，而是先补：约束 → 契约 → 测试 → 指标 → 运维 这一条最小骨干。

接下来你若真想补课，可以先自查：我是否已经具备“最小 Vibe 闭环 6 件套”？ 需求验收｜接口契约｜测试基线｜自动化集成｜观测指标｜回滚与成本护栏

缺哪一块，就从哪一块补起；否则所有“AI 赋能”都是伪加速。

所以要反过来： 用人的“结构化能力”，管住 AI 的“生成冲动”。 先约束，后生成；先测试，后集成；先回归，后上线。

Section titled “源于分歧的诞生：OpenAI 的哲学裂痕”

宪法式 AI（Constitutional AI）：价值观的编码

Section titled “宪法式 AI（Constitutional AI）：价值观的编码”

宪法式 AI（CAI）的目标是训练 AI 成为“有帮助、诚实且无害”。其核心是在训练中引入一份由人类撰写的“宪法”——一组原则，指导模型自我评估与修改输出。与传统在敏感问题上“回避/模糊”的处理不同，CAI 让模型在遵循原则的前提下更直接地回应请求，并在拒绝时说明理由。

训练过程分两阶段：监督学习阶段中，模型依据宪法原则批评并修正自身不当回答，生成更“无害”的数据集进行微调。

Section titled “基于 AI 反馈的强化学习（RLAIF）：对齐的自动化引擎”

相较常见的 RLHF，Anthropic 认为 RLAIF 更高效、更可扩展（减少人工标注依赖），且通过以自然语言编码目标提升可解释性与透明度，同时在一定程度上降低人类主观偏见的影响。批评者指出，宪法本身仍由人类撰写，难免主观性；同时减少人工干预也带来问责与监督的讨论。这种“自动化”与“人工控制”的张力，是理解 Anthropic 方法论的关键。

引言

Section titled “引言”

Section titled “TL;DR 清单（先拿走）”
  • 工程化/扩展性：Claude Code 胜，子代理（Subagents）、自定义 Slash 命令、Hooks 三件套非常成熟。

引言｜“不可能三角”的真相

Section titled “引言｜“不可能三角”的真相”

过去两年，从 bot 到 agent，气氛拉满，但项目却慢半拍。原因往往不是模型不够强，而是组织协作没对齐：老板想讲故事且要三个月见 ROI，技术想多模态+合规+可扩展，业务只认 KPI 和转化。 现实数据也在提醒我们：企业级 ROI 通常要更久，不要指望一口吃成胖子。Forrester 与一线 CIO 的复盘指出，18～24 个月更常见；急于求成反而更容易“烧钱不见效”。 IBM 委托的 2024/25 年度调研也显示，不到一半的企业 AI 项目当年盈利，多数把三年内见到成本节省当作合理预期。

AI 落地不是技术赛，是组织工程。MVP、工作流重构、治理与 KPI 追踪，哪个都不能少。

Section titled “一、组织先行：把三方拉到同一张桌子”

老板要做什么？

  • 任命“一把手工程”挂帅，CEO/总经理亲自盯治理与跨部门协同。
  • 明确“三张清单”：投入边界、风险红线、90 天内要看的指标。

McKinsey 的全球调研表明，高层直接负责 AI 治理、跟踪明确 KPI，与更高的实际利润改变量显著相关。

技术要做什么？

  • 不摆“工具展”，改打“闭环工具链”：数据→RAG/Agent→埋点→灰度→A/B。
  • 把合规与可解释并入流水线，数据血缘/审计随查随有。

业务要做什么？

  • 业务目标要可量化且能被 AI 直接影响：AHT、FCR、转化率、处理时长、人天节省。
  • 业务同学参与Prompt 共创。最新研究也强调：提示词质量与模型本身一样重要。

Section titled “二、90 天最小闭环打法（可直接执行）”

Day 8–21｜搭底座

  • 数据清洗+权限治理+RAG 索引，先把“能被问到的”整理干净。
  • 设三类埋点：质量（准确率/召回）、效率（时长/并发）、效果（转化/节省）。

Day 22–45｜做 MVP

  • 小队结构：业务 PO+技术 TL+数据治理+法务/合规+AI Ops。
  • 业务与技术共写系统提示与 Few-shot 案例，周迭代两次。
  • 上影子模式（Shadow）：AI 给建议，人来执行。

Day 46–60｜A/B 验证

  • 切 20–30% 流量做对照，盯三项：AHT、一次解决率、满意度。

Day 61–90｜灰度放量

  • 达到门槛就进半自动：常规问题 AI 先答，复杂问题人兜底。
  • 建立变更与回滚机制，指标跌破阈值自动降级。

先用 Claude Code 快速“跑通原型”，把需求打磨清楚；再回到工程化设计与实现，把质量与稳定性做到位。效率和质量，都要。

引言

Section titled “引言”

做产品最难的，不是写代码。 是把“到底要做啥”讲清楚。

所以我更推荐这套两段式节奏： 第一版只做原型，第二版认真做工程。 Claude Code 很适合这个打法：一边和你在终端里并肩作战，一边把杂活脏活先铺开，再回头精修。

原型开发法的 2 个阶段

Section titled “原型开发法的 2 个阶段”

阶段 1：只做原型，追求“跑通”

Section titled “阶段 1：只做原型，追求“跑通””
  • 目标：把功能跑起来，让需求说人话。
  • 不关注：设计、性能、安全、代码质量。
  • 做法：抛弃型代码，完全交给 AI 主导，快速探索可行路径（类似“Spike/Throwaway Prototyping”的思路）。

为什么这样干？ 因为很多模糊点，只有跑起来，才会暴露。截图、动图、可点的原型，比十页需求文档有用多了。

Claude Code 在这一步的优势：

  • 直接在本地项目里读、改、跑代码，对话即行动。
  • 可把高频动作做成 Slash Commands，一键复用，比如“起后端 + 假数据 + 本地跑”。

边界提醒：如果你用的是个人订阅版本，近期 Anthropic 有数据用于训练的选择机制，涉及 Claude Code 会话。做公司项目原型前，先确认隐私设置。

阶段 2：重新设计，认真做工程

Section titled “阶段 2：重新设计，认真做工程”
  • 触发点：需求基本确定，范围（Scope）稳定。
  • 输出物：数据库 Schema、API 协议、状态管理、模块边界、测试策略等。
  • 节奏：人主导，AI 辅助。由你下设计锚点，再让 AI 按图施工。

Claude Code 在这一步的用法：

  • 让它按你的设计搭脚手架、补样板、补测试。
  • 在 PR/Issue 里 @claude 自动出 PR、修小问题（GitHub Actions 集成）。
  • 将工程约定写进 CLAUDE.md / Slash 命令，让 AI 按项目规则执行。

安全/非功能建议：把安全、性能、可运维列为“技术 Enablers”，在二阶段逐步纳入 DoD（完成定义）。必要时先做 Spike 验证。
