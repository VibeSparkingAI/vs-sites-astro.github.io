Chat UI Swift：展示 AnyLanguageModel 和 Swift-HuggingFace 的 SwiftUI 聊天应用

2025年12月29日
Vibe Sparking

Chat UI Swift：展示 AnyLanguageModel 和 Swift-HuggingFace 的 SwiftUI 聊天应用

Section titled “Chat UI Swift：展示 AnyLanguageModel 和 Swift-HuggingFace 的 SwiftUI 聊天应用”

在 Apple 平台上构建 AI 驱动的聊天应用程序，传统上需要面对多个 SDK、API 和认证流程的复杂性。mattt/chat-ui-swift 通过一个精美的 SwiftUI 参考实现简化了这一局面，展示了如何通过统一的 API 集成多个语言模型提供商。

什么是 Chat UI Swift？

Section titled “什么是 Chat UI Swift？”

Chat UI Swift 是由传奇开发者 Mattt Thompson 创建的 SwiftUI 聊天应用，他也是著名的 AFNetworking 和 NSHipster 的作者。这个应用作为两个强大的 Swift 库的演示项目：

  1. AnyLanguageModel - Apple Foundation Models 框架的替代方案，支持自定义语言模型提供商
  2. swift-huggingface - Hugging Face API 的 Swift 客户端

GitHub 仓库： mattt/chat-ui-swift 作者： Mattt Thompson 许可证： MIT

核心特性

Section titled “核心特性”

已支持的功能

Section titled “已支持的功能”
  • Apple Intelligence - 与 Apple Foundation Models 的原生集成（macOS 26+）实现本地 AI
  • Hugging Face 集成 - 通过 OAuth 2.0 认证连接 Hugging Face
  • 流式响应 - AI 响应的实时流式传输，提供流畅的用户体验

BMAD Method：如何把大模型系统提示词，变成一支可复制的 AI 开发团队

2025年12月27日
Vibe Sparking

BMAD Method 是一个开源的 AI 驱动敏捷开发方法论与工具集，基于 BMAD Core 框架，内置 19 个专业 AI 角色（Agents） 和 50+ 条工作流（Workflows） ，覆盖从需求分析、规划、架构设计到开发与测试的完整生命周期，已经在 GitHub 上获得了 23.6k 颗 Star （截至 2025 年 12 月）。它把「系统提示词 + 角色定义 + 流程指令」打包成结构化资产，让你不再零碎地复制粘贴 prompt，而是像调度一个真正的软件团队那样，调度一群 AI 专家协同工作。

你为什么需要一个成体系的系统提示词仓库？

Section titled “你为什么需要一个成体系的系统提示词仓库？”

大部分 AI 学习者、开发者现在在用大模型时，都会遇到这些痛点：

  • 系统提示词乱七八糟 今天在 Claude 里写了一段不错的系统 prompt，明天在 ChatGPT 里又改了一版，结果散落在各个对话和文档中，根本找不到最佳版本。
  • AI 的角色不稳定 有时你希望它是产品经理，有时是架构师，有时是 QA，但每次都要重新解释，且每次解释都不完全一样，输出质量很难稳定。
  • 缺乏从需求到代码的闭环 即便你写了不少 prompt，更多是一次性的：需求文档一套写法，架构讨论一套写法，开发与测试又是一套，这些提示词之间没有严格的 artifact 流转关系，很难形成可复制的工程流程。
  • 协作成本高 当你想把自己整理的一堆 prompt 分享给同事时，对方往往只能照着一篇长文或一个 Notion 页面挨个复制，完全没有工具化。

如果你也有类似经历，那 BMAD Method 做的事情，可以理解为：

把「AI 系统提示词 + 角色分工 + 流程步骤」工程化、版本化、可移植化，变成一个可以在各种 IDE、大模型工具中反复复用的 AI 团队框架。

BMAD Method 本质在做什么？

Section titled “BMAD Method 本质在做什么？”

BMAD Method 的定位可以概括成一句话：

Breakthrough Method for Agile AI-Driven Development —— 一个让 AI 以敏捷团队的方式参与开发的框架。

它的核心有三层：

  • BMAD Core：一个为人类 + 多智能体协作设计的基础框架，里面包含：

    • 以 Markdown 编写的 Agents（专业角色 persona）
    • 由多个 Agent 组成的 Agent Teams
    • 用 YAML 定义的 Workflows（战略流程）
    • 带 LLM 指令的 Templates（PRD、架构文档、Story 文件等）

  • BMAD Method：站在 BMAD Core 之上，专注于软件开发场景的工作流，把分析 → 规划 → 方案 → 实施整个链路拆解成多个阶段与工件，由不同的 Agent 接力完成。

  • BMAD Builder / Creative Intelligence Suite：

    • BMAD Builder：让你可以在同一套框架上，设计自己的专业 Agent 与模块（如法律、金融、教育等领域）。
    • Creative Intelligence Suite：提供创意思考、头脑风暴类工作流，辅助产品创新、叙事设计等。

换句话说，BMAD Method 不是简单的 prompt 集合，而是把这些系统提示词按：

  • 角色（谁来做）
  • 阶段（何时做）
  • 工件（产出什么）
  • 工具（在哪儿用）

全部结构化整理成了一个可以移植到你项目里的工程方法论与代码仓库。

核心功能特性

Section titled “核心功能特性”

1. 19 个专业 AI 角色，组建虚拟开发团队

Section titled “1. 19 个专业 AI 角色，组建虚拟开发团队”

BMAD 内置了一个小型虚拟公司：

  • Analyst（分析师）
  • Product Manager（产品经理）
  • Architect（架构师）
  • Scrum Master（敏捷教练）
  • Product Owner
  • Developer（开发）
  • QA Engineer（测试）
  • UX Designer、Game Designer、Tech Writer 等等

每个角色都有：

  • 清晰的职责
  • 对应的命令（如 @analyst、@pm、@dev）
  • 对应要产出的文档类型（PRD、架构说明、Story 文件、QA 报告）

这些定义都是通过 Markdown + 系统提示词写好的，你无需重新发明如何让模型扮演一个合格的产品经理——直接调用现有 persona 即可。

2. 50+ 工作流覆盖完整开发生命周期

Section titled “2. 50+ 工作流覆盖完整开发生命周期”

BMAD Method 把整个软件交付过程拆成 4 个阶段：

  1. Analysis（分析）：需求挖掘、市场调研、方案对比
  2. Planning（规划）：PRD、技术规格说明、游戏设计文档等
  3. Solutioning（方案设计）：系统架构、数据流、API 设计、UX 草图
  4. Implementation（实施）：Story 驱动开发、代码实现、测试验证

针对每个阶段，框架都提供了多条工作流，以 YAML 形式定义流程与调用顺序。例如：

  • 用 Analyst 生成 Project Brief
  • 用 PM 生成 PRD
  • 用 Architect 生成 Architecture 文档
  • 用 PO 做文档分片
  • 用 SM 生成 Story 文件
  • 用 Developer/QA 实现与验证代码

这本质上是一套可执行的系统提示词流程，而不是一堆零散的 prompt 范例。

3. Scale-Adaptive Intelligence：三种智能轨道

Section titled “3. Scale-Adaptive Intelligence：三种智能轨道”

BMAD 提供了三种智能轨道：

  • Quick Flow：适合修 Bug、小需求，生成轻量级技术规格，几分钟即可开始写代码。
  • BMad Method 标准轨道：用于新功能、新产品或中等规模平台，从 PRD 到架构再到 UX。
  • Enterprise 轨道：面向合规、治理要求高的项目，提供更完整的文档、质量门槛与流程。

在实际使用时，你可以跑 *workflow-init 让 BMAD 自动根据目标与上下文推荐合适的轨道。

4. Story / Sharding 体系：文档当工程一等公民

Section titled “4. Story / Sharding 体系：文档当工程一等公民”

BMAD 里特别强调文档分片与 Story 文件体系：

  • 将大型 PRD 拆成多个 Epic 文件
  • 再从 Epic 生成带完整上下文的 Story 文件（如 1.1.user-registration.story.md）
  • Story 文件中包含：
    • 架构背景
    • 实现细节指引
    • 验收标准
    • 相关约束

最后，Developer 和 QA 都是围绕 Story 文件来工作——这就让系统提示词 + 文档 + 代码形成严格映射，方便版本控制、审计与回溯。

Section titled “5. 基于 BMAD Core 的模块化架构”

  • Agents：Markdown persona 定义（本质上是结构化系统提示词）
  • Agent Teams：可复用的团队组合
  • Workflows：YAML 流程定义
  • Templates：嵌入 LLM 指令的文档模版
  • Expansion Packs：针对特定领域（创作、商业、教育、健康等）的扩展

这意味着：如果你正在做整理大模型系统提示词的工作，可以直接沿用它的这套目录结构和抽象方式，把你自己的 prompt 仓库建设在同一套架构之上。

Section titled “7. Token 节省与更新安全”

六大核心素质深度解读

1. 责任心强

Section titled “1. 责任心强”

含义：你不只是写代码——你对结果负责。当凌晨 2 点系统出问题时，你已经在调查了，而不是等着别人给你分配工单。

实践中的体现：

  • 跟进问题完整的生命周期，而不是”把代码扔过墙”就完事
  • 主动监控你构建的系统
  • 对成功和失败都承担责任

为什么 AI 无法替代：AI 可以生成代码，但它感受不到生产事故的压力，也体会不到为用户交付真正价值的满足感。

2. 好奇心与学习能力

Section titled “2. 好奇心与学习能力”

含义：AI 领域发展速度极快。六个月前的最先进技术现在已经是基线。你必须有内在动力去学习，而不是因为经理安排了培训课程。

实践中的体现：

  • 周末自己尝试新工具（Claude Code、Cursor、GitHub Copilot）
  • 阅读研究论文，而不仅仅是文档
  • 问”这个为什么有效？“而不只是”我怎么用它？”

复利效应：Anthropic 的最佳实践强调，与 AI Agent 协作是一项随时间发展的技能。持续迭代和改进的好奇心将新手与专家区分开来。

3. 实践与执行能力

Section titled “3. 实践与执行能力”

含义：没有执行的想法毫无价值。你能在几天而不是几个月内把一个概念变成可工作的原型吗？

实践中的体现：

  • 快速构建 MVP 来验证假设
  • 知道什么时候”足够好”真的就足够好
  • 迭代交付而不是等待完美

AI 放大效应：借助 AI 编码助手，执行速度已成为超能力。根据 JetBrains 的指南，能有效引导 AI Agent 的开发者比手动编写所有代码的人快 10 倍。

4. 规划能力

Section titled “4. 规划能力”

含义：将复杂问题分解成可管理的小块，有效排序工作，并现实地管理时间。

实践中的体现：

  • 将一个功能分解成具体的、可测试的任务
  • 了解依赖关系和关键路径
  • 现实地估算（而不是乐观地）

为什么在 AI 时代这很重要：Medium 上关于 Agent 工作流的指南强调使用”计划-行动-反思”工作流。结构越清晰，AI Agent 表现越好。

5. 沟通与分享意识

Section titled “5. 沟通与分享意识”

含义：孤立的技术卓越毫无价值。你能向非技术利益相关者解释你的决策吗？你能指导初级开发者吗？你能写出别人真正想读的文档吗？

实践中的体现：

  • 写清晰、简洁的文档
  • 建设性地参与代码评审
  • 在团队会议或博客文章中分享学习成果

乘数效应：能有效沟通的开发者将其影响力倍增到整个组织。分享的知识是被放大的知识。

Section titled “6. 隐藏的第六素质：适应能力”

🎚️ CodexBar：在 macOS 菜单栏实时监控 Claude Code 和 OpenAI Codex 用量

2025年12月25日
Vibe Sparking

CodexBar 是由 Peter Steinberger (@steipete) 开发的一款轻量级 macOS 15+ 菜单栏应用，让你的 AI 编程助手使用限额始终可见。同时监控 OpenAI Codex 和 Claude Code 的限制——包括 5 小时会话窗口和每周配额——无需登录任何仪表板。

  • 双服务商支持：同时追踪 Codex（5 小时/每周 + 积分）和 Claude Code（会话/每周 + Sonnet 专属限制）
  • 零登录要求：直接读取本地 CLI 输出——无需密码，无需浏览器会话
  • 实时可视化：从菜单栏查看剩余 Token、重置计时器和账户信息

🧩 为什么需要用量监控

Section titled “🧩 为什么需要用量监控”

如果你正在积极使用 Claude Code 或 OpenAI Codex 进行开发，很可能遇到过意外触发速率限制的情况。以下是 CodexBar 必不可少的原因：

  1. 避免工作流中断 在开始大型编码会话之前了解剩余配额。不再在复杂重构过程中遭遇”速率限制”的突然惊吓。

  2. 追踪多重时间窗口 两个服务都使用重叠的 5 小时和每周窗口。CodexBar 可视化显示两者，让你准确了解限制何时重置。

  3. 监控 Sonnet 专属限制（Claude） Claude Code 的 Sonnet 模型有自己的每周上限。CodexBar 单独显示，让你能策略性地选择何时使用哪个模型。

  4. 积分可见性（Codex） 在速率限制旁查看付费积分余额。了解你是在消耗免费层还是付费用量。

🚀 工作原理

Section titled “🚀 工作原理”

读取 Claude Code 用量

Section titled “读取 Claude Code 用量”

CodexBar 在本地 PTY（伪终端）中运行 claude /usage 和 claude /status 来提取：

  • 会话使用百分比
  • 每周使用百分比
  • Sonnet 专属每周限制
  • 重置倒计时
  • 账户邮箱、组织和登录方式

隐私说明：一切都在本地完成。除 Claude CLI 本身的网络调用外，无任何额外网络请求。

读取 Codex 用量

Section titled “读取 Codex 用量”

CodexBar 优先使用 Codex app-server RPC 获取速率限制和积分。如果 RPC 不可用，则回退到 PTY 抓取 codex /status。

可选的”通过网页访问 OpenAI”设置可额外显示：

  • 代码审查剩余次数
  • 用量分解
  • 积分使用历史

这会重用已登录的 chatgpt.com 会话（通过 Safari/Chrome cookie 导入）——不存储密码。

Section titled “📊 可视化状态指示器”

条形显示内容
顶部条5 小时窗口配额（如果每周配额用尽则显示积分条）
底部条每周窗口配额（细线）

图标变体：

  • Codex 模式：标志性的”眼睑眨动”设计
  • Claude 模式：显示 Claude 的缺口/腿部变体
  • 错误/未知状态：图标变暗（无混乱文字）
