Dev Browser：Claude Code 的有状态浏览器自动化技能

Section titled “Dev Browser：Claude Code 的有状态浏览器自动化技能”

随着 AI 代理编写代码的能力日益增强，下一个逻辑步骤就是让它们能够与真实的网页进行交互。Dev Browser 是一个开源的 Claude Code 技能，它正是为此而生——并且在每次调用之间保持浏览器的所有状态。

为什么 Dev Browser 如此重要

Section titled “为什么 Dev Browser 如此重要”

Dev Browser 解决了 AI 代理能力中的一个关键缺口：持久化浏览器自动化。传统的浏览器自动化工具要么一次执行一个操作（慢），要么运行每次都从头开始的完整脚本（脆弱）。Dev Browser 采用了一种不同的方法。

核心特性

Section titled “核心特性”
  • 持久化页面 – 导航到页面一次后将其保存在内存中。无需为每个运行的脚本重新加载 DOM。
  • 灵活执行 – 可以一次性运行完整脚本，或在探索时逐步执行操作。
  • 对 LLM 友好的 DOM 快照 – 结构化、轻量级的页面表示，Claude 可以快速解析。
  • Chrome 扩展 – 控制现有的 Chrome 配置文件，利用已登录的会话、Cookie 和扩展。

这些特性为 Claude Code 代理在测试、数据提取和交互式演示方面提供了实际优势。

技术架构

Section titled “技术架构”

Dev Browser 的创新在于其混合架构：

  1. 有状态服务器：在操作之间维护浏览器内存中的页面
  2. 代理执行：自信时运行完整脚本，探索时回退到逐步执行
  3. 优化的 DOM 快照：页面表示专为 AI 消费而结构化，而不仅仅是原始 HTML

这种设计使其非常适合：

  • 端到端测试：验证应用程序的端到端工作流程
  • 数据提取：通过智能解析抓取网页内容
  • 调试：让 Claude 交互式地调查 UI 问题

与类似工具的对比

Section titled “与类似工具的对比”

vs. Playwright MCP

Section titled “vs. Playwright MCP”

Playwright MCP 使用传统的观察-思考-行动循环，每个浏览器操作都需要单独的 API 调用。这很简单，但会导致多次往返。Dev Browser 智能地批量处理操作。

vs. Selenium

Section titled “vs. Selenium”

Selenium 是一个成熟的浏览器自动化框架，但缺乏 AI 优化功能。Dev Browser 专为 LLM 消费而设计，具有结构化的页面表示。

vs. Puppeteer

Section titled “vs. Puppeteer”

Dev Browser 代表了 AI 代理能力的重大进步。通过将持久浏览器状态与智能执行模式相结合，它使 Claude Code 能够以比以前更快、更可靠的方式与网页交互。

无论你是自动化测试、提取数据还是调试 UI 问题，Dev Browser 都提供了一个实用的解决方案，弥合了 AI 代码生成与真实 Web 交互之间的差距。

Section titled “简介”

开源、快速、为建造者而生 — 这是 UI-TOOLS 的承诺，这是一套由 Naymur Rahman 开发的综合性创意工具箱，为设计师和开发者创建令人惊叹的现代 Web 界面视觉效果提供了强大支持。

UI-TOOLS 是一组专门的生成器集合，帮助您在一个平台上创建和定制阴影、SVG 裁剪路径、网格渐变、背景图案和配色方案。该项目使用 TypeScript、React、Next.js 和 Tailwind CSS 构建，代表了设计工程的现代方法。

引言：为什么需要多代理

Section titled “引言：为什么需要多代理”

构建强大AI应用的挑战不在于让单个代理变得更聪明，而在于让多个代理高效协同。单个代理很快会遇到天花板：上下文窗口溢出、专业化不足、串行处理成为瓶颈。

多代理系统通过将复杂任务分解给专业化的代理来解决这个问题，就像组建专家团队而非依赖一个通才。

  • 并行处理：缩短项目周期
  • 可维护性：快速定位问题
  • 可扩展性：通过添加新代理增加新能力

代价是：协调开销。代理需要通信、共享信息、避免相互踩踏。理解并选择正确的”团队管理模式”——即编排模式（Orchestration Pattern）——至关重要！

第一部分：三种核心团队管理模式

Section titled “第一部分：三种核心团队管理模式”

经过业界验证，目前主流的多代理编排模式有三种。你应该根据任务的协调需求来选择，而不是看哪个听起来最酷！

1. 主管模式（Supervisor Pattern）- 中心化控制

Section titled “1. 主管模式（Supervisor Pattern）- 中心化控制”

这是最直观、也最常见的模式。想象一个项目经理协调所有工作：一个”主管”代理接收任务，将其分解为多个子任务，分配给不同的”工作”代理，然后监督、验证并最终整合所有结果。

工作流程：用户请求 → Manager分解 → 分配给Sub-Agent → S-Agent执行 → Manager验证 + 综合 → 最终输出

优缺点：优点是控制流程清晰，易于调试和审计。缺点是主管容易成为性能瓶颈，所有协调步骤都是串行的，增加了延迟和Token成本。

2. 流水线模式（Sequential / Pipeline）

Section titled “2. 流水线模式（Sequential / Pipeline）”

这种协作方式的核心是，Sub-Agent按照固定顺序执行，前一个的Output是后一个Sub-Agent的Input。

优缺点：优点是具有极高的稳定性，而且易调试；缺点也同样明显——环节卡死则全线崩溃。

3. 并行模式（Concurrent/Map-Reduce）

Section titled “3. 并行模式（Concurrent/Map-Reduce）”

这一编排模式其实理解起来非常简单，相当于你同时打开几个不同的Claude Code命令窗口，然后分别给他们任务来完成。

2026年1月16日

引言

Section titled “引言”

Cursor 团队最近发表了一篇优秀的博客文章，分享了他们使用 AI Agent 编程的最佳实践。随着 AI 驱动的开发工具成为主流，理解如何有效地与 AI Agent 协作，正成为开发者必备的核心技能。

这些见解不仅适用于 Cursor，对于任何使用 Claude Code、GitHub Copilot 或类似 AI 编程助手的开发者都极具价值。

8 大核心原则

Section titled “8 大核心原则”

1. 先计划再动手

Section titled “1. 先计划再动手”

让 Agent 先列出步骤、提出问题、确认思路，然后再写代码。虽然这看起来会更慢，但实际上更快。

核心洞察： 如果做出来的结果不对，与其反复修补，不如回退代码、修改计划、重新执行一遍。

2. 上下文别给太多

Section titled “2. 上下文别给太多”

Agent 自己会搜索信息，不需要手动塞一堆文件进去。事实上，提供太多上下文反而会干扰 Agent 的判断。

  • 如果知道具体需要哪些文件，直接指定
  • 如果不确定，让 Agent 自己去找

3. 长对话会失焦

Section titled “3. 长对话会失焦”

换任务时开新对话。如果发现 Agent 反复犯同样的错误，也重新开始。

提示： 可以引用历史工作，不需要复制粘贴所有内容。

4. 规则按需加

Section titled “4. 规则按需加”

只在 Agent 反复犯同一个错误时才写规则，不要提前优化。

保持规则精简，引用文件而不是复制内容。

5. 给可验证的目标

Section titled “5. 给可验证的目标”

写测试、配置 linter、使用类型语言。Agent 需要明确的对错信号才能有效迭代。

TDD 流程特别适合 Agent： 先写测试，再让 Agent 写代码直到测试通过。

6. 图片比文字快

Section titled “6. 图片比文字快”

对于 UI 问题或设计规范，截图和设计稿比文字描述高效得多。

7. Prompt 要具体

Section titled “7. Prompt 要具体”

不要说：“给 auth.ts 加测试”

试试：“给 auth.ts 写测试覆盖 logout 边界情况，使用现有测试模式，不要使用 mock”

指令越具体，结果越好。

8. 认真审查

Section titled “8. 认真审查”

AI 生成的代码可能看起来正确但包含微妙错误。Agent 跑得越快，审查就越重要。

实践方法： 结合 Review Agent 使用。可以使用 Cursor 内置的 Review → Find Issues 功能，或集成外部审查工具。

核心理念

Section titled “核心理念”

把 Agent 当作有能力的协作者，而不是代码生成器。 给它计划、目标、反馈，它才能真正发挥作用。

这些技巧不仅适用于 Cursor，也适用于任何 Agent 工具。

推荐工作流程

Section titled “推荐工作流程”
  1. 规划阶段： 让 Agent 列出实施方案
  2. 目标设定： 定义可验证的成功标准（测试、类型）
  3. 迭代开发： Agent 写代码，你审查
  4. 纠偏调整： 需要时回退并修改计划
  5. 最终审查： 仔细检查生成的代码

结语

Section titled “结语”

软件开发的未来是人机协作。通过遵循这些最佳实践，你可以在保持代码质量的同时，大幅提升开发效率。

记住：Agent 是强大的工具，但你仍然是架构师。你的角色从编写每一行代码，转变为提供愿景、反馈和验证。

Q3：生成的视频质量如何？

Section titled “Q3：生成的视频质量如何？”

A：音频质量取决于 TTS 模型，图像质量取决于 Stable Diffusion 模型。使用默认模型可获得不错的效果。

Section titled “Q4：需要付费吗？”

  • 适当添加标点符号控制节奏

Section titled “Q6：处理速度如何？”

服务启动失败

Section titled “服务启动失败”

问题：程序停止执行并显示错误信息

解决方案：

  1. 检查 Ollama 服务是否运行：ps aux | grep ollama
  2. 检查 DrawThings 是否开启 HTTP 访问
  3. 检查 IndexTTS2 服务是否在 7860 端口
  4. 确认端口未被占用

Section titled “图像生成失败”

问题：某些章节没有生成图像

解决方案：

  1. 检查 DrawThings 服务状态
  2. 查看章节内容是否足够生成提示词
  3. 检查磁盘空间
  4. 查看 logs 获取详细错误信息

  2. 检查参考音频质量
  3. 调整采样率设置
  4. 尝试不同的 TTS 模型
