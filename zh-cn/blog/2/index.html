BMAD Method：如何把大模型系统提示词，变成一支可复制的 AI 开发团队

2025年12月27日
Vibe Sparking

BMAD Method 是一个开源的 AI 驱动敏捷开发方法论与工具集，基于 BMAD Core 框架，内置 19 个专业 AI 角色（Agents） 和 50+ 条工作流（Workflows） ，覆盖从需求分析、规划、架构设计到开发与测试的完整生命周期，已经在 GitHub 上获得了 23.6k 颗 Star （截至 2025 年 12 月）。它把「系统提示词 + 角色定义 + 流程指令」打包成结构化资产，让你不再零碎地复制粘贴 prompt，而是像调度一个真正的软件团队那样，调度一群 AI 专家协同工作。

你为什么需要一个成体系的系统提示词仓库？

Section titled “你为什么需要一个成体系的系统提示词仓库？”

大部分 AI 学习者、开发者现在在用大模型时，都会遇到这些痛点：

  • 系统提示词乱七八糟 今天在 Claude 里写了一段不错的系统 prompt，明天在 ChatGPT 里又改了一版，结果散落在各个对话和文档中，根本找不到最佳版本。
  • AI 的角色不稳定 有时你希望它是产品经理，有时是架构师，有时是 QA，但每次都要重新解释，且每次解释都不完全一样，输出质量很难稳定。
  • 缺乏从需求到代码的闭环 即便你写了不少 prompt，更多是一次性的：需求文档一套写法，架构讨论一套写法，开发与测试又是一套，这些提示词之间没有严格的 artifact 流转关系，很难形成可复制的工程流程。
  • 协作成本高 当你想把自己整理的一堆 prompt 分享给同事时，对方往往只能照着一篇长文或一个 Notion 页面挨个复制，完全没有工具化。

如果你也有类似经历，那 BMAD Method 做的事情，可以理解为：

把「AI 系统提示词 + 角色分工 + 流程步骤」工程化、版本化、可移植化，变成一个可以在各种 IDE、大模型工具中反复复用的 AI 团队框架。

BMAD Method 本质在做什么？

Section titled “BMAD Method 本质在做什么？”

BMAD Method 的定位可以概括成一句话：

Breakthrough Method for Agile AI-Driven Development —— 一个让 AI 以敏捷团队的方式参与开发的框架。

它的核心有三层：

  • BMAD Core：一个为人类 + 多智能体协作设计的基础框架，里面包含：

    • 以 Markdown 编写的 Agents（专业角色 persona）
    • 由多个 Agent 组成的 Agent Teams
    • 用 YAML 定义的 Workflows（战略流程）
    • 带 LLM 指令的 Templates（PRD、架构文档、Story 文件等）

  • BMAD Method：站在 BMAD Core 之上，专注于软件开发场景的工作流，把分析 → 规划 → 方案 → 实施整个链路拆解成多个阶段与工件，由不同的 Agent 接力完成。

  • BMAD Builder / Creative Intelligence Suite：

    • BMAD Builder：让你可以在同一套框架上，设计自己的专业 Agent 与模块（如法律、金融、教育等领域）。
    • Creative Intelligence Suite：提供创意思考、头脑风暴类工作流，辅助产品创新、叙事设计等。

换句话说，BMAD Method 不是简单的 prompt 集合，而是把这些系统提示词按：

  • 角色（谁来做）
  • 阶段（何时做）
  • 工件（产出什么）
  • 工具（在哪儿用）

全部结构化整理成了一个可以移植到你项目里的工程方法论与代码仓库。

Section titled “2. 50+ 工作流覆盖完整开发生命周期”

BMAD Method 把整个软件交付过程拆成 4 个阶段：

  1. Analysis（分析）：需求挖掘、市场调研、方案对比
  2. Planning（规划）：PRD、技术规格说明、游戏设计文档等
  3. Solutioning（方案设计）：系统架构、数据流、API 设计、UX 草图
  4. Implementation（实施）：Story 驱动开发、代码实现、测试验证

针对每个阶段，框架都提供了多条工作流，以 YAML 形式定义流程与调用顺序。例如：

  • 用 Analyst 生成 Project Brief
  • 用 PM 生成 PRD
  • 用 Architect 生成 Architecture 文档
  • 用 PO 做文档分片
  • 用 SM 生成 Story 文件
  • 用 Developer/QA 实现与验证代码

这本质上是一套可执行的系统提示词流程，而不是一堆零散的 prompt 范例。

  • BMad Method 标准轨道：用于新功能、新产品或中等规模平台，从 PRD 到架构再到 UX。
  • Enterprise 轨道：面向合规、治理要求高的项目，提供更完整的文档、质量门槛与流程。

在实际使用时，你可以跑 *workflow-init 让 BMAD 自动根据目标与上下文推荐合适的轨道。

一个从需求到上线的系统提示词闭环示例

根据官方技术文章中的电商示例，BMAD 的完整流程大致如下：

  1. Analyst：项目构想与市场分析

    • 命令示例：@analyst Create project brief for modern e-commerce platform

    • 使用 Analyst 的 Brief 作为输入
    • 产出：PRD.md

  3. Architect：系统架构设计

    • 产出：architecture.md，包含技术栈、系统组件、数据流与 API 规范等。

  4. Product Owner：文档分片

    • 将 PRD + 架构文档拆成聚焦的 Epic 文件，例如：
      • epic-1-user-auth.md
      • epic-2-products.md

  5. Scrum Master：生成 Story 文件

    • 针对某个 Epic 生成 Story 文件，如：
      • 1.1.user-registration.story.md
    • Story 中包含上下文、实现细节、验收标准等。

  6. Developer：代码实现

    • 围绕 Story 文件来实现代码（Controller、组件、数据库迁移等），并生成测试用例。

  7. QA：质量验证与追踪

    • 产生包括测试策略、追踪矩阵、质量门报告在内的一系列 QA 文档。

在这一整套过程里，你可以看到：

  • 每个阶段对应一个或多个 Agent
  • 每个 Agent 背后是一整段高度结构化的系统提示词
  • Artifacts（文档 & 代码）通过文件名和目录结构形成严格的追踪链路

Section titled “🧩 为什么需要用量监控”

如果你正在积极使用 Claude Code 或 OpenAI Codex 进行开发，很可能遇到过意外触发速率限制的情况。以下是 CodexBar 必不可少的原因：

教训：观察用户正在用你现有功能拼凑出什么——那才是产品市场契合点的藏身之处。

“更好的工程是工程师拓展人脉和获得影响力的最简单方式。“

Section titled “4. 成为打破专业泳道的通才”

“仅仅因为你是产品工程师，不代表你不能做基础设施。“
