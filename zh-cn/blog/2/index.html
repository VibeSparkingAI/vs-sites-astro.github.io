BMAD v6 安装更新：模块化 + AgentVibes “会说话”的开发体验

2025年12月15日
Vibe Sparking

v6 的核心变化就是“目录/工件体系重做” + “语音能力（AgentVibes）正式集成进 BMAD 工作流”。

1）先说结论：v6 相比 v4，最“能感知”的两点变化

Section titled “1）先说结论：v6 相比 v4，最“能感知”的两点变化”

变化 A：目录从“单核”升级为“Core + Modules + Manifests”的工程化结构。 v4 的用户指南里，个性化偏好（比如技术栈、命名规则）是放在 .bmad-core/data/technical-preferences.md 里管理的。

而 v6（Alpha）的官方定位是“Agent-as-Code + 可复现流水线”，并明确给出 workspace anatomy：.bmad/ 用来放 profiles/blueprints/runs 等“可追踪工件”。

变化 B：AgentVibes 被当成“第一方体验层”接进来：让 BMAD agent 能在 Claude Code 里说话、切声线、用 slash command / MCP 控制。 AgentVibes 的官方说明就直白：最初就是为了给 Claude Code “加声音”，并且提供 MCP（自然语言控制）和 slash commands 两套控制方式。

2）目录结构：v4 的“.bmad-core” vs v6 的“.bmad + modules + artifacts”

Section titled “2）目录结构：v4 的“.bmad-core” vs v6 的“.bmad + modules + artifacts””

按日志里的安装结果来拆

v4（用户常见心智）

Section titled “v4（用户常见心智）”
  • 核心目录：.bmad-core/ 用来放偏好、数据、模板等；例如 v4 用户指南明确提到偏好文件位置：.bmad-core/data/technical-preferences.md。
  • Web Bundle 玩法：dist/teams/... 典型是复制 dist/teams/team-fullstack.txt 去 Web UI/Claude/GPT 里当“团队提示词”。
  • 更像“提示词框架 + 目录约定”，工程化程度够用，但**“产物可追踪/可复现”不是第一目标**。

v6（你这次安装体现出来的结构变化）

Section titled “v6（你这次安装体现出来的结构变化）”

从日志看，v6 安装完成后最关键的“落盘结构”是：

  • 根目录统一落在：<project>/.bmad（你这次是 /Users/vsai/.../vs-ai-agents/.bmad）

  • 模块化安装：Core + bmb + bmgd + bmm + cis

    • Core：安装核心 agent（比如 bmad-master.md）
    • bmb：Builder/Factory（自定义 agent、workflow、module 的“工厂”）
    • bmm：Method（Analyst/PM/Dev/SM/UX 等完整团队）
    • bmgd：Game Dev（并且会把跨模块 workflows “vendoring”进来）
    • cis：Creative Innovation Suite

同时，它还做了 v4 时代不太“显性”的几件事：

  • 编译/生成工件（manifests）：日志里直接报了 “61 workflows、21 agents、5 tasks、1 tools…”——这就是 v6 走向“可版本化/可追踪”的证据之一（工程化明显更强）。这一点也符合 v6 Alpha 强调的“可复现、可审计、可 diff”的方向。

  • Sprint 产物目录变成安装期配置项：你选择了 docs/sprint-artifacts，安装器还会自动创建（便于团队落地敏捷节奏）。

  • 自定义扩展目录被显式分离出来

    • bmad-custom-src：自定义 agents / workflows
    • bmad-custom-modules-src：自定义 modules 这其实是在逼你把“官方模块”和“自定义资产”隔离，方便升级/合并。

  • Sidecar memory 目录：日志里还替换了 {agent_sidecar_folder} → {project-root}/.bmad-user-memory（典型用于 agent 的“旁路记忆/运行痕迹”，也更符合 v6 的“工件化”理念）。

v4 更像“提示词框架 + 目录规范”；v6 更像“带构建产物与可追踪工件的 agent 工程系统”。

3）AgentVibes 集成：从“能用”到“好用”的关键升级

Section titled “3）AgentVibes 集成：从“能用”到“好用”的关键升级”

日志里最“出圈”的部分其实是这一段：

  • 安装器提示 “AgentVibes TTS Injection Summary”
  • 对 .../.bmad/core/agents/bmad-master.md 做了 TTS 注入
  • 备份放在 ~/.bmad-tts-backups/
  • 然后一键拉起 AgentVibes 安装器，往项目的 .claude/ 里塞了 slash commands、TTS hooks、personality templates、背景音乐等

这套设计跟 AgentVibes 的官方说明高度一致：

AgentVibes 到底解决了什么？

Section titled “AgentVibes 到底解决了什么？”
  • 它提供 MCP（自然语言控制）：让助手不必记 slash commands，也能“说：换成 Aria 声线”。
  • 同时也提供 slash commands（零 token 开销），例如 /agent-vibes:list、/agent-vibes:switch Aria。
  • 多 TTS provider 抽象：Piper、macOS Say、ElevenLabs 都在同一个能力层里（你的日志里也体现了 macOS Say 的快速安装路径）。
  • 安装结构明确写了：会把 commands/hooks/personalities/plugins 放进 .claude/。

为什么这对 BMAD v6 特别重要？

Section titled “为什么这对 BMAD v6 特别重要？”

因为 v6 的方向是“多 agent、多 workflow、多工件”，人的注意力会被大量“过程信息”打散。TTS 反而能带来两个好处：

  1. 状态反馈更强：开始做什么、做完了什么，耳朵一听就知道，不用盯屏幕。
  2. 角色感更强：PM/SM/Architect/Dev 这类角色切换，会更有“团队协作”的沉浸感（AgentVibes 的 BMAD 集成也在强调这点）。

4）给从 v4 到 v6 的总结

Section titled “4）给从 v4 到 v6 的总结”

  1. 把 v4 的偏好文件迁移思路定好 v4 偏好在 .bmad-core/data/technical-preferences.md。 v6 更强调 workspace + policies + manifests（.bmad/ 下的结构）。 建议你做法是：保留 v4 偏好内容作为“团队约定”，再逐步映射到 v6 的 profile/policy 体系（不要一次性硬迁，容易丢语义）。

  2. 把“自定义资产”放进你已经配置好的两个目录

  • agent / workflow → bmad-custom-src
  • module → bmad-custom-modules-src 这样未来升级 v6 alpha/beta 时，冲突会少很多。
  1. AgentVibes 用法上，建议默认“slash commands 优先” AgentVibes 官方也强调：MCP 更方便，但会吃上下文 token；slash commands 更省。 也就是说：日常用 / 命令，偶尔再用自然语言 MCP，体验和成本会比较平衡。

破局“无设备复现”难题：Antigravity 如何通过代码推理精准定位复杂 Bug

2025年12月11日
Vibe Sparking

在软件开发中，最让工程师头疼的问题莫过于——“用户那里有问题，但我这边复现不出来”。

特别是在涉及特定硬件设备、特定用户角色或特定网络环境的场景下，复现问题往往需要昂贵的设备支持或复杂的环境搭建。然而，作为智能编码 Agent，Antigravity 展示了一种全新的解题思路：在没有物理设备的情况下，通过纯粹的代码逻辑推理和全链路数据验证，精准定位并修复 Bug。

本文将以最近解决的一个真实案例（Issue #181：医生查看患者数据时出现异常记录）为例，复盘这一“盲态解题”过程。

挑战：看不见的现场

Section titled “挑战：看不见的现场”

问题描述：某医院医生在使用 PC/平板查看特定患者数据时，意外看到了其他患者的记录。

难点：

  1. 无环境：开发环境无法连接医院内网，无法登录该特定医生的账号。
  2. 无设备：无法操作医生使用的特定终端设备。
  3. 信息有限：只有一张截图和一句简短的描述：“查房时看到了别人的数据”。

传统做法通常是：借账号、找设备、连内网、尝试复现。如果复现失败，Bug 往往会被搁置。

他们拉了个会，专门在白板上写—— “什么事最耗精力？什么最重复？哪些活根本不配占我们这么多时间？”

答案出来之后，他们没急着谈“AI 能做什么”，而是只盯一个问题：

哪些事情，我们最想再也不用亲手做一遍？

二、法务工作流怎么拆：4 个真实场景

Section titled “二、法务工作流怎么拆：4 个真实场景”

Anthropic 法务团队花了几个月，把痛点一点点变成了“可复制工作流”。

1）营销素材自助审核：先过 AI 这一关

Section titled “1）营销素材自助审核：先过 AI 这一关”

以前的流程，你肯定见过：

市场部在发布前一晚，把落地页、Push、海报一股脑甩给法务： “帮忙今天之内过一下，明天要上线。”

法务只好：打开文档 → 从头看一遍 → 留一堆 comment → 来回几轮修改。 经常一拖就是几天。

Anthropic 的改造方式很简单，但很狠：

他们在 Slack 里做了一个**“营销素材自查工具”**（Self-Review Tool）：

  • 市场同学把文案粘进去

  • Claude 调用一个专门的 “法务 Skill 文件”

    • 里面是：历史审核意见、常见风险点、团队约定的判断标准

  • Claude 自动标出：

    • 夸大宣传？
    • 数据是否有来源？
    • 是否侵犯肖像权、名誉权？
    • 有没有暗示不该承诺的东西？

  • 每个问题打风险等级：低 / 中 / 高

  • 顺带给出“修改建议版本”

只有过了这一关，才会真的创建工单，进法务队列。

StartupHub 的报道里提到，这一套下来，原来要几天的审核，现在可以压缩到几个小时，而且 Slack 里还能自动生成一条总结消息，帮法务把风险点捋清楚。

这件事本质是：

把“问诊前的表格”和“基础体检”交给 AI 做， 真正的诊断，还是医生说了算。

2）兼职 / 外部活动审批：AI 先问清楚细节

Section titled “2）兼职 / 外部活动审批：AI 先问清楚细节”

很多公司都有类似流程：

员工想去接个顾问、搞个副业、去非营利机构当个董事，都要走“外部商业活动审批”。

以前的做法：法务要一对一问清楚背景、利益冲突、时间占用、竞业风险……来回问半天。

现在 Anthropic 是这么玩：

  • 员工先填一个结构化表单
  • Claude 读完表单内容
  • 如果信息不够，它会自动追问补充问题
  • 再根据公司规则，给出一个建议结论 + 风险说明
  • 最后进法务队列，由律师确认通过与否

律师们现在主要盯两类：

  • 真·疑难杂症
  • 风险比较高的边界情况

把“90% 标准流程 + 10% 极值判断”拆开了。

3）隐私影响评估（PIA）：让 AI 学会“写模板”

Section titled “3）隐私影响评估（PIA）：让 AI 学会“写模板””

隐私类项目，每上一个新功能，就要重新写一份 PIA。

痛点在于：

  • 模板类似，但每次都得重写
  • 既要贴合具体场景，又不能漏掉必写模块

Anthropic 的做法有两步：

  1. 用 MCP（Model Context Protocol）把 Claude 接入一个 Google Drive 文件夹

    • 里面是他们历史所有 PIA 文档

  2. 给 Claude 配一个“写 PIA 的 Skill”，告诉它：

    • 文档结构
    • 每一块重点关注什么
    • 公司特别在意的隐私点

律师只要说一句：

“根据之前这些案例，帮我写一个这次新功能的 PIA 草稿。”

Claude 会：

  • 先理解历次 PIA 的风格
  • 再按格式写出初稿
  • 把历史关注点迁移到新场景

律师做的工作，从“从 0 写一遍”，变成“改一篇已经 80 分的稿子”。

4）Legal 玩出了“摩斯密码灯”，但重点在于“门槛降到了人话”

Section titled “4）Legal 玩出了“摩斯密码灯”，但重点在于“门槛降到了人话””

有个细节挺好玩：

Mark 用 Claude Code 做了个“法律台灯”，能闪烁摩斯密码来收发信息——纯属个人兴趣项目。

但这个故事背后的点很重要：

连一个不会写代码的律师，都能靠“跟 AI 说人话”，做出一个完整的小应用。

对企业来说，这意味着：

  • 法务、合规、运营这些本来“不会写代码的人”，可以自己动手搭工作流
  • 不用一上来就排工程资源、立大项目
  • 他们只要把规则讲清楚，就能让 AI 帮忙落地成工具

这就从根上改变了： “只有工程师才能做工具”这件事。

三、这套法务工作流的“底层积木”

Section titled “三、这套法务工作流的“底层积木””

Anthropic 法务这套体系背后，其实就三块积木：

1）从“技术能做什么”换成“我不想干什么”

Section titled “1）从“技术能做什么”换成“我不想干什么””

Mark 给的建议很直接：

不要上来问“AI 能干嘛”， 先问“我们最想摆脱的活是什么”。

他们就是按这个思路，白板上写：

  • 哪些是重复文书
  • 哪些是每天都在做的判断
  • 哪些是完全可以标准化的流程

确认完，再看：

  • 能不能先让 AI 做第一轮
  • 把人类的时间保留给争议点、例外情况、战略判断

2）Skills：把团队经验打包成“可调用资产”

Section titled “2）Skills：把团队经验打包成“可调用资产””

官方博客里把 Skills 定义成：一份文件，里面写着指令、脚本、参考资料，让 Claude 在做某个任务时能“变成行家”。

对法务来说，Skills 用在两件事：

  • 保证规则统一

    • 营销审核 Skill：把“团队历史审核意见”固化进去
    • 不同领域有不同 Skill：雇佣、商业、隐私、公司治理……

  • 保持文风统一

    • Mark 让 Claude 读了自己 10 篇备忘录
    • 让它学会自己的写法、结构、口吻
    • 以后让 Claude 起草 memo，可以直接“写出他的风格”

你可以把 Skill 想象成：

团队里那位资深律师的“外挂记忆卡”。

用的时候插上，不用的时候拔掉。

3）MCP：把分散的信息接成一张“法务知识网”

Section titled “3）MCP：把分散的信息接成一张“法务知识网””

另一个关键积木是 MCP（Model Context Protocol）。

Anthropic 法务用它把 Claude 接到了：

  • Google Drive：历史合同、PIA、模板、政策
  • JIRA：需求与工单
  • Slack：沟通记录
  • 日历：关键会议、审批节点

这样做有两个效果：

  • AI 不再“瞎编”，因为它有你自己的资料可查
  • 每个工作流，都能拿到刚好够用的上下文

你可以简单理解：

MCP = 给 AI 接上“公司内部网”。

Skills = 教它“怎么用这些资料办事”。

四、整套方案最关键的一条：人绝对不能缺席

Section titled “四、整套方案最关键的一条：人绝对不能缺席”

无论是 Anthropic 自己的博客，还是 StartupHub 的报道，都在反复强调一句话：“法务一定要在人在环中。”

他们的实际做法是：

  • AI 负责：

    • 首轮审稿、找问题
    • 初步打风险等级
    • 草拟修改建议和文档

  • 律师负责：

    • 核查事实与引用
    • 判断“这个风险公司能不能接受”
    • 对外最后拍板

Anthropic 明说了：

AI 会幻觉，这是客观事实。

所以他们的设计就是：AI 做“加速器”，人做“刹车 + 方向盘”。

对任何想在公司里推 AI 的人来说，这句话很有用：

你不是在用 AI 替代法务， 你是在用 AI 让法务不再做“改格式、找错别字”的机器。

五、这套玩法，国内团队怎么“抄作业”？

Section titled “五、这套玩法，国内团队怎么“抄作业”？”

说完 Anthropic，我们来聊“怎么抄”。 不管你现在用的是 Claude、ChatGPT、DeepSeek 还是自家的模型，其实都能借这套思路。

你可以这样开局：

1）列出你团队的“法务地狱清单”

Section titled “1）列出你团队的“法务地狱清单””

找一天，拉上法务 / 合规 / 市场 / 产品：

  • 写下最不想亲自再做一百遍的任务
  • 按“重复度 × 风险可控度”来排优先级
  • 选 1～2 个“现在就可以试水”的

典型候选：

  • 营销素材初审
  • 合同标准条款比对
  • 数据保护、隐私类模板
  • 标准问卷、合规 Q&A

2）先搭“人 + AI 双层流程”，再谈自动化

Section titled “2）先搭“人 + AI 双层流程”，再谈自动化”

一上来就做“端到端自动审批”，往往会死在风控和信任上。

更稳的方式是：

  • 让 AI 做“内部预审 + 辅助草稿”

  • 真正的决策流程不变

  • 用 1～2 个月，验证：

    • 它有没有明显错误
    • 速度是不是真的提升
    • 团队有没有觉得更好用

过了这一关，再看要不要升级成：

  • 自助工具
  • 与内部系统打通的工作流
  • 加入模板、Skills、团队规范

3）把“规则”和“风格”慢慢喂给 AI

Section titled “3）把“规则”和“风格”慢慢喂给 AI”

可以学 Anthropic 的两步法：

  • 规则

    • 整理一份“本公司法务审核 checklist”
    • 例如：不可使用的词、必加免责声明、必须核实的数据项
    • 编成一个长期维护的“AI 提示文档”或 Skill

  • 风格

    • 丢给 AI 你们团队写得比较满意的几份合同评审意见、备忘录
    • 让它总结：常见结构、表达习惯、措辞偏好
    • 以后生成草稿时，都要求“按这个风格写”

慢慢地，你会得到一个：

会按你们自己的习惯做事的 AI 小同事。

4）别忘了收益要算清楚

Section titled “4）别忘了收益要算清楚”

做这类自动化，到底值不值？ Anthropic 在另一份《How Anthropic teams use Claude Code》文档中提到，他们内部很多团队都用这套方式来评估效率提升，而法务是收益特别明显的一块。

你可以简单量化：

  • 每条营销素材的平均审核时间
  • 每份合同从“发给法务”到“给出意见”的周期
  • 每个律师每周花在重复文书上的小时数

做个前后对比，很容易说服老板继续投资源。

六、一点小总结

Section titled “六、一点小总结”

用一句话概括 Anthropic 法务这次的实践：

用 AI 做“懂规则的初级助理”， 把资深律师从体力活里解放出来，去做真正需要判断力的事。

这件事对所有“又懂业务又懂一点点 AI 的人”来说，都是一个巨大的机会：

  • 你可以成为那个把团队经验沉淀成 Skills 的人
  • 你可以搭起第一个“AI + 法务工作流”
  • 你可以把 AI 从“聊天玩具”，变成真正的“流程基础设施”

给 Claude 装上「长期记忆」：claude-mem 实战体验与使用建议

2025年12月9日
Vibe Sparking

claude-mem 这玩意儿，能让 Claude 真正「记住你」

你有没有这种感受：

每次打开 Claude / ChatGPT 写代码， 都要从头给它讲一遍项目背景：

「我在做个后台管理系统，技术栈是……」 「上次我们做到用户登录，这次要加的是……」

讲着讲着，人都烦了。

模型不是真的笨，只是没记忆。

今天这篇，就带你看看 GitHub 上一个很有意思的小项目： claude-mem——给 Claude Code 装上「长期记忆」的大脑。

一、claude-mem 到底干嘛的？

Section titled “一、claude-mem 到底干嘛的？”

一句话版：

它会自动记录你在 Claude Code 里的操作，自己总结、压缩， 下次再开同一个项目，Claude 会「主动想起」之前的上下文。

更细一点说，它是：

  • 一个 Claude Code 插件
  • 会自动捕获：你的指令、工具调用、读写文件等操作
  • 后台用 Claude 的 Agent SDK 做总结、提炼要点
  • 存进本地数据库和向量库，支持全文 + 语义检索
  • 下次你再开 Claude Code，它会把「最相关」的历史内容 当作上下文自动塞回新对话里

所以体验上就是：

你继续写代码，Claude 变成了那个 「对项目背景比你还熟」的搭档。

二、它是怎么帮 Claude「开挂记忆」的？

Section titled “二、它是怎么帮 Claude「开挂记忆」的？”

官方 README 里画了一个很清晰的流程图，我给你翻译成人话版：

1）会话开始：先喂点「最近发生的事」

Section titled “1）会话开始：先喂点「最近发生的事」”

每次你在 Claude Code 里开一个新 session， claude-mem 会先从数据库里挖出「最近的关键信息」， 整理成一段简短上下文， 自动注入到这次对话开头。

就好像跟朋友见面前，先在脑子里过一遍：

「上次我们聊到登录注册，中间踩了哪些坑？」

2）你在写代码：它在后台默默记笔记

Section titled “2）你在写代码：它在后台默默记笔记”

在整个开发过程中，它会持续记录：

  • 你发给 Claude 的重要指令
  • Claude 调用工具时的「观察结果」（读了哪个文件、写了什么）
  • 一些关键的中间过程

这些原始信息不会直接拿来当上下文那样粗暴重放， 而是先交给 Worker Service，用 Claude 自己的 Agent SDK 来做总结：

「这次 session 里有哪些值得记住的‘经验’？」

比如：

  • 我们决定弃用某个第三方库
  • 重构了某个模块的接口设计
  • 修了一个阴间 Bug，并总结了成因

3）会话结束：生成一份「学习笔记」

Section titled “3）会话结束：生成一份「学习笔记」”

session 结束时，它会做一件事：

把这整个会话浓缩成一份「项目变更日志 + 决策摘要」。

然后存进本地 SQLite（支持全文搜索）， 同时也写入 Chroma 这类向量数据库，方便走语义检索。

下一次 Claude 要查历史，就不用「全量翻旧账」， 而是通过智能搜索 + 分层展开（progressive disclosure） 把少量但关键的上下文塞回对话中。

三、对开发者来说，爽点在哪？

Section titled “三、对开发者来说，爽点在哪？”

1）长周期项目：不用每次都复盘一遍

Section titled “1）长周期项目：不用每次都复盘一遍”

想象一个真实场景。

你在做一个 SaaS 管理后台，跨度 3 个月。 中间还要被各种会议、需求变更打断。

没装 claude-mem 的时候，每次回到项目是这样的：

  • 先翻 Git log 和文档
  • 再用大段提示把背景讲一遍给 Claude
  • Claude 才慢慢进入状态

装上 claude-mem 之后：

  • 新 session 一开，Claude 自己就知道：

    • 项目在做什么
    • 最近改了哪些模块
    • 哪些坑已经踩过、别再踩第二次

你只需要说一句：

「继续上次我们做到的权限管理，把导出功能补完。」

Claude 就能「接得上话」，而不是一脸懵逼。

2）mem-search：给你一个「项目时光机」

Section titled “2）mem-search：给你一个「项目时光机」”

claude-mem 还内置了一个很有意思的技能：mem-search。

你可以直接用自然语言问它：

  • 「帮我找下我们之前是怎么设计登录接口的？」
  • 「之前那个 file upload 的 Bug，我们当时是怎么修的？」

Section titled “拆解真正的 AI 芯片战争”

最近，市场上最流行的一句故事是：

“Google TPU 崛起，NVIDIA 要被干掉了；AWS 有 Trainium，微软有 Maia，自研 ASIC 要把 GPU 挤下王座。”

但如果你看一下摩根士丹利互联网分析师 Brian + 半导体分析师 Joe Moore 的那种“内部视角”，再对照公开数据，会发现一个更接近真实产业逻辑的图景：

现在不是“谁替代谁”，而是“只要能算，什么都要”。 短期是算力饥荒，长期是利润向少数掌握标准和成本优势的玩家集中。

下面这篇文章，我按四个问题来讲：

  1. 真的进入了“无限算力时代”吗？

  • 以及站在人性和利益上的“谁在忽悠谁”。

一、算力洪水：是“永远不够”，还是“短期饥荒”？

Section titled “一、算力洪水：是“永远不够”，还是“短期饥荒”？”

1. 有多“洪水”？先看几组硬数据

Section titled “1. 有多“洪水”？先看几组硬数据”

（1）数据中心 CapEx 的爆发

  • Dell’Oro 的报告显示：2024 年二季度，全球数据中心资本开支同比增长 46%，主要由部署 AI 加速服务器（GPU / TPU / 自研加速卡）驱动。
  • 同一机构后续统计：2024 年全年全球数据中心 CapEx 跳升到 4550 亿美元，增速 51%，并预计 2025 年仍保持两位数增长，核心动力仍然是 AI 加速服务器。

这些数字基本等于在说：

“大家在数据中心上砸钱的速度，确实到了历史罕见水平，而且就是为了 AI。”

（2）GPU 这边：NVIDIA 的“吸金黑洞”

  • NVIDIA 最新财报：2025 财年 Q4（截至 2025-01-26）单季营收 393 亿美元，同比 +78%，主要由数据中心业务拉动。
  • 行业分析估算：2025 年 Q3，NVIDIA 数据中心部门单季就已突破 500 亿美元收入，毛利率接近 75%。
  • 针对“数据中心 GPU”单独的市场，研究机构预计：2025 年约 105 亿美元，2035 年有望扩张到 770 亿美元，年复合增速超 22%。

（3）Blackwell 的真实状态：不是“没人要”，而是“根本不够卖”

NVIDIA 管理层在 2024 年底、2025 年的多次财报会上反复强调：

  • Hopper 和 Blackwell 平台都存在供应约束；
  • Blackwell 的需求预计在 2026 财年连续多个季度将“超过供给”，目前很多明年的产能已经被提前锁定。

各家研究报告和媒体基本达成共识：

高端 AI GPU（H100/H200/Blackwell）处于“排队买 + 配给制”的状态。

所以，你原来那句：

“行业不是卷架构，而是卷能不能拿到卡”

在当下 1～3 年的窗口期里，和公开数据是吻合的。

2. 但“算力永远不够”是情绪，不是物理定律

Section titled “2. 但“算力永远不够”是情绪，不是物理定律”

这一点对投资者很重要。

（1）物理和基础设施约束

  • AI 数据中心不是只买卡，还要：

    • 电力：大规模 AI 机房功耗动辄几百兆瓦；
    • 铜和材料：分析指出 AI 数据中心每 MW 用铜 27–33 吨，按当前规划，到 2035 年现有和在建铜矿产能只能覆盖约 70% 需求，2025 年就可能出现 30 万吨级铜缺口。

这些约束在告诉我们：

“无限算力”更像是需求侧的愿望，现实世界会用电网、材料和审批给你踩刹车。

（2）现金流和股东约束

  • 云厂商如今每年在 AI 基建上的投资都几十上百亿美元，
  • 但股东会问：“这些 GPU / TPU 多久能回本？AI 产品赚的钱跟得上吗？”

如果未来两三年“大模型赚钱速度”不及市场预期， CapEx 的增速一定会从 50%+ 降下来，这跟任何一轮基础设施周期（4G、云计算、光纤）都类似。

总结这一点：

  • 事实层：

    • 现在确实是算力饥荒 + 高端 GPU 极度紧缺的阶段；

  • 观点层：

    • “算力永远不够”是被情绪放大的说法；

  • 投资含义：

    • 短期可以用“抢卡逻辑”解释估值，
    • 中长期必须盯住：电力 / 材料 / 回报率这三把剪刀什么时候开始剪掉增速。

二、“谁有我用谁”：多架构并存是被逼出来的，不是优雅选择

Section titled “二、“谁有我用谁”：多架构并存是被逼出来的，不是优雅选择”

你总结的第二点是：

1. 事实面：头部玩家确实在走“多架构路线”

Section titled “1. 事实面：头部玩家确实在走“多架构路线””

（1）云厂商 / Hyperscaler 的结构

多家研究机构（Kearney、MLQ.ai 等）在 2025 年的报告里都指出：

  • 目前数据中心 AI 加速卡市场仍由 NVIDIA 主导，
  • 但 Google TPU、AWS Trainium、Microsoft Maia 这类自研芯片， 预计在 2020s 后半段能拿到 15–20% 的内部部署份额（主要是各家自己内部工作负载）。

换句话说：

  • 对 Google/AWS/Microsoft 这些巨头来说：

    “GPU + 自研 ASIC 混用”已经是被公开写进路线图的现实。

（2）具体厂商怎么做？

  • Google Cloud：AI Hypercomputer

    • 一边在推出新一代 Trillium TPU（第六代），
    • 一边在官方博客里明确宣布：会把 NVIDIA Blackwell 平台（B200 / GB200）作为 Hypercomputer 架构的一部分，客户可以在 TPUs 和 NVIDIA GPUs 之间选择。

  • AWS：Trainium + NVIDIA GPU 同时卖

    • re:Invent 2024 / 2025 上，AWS 一边发布 Trainium2/3/4 自研 AI 芯片和 UltraServer，定位为“更便宜的训练 + 推理算力”；

  • Microsoft：Maia 内部用，NVIDIA 对外卖

    • 同时依然在 Azure 上大规模部署并宣传最新的 NVIDIA GPU（包括 GB300 NVL72 集群），还发布联合博客说明如何在 Azure 上“挖尽每一块 GPU 的价值”。

这些都说明：

在最头部的那圈玩家里，“多架构并存”已经是事实。 他们真的在做你说的那句话：“谁能给我算力，我就用谁。”

2. 但这不是整个行业的常态，而是“巨头的 privilege”

Section titled “2. 但这不是整个行业的常态，而是“巨头的 privilege””

这里的逻辑陷阱在于：

看到前沿模型公司可以同时跑 4–5 种架构，就以为所有 AI 公司都可以这么玩。

现实更接近：

  • 只有云巨头 / 顶级模型公司，才有能力：

    • 维护多套底层硬件（GPU / TPU / Trainium / Maia / 其他 ASIC）；
    • 投入工程资源把模型移植到各种后端（CUDA、ROCm、XLA、自研 SDK）；
    • 搭建统一调度 / 编译 / MLOps 层去屏蔽异构硬件的差异。

  • 大多数 AI 创业公司、传统企业客户的行为其实是：

    “只要能租到足够的 NVIDIA GPU，我不会主动去折腾别的架构。”

这点从生态能看得很清楚：

  • CUDA 在框架、工具链、教程、开源项目中的支持度，仍然远高于其他体系；
  • 很多三方推理/训练平台，做“多后端适配”时，依然是把 NVIDIA 当默认第一优先级。

所以：

  • 你那句“多架构成了常态”，

    • 对 Google/AWS/Microsoft/OpenAI/Anthropic 这几家，是对的；
    • 对 普通 AI 公司，更多仍是“以 NVIDIA 为主，其他做补充选项”。

3. 人性 & 利益：多架构背后是谁在算计谁？

Section titled “3. 人性 & 利益：多架构背后是谁在算计谁？”

把这件事从“技术理想主义”拉回到“利益现实”：

  • 对大模型公司 / 云厂商来说，多架构的真正动机只有三个：

    1. 活命：在算力短缺时期，“谁有卡谁是亲爹”；
    2. 压价：用 TPU / Trainium / Maia 在谈判桌上给 NVIDIA 施压；
    3. TCO：把部分推理负载迁到更便宜、更节能的 ASIC 上，降低长期成本。

  • 对NVIDIA来说，最大的目标是：

    “让 CUDA + 平台生态成为‘行业默认标准’， 这样即使别人造出便宜芯片，迁移成本也足够高。”

结论：

多架构并不是“技术多元之美”， 而是一场供应链博弈 + 成本博弈 + 议价权博弈。

Section titled “1. Google 自己怎么说？”

Section titled “2. 逻辑上为什么“不能只用 TPU”？”

（1）业务结构差异

  • 自家核心业务（搜索、广告、YouTube 推荐、GWS 等）：

    • 可以强行按自己的节奏迁到 TPU 上，这里是“内部强控 + 长期 ROI”。

  • 对外的云业务：

    • 客户有大量基于 CUDA 的代码、模型和工具链，
    • 如果 Google 说“只能用 TPU，不给你 GPU 选项”，那等于把客户往 AWS / Azure 赶。

所以对 Google 来说，GPU 和 TPU 分工是：

  • TPU：内部大规模生产 + 部分对外，重点是拉低自己的长期算力成本；
  • NVIDIA GPU：生态标配 + 客户需求，是留住云客户的必需品。

（2）路径依赖和迁移成本

就算 TPU 在某些任务上更便宜、更高效， 但对大部分已经被 CUDA 教育过的开发者来说：

“重写一堆代码 + 重建全套工具链” 的成本，往往大于“多付一点 NVIDIA 税”的痛感。

  • 自研 ASIC 的本质，是给巨头自己“减轻长期算力成本 + 提升供应链安全”的工具；

  • 它不是为了“在公开市场上卖到把 NVIDIA 挤死”，

  • 而是为了在谈判桌上说：

    “我不是只能用你家卡，我自己也能做，价格别太离谱。”

所以你那句 —— “TPU 不是替代，而是补充” —— 在产业结构层面是完全成立的。

从现有数据大概能看出：

  • NVIDIA：

    • 单季数据中心收入 500 亿美元量级，毛利率约 75%，这已经是典型“印钞机”水平。

  • 其他厂商（包括部分 GPU 厂 / ASIC 厂）：

    • 收入在几十亿到十几亿美元之间，
    • 毛利率、议价权、生态粘性都难以和 NVIDIA 同日而语。

行业分析普遍观点是：

目前 AI 加速芯片利润池，高度集中在少数几家头部厂商， 其他更多是“有营收、有故事，但未必有厚利”。

对散户来说， “大家都赢”的叙事很危险 —— 它会诱导你去买一篮子边缘玩家， 而真正挣钱的，可能只有那几家掌握标准 + 生态 + 成本曲线的公司。

Section titled “2. 自研 ASIC 是长期威胁，主要对谁？”

自研芯片的长期影响，不是“马上干翻 NVIDIA”， 而是逐渐改变**“谁吃掉多少利润”的分配**。

  • 对 NVIDIA：

    • 短期：依然最受益，所有云厂商还在疯狂买它的卡；
    • 中长期：一旦 TPU / Trainium / Maia 在各家内部占到 15–25% 份额， 就会开始削弱 NVIDIA 的终极定价权（也就是大家吐槽的“GPU 税”）。

  • 对二线芯片厂 / 小厂：

    • 上有自研 ASIC 压价，下有 NVIDIA 生态压制，

    • 它们反而处在最难受的位置——

      “性能差一些、生态弱很多、价格又不便宜到极致”，

    • 很可能在淘汰赛中先出局。

所以：

“自研 ASIC 的子弹，未必先打在 NVIDIA 身上， 很可能先打中的是一圈缺乏生态、规模和成本优势的中小供应商。”

3. “无限算力时代”的三把剪刀：电力、资本、监管

Section titled “3. “无限算力时代”的三把剪刀：电力、资本、监管”

再强调一遍，算力不可能真的无限扩张：

  1. 电力和材料：

    • AI 数据中心对电力、变压器、铜、冷却的需求，开始和全社会的能源基础设施抢资源，
    • 铜的长周期供给已经被机构预警会在 2030 年前后出现更严重的缺口。

  2. 资本回报率：

    • 如果 AI 应用的商业变现赶不上 CapEx 投入的速度，
    • 股东会逼云厂商下调“砸钱曲线”，这对上游芯片厂就是直接刹车。

  3. 政策与监管：

    • 能耗、碳排、数据中心选址、AI 风险监管等，一旦上量，都会变成硬约束，
    • 这方面现在还在酝酿阶段，一旦落地，会立刻影响算力扩张节奏。

Section titled “五、普通投资者该怎么用这套认知？”

  1. 算力芯片 / 模块提供者

    • NVIDIA、AMD、自研 ASIC（TPU / Trainium / Maia）、部分新型架构厂商；

2025保姆级教程：手把手教你申请Google开发者账号（含邓白氏/支付攻略）

2025年12月5日
Vibe Sparking

【导语】 随着国内移动互联网流量见顶，“出海”已成为众多开发者的必选项。而作为全球最大的安卓应用分发平台，Google Play 无疑是出海的第一站。

然而，从2024年开始，Google对开发者账号的审核力度空前严格。很多小伙伴卡在了注册环节：要么网络环境不过关被秒封，要么卡在邓白氏编码申请，甚至连25美元都付不出去。

今天，我们就来拆解注册Google Play开发者账号最关键的三个环节，助你顺利开启出海掘金之路！

第一关：稳定的“网络环境” (Network)

Section titled “第一关：稳定的“网络环境” (Network)”

—— 账号防关联的第一道防线

很多新手在注册时最大的误区是：只要能打开Google网站就行。大错特错！

Google的风控系统非常强大，它会记录你的IP地址、设备指纹和浏览器Cookies。对于开发者注册来说，“能访问”只是基础，“干净且稳定”才是核心。

💡 避坑指南：

  1. 固定IP很重要： 注册过程中（以及后续登录后台），尽量保持IP地址的固定。如果你的IP一会儿在跳到美国，一会儿跳到新加坡，Google会认为账号存在安全风险或被盗，直接导致封号或验证失败。
  2. 浏览器隔离： 建议使用Chrome的无痕模式，或者专门创建一个新的浏览器配置文件（Profile）用于注册。坚决不要在一个浏览器内同时登录多个不同的开发者账号，一旦其中一个出问题，很容易因为Cookies关联而导致“连坐”。
  3. 物理环境： 如果条件允许，使用一台从未注册过Google账号的干净电脑进行操作是最稳妥的。

⚠️ 警告： 网络环境的纯净度直接决定了账号的“存活率”。很多“秒封”的情况，都是因为IP地址曾被列入黑名单。

第二关：邓白氏编码 (D-U-N-S Number)

Section titled “第二关：邓白氏编码 (D-U-N-S Number)”

—— 企业身份的“全球身份证”

自2023年底起，Google Play强制要求注册“组织/公司”类型的账号必须提供邓白氏编码（D-U-N-S Number）。这成为了很多中小团队最大的拦路虎。

什么是邓白氏编码？ 它是一个由邓白氏公司（Dun & Bradstreet）分配的9位数字，用于全球范围内识别企业的身份。简单来说，就是验证你家公司是不是真实存在的。

🛠️ 申请攻略：

  1. 准备材料： 营业执照原件及英文翻译件（注意：公司名称的英文翻译必须标准，因为这将显示在Google Play商店的开发者信息中）、公章、联系人信息。
  2. 申请渠道：
    • 推荐渠道： 通过 Google 提供的专用链接 或 Apple开发者支持页面 进入邓白氏申请入口。通常这些巨头合作通道的处理速度比直接去邓白氏官网申请要快，且通常免费。
  3. 时间预期： 申请提交后，通常需要5-14个工作日。拿到编码后，可能还需要等待Google数据库同步（最长可能需要48小时），之后才能在Google Play后台填入。

📝 划重点： 申请邓白氏编码时的公司英文名、地址，必须与你在Google Play后台填写的完全一致！任何一个字母的差异都可能导致验证失败。

第三关：支付注册费用 (Payment)

Section titled “第三关：支付注册费用 (Payment)”

—— 25美元的入场券

不同于Apple Developer每年99美元的订阅制，Google Play目前实行的是一次性支付25美元（约合人民币180元左右）的注册费。

听起来很简单？但很多国内开发者倒在了“支付失败”这一步。

💳 支付难点与对策：

  1. 卡种限制： Google Play不支持支付宝或微信支付。你需要一张支持外币支付的信用卡，通常是 Visa 或 MasterCard。

引言：AI 真的让工程师更高效了吗？

Section titled “引言：AI 真的让工程师更高效了吗？”

这一年，身边做开发的朋友，都在问同一个问题：

“我天天在用 Claude / Copilot，但到底值不值？有没有被厂商忽悠？”

刚好，Anthropic 自己把内部 132 名工程师、研究员的真实使用情况，做成了一份很详细的调查和访谈，外加好几张数据图。

这就像是AI 厂商给自己做了一次体检报告。 谁在真香，谁在踩坑，数据都写在脸上了。

今天就带你用“打工人视角”，拆一下这份报告，顺便聊聊： 在软件工程里，哪些任务用 AI ROI 最高，哪些地方别急着全交给模型。

一、这是一份什么样的内部调查？

Section titled “一、这是一份什么样的内部调查？”

Anthropic 找了 132 名工程师和研究员，问了他们一堆非常具体的问题：

  • 你每天有多少工作是和 Claude 一起做的？
  • 主要用它干什么？调试、写新功能、重构、写测试、写文档，还是做设计和规划？
  • 对每一类任务，用 Claude 后花的时间变多还是变少？
  • 产出的量（代码、文档、可交付成果）变多还是变少？

他们还做了对比：

  • 一年前，员工觉得自己大概有 28% 的工作在用 Claude，自报生产力 +20%。
  • 到了现在，这个数字涨到 59% 的工作用 Claude，平均自报生产力 +50%，其中大约 14% 的“重度玩家”认为自己提高 超过 100%。

另外一条独立研究也挺猛：Anthropic 用 10 万条真实对话估算，中位任务的完成时间减少了 80% 左右。

听起来很夸张，但工程师都懂——自报生产力要打个折。 所以更有价值的是：按任务细分，哪些地方真的收益大，哪些地方水分多。

二、整体现象：时间只省一点点，产出涨得很猛

Section titled “二、整体现象：时间只省一点点，产出涨得很猛”

那张你看到的 Figure 就是核心结论之一：

  • 左边：用 Claude 后，做某类任务 花的时间 是减少、没变，还是增加
  • 右边：在同样时间里，产出的工作量 是变多了多少

把所有任务拉开来看，有两个有意思的模式：

  1. 大多数任务，时间是略微下降的

    • 很多点落在 “-1 左右”（轻微减少），
    • 意味着：并不是每个人都节省了一半时间，更多是省一点点。

  2. 但产出量普遍涨得更明显

    • 右图里，大部分点在 “+1 ~ +2” 区间，
    • 说明即便时间只省了一点，写出来的东西多很多。

还有一个隐藏细节： 原始数据里，有一小撮人，在某些任务上 时间反而变多了。

访谈里，他们的解释特别真实：

  • 要帮 Claude 收拾它写出来的烂摊子，重搞调试和清理
  • 要花精力理解自己没写过的代码
  • 或者因为有了 Claude，干脆做得更细、更认真，比如更多测试、更多探索

一句话总结：

时间并不是凭空消失，而是被“再投资”到了更多工作上。

三、从 ROI 看任务类型：前端最香，设计和 Infra 最吃亏

Section titled “三、从 ROI 看任务类型：前端最香，设计和 Infra 最吃亏”

结合图里的分布，再对照我自己的使用体验，能看到一个很清晰的 ROI 排名。

1）ROI 天花板：前端、新功能、调试、测试、文档

Section titled “1）ROI 天花板：前端、新功能、调试、测试、文档”

这些任务，有几个共同点：

  • 结果非常容易验证

    • 前端：跑起来一看 UI 就知道有没有问题
    • 调试：复现 bug、跑测试，要么过要么挂
    • 测试和文档：格式、覆盖范围一下子就能看出来

  • 任务边界比较清晰

    • “把这个组件改成响应式布局”
    • “为这段代码补一组单测”
    • “帮我写一段 README，目标读者是新同事”

这类工作里，人类最烦的是： 机械、重复、写起来枯燥，但好不好一眼就能看出来。

这刚好踩中了 Anthropic 员工总结的一个关键词：

“我更愿意把容易验收的任务丢给 Claude。”

报告还提到，工程师越来越常用 Claude 来做前端可视化、实现 UI、做小工具， 甚至让后端工程师也敢去写前端，把自己变成“准全栈”。

从 ROI 角度看，这些任务就像短平快项目：

  • 模型干活
  • 人类验收
  • 出问题直接回滚重来
  • 新手也可以监督

前端在这波里是明显受益者——画面、交互、样式都能快速迭代， 你只要盯着用户体验本身，而不是纠结 CSS 细节。

2）ROI 较低：设计 & 规划、Infrastructure / DevOps

Section titled “2）ROI 较低：设计 & 规划、Infrastructure / DevOps”

再看图的另一端：高层设计、系统规划、Infra / DevOps。

从 Anthropic 的报告里可以看到两个现象：

  • 员工明确表示：

    • “我通常保留高层思考和设计在自己手里，把实现、调试类任务交给 Claude。”
    • 设计和规划类任务，在问卷里生产力提升最小。

  • Infra 和 DevOps 相关任务，用得没那么频繁，时间和产出提升也比较有限。

这就和我们的“软件工程常识”高度吻合：

越依赖经验、上下文、品味、组织策略的工作，越难完全交给 AI。

比如设计一个新系统架构，你要考虑：

  • 五年后的业务形态
  • 团队现在的技术栈
  • 谁来维护、谁来值班
  • 成本预算、合规、安全

Claude 可以帮你：

  • 生成几版架构图
  • 帮你列风险清单和 trade-off
  • 写设计文档初稿

但真正拍板的人，还是你。 AI 在这里更像一个聪明顾问，不是拍板的 CTO。

Infra / DevOps 也是一样：

  • 真正困难的，是

    • 熟悉公司复杂的网络拓扑
    • 一堆历史债务的脚本和配置
    • 出事时谁负责背锅

  • AI 可以帮你写脚本、解释 log、起草 runbook

  • 很难一个命令直接上生产，不看就执行

所以从 ROI 来看：

前端、新功能、调试、测试、文档 → 投入少、收益快

设计、规划、Infra → 价值高，但 AI 很难“直接帮你省大量时间”

四、可验证性：决定 AI 效益的那条“分水岭”

Section titled “四、可验证性：决定 AI 效益的那条“分水岭””

Anthropic 在访谈里，总结了一套很实用的“委托原则”：工程师更愿意把这些任务交给 Claude：

  • 容易验证
  • 复杂度不高或边界清晰
  • 质量要求没那么极端严苛（比如一次性的 debug 脚本）
  • 重复、无聊，但又不能不做的活

这套原则其实可以抽象成一个词：可验证性。

你可以这样想象：

  1. 有单测 / 有 demo 的任务，适合 AI 冲锋

    • 比如前端页面、后端接口、数据处理脚本
    • 写完就跑一遍，看结果是不是在预期范围内

  2. 靠“味道”和“判断”的任务，要人类把关

    • 例如系统设计、体验设计、安全策略
    • 很多时候没有标准答案，只有 trade-off

  3. 局部问题 vs 全局问题

    • 改一个组件、写一段文档，是局部问题
    • 规划跨团队的技术路线，是全局问题

Claude 在“局部、可验证”的任务上 ROI 爆表， 在“全局、模糊”的任务上，更适合帮你想思路、列清单、做对比， 而不是直接给最终方案。

五、给普通团队的落地建议：别急着让 AI 当架构师

Section titled “五、给普通团队的落地建议：别急着让 AI 当架构师”

如果你在带一个工程团队，或者自己就是个人开发者，可以参考 Anthropic 的玩法，给 AI 划一条清晰的“势力范围”。

1）先把 AI 当成“苦力 + 记录员”

Section titled “1）先把 AI 当成“苦力 + 记录员””

优先让 Claude 负责这些：

  • 调试相关

    • 让它帮你定位栈追踪、解释报错、重现 bug
    • 写临时脚本做验证

  • 测试 & 文档 & 注释

    • 自动补单测、生成 mock
    • 总结 PR 变更、生成 README、接口说明

  • 前端和可视化

    • 生成页面骨架、样式方案
    • 做内部工具、仪表盘、数据可视化

  • 小型重构 & “去纸割”

    • 帮你把一坨屎山拆成函数
    • 改变量名、提取公共逻辑

这些地方，你只要多写几个验收条件， 效果会远好于让它“从 0 思考一个架构”。

2）设计和规划：把 AI 当头脑风暴伙伴

Section titled “2）设计和规划：把 AI 当头脑风暴伙伴”

在高层设计上，Claude 更适合这么用：

  • 让它列出多种架构方案和优缺点
  • 帮你写 RFC / 技术方案初稿
  • 模拟不同角色视角的 review（比如“安全团队会问什么问题？”）

决策权、取舍、最终署名，还是掌握在你手里。

3）Infra / DevOps：脚本和说明书可以交，生产命令要自己敲

Section titled “3）Infra / DevOps：脚本和说明书可以交，生产命令要自己敲”

在 Infra 场景里，可以放心交给 AI 的，是这些东西：

  • 解释复杂的 Kubernetes YAML、Terraform 配置
  • 写 CI/CD pipeline 模板
  • 帮你阅读日志，先做一版归因分析

真正执行到生产环境的命令、变更计划，需要：

  • 人类自己再看一遍
  • 多一道同事 review
  • 必要时先在 staging 走一遍流程

可以给团队定一个简单规则：

“凡是直接影响生产的命令，必须由人类亲手改、亲手敲。”

4）给自己做一份“AI 任务地图”

Section titled “4）给自己做一份“AI 任务地图””

结合 Anthropic 的做法，你完全可以给团队画一张表：

  • 行：Debug / 前端 / 新功能 / 测试 / 文档 / 重构 / 设计 / Infra …

  • 列：

    • 是否交给 AI 做初稿？
    • 是否允许 AI 独立完成？
    • 最终需要谁 review？

这张表就是你们自己的 “AI 使用策略”， 可以每季度调整一次，等模型升级了，再把更多任务划进来。

六、未来工程师的样子：更全栈，也更需要判断力

Section titled “六、未来工程师的样子：更全栈，也更需要判断力”

调查里，有一条我很有共鸣：

  • 员工估计，27% 的 Claude 辅助工作，在没有 AI 的情况下根本不会做， 比如更多的重构、更多仪表盘、更多 internal 工具。

也就是说，Claude 不只是帮你“把本来的工作做快一点”， 而是让你多做了一堆原本“来不及做”的好东西。

同时，很多人也在担心：

  • 自己写代码的时间变少了，会不会手生？
  • 监督 AI 需要经验，可经验又在一点点被 AI 抢走。

这就是所谓的“监督悖论”：

用好 AI 需要更强的判断力，而判断力本身又需要亲自下场练级。

所以比较健康的做法，可能是：

  • 在 ROI 高的任务里大胆用 AI 提速
  • 但给自己保留一部分“手写训练场”
  • 把 AI 当成你升级技能的加速器，而不是替代品

长期看，工程师会变成怎样？

  • 更像 产品 + 架构 + 项目经理 + AI 驯兽师 的混合体
  • 更关注“要做什么”和“为什么做”， 而不是“这一行代码具体写成什么样”。

结语：把 AI 用对地方，你就是团队里的“生产力套利者”

Section titled “结语：把 AI 用对地方，你就是团队里的“生产力套利者””

Anthropic 的这份内部调查，其实给我们提供了一个很清晰的思路：

  • 别期待 AI 在所有任务上都 10 倍提速
  • 真正的差别在于：你敢不敢、会不会挑对任务去交给它

如果你是工程师、技术负责人， 现在就可以拿出纸和笔，列一个自己的任务清单：

  • 哪些事容易验证、重复又枯燥，可以马上交给 Claude？
  • 哪些事涉及高层设计和团队策略，只用它做“智囊团”，而不是“代写”？

把这张“AI 任务地图”画出来，你在团队里的生产力，就已经领先一大截了。

从前端到监控，这就是一套可落地的 Agentic AI 工程地图

2025年12月2日
Vibe Sparking

引言：

Section titled “引言：”

这两年，「Agentic AI」「AI Agent」这几个词，已经被说烂了。

很多人一听“智能体”，脑子里就是：

“我也能做一个，用大模型 + Prompt + LangChain 就行了嘛。”

结果真上项目时才发现： 一个能跑在生产环境里的 Agentic AI，远远不是一条 llm(prompt) 这么简单。

你今天看到的这张“冰山图”，其实就把一套可落地的 Agentic AI 技术栈摊开给大家看了个明白： 上面露出水面的，是你能看到的聊天界面、Web App； 下面沉在水里的，是一整套文档处理、向量数据库、检索、监控和部署体系。

这篇文章，我想用尽量“人话”的方式，带你从上往下，把这座“Agentic AI 冰山”走一遍。

一、露在水面上的，是你看到的那层“壳”

Section titled “一、露在水面上的，是你看到的那层“壳””

1. 前端：你以为是“界面”，其实是“体验引擎”

Section titled “1. 前端：你以为是“界面”，其实是“体验引擎””

图里放的是 React、Next.js、Streamlit 这类前端框架，配上 Azure App Service 这类托管环境。

很多 AI 项目一开始，只是给模型套个对话框，看起来能用； 但等到真要给内部同事、客户用，问题就来了：

  • 要支持多轮对话历史
  • 要做文件上传、结果导出、可视化
  • 要做权限控制、组织维度统计

这些都落在前端 + 简单后台上。

一句话：前端不是“顺便写一下 UI”，而是你整套 Agent 体验的“控制台”。 你想的不清楚，用户就只会觉得：

“这不就是个加了皮肤的 ChatGPT 吗？”

二、冰山中部：让 Agent “读得懂世界”的知识管线

Section titled “二、冰山中部：让 Agent “读得懂世界”的知识管线”

Agentic AI 真正的威力，在于“会理解你自己的数据”。 也就是图里这几层：文档摄取 → 切块预处理 → 向量表示 → 向量库 → 检索层。

1. 文档摄取：给 Agent 喂干净的“原材料”

Section titled “1. 文档摄取：给 Agent 喂干净的“原材料””

像 Microsoft Fabric、Azure AI OCR、Apache Tika、LangChain 这些组件，做的事情很朴素：

  • 把 PDF、Word、网页、截图里的内容统一抽出来
  • 去掉噪音、广告、乱七八糟的格式
  • 做结构化，比如目录、标题、段落、表格

可以把它想象成：

给健身的人吃饭，先得保证“真肉真菜”，而不是垃圾食品。

你要是直接把一堆 PDF 生扔给模型，还指望它帮你做“企业搜索、知识助手”，基本是在玩火。

2. Chunking & 预处理：把知识切成模型能消化的小块

Section titled “2. Chunking & 预处理：把知识切成模型能消化的小块”

图里像 spaCy、Hugging Face、LangChain，以及 Azure AI Search 的一些处理能力，就是帮你做这件事：

  • 按段落、逻辑结构智能切块（chunking）
  • 识别实体、主题，做点轻量 NLP 预处理
  • 有的还会加上元数据标签（比如来源系统、时间、业务线）

做不好这一步，会发生什么？

  • 要么 chunk 太大：超长上下文，贵且不准
  • 要么 chunk 太碎：模型每次看到一堆“碎片”，答非所问

工程上比较常见的打法，是根据文档结构 + 字数 + 语义，做一个动态的切块策略，再配上重叠窗口，保证“一问不漏关键信息”。

3. Embeddings：把企业知识压成“向量指纹”

Section titled “3. Embeddings：把企业知识压成“向量指纹””

这层就是 Azure OpenAI、OpenAI、Cohere 等 Embedding 模型发挥作用的地方。

通俗讲，就是帮你把：

文本 → 高维向量

向量之间的距离，代表“语义相似度”。

为什么这一步关键？

因为企业里的问题基本长这样：

  • “帮我查一下某产品 2023 年退货率异常的原因？”
  • “把这几个版本的合同条款差异列出来？”

没有人会按原文去搜，大家都是“自然语言胡乱一问”， Agent 得靠语义检索，在你的知识库里快速找到真正相关的内容。

4. 向量数据库：给 Agent 一块“健身房专用硬盘”

Section titled “4. 向量数据库：给 Agent 一块“健身房专用硬盘””

图里列了 Azure AI Search、Cosmos DB、PostgreSQL + 向量插件，还有 Milvus、FAISS 等等。

现实世界里，你基本要考虑几件事：

  • 数据量多大的时候，检索还能保持毫秒级响应
  • 支不支持混合检索（向量 + 关键字 + 过滤条件）
  • 安全和隔离怎么做，比如按租户、按部门、按项目分库

这里其实就是在选你的“Agent 记忆中枢”。 选错了，你后面所有“RAG + Agent”能力都会卡在性能、成本上。

5. Retrieval Layer：让 Agent“先查再想”

Section titled “5. Retrieval Layer：让 Agent“先查再想””

这层才是大家常说的 RAG：

  • LangChain、LlamaIndex、Haystack 做的是检索编排
  • Azure AI Search 这种则提供内建的检索 + 排序能力

你可以给 Agent 设计一个固定套路：

用户提问 → 检索 N 条相关知识 → 让大模型在“证据”基础上作答 → 必要时把引用也返回给用户

这一步做得好， Agent 的“幻觉率”和“胡说八道”会下降一个数量级。

三、Prompt 工程层：从“写提示”升级到“设计策略”

Section titled “三、Prompt 工程层：从“写提示”升级到“设计策略””

图上放了 Prompty、LangChain、DSPy。

很多人对 Prompt 的理解，还停留在“想几句好点的英文描述”。

但 Agentic AI 里的 Prompt，更接近于一种：

“决策策略 + 工作流编排语言”

大概可以分几类：

  • 系统提示：定义角色、语气、边界
  • 工具调用策略：什么时候查知识库、什么时候调 API
  • 多 Agent 协作：规划、执行、复盘三个角色如何互相对话

像 DSPy 这类工具，还会尝试“自动优化 Prompt”， 有点类似把 Prompt 当“参数”， 通过数据集和反馈信号去调优。

当你开始把 Prompt 当“可实验、可迭代的产品配置”来管理，而不是随手写写，大模型应用的上限才会被打开。

四、LLM 层：不是“选个模型”，而是“选个大脑组合”

Section titled “四、LLM 层：不是“选个模型”，而是“选个大脑组合””

图里提到 Azure OpenAI，外加 LLaMA、Mistral 这些模型。

现实项目里，单模型一把梭已经不太现实了， 更多是在做“模型拼盘”：

  • 便宜模型跑大部分闲聊和简单任务
  • 专业模型跑代码、法律、医学等垂直任务
  • 企业内部还会混合开源模型 + 商业模型，做隐私和成本权衡

你可以把这层理解为：

Agent 的“多核大脑池子”

它要对上面那些“Prompt 策略”和“检索结果”负责， 做出尽可能可靠、符合业务逻辑的决策。

五、监控 & 评估：Agent 不监控，就会慢慢变成“定时炸弹”

Section titled “五、监控 & 评估：Agent 不监控，就会慢慢变成“定时炸弹””

图上有 Azure Foundry、OpenTelemetry、Grafana。

很多团队栽的一个坑，就是：

“应用上线了，大家能用，就算成功了。”

过几周你会发现：

  • 用户经常吐槽“这次回答跟上次不一样”
  • 模型升级之后，部分场景效果变差，却没人发现
  • 某些 Prompt 每天被调用上万次，成本直接炸穿预算

这时候，**可观测性（Observability）和评估（Evaluation）**就很关键：

  • 记录每一次调用的输入、输出、使用的工具、耗时、成本
  • 对关键场景，建立一套自动评估集，新版本一来就跑对比
  • 把这些指标挂在 Grafana 一类的仪表盘上，方便运营和技术一起盯

一句话：

没有监控的 Agent，就像没有后视镜的车。开得越快，越危险。

六、基础设施 & 部署：让 Agent 从“实验室玩具”变成“生产力工具”

Section titled “六、基础设施 & 部署：让 Agent 从“实验室玩具”变成“生产力工具””

最底层是 Azure Container Apps、AKS、Docker、Kubernetes 这些东西。

它们解决的问题其实就两个字：撑得住。

  • 撑得住高并发
  • 撑得住高可用要求（不能总挂）
  • 撑得住多环境（开发、测试、预发、生产）

很多人做 Demo 时觉得一切都好： “本地跑得挺快呀。”

但到生产环境，你要考虑：

  • 横向扩容、自动伸缩怎么做？
  • 模型服务和业务服务怎么拆分？
  • 敏感数据如何隔离、审计？

这些东西，都是冰山最底下那一层的工程细节。 你越早意识到它的存在，后面返工就越少。

七、给想做 Agentic AI 的你，一个落地路线建议

Section titled “七、给想做 Agentic AI 的你，一个落地路线建议”

如果你现在在公司里，正打算做一个“智能客服”“内部知识助手”“AI 运营小助手”，可以这样想路线：

  1. 先从一个清晰场景入手 比如“让运营能秒查任何一条历史活动数据”，而不是“做个啥都能聊的 Agent”。

  2. 用最简单的前端 + RAG 管线跑通 MVP 一开始不用上太复杂的多 Agent 调度， 能做到：“问业务问题 → 查知识库 → 给出引用答案”，就已经很能打。

  3. 从一开始就留好监控和评估的钩子 哪怕一开始只做最简单的日志 + 若干典型问题集， 后期调优会省非常多时间。

  4. 把这张冰山图当做“技术债清单” 想清楚：

    • 哪几层是现在必须做的
    • 哪几层可以后面逐步补上
    • 哪几层可以用云厂商/开源方案“先借用一下”

等你把第一版 Agent 做到团队日常离不开， 再慢慢往下挖这座冰山， 你会发现：

真正难的不是“会不会用大模型”， 而是“能不能把大模型嵌进业务，跑得稳、跑得久、跑得起”。

乔哈里视窗模型与人类–AI对话的四象限分析

2025年12月1日
Vibe Sparking

乔哈里视窗模型简介及四个象限

Section titled “乔哈里视窗模型简介及四个象限”

乔哈里视窗（Johari Window）是心理学中用于提升自我认知和改善人际沟通的模型，由心理学家约瑟夫·卢夫特（Joseph Luft）和哈里·英格哈姆（Harrington Ingham）于1955年提出。该模型将一个人的认知分为四个象限（又称四个”窗口”）:

  • 开放区（Open Area）: 自己和他人都清楚了解的信息。例如公开的行为、态度和特征。这是**“你知、我知”**的领域。

  • 盲区（Blind Area）: 他人看得见但自己未察觉的信息，如他人眼中你的行为特征而你本人不知道。对应**“你知、我不知”**，即自己的认知盲点。

  • 隐藏区（Hidden Area）: 自己知道但他人不知道的私密信息，例如个人隐私、内心想法等。这是**“你不知、我知”**的领域，又称”隐私区”。

  • 未知区（Unknown Area）: 自己和他人都不知道的潜在信息或特质，包括尚未发掘的潜能等。对应**“你不知、我不知”**的完全未知领域。

乔哈里视窗的核心思想是通过扩大开放区、减少盲区和隐藏区、探索未知区，来增进自我认知和促进沟通。

人类与AI对话中的四象限类比

Section titled “人类与AI对话中的四象限类比”

在将乔哈里视窗类比到人类与AI的对话中，我们可以将**“自己”对应为人类用户，将”他人”对应为AI系统。这样，“知道/不知道”这两个维度就变成了”人类是否知道”和”AI是否知道”**。约翰逊窗模型的四象限在这一语境下重新定义，为分析人机对话提供框架。下面的示意图展示了这一对应关系:

乔哈里视窗模型在人类与AI对话中的象限示意图。横轴表示AI是否掌握相关信息（右侧为AI知道），纵轴表示人类是否掌握相关信息（上方为人不知道）。每个象限对应不同的人机认知配置: 双方都知道（开放区）、AI知道但人类不知道（盲区）、人类知道但AI不知道（隐藏区）、双方都不知道（未知区）。

具体来说，人类与AI对话的四个象限含义如下:

  • 开放区（人类知道 + AI知道）: 对话涉及的信息或背景对双方都是透明的。这类问题在人机双方都有共同认知基础，任务或问题的前提清晰明确，相当于公共知识领域。

  • 盲区（人类不知道 + AI知道）: AI拥有而用户缺乏的信息领域。用户的问题属于自己的认知盲点，但AI可能具备相关知识或技能，可以提供用户不知道的答案。

  • 隐藏区（人类知道 + AI不知道）: 用户掌握着AI所不了解的信息，比如个人背景、私有数据或专业知识等。在对话中，这部分信息如果用户不提供，AI就处于信息缺失状态。

  • 未知区（人类不知道 + AI不知道）: 人机双方都没有现成答案的新问题领域。可能是创新性的开放探索，或无人掌握的全新难题，双方都需要共同探索。

上述象限重新刻画了人类与AI在知识/信息方面的关系。基于这样的划分，我们可以更有针对性地设计提问方式和互动策略，使人机对话更高效、智能。

四个象限的对话策略分析

Section titled “四个象限的对话策略分析”

针对以上四个象限，我们分别探讨适合的问题类型、提问和应答策略以及AI在该象限下的局限性。不同象限对应不同的人机认知关系，需要调整对话策略来获得最佳效果。

开放区: 双方知识公开透明

Section titled “开放区: 双方知识公开透明”

开放区是**“人类知晓，AI也知晓”的领域，双方对问题背景都有共同认知。例如让AI润色一段用户自己写的文章、翻译一句话，或询问已知范围内的数据。这类场景下AI可以充当”高效助手”**。

适用的问题类型: 有清晰上下文且人机共享背景的信息处理任务。例如文本润色、翻译、格式转换、简单问答等常规任务。在这些任务中，人类和AI对要求理解一致，沟通成本最低。

人类提问方式: 直接明确地提出任务和要求即可。用户应提供清晰具体的指示，包括要完成的任务、期望格式或其他约束。例如: “请用简洁语言总结下面这段文字的要点。” 提问时不必赘述过多背景，因为AI已经”知道”相关知识，过多解释反而可能画蛇添足。

AI应答策略: AI在开放区主要扮演执行者的角色，即按指令高效率完成任务。由于上下文明确，AI可以直接利用训练中掌握的知识或技能给出回答，通常准确性高、效率高。

AI局限性: 在开放区AI的表现往往很好，局限性较小。但需要注意，如果用户过度描述已知信息，可能使AI困惑或输出冗长（因为AI试图处理用户的冗余指令）。相反，如果用户指令不清，即使双方知识重合，AI也可能难以把握意图。因此开放区的主要挑战在于清晰沟通需求，而非知识本身的欠缺。

盲区: AI知道人类未知

Section titled “盲区: AI知道人类未知”

盲区对应**“人类不懂，AI懂”的情况。在这种对话中，用户在向AI寻求自己欠缺的知识或答案。例如向AI请教专业领域的概念、让AI提供某方面的数据或见解等。这时AI相当于”加速学习者”**或者说知识提供者。

推荐的问题类型: 知识性问答和解释性问题。比如询问专业术语的含义、让AI科普某个概念、分析一个领域的现状等。这类问题针对的是用户的认知盲点，AI往往在训练数据中涵盖了相关信息，可以弥补用户知识不足。

人类提问技巧: 关键是善于提问、循序渐进。由于用户对该领域不熟悉，可能一次提问无法涵盖全部信息。最佳做法是将大问题拆解成一系列子问题，层层深入。先从基础定义问起，再逐步追问细节、例子和应用场景，让AI逐步展开知识。例如用户可以分层提问: “首先，这个概念是什么？解决了什么问题？有何核心原理？能否举例说明？” 这样的迭代对话有助于全面挖掘AI掌握的信息。

AI应答策略: 面对盲区提问，AI应尽量详细且循序地回答，帮助用户理解未知内容。这包括提供定义、原理解释、举例和进一步的拓展。理想情况下，AI会根据用户的每步提问进行教学式回答，就像导师引导学生逐步深入知识领域。但AI也需要避免一次性给出过量信息，应适应用户逐步深化的问题节奏。

AI局限性: 在盲区对话中，主要挑战在于用户不知道自己不知道什么。用户可能提问不明确或遗漏关键点，导致AI回答不具针对性或过于泛泛。研究指出，当用户仅用非常简单的查询时，AI可能难以判断具体需求，从而给出笼统回答，甚至不如传统搜索引擎。此外，如果AI的知识训练数据有限或有偏差，也可能提供不准确甚至错误的信息，而用户由于不懂该领域，难以及时识别错误。因此，在盲区象限下，人类应注意多角度提问和交叉验证信息来源，而AI应该在不确定时坦诚其局限，或提供信息出处以增强可信度。

隐藏区: 人类知道AI未知

Section titled “隐藏区: 人类知道AI未知”

隐藏区对应**“人类掌握，AI未掌握”的信息，例如用户的个人经验、私有数据或上下文背景。这里AI相当于在相关知识上一片空白**，需要由用户补充喂给必要的信息后才能完成任务。这种场景下AI的角色类似**“新手”或”被指导者”，人类则是向AI提供知识的专家**。

典型对话场景: 定制化、个性化的任务。例如让AI分析公司内部报告并给出摘要，或者根据用户提供的特定数据进行推理。在这些情形中，任务所需的信息不在AI的训练集中（因为是用户独有的），必须由用户提供给AI。例如: “这是我们公司的销售数据，请根据这些数据做业绩分析。”

人类提问方式: 在隐藏区，人类需要先提供充分的背景知识给AI，然后再提出问题。也就是说，“先喂料，后提问”。提供信息时要结构化、清晰: 可以包括背景说明、关键数据、术语解释以及期望AI完成的任务说明。例如，用户可以先输入一段自己的专属资料（如产品文档或会议记录），接着提问: “请根据以上资料总结主要结论并给出建议。” 如果用户只笼统地提一句”我们公司有个新项目X，你帮忙分析一下”，但不提供详细信息，AI由于缺乏必要背景将无法有效回答。因此提问时应毫不吝啬地把AI未知的部分讲清楚。

AI应答策略: 在获取用户提供的信息后，AI应充分利用这些新知识来完成任务。它可以运用自身的语言能力，将用户输入的专有内容与已有常识结合，进行总结、分析或创作。例如，AI拿到一份用户提供的内部报告后，可以提取关键信息并给出总结，或者根据用户给的事实回答问题。AI在这一象限主要扮演**“知识整合器”和”学习者”的角色。必要时，AI也可以通过澄清问题**向用户提问，以确保对用户提供资料的正确理解。

AI局限性: 隐藏区中AI最大的局限在于**“巧妇难为无米之炊”——没有用户输入的专有信息，AI就无法凭空解决定制化问题。另外，即使提供了信息，AI可能无法完全理解复杂专业数据的细微含义**，或者受模型自身知识局限，对新信息的整合能力有限。例如，AI可能误解用户提供的数据含义，或者遗漏重要背景细节。因此，用户在这一象限下需要对AI的回答保持监督，验证AI是否正确利用了提供的信息。为缩小隐藏区带来的劣势，一些先进的方法如**检索增强生成（RAG）**允许将外部知识库接入AI，使模型在对话中可以查询相关资料。通过引入这些技术，未来AI可以在隐藏区更好地获取用户领域知识，从而提升回应质量。

未知区: 双方都未知领域

Section titled “未知区: 双方都未知领域”

未知区是**“人类不知道，AI也不知道”的象限。这通常对应开放式的创造性对话或前沿探索性问题——没有现成标准答案，需要人机共同脑暴。在这里，AI和人类处于同一起跑线**，可以把AI看作**“共创伙伴”**。双方可以各自贡献思路，碰撞出新的想法。

适合的问题类型: 开放性创意、战略规划、前沿研究猜想等。例如，头脑风暴新的商业模式，提出科幻创意点子，或讨论尚无定论的科学问题。此类话题没有对错之分，重在广泛探索和创新。

人类提问方式: 鼓励采取开放式提问，给AI一定的创作自由，同时设置适当的约束来保证输出方向有意义。例如，可以提出假设性的问题: “如果要在火星上建立城市，我们需要哪些创新技术？” 并要求AI给出多角度的思考方案。也可以要求AI发散思维，如”请提出10种可能的解决思路”，然后人类再根据可行性筛选。总之，人类在未知区应扮演引导者和评估者的角色: 引导AI朝某个方向尝试，并不断根据AI的回答进行追问或调整。

AI应答策略: 在未知区，由于没有固定答案，AI应发挥创造力，提供多样化的观点或设想。同时，AI也要接受人类的反馈，共同收敛出更合理的方案。例如AI可能先天马行空地给出一些想法，人类据此指出哪些有价值、哪些不现实，然后AI再基于人类的判断继续深化或改变方向。人机在此象限下是一种**“协作探索”关系: AI提供原始创意，人类进行评估和综合**，两者反复互动以逼近问题的解决方案。

AI局限性: AI在未知区最大的局限在于缺乏真实世界验证能力。它提出的点子只是基于训练语料的类比和组合，可能新颖有趣，但也可能不切实际甚至荒诞。由于AI并不真正理解物理世界的因果关系或商业可行性，其创意需要人类来把关评估。另外，在无人知晓答案的问题上，AI可能倾向于编造看似合理的解释（即”幻觉”），给出毫无根据的结论。因此，人类应意识到AI输出的不确定性，善于识别和质疑其中的不可靠部分。总的来说，在未知区，人类主要依靠AI激发灵感，但最终的判断和决策仍需由人来做。正如有研究指出的，人类独有的直觉、情感和价值判断是AI无法取代的优势——这在面对未知问题时尤为重要。

人类视角: 设计高效提问以获取高质量回应

Section titled “人类视角: 设计高效提问以获取高质量回应”

从提问者（人类）的角度看，充分利用乔哈里视窗模型可以极大提升与AI对话的有效性和质量。核心在于: 了解自己与AI分别知道什么，并针对不同象限调整提问策略。以下是人类设计更有效问题的一些建议:

明确识别问题所属象限: 在发问前，先判断这个问题对于你和AI来说属于哪个象限。如果是你俩都熟悉的（开放区），就直接下达清晰指令；如果是你不了解但AI可能知道的（盲区），就准备分步提问、循序渐进挖掘知识；如果需要AI处理你提供的专有信息（隐藏区），就先提供背景再提问；如果是开放式探索（未知区），就提开放性问题并鼓励多角度回答。这种有意识的象限识别能帮助选择合适的提问方式，避免”一刀切”式发问导致的效果不佳。

提供充分且必要的上下文: 信息是高质量回答的燃料。根据象限不同，提供上下文的策略也不同: 在开放区，只需简明提供任务要求，不要堆砌多余信息干扰AI；在隐藏区，则应尽可能提供AI所缺少的背景，使AI有料可用。总之，提问时站在AI视角想一想: “它需要哪些信息才能准确回答？” 然后主动提供这些信息而非让AI猜测。

善用迭代对话: 不一定指望一次性提出”大而全”的问题。相反，逐步深入往往更有效。在盲区象限，将复杂问题拆解是常用策略；在未知区，与其一次要AI给出终极答案，不如让AI先生成思路，再根据回答继续提问或要求elaboration。这种持续对话可以引导AI不断朝着人类需求的方向优化输出，也让人类有机会随时校正AI的理解偏差。

强调具体期望和输出格式: 对于很多任务，明确你希望AI如何作答，包括回答的长度、格式、风格等，可以显著提高回答质量。例如，注明”请用100字以内回答”或者”列出要点”。在开放区这类提示尤为重要，让AI输出恰到好处的信息量；在盲区也有用，防止AI给出过于庞杂的解释。

保持反馈与引导: 人机对话不是单向的，提问者可以根据AI的初步回答做出反馈，决定下一步问什么。这在未知区和盲区尤其关键。如果AI回答不符合预期，不妨指出问题并要求修正，例如: “你刚才的回答遗漏了X方面，请详细补充”。通过反馈引导，用户可以逐步将AI的输出逼近自己想要的结果。

总之，人类提问者要有主动设计对话的意识，而不仅是随意发问。正如爱因斯坦所言: “问题比答案更重要”。在AI时代，设计高质量问题的能力已成为人机协作的核心竞争力。掌握了提问的艺术，即使面对日益智能的AI，我们仍能主导对话节奏，获得更理想的回答——会提问的人将不会被AI所取代。

AI系统视角: 理解人类需求与盲区的改进方向

Section titled “AI系统视角: 理解人类需求与盲区的改进方向”

从AI系统的角度，为了更好地服务于人类，它需要不断提升对人类需求、意图以及自身认知盲区的理解。结合乔哈里视窗的框架，我们可以提出以下改进思路:

增强对用户背景的获取和理解: 当前的大模型对用户的隐藏信息无法直接获知，但可以通过技术改进来缩小”隐藏区”。例如，引入检索增强生成（RAG）等机制，允许AI在对话时检索用户的专属知识库或实时信息。这样，当用户提问涉及私有领域时，AI能及时获取相关资料，从而理解人类提供的背景。在实际应用中，这意味着AI系统应当支持让用户上传文档、数据等，并记忆对话上下文，逐步构建对特定用户的知识图景。在保护隐私的前提下，AI对用户背景掌握越充分，越能缩小因信息不对称带来的误解。

学会提出澄清性问题: 人类有时提问不清或藏有隐含前提，这是AI的”人类盲区”。理想的AI应能够检测出用户提问中的歧义或不完整之处，并主动询问澄清，而不是贸然给出可能偏离需求的回答。例如，如果用户问”请总结一下报告”，AI可以回应: “请问是哪份报告？您希望我关注哪些方面？” 通过模拟人类对话中的澄清行为，AI能更准确抓住人类真正的意图。这种改进可减少由于误解问题而产生的无效回答，使对话更加高效。

识别并提示自身知识局限: 在盲区和未知区，当AI识别出用户的问题超出了自己的知识范围，或回答中存在高度不确定性时，应该坦诚地向用户提示。例如，AI可以直言”关于这个最新研究，我没有足够信息，是否需要我搜索最新资料？” 甚至在接口上提供搜索引擎模式供用户选择。通过这种方式，AI不至于自信地输出错误信息，让人类误以为AI已经”全知全能”。相反，承认局限并寻求合作（如请求额外信息或网络查询）反而会提升对话质量和用户信任。

持续学习用户反馈以缩小盲区: 每次人机对话都是AI了解人类需求和盲区的机会。AI系统应能从用户的反馈、纠正中学习: 当用户多次修改提问或对AI回答表示不满时，AI可以尝试分析原因，例如是没有理解某个词的真实含义，还是忽略了用户真正关心的点。未来的AI或许可以在对话过程中动态调整推理策略，例如切换不同知识模块，来更贴近用户期望。这种交互式学习能让AI逐步适应个人化的提问风格，弥补模型通用性与个性化需求之间的差距。

拓展知识边界并更新模型: AI的”已知”在不断变化。随着时间推移，新知识不断产生，今天的未知区明天可能变成盲区甚至开放区。当下的大模型通过插件、联网检索等方式已经在尝试扩充知识库。未来，开发者应定期更新AI模型的训练，使其掌握最新的公共知识，推动横轴（AI知识）不断右移。同时，通过强化学习等手段，让模型更好地保留高质量的用户交互经验。这将逐渐缩小AI的知识盲区，使其在更多领域达到专业水准。不过，无论AI多么进步，人类的直觉、情感和价值判断仍是AI难以替代的。因此AI也应被设计得能够理解和尊重这些人类特有需求，例如在回答涉及伦理、情感的问题时表现出适当的同理心和谨慎。

综上，从AI系统一侧看，优化的方向在于更充分地获取并利用人类提供的信息，更智能地处理人类的模糊提问，以及更谦逊地面对自身知识盲点。通过这些改进，人机对话将更趋向真实伙伴式的沟通: AI更深刻地理解人类意图和背景，人类也能更信任AI的回应质量。这正是人机共生的目标——双方在沟通中不断缩小误解的窗格，拓展知识与创意的共享窗口，最终实现协作共创而非简单的工具问答。当我们充分运用乔哈里视窗思维去审视人机交流，每一次对话都将不仅是获取答案，而是一次认知互相启发的旅程。

  • 上涨概率接近 80%

  • 美联储刚刚结束 QT
  • 日本可能加息、美国还在讨论再降几次息

所以更现实的看法是：

如果今年仍然出现一波圣诞+跨年小反弹，可以视作「在降息预期与流动性缓和下的一次情绪修复」。

但它并不能掩盖：

  • AI 估值是否合理
  • 就业是否明显走弱

从探索到修复：Claude Code 如何思考与推理，排查 SQLAlchemy 事务回滚问题

2025年10月28日

这是一篇基于一次真实的生产问题排查记录而整理的技术博客，焦于 Claude Code 如何“探索 → 思考 → 推理”，快速定位根因并实施最小可行修复，同时给出进一步的结构性优化建议。

问题发生在 mailflow 服务的生产环境，核心报错如下：

对应日志上下文显示，错误发生在向 emails 表执行 INSERT 时，随后一系列数据库操作进入“级联失败”状态。

1. 场景与目标

Section titled “1. 场景与目标”
  • 场景：mailflow 在处理入站邮件（解析、生成记录、转发、移动 IMAP 文件夹）时，偶发整条处理链条失败。
  • 目标：
    • 明确事务失败的首要触发点；
    • 解释为何后续所有 DB 操作都报相同错误；
    • 以最小风险改动尽快恢复处理能力；
    • 提供结构性改进建议，提升系统鲁棒性。

2. 探索路径：从现象到证据

Section titled “2. 探索路径：从现象到证据”

基于 Claude Code 的代码搜索与上下文关联能力，我们按“从症状到根因”的顺序展开：

  • 锁定日志与触发点

    • 从日志可见异常由 SQLAlchemy 会话（Session）状态所致：之前事务失败时未进行 rollback()，当前 Session 进入 invalid 状态。

  • 关键代码路径定位

    • mailflow/app/db.py：连接与会话工厂定义。
    • mailflow/app/mail.py：实际的业务主流程、创建记录、提交事务的位置。
    • mailflow/app/utils.py：日志与告警（微信）入口 notify_and_log。

  • 证据一：全局（模块级）会话

    • mail.py 顶部存在：
    • 一旦某处 flush/commit 失败，该全局 Session 便进入 invalid 状态，后续所有依赖它的 DB 操作都会失败，直至显式 session.rollback()。

  • 证据二：缺少异常处理与回滚

    • 多处 session.commit() 调用无 try/except/rollback 保护：
      • 创建邮件记录（create_mail）
      • 建立邮件与附件关联（_relation_attachment_and_mail）
      • 处理请求结果并更新状态（deal_requests_result）
      • 主流程 process_mail 的大段逻辑

结论：首个失败的 DB 操作将全局 Session 置为 invalid，因缺少 rollback()，其后路径上的所有提交与查询都会以相同错误失败，形成“级联故障”。

3. 推理链路：从假设到验证

Section titled “3. 推理链路：从假设到验证”

  1. 假设：出现一次 INSERT/UPDATE 异常（可能由数据约束、网络抖动、连接池等引发），全局 Session 进入 invalid；由于没有 rollback()，后续流程（更新状态、关联附件、移动邮件）全部失败。

  2. 代码验证：

    • 搜索到全局 session = session_factory()；
    • 搜索 session.commit() 的使用点，基本均无 except: session.rollback()；
    • 与报错语义完全吻合。

  3. 策略：小步快跑，先以最小改动恢复稳定性，再评估结构性重构。

4. 修复策略与实施

Section titled “4. 修复策略与实施”

本次修复分三层优先级（P0 → P2）：

  • P0（立即生效，最小改动）：在异常捕获处补齐 session.rollback()，让 Session 状态可自愈。
  • P1（全面防护）：所有 commit() 外围增加 try/except/rollback，避免任何单点失败拖垮后续流程。
  • P2（结构优化）：用 scoped_session 替换模块级全局会话，隔离不同处理流的会话状态（建议，非本次强制上线）。

4.1 关键修复片段

Section titled “4.1 关键修复片段”
  • 主流程增加回滚（mailflow/app/mail.py）
  • 处理请求结果时的状态更新，增加防护（成功与失败分支都包裹）
  • 建立邮件与附件关联时增加防护
  • 创建邮件记录 create_mail 的提交保护

4.2 结构性优化建议（P2）

Section titled “4.2 结构性优化建议（P2）”

从根上避免“全局会话拖累所有流程”的风险，推荐使用 scoped_session：

说明：scoped_session 能在并发/多线程环境中提供更好的隔离性，减少“跨请求污染”。本次线上紧急修复优先保证回滚语义正确与完整性，再评估迁移成本与收益。

5. 验证与回归

Section titled “5. 验证与回归”

建议的验证思路：

  • 单条可复现邮件回放：重放出现过错误的邮件，观察处理链是否恢复；
  • 观察日志：不再出现 “Can’t reconnect until invalid transaction is rolled back”；
  • 人工 spot-check：核对 IMAP 文件夹移动、转发邮件、数据库状态；
  • 受控压力：增加一定数量的邮件样本，确保多处提交点都能正确回滚与继续。

6. 经验总结与最佳实践

Section titled “6. 经验总结与最佳实践”
  • 反模式：模块级全局 Session 一旦进入 invalid，会引发全链条“级联失败”。
  • 原则：任何 commit() 都应具备 try/except + rollback；在主流程捕获顶层异常时也要 rollback()。
  • 设计：面向故障恢复的会话模型（scoped_session 或每次处理获取/释放会话）。
  • 运维：第一时间记录和告警（本项目已通过企业微信 notify_and_log 实现），并在消息里携带关键信息（uid、subject、时间、to_list）。

7. Claude Code 的方法论

Section titled “7. Claude Code 的方法论”

  1. 快速建立问题边界：从错误语义入手，匹配最可能的层（事务/会话管理）。
  2. 证据驱动的搜索：定位涉及的文件、变量、函数（session_factory、commit() 使用点）。
  3. 假设 → 证伪/证成：提出“全局 Session + 无 rollback 导致级联失败”的假设，并在代码中找到直接证据。
  4. 最小改动优先：先补齐 rollback() 使 Session 可恢复，快速止血。
