VoiceFlow 由 infiniV 开发，提供了一个引人注目的替代方案。这款开源 Windows 应用将 OpenAI 强大的 Whisper 语音识别模型直接带到你的本地计算机。你说的每一个字都在你的硬件上完全处理——你的语音数据永远不会离开你的设备。

Section titled “什么是 VoiceFlow？”

VoiceFlow 是一款免费的、开源的 Windows 10/11 语音听写应用，利用 OpenAI 的 Whisper 模型进行语音转文字转换。与基于云端的替代方案不同，VoiceFlow 在你的计算机上 100% 本地运行，确保完全隐私并消除持续成本。

这并不是关于 AI 取代开发者。而是关于 AI 作为协作伙伴 —— 转录记录展示了美妙的一面：AI 发现人类容易忽略的问题，人类提供方向和上下文，两者共同构建出任何一方都无法单独完成的东西。

什么是 “Vibe Coding”？

Section titled “什么是 “Vibe Coding”？”

Vibe coding 是一种对话式的软件开发方法，开发者与 AI 通过自然语言对话一起工作。与其将 AI 视为命令执行工具，不如将其视为结对程序员 —— 一起讨论、探索和迭代。

在 Mitchell 的会话中（从 AmpCode 上的真实编码对话捕获），流程是这样的：

  1. 咨询预言家 — 让 Claude 分析代码以寻找改进点
  2. 讨论 — 一起讨论发现，共同做出决策
  3. 实施 — 在 Claude 的协助下编写代码
  4. 迭代 — 从失败中学习，调整方法，再次尝试
  5. 测试 — 验证更改有效，添加测试以建立信心

关键安全问题

Section titled “关键安全问题”
  1. 循环引用 — UpdateDriver.showUpdaterError 中的重试闭包强引用了 self，造成内存泄漏
  2. 线程问题 — UI 面向类型应标记为 @MainActor
  3. 进度值错误 — 进度值需要限制在 [0, 1] 范围内以防止视觉故障
  4. 不可取消的定时器 — “无更新”自动关闭任务可能触发过期更新
  5. 吞没的错误 — try? 隐藏了 startUpdater() 中的潜在失败

  3. ViewModel 格式化 — 每个状态的文本/图标/颜色映射
  4. 进度数学 — 除零、nil 值、超调场景

规划：始终从计划开始

Section titled “规划：始终从计划开始”

花五分钟规划，能节省十分钟的审查时间。

无论任务大小，都要从计划开始。无论你的代理是否有内置的”计划模式”，你都可以直接要求：

“先制定一个计划，在做出更改之前与我确认。”

规划可能会感觉像是一种开销，但它将不可避免的工作提前了。审查计划比审查代码差异要快得多。

规划决定架构形态

Section titled “规划决定架构形态”

代理的目标是以最少的编辑次数让代码工作起来。一旦某个版本存在，后续的更改只会修补已有的内容，而不是重新思考方法。计划是你有机会在代码围绕错误架构固化之前选择正确形态的机会。

异步工作流：同时处理多个任务

Section titled “异步工作流：同时处理多个任务”

与编码代理异步工作（有时称为”后台运行”）让你能够：

  • 同时处理多个任务线程
  • 避免陷入拖延
  • 让代理持续运行较长时间，而你处理其他事务

典型工作流：让代理持续运行，同时你处理其他任务，定期检查进度。

使用YOLO模式实现真正的异步

Section titled “使用YOLO模式实现真正的异步”

只有在你不批准每个工具调用的情况下，异步才能发挥作用。要么使用代理的YOLO模式，要么给它一个明确的命令允许列表，覆盖它需要的几乎所有操作。

否则你就重新发明了结对编程——而且更慢。

不同平台的命名不同：‘yolo’、‘危险地跳过权限’或类似名称。

让代码库可以进行质量验证

Section titled “让代码库可以进行质量验证”

如果代理无法测试自己的更改，它就会要求你来做——这不是对你时间的很好利用。

**投入精力让更改可以通过开发或测试命令进行验证。**并非所有内容都可以由代理测试（前端仍然需要人眼查看），但大多数更改应该以”测试通过”结束，而不是”你能试试这个吗？“

解决开发服务器问题

Section titled “解决开发服务器问题”

代理不擅长启动和清理开发服务器，这会导致：

  • 端口冲突
  • 浪费时间排查”该端口上已有内容在运行”的错误

像 Dev Manager MCP 这样的解决方案可以自动分配端口并终止僵死的服务器。

添加虚拟数据以支持离线测试

Section titled “添加虚拟数据以支持离线测试”

使用真实的种子数据让项目易于离线运行。如果代理花费一半时间让应用进入可测试状态，你将得到：

  • 不稳定的结果
  • 更慢的进度

虚拟数据还能支持并行代理——如果它们共享真实数据库，一个可能使用与另一个不同的架构版本。

不追求向后兼容

Section titled “不追求向后兼容”

每个编码代理默认都保持向后兼容。这部分是过度谨慎，部分是懒惰——保留旧接口意味着不必重构所有调用点。

将此添加到你的系统提示词中：

我们想要尽可能简单的更改。我们不关心迁移。代码可读性最重要，我们愿意做出更大的更改来实现它。

禁用禁用lint规则的功能

Section titled “禁用禁用lint规则的功能”

代理喜欢通过禁用lint规则来消除错误，而不是真正修复它们。eslint-disable-next-line 和类似模式出现在 TypeScript、Rust 以及可能的其他语言中。

使用插件禁止这种行为，如 eslint-comments/no-restricted-disable。

Kestrel 解决的问题

Section titled “Kestrel 解决的问题”

在进行深度研究工作时，你经常会遇到耗时的机械性任务：

  • 比较工具和框架
  • 收集真实世界的使用案例

  • 汇编综合对比

这些任务会打断你在实际创新工作上的流程。Kestrel 处理这些背景研究，在需要时提供组织好的证据。

使用场景示例

Section titled “使用场景示例”

与其花费数小时手动研究，你可以告诉 Kestrel：

“比较 Polars、pandas 和 Spark 的数据处理能力。收集真实世界的使用情况、性能基准测试和用例推荐。”

Kestrel 然后在后台运行：

  1. 搜索文档和基准测试
  2. 筛选相关来源

  4. 汇编可审查的对比

几小时后，你有一个准备好的组织报告——而不是散落在浏览器标签页中。

  • 昂贵的 API 订阅
  • 云服务的隐私顾虑
  • 依赖互联网连接
