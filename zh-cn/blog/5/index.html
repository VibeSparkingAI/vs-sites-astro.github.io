简介

Section titled “简介”

Design and Refine 是一款 Claude Code 插件，通过快速迭代帮助开发者做出自信的 UI 设计决策。

一句话理解：生成多个 UI 变体 → 浏览器并排比较 → 收集反馈 → 合成优化版本 → 重复直到满意

如果你曾经为一个按钮的样式纠结半天，或者在多个设计方案之间来回修改却始终不满意，这个插件就是为你准备的。

使用流程

Section titled “使用流程”

启动会话：

接下来会发生什么：

  1. 预检 — 检测你的框架（Next.js、Vite 等）和样式系统（Tailwind、MUI 等）

  2. 样式推断 — 从 Tailwind config、CSS 变量或主题文件读取现有设计规范

  3. 访谈 — AI 会询问以下问题：

    • 设计目标（组件 vs 页面，新建 vs 重设计）
    • 痛点和需要改进的地方
    • 视觉和交互灵感参考
    • 目标用户和关键任务

  4. 生成 — 创建 5 个不同变体，探索不同方向：

    • 信息层级
    • 布局模型（卡片、列表、表格、分栏）
    • 密度（紧凑 vs 宽松）
    • 交互模式（模态框、内联、抽屉）
    • 视觉表现

  5. 评审 — 打开 http://localhost:3000/__design_lab（或你的开发服务器端口）并排查看所有变体

  6. 反馈 — 告诉 Claude：

    • 如果某个已经很好 → 选择它，做小调整
    • 如果喜欢不同变体的部分 → 描述你喜欢每个变体的哪些方面，获得合成版本

  7. 迭代 — 重复直到满意

  8. 完成 — 所有临时文件被删除，生成 DESIGN_PLAN.md 包含实施步骤

最佳实践

Section titled “最佳实践”

在访谈中要具体。 你给出的痛点、目标用户和灵感的上下文越多，生成的变体就越有特色和有用。

参考你欣赏的产品。 “像 Linear 的密度” 或 “Stripe 的清晰度” 给 Claude 具体的方向。

不要满足于第一轮。 合成步骤才是精彩之处——描述你喜欢每个变体的哪些方面，让它们结合起来。

保持开发服务器运行。 插件不会为你启动（那会阻塞）。在另一个终端保持运行即可。

检查 DESIGN_PLAN.md。 完成后，这个文件包含实施步骤、组件 API、可访问性检查清单和测试指导。

UI Skills：帮助 AI 代理构建更好界面的意见性约束

简介

Section titled “简介”

随着 AI 编码助手在构建用户界面方面变得越来越强大，一个新的挑战出现了：一致性。虽然代理可以编写功能正常的代码，但它们经常产生设计模式各异、样式不一致、存在无障碍问题的界面。UI Skills 应运而生——这是一套意见性约束，旨在帮助代理交付更好、更一致的界面。

什么是 UI Skills？

Section titled “什么是 UI Skills？”

UI Skills 是一组意见性 UI 约束，作为 AI 代理构建用户界面的护栏。当被调用时，这些约束确保：

  • 界面遵循一致的设计模式
  • 默认内置无障碍最佳实践
  • 性能默认优化
  • 代码保持可维护和可扩展

该项目支持多个 AI 编码环境，包括 Claude Code、Cursor、OpenCode 以及其他接受技能定义的工具。

为什么意见性约束很重要

Section titled “为什么意见性约束很重要”

在 AI 生成 UI 的世界中，意见性约束具有几个关键作用：

1. 跨代理一致性

Section titled “1. 跨代理一致性”

没有约束，不同的代理（甚至同一代理在不同时间）可能会产生截然不同的实现。UI Skills 为设计决策建立了单一的事实来源。

2. 默认无障碍

Section titled “2. 默认无障碍”

许多 AI 生成的界面会忽略无障碍性。UI Skills 从一开始就强制使用无障碍模式，确保符合 WCAG 标准，无需人工干预。

3. 性能优化

Section titled “3. 性能优化”

代理通常选择最简单的实现路径，而不是性能最佳的路径。UI Skills 将选择限制在经过验证的性能模式上。

1. 组件

Section titled “1. 组件”

UI Skills 强制执行以下组件级规则：

规则理由
为仅图标按钮添加 aria-label屏幕阅读器可访问性
对破坏性操作使用 AlertDialog防止意外数据丢失
使用结构骨架进行加载状态感知性能提升
永不使用 h-screen，使用 h-dvh移动视口兼容性
为固定元素遵守 safe-area-inset刘海/设备兼容性
在操作旁边显示错误更好的用户反馈
永不在输入中阻止粘贴用户体验增强

2. 动画

Section titled “2. 动画”

动画约束专注于性能和用户体验：

规则理由
除非明确要求，否则不添加动画减少晕动症
仅动画合成器属性（transform, opacity）60fps 性能
永不动画布局属性昂贵的重排
避免动画绘制属性（background, color）GPU 优化
入场使用 ease-out自然的运动感
交互反馈永不超过 200ms响应式感觉
屏幕外时暂停循环动画电池/资源节约
遵守 prefers-reduced-motion无障碍合规
避免动画大图性能优化

3. 排版

Section titled “3. 排版”

排版约束确保可读性：

规则理由
标题使用 text-balance改善视觉居中
正文/段落使用 text-pretty更好的换行
数据使用 tabular-nums防止布局偏移
密集 UI 使用 truncate/line-clamp内容适配
除非要求，否则不修改 letter-spacing字体设计师意图

Section titled “4. 布局”

规则理由

Section titled “5. 性能”

性能约束防止常见反模式：

规则理由
永不动画大型 blur() 表面GPU 内存
永不在活动动画外应用 will-change浏览器优化
永不将 useEffect 用于渲染逻辑防止不必要的渲染

Section titled “6. 设计”

设计约束确保视觉一致性：

规则理由
除非要求，否则不使用渐变更简洁的美学

永不将发光效果作为主要示能性可访问性
除非要求，否则使用 Tailwind 默认阴影一致性
为空状态提供一个明确的下一步操作用户引导
每个视图限制强调色为一个视觉层次
在新颜色之前使用现有主题颜色设计系统完整性

BMAD Method（Breakthrough Method of Agile AI‑Driven Development）是一套 AI 驱动的软件交付方法论与工具体系，它要解决的不是“让 AI 更会回答问题”，而是把你从“临时写 prompt”带到“可重复的交付流程”。

它通过把下面这些东西工程化、资产化：

  • 专业分工的 Agents（PM / 架构师 / 开发 / QA / 文档…）
  • 可执行的 Workflows（发现 → 规划 → 实施 → 验证）
  • 可追踪的 Artifacts（brief / PRD / story / checklist）

让项目上下文、质量门槛和决策理由都沉淀在仓库里，而不是散落在聊天记录里。

Section titled “1）亮点速览（你会得到什么）”

从“随缘问 AI”到“角色分工”

Section titled “从“随缘问 AI”到“角色分工””

BMAD 把常见的团队角色做成可复用的 persona，你可以稳定地获得：

  • PM 视角的目标/边界/验收口径
  • 架构师视角的方案比较与技术决策
  • 开发视角的落地实现与改代码
  • QA / Test Architect 视角的验证方案与回归策略
  • Tech Writer 视角的文档与发布说明

过程可伸缩：小事快做，大事不乱

Section titled “过程可伸缩：小事快做，大事不乱”

BMAD 的思路是“按任务规模给出合理流程深度”：

  • Quick Dev / Quick Flow：修 bug、小需求、快速迭代
  • 标准轨道：中等规模功能/产品，PRD → 架构 → Stories
  • 企业轨道：强治理与合规场景，质量门槛更完整

Section titled “形成从想法到验证的闭环”

BMAD 强调“产出工件（Artifacts）再实施”，而不是只做一次性对话。
