Claude Code 提示词模板

Section titled “Claude Code 提示词模板”
  • Android productFlavors：在 Gradle 中声明不同环境（applicationId 后缀、资源替换、签名、图标）。
  • iOS Schemes + Build Configurations：为 dev/staging/prod 建独立 Scheme，并指向对应的 Build Configuration 与 Bundle ID。
  • Dart --dart-define：把环境变量注入到代码里（如 BASE_URL、SENTRY_DSN）。
  • 多入口文件：lib/main_dev.dart、lib/main_staging.dart、lib/main_prod.dart，各自初始化。

常见坑与排查

Section titled “常见坑与排查”
  • iOS 运行报找不到 Scheme：确认 Xcode 中的 Scheme 名与 --flavor 完全一致。
  • Android 构建找不到任务：assemble<Flavor><BuildType> 大小写要对（如 assembleDevDebug）。
  • 图标/名称无变化：检查是否使用了 @string/app_name，以及各 flavor 的 resValue 或资源目录是否生效。
  • AGP 8+ 报 namespace：在 android { namespace "..." } 指定 namespace。
  • 机密管理：dart-define 用在 CI 时避免把密钥写进仓库；本地 .env 不要入库。

用 Claude Code 生成 Flutter Android 编译构建的 APK 命名规则

2025年9月17日
Vibe Sparking

为什么要改 APK 文件名？

Section titled “为什么要改 APK 文件名？”

默认名字是 app-release.apk。 发给测试同学、上 CI 制品库，全都叫一个名。 找版本靠猜，非常痛苦。

而改名后，一眼知道：

  • 是哪个分支出的包；
  • 版本号是多少；
  • 什么时候打的。

效率直接起飞。

你要准备什么？

Section titled “你要准备什么？”

  • 正常的 Flutter/Android 工程。

  • Android Gradle Plugin（AGP）版本任意，但改名方式会随 AGP 变。 更稳妥的做法是：打完包再拷贝并改名（下文代码就是这样做）。 这是社区在 AGP 新 API 下的通行实践。

  • Flutter 的 --dart-define 参数会以 Base64 传给 Gradle 的 -Pdart-defines，我们可以在 build.gradle 里解码读取。

一键上手：把这段加入 android/app/build.gradle

Section titled “一键上手：把这段加入 android/app/build.gradle”

放在 android { ... } 之后的 applicationVariants 逻辑里即可（Groovy 语法）。 思路：拿到分支名 + 解码 --dart-define + 生成新文件名 + assemble 后拷贝重命名。 这种“拷贝改名”的方式跨 AGP 版本更稳定。

说明：

  • Git 分支在本地用 rev-parse，在 CI 可通过 GIT_BRANCH 环境变量兜底。
  • 新版 AGP 的“直接改产物名”方案经常不稳定；拷贝改名是更通用的路线。
  • Flutter 的打包配置位置与打包方式，详见官方文档。

常见坑位（我帮你踩过了）

Section titled “常见坑位（我帮你踩过了）”

  • 直接改产物名偶尔失效 新版 AGP 的 Variant API 有改动。和社区经验一致，推荐 assemble 后“拷贝+改名”。

  • Flutter 产物目录差异 部分情况下会出现 flutter-apk/ 目录或忽略 outputFileName 的现象。遇到就用拷贝改名兜底。

  • CI 是 detached HEAD git rev-parse 取不到分支，记得用环境变量兜底（如 GIT_BRANCH 或 CI 专用变量）。
