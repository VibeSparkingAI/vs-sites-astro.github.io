BMAD Method 入门指南：用 Quick Dev 工作流更快、更稳地交付

2026年1月14日
Vibe Sparking

BMAD Method（Breakthrough Method of Agile AI‑Driven Development）是一套 AI 驱动的软件交付方法论与工具体系，它要解决的不是“让 AI 更会回答问题”，而是把你从“临时写 prompt”带到“可重复的交付流程”。

它通过把下面这些东西工程化、资产化：

  • 专业分工的 Agents（PM / 架构师 / 开发 / QA / 文档…）
  • 可执行的 Workflows（发现 → 规划 → 实施 → 验证）
  • 可追踪的 Artifacts（brief / PRD / story / checklist）

让项目上下文、质量门槛和决策理由都沉淀在仓库里，而不是散落在聊天记录里。

1）亮点速览（你会得到什么）

Section titled “1）亮点速览（你会得到什么）”

从“随缘问 AI”到“角色分工”

Section titled “从“随缘问 AI”到“角色分工””

BMAD 把常见的团队角色做成可复用的 persona，你可以稳定地获得：

  • PM 视角的目标/边界/验收口径
  • 架构师视角的方案比较与技术决策
  • 开发视角的落地实现与改代码
  • QA / Test Architect 视角的验证方案与回归策略
  • Tech Writer 视角的文档与发布说明

过程可伸缩：小事快做，大事不乱

Section titled “过程可伸缩：小事快做，大事不乱”

BMAD 的思路是“按任务规模给出合理流程深度”：

  • Quick Dev / Quick Flow：修 bug、小需求、快速迭代
  • 标准轨道：中等规模功能/产品，PRD → 架构 → Stories
  • 企业轨道：强治理与合规场景，质量门槛更完整

形成从想法到验证的闭环

Section titled “形成从想法到验证的闭环”

BMAD 强调“产出工件（Artifacts）再实施”，而不是只做一次性对话。

2）前置条件、安装与配置

Section titled “2）前置条件、安装与配置”

你需要什么

Section titled “你需要什么”

BMAD 适用于任何支持“自定义系统提示词 / 项目上下文”的 AI 编程助手，例如：

  • Claude Code（CLI）
  • Cursor
  • Windsurf
  • Roo Code（VS Code）
  • GitHub Copilot（VS Code）

安装（典型方式）

Section titled “安装（典型方式）”

建议安装到项目目录而不是用户主目录，这样配置才能版本化、可迁移：

安装后会生成标准化的模块结构（agents/workflows/templates），随后即可在 IDE/助手里运行工作流。

3）Quick Dev 工作流（最快的上手方式）

Section titled “3）Quick Dev 工作流（最快的上手方式）”

如果你的 BMAD 安装已暴露斜杠命令，可以直接从这里开始：

如果你的目标是“尽快交付，但别埋坑”，Quick Dev 的核心节奏是：

  1. 明确目标与约束
  2. 形成轻量计划与验收标准
  3. 快速实现 + 紧凑验证

一个实战可复用的跑法：

Step A — 先写清楚“什么叫完成”

Section titled “Step A — 先写清楚“什么叫完成””

至少包含：

  • 验收标准（done 的判定）
  • 不要改动的边界（guardrails）
  • 验证方式（build / tests / 手动 smoke）

Step B — 用工作流生成“用得上的轻量工件”

Section titled “Step B — 用工作流生成“用得上的轻量工件””

Quick Dev 不是为了写大文档，而是为了产出能指导实现与验证的东西，例如：

  • 任务拆分（足够小、可串行）
  • 实施 checklist
  • 验证 checklist

Step C — 实施并验证

Section titled “Step C — 实施并验证”

让 Dev 快速落地，让 QA/Test Architect 给出：

  • 针对性回归点
  • 关键边界用例
  • build / check / preview 验证

最大的收益是：这些输出会沉淀为下一次迭代的可复用基线。

4）推荐使用场景与日常工作流示例

Section titled “4）推荐使用场景与日常工作流示例”

场景 1：修 bug 不走神

Section titled “场景 1：修 bug 不走神”

Quick Dev 适合：

  • 复现问题 → 定位根因
  • 加最小回归用例
  • 修复 → 验证构建

场景 2：小功能可控落地

Section titled “场景 2：小功能可控落地”

Quick Dev 适合：

  • 明确边界与异常分支
  • 定义验收标准
  • 分步实现
  • 快速验证体验与行为

场景 3：知识工作（写文档 / 写博客 / 学习）

Section titled “场景 3：知识工作（写文档 / 写博客 / 学习）”

BMAD 不只写代码。一个很实用的模式：

  • Tech Writer 先出结构（标题/大纲/关键段落）
  • Dev/Architect 校对技术准确性
  • QA 做“步骤可复现性”与遗漏检查

特别适合“教程 + 故障排查 + 参考链接”这种内容体裁。

5）注意事项、FAQ 与排障

Section titled “5）注意事项、FAQ 与排障”

“输出太啰嗦 / 太浅”

Section titled ““输出太啰嗦 / 太浅””
  • 检查安装/配置时的经验等级（expert vs beginner）。
  • 用验收标准收敛范围。
  • 优先让模型输出可复用工件（brief/spec/story），别只聊。

“上下文太长 / Token 不够”

Section titled ““上下文太长 / Token 不够””

利用 BMAD 的分片与 story 驱动，把信息拆小、按需喂给下一个角色。

“团队不想搞重仪式”

Section titled ““团队不想搞重仪式””

从 Quick Dev 开始；当你感受到收益，再把规划深度往上加。

6）协作与对比

Section titled “6）协作与对比”

BMAD 和一些“规范化 / spec-first”的方法（如 OpenSpec、Spec Kit）属于同一类思路，但 BMAD 的差异点在于：

  • 更明确的角色分工（Agents）
  • 可执行的工作流定义（Workflows）
  • 更强调**工件流转（Artifacts flow）**与可复制性

2025年12月29日
Vibe Sparking

深入探索 Code Forge Temple 开发的 Agentic Signal — 一个可视化的拖拽式 AI 工作流自动化平台，通过 Ollama 集成在本地运行所有功能。无需编写代码即可构建智能工作流，无需依赖云服务。

简介

Section titled “简介”

Agentic Signal 是由 Code Forge Temple 开发的开源可视化 AI 工作流自动化平台。它使用拖拽式界面让用户能够构建由 AI 驱动的智能工作流，同时所有操作都在本地机器上运行。

Agentic Signal 的独特之处在于其对隐私优先的本地执行的承诺。通过与 Ollama 集成，它允许您利用大语言模型的力量，而无需将数据发送到云端。这使得它非常适合处理敏感文档、构建个人自动化工作流，或者只是希望完全掌控 AI 交互的场景。

仓库地址： code-forge-temple/agentic-signal 官方网站： agentic-signal.com

BMAD Method：如何把大模型系统提示词，变成一支可复制的 AI 开发团队

2025年12月27日
Vibe Sparking

BMAD Method 是一个开源的 AI 驱动敏捷开发方法论与工具集，基于 BMAD Core 框架，内置 19 个专业 AI 角色（Agents） 和 50+ 条工作流（Workflows） ，覆盖从需求分析、规划、架构设计到开发与测试的完整生命周期，已经在 GitHub 上获得了 23.6k 颗 Star （截至 2025 年 12 月）。它把「系统提示词 + 角色定义 + 流程指令」打包成结构化资产，让你不再零碎地复制粘贴 prompt，而是像调度一个真正的软件团队那样，调度一群 AI 专家协同工作。

你为什么需要一个成体系的系统提示词仓库？

Section titled “你为什么需要一个成体系的系统提示词仓库？”

大部分 AI 学习者、开发者现在在用大模型时，都会遇到这些痛点：

  • 系统提示词乱七八糟 今天在 Claude 里写了一段不错的系统 prompt，明天在 ChatGPT 里又改了一版，结果散落在各个对话和文档中，根本找不到最佳版本。
  • AI 的角色不稳定 有时你希望它是产品经理，有时是架构师，有时是 QA，但每次都要重新解释，且每次解释都不完全一样，输出质量很难稳定。
  • 缺乏从需求到代码的闭环 即便你写了不少 prompt，更多是一次性的：需求文档一套写法，架构讨论一套写法，开发与测试又是一套，这些提示词之间没有严格的 artifact 流转关系，很难形成可复制的工程流程。
  • 协作成本高 当你想把自己整理的一堆 prompt 分享给同事时，对方往往只能照着一篇长文或一个 Notion 页面挨个复制，完全没有工具化。

如果你也有类似经历，那 BMAD Method 做的事情，可以理解为：

把「AI 系统提示词 + 角色分工 + 流程步骤」工程化、版本化、可移植化，变成一个可以在各种 IDE、大模型工具中反复复用的 AI 团队框架。

BMAD Method 本质在做什么？

Section titled “BMAD Method 本质在做什么？”

BMAD Method 的定位可以概括成一句话：

Breakthrough Method for Agile AI-Driven Development —— 一个让 AI 以敏捷团队的方式参与开发的框架。

它的核心有三层：

  • BMAD Core：一个为人类 + 多智能体协作设计的基础框架，里面包含：

    • 以 Markdown 编写的 Agents（专业角色 persona）
    • 由多个 Agent 组成的 Agent Teams
    • 用 YAML 定义的 Workflows（战略流程）
    • 带 LLM 指令的 Templates（PRD、架构文档、Story 文件等）

  • BMAD Method：站在 BMAD Core 之上，专注于软件开发场景的工作流，把分析 → 规划 → 方案 → 实施整个链路拆解成多个阶段与工件，由不同的 Agent 接力完成。

  • BMAD Builder / Creative Intelligence Suite：

    • BMAD Builder：让你可以在同一套框架上，设计自己的专业 Agent 与模块（如法律、金融、教育等领域）。
    • Creative Intelligence Suite：提供创意思考、头脑风暴类工作流，辅助产品创新、叙事设计等。

换句话说，BMAD Method 不是简单的 prompt 集合，而是把这些系统提示词按：

  • 角色（谁来做）
  • 阶段（何时做）
  • 工件（产出什么）
  • 工具（在哪儿用）

全部结构化整理成了一个可以移植到你项目里的工程方法论与代码仓库。

核心功能特性

Section titled “核心功能特性”

1. 19 个专业 AI 角色，组建虚拟开发团队

Section titled “1. 19 个专业 AI 角色，组建虚拟开发团队”

BMAD 内置了一个小型虚拟公司：

  • Analyst（分析师）
  • Product Manager（产品经理）
  • Architect（架构师）
  • Scrum Master（敏捷教练）
  • Product Owner
  • Developer（开发）
  • QA Engineer（测试）
  • UX Designer、Game Designer、Tech Writer 等等

每个角色都有：

  • 清晰的职责
  • 对应的命令（如 @analyst、@pm、@dev）
  • 对应要产出的文档类型（PRD、架构说明、Story 文件、QA 报告）

这些定义都是通过 Markdown + 系统提示词写好的，你无需重新发明如何让模型扮演一个合格的产品经理——直接调用现有 persona 即可。

2. 50+ 工作流覆盖完整开发生命周期

Section titled “2. 50+ 工作流覆盖完整开发生命周期”

BMAD Method 把整个软件交付过程拆成 4 个阶段：

  1. Analysis（分析）：需求挖掘、市场调研、方案对比
  2. Planning（规划）：PRD、技术规格说明、游戏设计文档等
  3. Solutioning（方案设计）：系统架构、数据流、API 设计、UX 草图
  4. Implementation（实施）：Story 驱动开发、代码实现、测试验证

针对每个阶段，框架都提供了多条工作流，以 YAML 形式定义流程与调用顺序。例如：

  • 用 Analyst 生成 Project Brief
  • 用 PM 生成 PRD
  • 用 Architect 生成 Architecture 文档
  • 用 PO 做文档分片
  • 用 SM 生成 Story 文件
  • 用 Developer/QA 实现与验证代码

这本质上是一套可执行的系统提示词流程，而不是一堆零散的 prompt 范例。

3. Scale-Adaptive Intelligence：三种智能轨道

Section titled “3. Scale-Adaptive Intelligence：三种智能轨道”

BMAD 提供了三种智能轨道：

  • Quick Flow：适合修 Bug、小需求，生成轻量级技术规格，几分钟即可开始写代码。
  • BMad Method 标准轨道：用于新功能、新产品或中等规模平台，从 PRD 到架构再到 UX。
  • Enterprise 轨道：面向合规、治理要求高的项目，提供更完整的文档、质量门槛与流程。

在实际使用时，你可以跑 *workflow-init 让 BMAD 自动根据目标与上下文推荐合适的轨道。

4. Story / Sharding 体系：文档当工程一等公民

Section titled “4. Story / Sharding 体系：文档当工程一等公民”

BMAD 里特别强调文档分片与 Story 文件体系：

  • 将大型 PRD 拆成多个 Epic 文件
  • 再从 Epic 生成带完整上下文的 Story 文件（如 1.1.user-registration.story.md）
  • Story 文件中包含：
    • 架构背景
    • 实现细节指引
    • 验收标准
    • 相关约束

最后，Developer 和 QA 都是围绕 Story 文件来工作——这就让系统提示词 + 文档 + 代码形成严格映射，方便版本控制、审计与回溯。

5. 基于 BMAD Core 的模块化架构

Section titled “5. 基于 BMAD Core 的模块化架构”

在框架层面，BMAD 明确把自身拆成若干模块：

  • Agents：Markdown persona 定义（本质上是结构化系统提示词）
  • Agent Teams：可复用的团队组合
  • Workflows：YAML 流程定义
  • Templates：嵌入 LLM 指令的文档模版
  • Expansion Packs：针对特定领域（创作、商业、教育、健康等）的扩展

这意味着：如果你正在做整理大模型系统提示词的工作，可以直接沿用它的这套目录结构和抽象方式，把你自己的 prompt 仓库建设在同一套架构之上。

Section titled “6. IDE & 工具链集成”

对于习惯在 IDE 里边写边问 AI 的开发者来说，这比在浏览器 Tab 里复制粘贴 prompt 舒服太多。

7. Token 节省与更新安全

Section titled “7. Token 节省与更新安全”

  • 官方宣称可以带来高达 90% 左右的 Token 节省（针对大型项目）

  • 升级 BMAD 版本时，保留你的个性化 Agent 与工作流设置

    • 使用 Analyst 的 Brief 作为输入
    • 产出：PRD.md

  4. Product Owner：文档分片

      • epic-1-user-auth.md
      • epic-2-products.md

  5. Scrum Master：生成 Story 文件

    • 针对某个 Epic 生成 Story 文件，如：
      • 1.1.user-registration.story.md
    • Story 中包含上下文、实现细节、验收标准等。

  6. Developer：代码实现

    • 围绕 Story 文件来实现代码（Controller、组件、数据库迁移等），并生成测试用例。

  7. QA：质量验证与追踪

    • 产生包括测试策略、追踪矩阵、质量门报告在内的一系列 QA 文档。

在这一整套过程里，你可以看到：

  • 每个阶段对应一个或多个 Agent
  • 每个 Agent 背后是一整段高度结构化的系统提示词
  • Artifacts（文档 & 代码）通过文件名和目录结构形成严格的追踪链路

这就是把系统提示词当成工程资产来管理的具体体现。

  • PromptSource 更偏向给不同 NLP 数据集配合不同 prompt 的开发工具
  • BMAD Prompt MCP 是把 BMAD 方法做成一个 Prompt 工程化服务的补充
  • Prompt Commons 更像是未来城市 / 机构级 Prompt 治理的设想与框架

而 BMAD Method 的独特之处在于：

  • 它把系统提示词 + 角色 + 工作流 + 文档作为一个整体来设计
  • 提供了从命令行到 IDE、从 Web UI 到 Git 仓库的完整落地路径
  • 已经在 GitHub 上形成较大的开源生态与使用者群体（23.6k Star、3.4k Fork）

小结

Section titled “小结”

如果把这篇文章压缩成一句话：

BMAD Method 展示了一种把系统提示词工程化的典型路径：用 Agents 抽象角色，用 Workflows 抽象流程，用 Templates 抽象产出物，再用 Git 抽象整个协作历史。

对于运营技术公众号、做 AI 教学或日常做项目的你来说，可以从这个仓库中学习到的，不只是用哪个 prompt 更香，而是：

  • 如何为不同角色设计结构化 persona
  • 如何让提示词与文档、代码形成严格映射
  • 如何通过文件结构与版本控制管理整个 AI 协作过程

即便你不直接使用 BMAD Method 的全部功能，也完全可以把它当成一个系统提示词工程实践教科书，照着它的目录结构和工作流方式，打造属于你自己的大模型系统提示词仓库。
