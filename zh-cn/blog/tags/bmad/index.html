BMAD v6 安装更新：模块化 + AgentVibes “会说话”的开发体验

2025年12月15日
Vibe Sparking

v6 的核心变化就是“目录/工件体系重做” + “语音能力（AgentVibes）正式集成进 BMAD 工作流”。

1）先说结论：v6 相比 v4，最“能感知”的两点变化

Section titled “1）先说结论：v6 相比 v4，最“能感知”的两点变化”

变化 A：目录从“单核”升级为“Core + Modules + Manifests”的工程化结构。 v4 的用户指南里，个性化偏好（比如技术栈、命名规则）是放在 .bmad-core/data/technical-preferences.md 里管理的。

而 v6（Alpha）的官方定位是“Agent-as-Code + 可复现流水线”，并明确给出 workspace anatomy：.bmad/ 用来放 profiles/blueprints/runs 等“可追踪工件”。

变化 B：AgentVibes 被当成“第一方体验层”接进来：让 BMAD agent 能在 Claude Code 里说话、切声线、用 slash command / MCP 控制。 AgentVibes 的官方说明就直白：最初就是为了给 Claude Code “加声音”，并且提供 MCP（自然语言控制）和 slash commands 两套控制方式。

2）目录结构：v4 的“.bmad-core” vs v6 的“.bmad + modules + artifacts”

Section titled “2）目录结构：v4 的“.bmad-core” vs v6 的“.bmad + modules + artifacts””

按日志里的安装结果来拆

v4（用户常见心智）

Section titled “v4（用户常见心智）”
  • 核心目录：.bmad-core/ 用来放偏好、数据、模板等；例如 v4 用户指南明确提到偏好文件位置：.bmad-core/data/technical-preferences.md。
  • Web Bundle 玩法：dist/teams/... 典型是复制 dist/teams/team-fullstack.txt 去 Web UI/Claude/GPT 里当“团队提示词”。
  • 更像“提示词框架 + 目录约定”，工程化程度够用，但**“产物可追踪/可复现”不是第一目标**。

v6（你这次安装体现出来的结构变化）

Section titled “v6（你这次安装体现出来的结构变化）”

从日志看，v6 安装完成后最关键的“落盘结构”是：

  • 根目录统一落在：<project>/.bmad（你这次是 /Users/vsai/.../vs-ai-agents/.bmad）

  • 模块化安装：Core + bmb + bmgd + bmm + cis

    • Core：安装核心 agent（比如 bmad-master.md）
    • bmb：Builder/Factory（自定义 agent、workflow、module 的“工厂”）
    • bmm：Method（Analyst/PM/Dev/SM/UX 等完整团队）
    • bmgd：Game Dev（并且会把跨模块 workflows “vendoring”进来）
    • cis：Creative Innovation Suite

同时，它还做了 v4 时代不太“显性”的几件事：

  • 编译/生成工件（manifests）：日志里直接报了 “61 workflows、21 agents、5 tasks、1 tools…”——这就是 v6 走向“可版本化/可追踪”的证据之一（工程化明显更强）。这一点也符合 v6 Alpha 强调的“可复现、可审计、可 diff”的方向。

  • Sprint 产物目录变成安装期配置项：你选择了 docs/sprint-artifacts，安装器还会自动创建（便于团队落地敏捷节奏）。

  • 自定义扩展目录被显式分离出来

    • bmad-custom-src：自定义 agents / workflows
    • bmad-custom-modules-src：自定义 modules 这其实是在逼你把“官方模块”和“自定义资产”隔离，方便升级/合并。

  • Sidecar memory 目录：日志里还替换了 {agent_sidecar_folder} → {project-root}/.bmad-user-memory（典型用于 agent 的“旁路记忆/运行痕迹”，也更符合 v6 的“工件化”理念）。

v4 更像“提示词框架 + 目录规范”；v6 更像“带构建产物与可追踪工件的 agent 工程系统”。

3）AgentVibes 集成：从“能用”到“好用”的关键升级

Section titled “3）AgentVibes 集成：从“能用”到“好用”的关键升级”

日志里最“出圈”的部分其实是这一段：

  • 安装器提示 “AgentVibes TTS Injection Summary”
  • 对 .../.bmad/core/agents/bmad-master.md 做了 TTS 注入
  • 备份放在 ~/.bmad-tts-backups/
  • 然后一键拉起 AgentVibes 安装器，往项目的 .claude/ 里塞了 slash commands、TTS hooks、personality templates、背景音乐等

这套设计跟 AgentVibes 的官方说明高度一致：

AgentVibes 到底解决了什么？

Section titled “AgentVibes 到底解决了什么？”
  • 它提供 MCP（自然语言控制）：让助手不必记 slash commands，也能“说：换成 Aria 声线”。
  • 同时也提供 slash commands（零 token 开销），例如 /agent-vibes:list、/agent-vibes:switch Aria。
  • 多 TTS provider 抽象：Piper、macOS Say、ElevenLabs 都在同一个能力层里（你的日志里也体现了 macOS Say 的快速安装路径）。
  • 安装结构明确写了：会把 commands/hooks/personalities/plugins 放进 .claude/。

为什么这对 BMAD v6 特别重要？

Section titled “为什么这对 BMAD v6 特别重要？”

因为 v6 的方向是“多 agent、多 workflow、多工件”，人的注意力会被大量“过程信息”打散。TTS 反而能带来两个好处：

  1. 状态反馈更强：开始做什么、做完了什么，耳朵一听就知道，不用盯屏幕。
  2. 角色感更强：PM/SM/Architect/Dev 这类角色切换，会更有“团队协作”的沉浸感（AgentVibes 的 BMAD 集成也在强调这点）。

4）给从 v4 到 v6 的总结

Section titled “4）给从 v4 到 v6 的总结”

  1. 把 v4 的偏好文件迁移思路定好 v4 偏好在 .bmad-core/data/technical-preferences.md。 v6 更强调 workspace + policies + manifests（.bmad/ 下的结构）。 建议你做法是：保留 v4 偏好内容作为“团队约定”，再逐步映射到 v6 的 profile/policy 体系（不要一次性硬迁，容易丢语义）。

  2. 把“自定义资产”放进你已经配置好的两个目录

  • agent / workflow → bmad-custom-src
  • module → bmad-custom-modules-src 这样未来升级 v6 alpha/beta 时，冲突会少很多。
  1. AgentVibes 用法上，建议默认“slash commands 优先” AgentVibes 官方也强调：MCP 更方便，但会吃上下文 token；slash commands 更省。 也就是说：日常用 / 命令，偶尔再用自然语言 MCP，体验和成本会比较平衡。

AI 时代的文档屎山？BMAD、Spec-Kit、OpenSpec 等面向文档AI编程的利弊

2025年10月23日
Vibe Sparking

大家还记得曾经让人头大的”屎山代码💩“吗？如今在 AI Claude Code, Codex, Gemini 等加持下，我们又多了一个新坑——“文档屎山💩“。面向文档AI编程最近很火，说白了就是先写好文档（规范说明），再让 AI 根据文档写代码。听起来很美好：不用再瞎猜 AI 想要什么，写个清晰说明就能自动出代码。然而，真相是如果方法不对，文档反而更容易堆成一座新的屎山，让人啼笑皆非😅。

引言：BMAD、Spec-Kit、OpenSpec 是什么鬼？

Section titled “引言：BMAD、Spec-Kit、OpenSpec 是什么鬼？”

先说说这几个时下流行的面向文档AI编程工具吧，它们各有千秋：

BMAD 方法 – 一个”大而全”的框架，全名 Breakthrough Method for Agile AI-Driven Development。它野心很大，想 AI 模拟完整敏捷团队。里面有分析员AI、架构师AI、开发AI、测试AI等等，各司其职，像搭戏台一样把规划、编码、测试都演一遍。BMAD 提供了 CLI 工具和网页界面，用特殊的”剧情文件”和上下文管理来让 AI 团队协作。它通常分两阶段：先由”分析员/产品经理/架构师”代理规划设计，再由”Scrum Master/开发”代理具体写代码。优点是过程完备、有条不紊，适合大型复杂项目，强调清晰的文档和分工。但也相当繁琐，上手门槛高，对底层模型理解你写的文档要求也高。

Spec-Kit – GitHub 开源的小工具，帮你在现有 AI 编码助手上加一层”流程”。开发者通过命令行引导 AI 按顺序执行 “规范 -> 计划 -> 任务 -> 实现” 四步走。简单讲，Spec-Kit 先写清需求规范，然后让 AI 按规范一步步拆解任务、写代码。它用 /specify、/plan、/tasks 这些口令把我们的规范文档变成”可执行文档”。优点是轻量灵活，不用大改现有工作流，也不绑定特定模型。适合个人开发者或小团队给 AI 编码加点结构，提升代码质量。

OpenSpec – 一个极简风格的新秀。它追求”能简单就不复杂”，让你用纯 Markdown 写规范文件，配合少量指令完成开发。流程只有三步：提出修改提案、AI 生成任务列表、AI 应用规范改代码。OpenSpec 的亮点是效率高，耗费 token 少，因为它的规范很精简，甚至没什么额外元数据。它适合个人开发者或小团队追求极致速度，小改动几分钟搞定。缺点是目前支持的 AI 模型有限，但胜在上手容易，基本没什么额外负担。

看出来了吗？这些工具理念都差不多：让开发先写好”文档规范”，再驱动 AI 写代码。初衷是减少 AI 瞎发挥（所谓 vibe coding）带来的不确定性，代码质量更可控。然而，理想很丰满，现实有点骨感。一不小心，我们写的文档反而可能变成另一坨屎山。为什么这么说？往下看。

文档更容易变成屎山的四大原因

Section titled “文档更容易变成屎山的四大原因”

1. 缺乏结构

Section titled “1. 缺乏结构”

写代码有语法约束，错一个逗号程序都跑不通；可写文档就自由得多了。AI 又不是人，面对随意发挥的长篇文档，理解起来经常跑偏。毕竟代码是确定的，规范文档却往往很模糊，不同的人和AI会有不同解读。你写的文档如果结构松散，AI 看了就像雾里看花，最后生成的代码南辕北辙也不稀奇。

2. AI 理解不一致

Section titled “2. AI 理解不一致”

就算文档写得还行，也难保 AI 每次都理解成同一种意思。有经验的开发者都知道，“一千个读者就有一千个哈姆雷特”——更何况 AI 模型和人脑思维差异巨大。今天 ChatGPT 也许按规范实现了功能，明天换个模型可能曲解你的意图。文档不像代码有编译器检查，AI 理解偏了你不一定立刻发现，这无形中埋下隐患。

3. AI 幻觉放大错误

Section titled “3. AI 幻觉放大错误”

大家知道大语言模型会”胡编乱造”——也就是幻觉，一本正经地瞎说。当 AI 用不完整或有歧义的文档去生成代码时，这种问题被放大了。举个例子，你文档里漏了一点细节，AI可能自作聪明编造出一个看似合理但实际上错误的实现。更糟的是，AI 写出的文档或代码带着谬误，下次另一个 AI 还当真引用，错误滚雪球越滚越大。就像谣言一样，一旦进了正式文档，后来的人和AI都会被误导。

4. 工具带来的新风险

Section titled “4. 工具带来的新风险”

最后，不得不提这些新工具本身的复杂度。为了用 Spec-Kit、BMAD 之类，你可能要安装额外的软件、学习一堆新流程。折腾半天工具，反而忘了初心——花大量时间写规范、调框架，而真正写代码的时间变少了。更尴尬的是，如果不深入理解这些框架，你可能按模板生搬硬套，反而让文档和实际代码脱节。就像拿着手术刀不会用的人，伤害可能更大。

如何避免屎山：文档工具分级使用指南

Section titled “如何避免屎山：文档工具分级使用指南”

听起来问题不少，不过别灰心，我们可以制定分级使用策略，把面向文档编程的坑变成更加平坦的路：

小改动直接问

Section titled “小改动直接问”

当你只是代码里改个小逻辑、调个样式，不需要大费周章写什么规范。这种情况下，直接用 Claude Code, Codex, Gemini， GitHub Copilot, Cursor 等这类助手提问最快。一两句话说明需求，让 AI 给出改动建议或代码片段就行。搞不好，还省下写文档的时间。正如有经验的人所说，小需求不用上纲上线搞全过程文档，能用脑子和对话解决的，就别兴师动众。

中等改动用 Spec-Kit/OpenSpec

Section titled “中等改动用 Spec-Kit/OpenSpec”

如果是增加一个中等规模的新功能，涉及多个步骤但不算庞大，这时可以考虑用 Spec-Kit 或 OpenSpec 这类轻量工具。先花十几分钟写个MVP级别的规范说明，列出主要需求和边界条件，然后交给 AI 去细化计划和任务清单。Spec-Kit 会帮你把任务逐个拆开，让你可以逐步实现；OpenSpec 则省去上下文切换，几条命令完成修改。关键是保持规范简明扼要，只覆盖必要细节。事实证明，简洁的规范更容易让 AI 正确执行，速度也更快。一旦功能完成，记得及时更新文档或将临时规范清理掉，避免堆积过时信息。

大改动用 BMAD

Section titled “大改动用 BMAD”

遇到重构大型模块、开发全新复杂子系统这种”大手术”，就需要更严谨的方案。BMAD 这类多代理框架虽然重，但在这时候派上用场：它能帮你全面规划——从需求分析、架构设计到任务分配，都有条不紊。相当于请了一支 AI 虚拟团队给你出谋划策，确保大项目不失控。当然，人要在回路中把关，不要盲目相信 AI 的每一步输出，要及时 review 和修正。启动 BMAD 前也要评估成本：它用时较长、对模型理解力要求高，但对于复杂项目可能是值得的投入。大型项目往往生命周期长，文档也要实时演进，BMAD 生成的规范、用户故事等应该和代码一起维护更新，别一股脑用完就扔，否则下次还得推倒重来。

此外，无论大改小改，都请牢记**“文档 MVP”原则：文档只覆盖当前必要且真实的情况**，不追求面面俱到。很多高手分享经验说，功能做完后他们往往会删除临时规范文档，因为代码本身已经说明一切。与其把过期文档留在那儿坑人，不如保持一个精简而准确的说明，然后随着代码和业务演进不断更新。记住，文档的价值在于及时反映真实情况，脱节再多也只是废纸一张。

结语

Section titled “结语”

面向文档编程是 AI 时代的一把双刃剑。用得好，它确实能让开发更高效、有条理；用不好，就会出现新的屎山，让人抓狂。对于追求效率和品质的年轻 IT 从业者来说，新工具值得探索，但不能迷信。代码也好，文档也罢，核心还是对需求的理解和及时维护。当我们选择何时写规范、用哪种工具时，不妨多问问自己：“这么做是为了更清楚地解决问题，还是在制造额外的复杂度？”

最后，希望每位开发者都能驾驭这些 AI 工具，写出既清爽又实用的文档，让文档真正成为代码的助力，而不是新的负担。毕竟，我们要的是更高的生产力，可不想真的在办公室里堆出一座”屎山”来！ 💩
