OpenClaw Telegram Topics: 一个群组运行多条并行任务流

Section titled “OpenClaw Telegram Topics: 一个群组运行多条并行任务流”

OpenClaw 是一个开源 AI 助手，支持 Telegram、Discord 等多个平台。它的”并发任务引擎”能力在配合 Telegram Topics 功能时效果尤为出色。本指南将带你在单个群组中配置多条独立任务流。

理解 OpenClaw 的并发模型

Section titled “理解 OpenClaw 的并发模型”

OpenClaw 通过 Sessions 处理任务：

  • 一条 Session = 一条独立任务流，拥有独立上下文
  • 多个 Session = OpenClaw 同时处理多条任务，互不干扰

在 Telegram 里，每个 Topic 就是一个 Session。一个群可以包含聊天、工作、资讯三个 Topic，每个都维护独立的对话和上下文。

Session Key 架构

Section titled “Session Key 架构”

OpenClaw 根据聊天位置生成 session key：

位置Session Key 格式
私聊共享主会话
群组agent:<agentId>:telegram:group:<chatId>
Topicagent:<agentId>:telegram:group:<chatId>:topic:<threadId>

:topic:<threadId> 后缀隔离了每个 Topic 的上下文。Topic A 的消息不会串到 Topic B。

设置并行任务通道

Section titled “设置并行任务通道”

推荐的 Topic 结构

Section titled “推荐的 Topic 结构”

从三个 Topic 开始，覆盖常见工作流：

Topic用途使用场景
Chat日常聊天随意提问、快速问答
Work办正事文档处理、数据分析
Feed资讯巡逻新闻聚合、监控追踪

第一步：让 Bot 能看到群消息

Section titled “第一步：让 Bot 能看到群消息”

Telegram 的 Privacy Mode 默认阻止 bot 看到群消息。解决方法：

  1. 打开 @BotFather
  2. 发送 /setprivacy
  3. 选择你的 bot
  4. 选择 Disable
  5. 把 bot 从群里踢掉再重新拉进来
  6. 授予管理员权限

管理员身份可以确保 bot 看到所有消息，不受隐私设置影响。

第二步：在群里开启 Topics

Section titled “第二步：在群里开启 Topics”
  1. 打开群设置
  2. 找到 Topics 或 论坛 模式
  3. 开启它
  4. 创建 Chat、Work、Feed 三个 Topic

第三步：配置每个 Topic 的行为

Section titled “第三步：配置每个 Topic 的行为”

编辑你的 openclaw.json 配置文件：

把 -1001234567890 替换成你的群 ID（把任意消息转发给 @userinfobot 就能获取）。把 123 和 456 替换成你的 Topic thread ID。

配置选项说明

Section titled “配置选项说明”
选项可选值效果
requireMentiontrue/falsebot 是否需要 @才回复
systemPrompt字符串给这个 Topic 的额外指令
skills数组限制该 Topic 可用的技能
allowFrom数组限制该 Topic 允许的发送者

Topic 会继承父群组的设置，除非显式覆盖。

测试你的配置

Section titled “测试你的配置”

快速验收清单：

  1. 在 Chat Topic 发消息：bot 应该只在被 @时回复（如果 requireMention: true）
  2. 在 Work Topic 发消息：bot 应该回复所有消息（如果 requireMention: false）
  3. 检查隔离性：用 /new 重置一个 Topic 的会话，其他 Topic 应该保持上下文

常见问题排查

Section titled “常见问题排查”

Bot 在群里完全没反应

Section titled “Bot 在群里完全没反应”

原因：Privacy Mode 还开着，或者 bot 没有管理员权限。

解决：在 @BotFather 重新运行 /setprivacy → Disable。确保 bot 是群管理员。

私聊正常但群里不正常

Section titled “私聊正常但群里不正常”

原因：隐私设置和 requireMention 配置的组合问题。

解决：

  1. 确认管理员身份
  2. 检查 requireMention 值是否符合你的意图
  3. 先试试显式 @bot 确认它能收到消息

回复出现在错误的 Topic

Section titled “回复出现在错误的 Topic”

原因：JSON 里的 Topic ID 配错了。

解决：用 @userinfobot 确认准确的 Topic thread ID，更新配置。

重置会话影响多个 Topic

Section titled “重置会话影响多个 Topic”

已知问题：在多 agent 配置中，对非默认 agent 使用 /new 可能同时重置它和默认 agent 的会话。

临时方案：每个群使用单 agent 配置，或关注 GitHub issue 等待修复。

Section titled “进阶：多 Bot 的多 Agent 配置”

每个账号维护完全独立的会话和配置。

最佳实践

Section titled “最佳实践”
  1. 从简单开始：在工作流稳定之前，最多用三个 Topic
  2. Topic 命名：清晰的名称帮助人和 bot 都理解上下文
  3. 选择性 @：高流量 Topic 保持 requireMention: true 避免刷屏
  4. 历史记录限制：根据 Topic 的上下文需求调整 historyLimit

什么是 Clawdis？

Section titled “什么是 Clawdis？”

Clawdis 是你个人 AI 助手的本地优先控制平面。你可以把它想象成一个基础设施，让你通过以下方式与 AI 代理交互：

  • WhatsApp — 发送消息，获得智能回复
  • Telegram — 通过 Telegram Bot API 进行机器人对话
  • WebChat — 基于浏览器的桌面界面
  • macOS 应用 — 菜单栏控制，支持语音唤醒和按住说话
  • iOS 节点 — Canvas 界面与语音触发转发

“Clawdis” 这个名字是一个巧妙的混成词：CLAW + TARDIS — 因为每只太空龙虾都需要一台时空机器！

核心特性

Section titled “核心特性”

1. 多平台收件箱

Section titled “1. 多平台收件箱”

Clawdis 将你的 AI 助手带到你已经在使用的通讯平台。无需切换到专用应用 — 只需从 WhatsApp 或 Telegram 发送消息，你的助手就会回复。

2. 语音唤醒和按住说话

Section titled “2. 语音唤醒和按住说话”

在 macOS 和 iOS 上，Clawdis 支持：

  • 语音唤醒 — 说出唤醒词开始对话
  • 按住说话叠加层 — 快速语音交互
  • 本地语音识别 — 隐私保护，设备端处理

3. 实时 Canvas

Section titled “3. 实时 Canvas”

一个可以从代理控制的可视化工作区。Canvas 渲染实时内容，支持 A2UI（代理到 UI）通信，提供丰富的交互体验。

4. 自动化就绪

Section titled “4. 自动化就绪”

Clawdis 内置以下功能：

  • 浏览器控制 — 自动化网页交互
  • 媒体处理 — 处理图片、音频和视频
  • 工具流式传输 — 实时工具执行反馈

用 Claude Code Hooks 把消息秒送到 Telegram

2025年8月14日
Vibe Sparking

写代码不想守着终端？ 让 Claude Code 自己在关键节点“叮咚”你。 用 Hooks + Telegram Bot，3 分钟搞定全流程，稳、准、快。

这事儿为什么靠谱？

Section titled “这事儿为什么靠谱？”
  • Hooks 是硬规则。不是“提示词祈祷”，而是在固定时机自动跑脚本，比如“需要你授权”“等你输入”“任务结束”。可控、可复用。
  • Notification 事件天然适配提醒。当 Claude 等你点权限，或闲置 60 秒，就会触发。接好这个“水龙头”，就能把提醒推到手机。
  • 实现超简单。一段 shell 脚本 + Telegram 官方 API，一把梭。

一张图看流程（你马上会配出来）

Section titled “一张图看流程（你马上会配出来）”

Claude Code → 触发 Notification Hook → 执行本地脚本 → 调 Telegram Bot API → 你的手机 Telegram 收到消息。

① 在 Telegram 创建机器人

Section titled “① 在 Telegram 创建机器人”

1）找 @BotFather，新建机器人，拿到 BOT_TOKEN。 2）给你的机器人发条消息，然后用 getUpdates 拿 CHAT_ID。

② 写一个 Hook 脚本（telegram_notify.sh）

Section titled “② 写一个 Hook 脚本（telegram_notify.sh）”

把它放进你的项目：$PROJECT/.claude/hooks/telegram_notify.sh

验证与排错

Section titled “验证与排错”

  • 触发验证：让 Claude 运行需要授权的 Bash，或挂着 1 分钟不回复，应该立即收到通知。

  • 调试：claude --debug 查看 Hook 执行日志；或在脚本里临时 echo 到文件看输入。

  • 常见坑

    • chat_id 不对 → 重新查 getUpdates。
    • 文本不换行 → URL 编码 %0A。
    • JSON/引号转义 → 设置里用 \"。

进阶玩法（可选）

Section titled “进阶玩法（可选）”
  • 更强的格式：用 HTML/Markdown 渲染代码块、链接预览开关。
  • 安全隔离：不想把 Token 放本地？用 MCP 的Telegram Notifications 服务，部署到 Cloudflare Workers，再让 Claude 走 MCP 工具发信。
  • 成品项目参考：有人做了“Claude Code Telegram 远程操控/通知”开源仓库，可以借鉴结构与权限设计。

安全清单（务必过一遍）

Section titled “安全清单（务必过一遍）”
  • Hooks 能执行任意命令。只放团队可读的脚本，脚本里严格引用变量，避免路径穿越。
  • 私密信息用环境变量或密钥管理，脚本不要 echo 出来。
  • 在单独项目先试跑，再推广到全局。

你可以这样落地

Section titled “你可以这样落地”
  • 团队协作流：需要权限时推送群组，谁在线谁处理。
  • 长任务：Stop 事件结束就发结果摘要链接。
  • 子任务并发：SubagentStop 单独提醒，方便分配。

小结

Section titled “小结”

把“等你一下”变成“手机震一下”。 Hooks 负责“什么时候”，Telegram 负责“到你手里”。 稳定、简单、不打扰专注。
