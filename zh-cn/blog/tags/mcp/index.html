用 Claude Code 做「微信聊天日报」：用 chatlog + MCP，3 步跑通

2025年9月18日
Vibe Sparking

把昨天所有群聊和私聊，一键变日报：top 关键词、重点人、待办、风险提醒，全自动。 本文手把手教你用 Claude Code + chatlog 把聊天数据榨干成洞察，还送上开箱即用的提示词。

你会得到什么

Section titled “你会得到什么”
  • 跑通流程：导出/解密 → 开 HTTP → 连接 Claude Code（MCP） → 一键出日报
  • 提示词模板：复制即用，覆盖拉取数据、分析聚合、生成日报（Markdown/HTML/图片）
  • 最佳实践：关键词雷达、敏感词预警、待办抓取、情绪曲线
  • 避坑清单：密钥、SIP、远程/本地连接、CSV/语音

chatlog 是一个开源工具，支持从本地微信数据库取数、自动解密、HTTP API、Webhook、以及 MCP Streamable HTTP，能直接被 Claude Code 当成「数据工具」来用。

一、准备工作（10 分钟）

Section titled “一、准备工作（10 分钟）”

1）安装并启动 chatlog（本地）

Section titled “1）安装并启动 chatlog（本地）”

启动后在 TUI 里选择 解密数据 → 开启 HTTP 服务（默认 http://127.0.0.1:5030）。 API 示例：

支持 YYYY-MM-DD 或 YYYY-MM-DD~YYYY-MM-DD 时间窗，format 支持 json/csv/text。

小贴士：需要先获取数据密钥/图片密钥；macOS 用户获取前需临时关闭 SIP（完事可再开启）。

2）把 chatlog 作为 MCP 服务器暴露给 Claude Code

Section titled “2）把 chatlog 作为 MCP 服务器暴露给 Claude Code”

chatlog 原生支持 MCP Streamable HTTP，启动 HTTP 后路径为：

然后在 Claude Code 输入 /mcp 管理与使用工具。

MCP 是 Anthropic 推出的开放标准，支持 Streamable HTTP 传输、OAuth/Token 等认证，便于把外部数据接入 AI 助手。

备注：Claude 的远程 MCP 支持已逐步放开，也可以走本地 STDIO；具体以你账户与版本为准。

二、最省心的日报结构（可直接让 Claude 生成）

Section titled “二、最省心的日报结构（可直接让 Claude 生成）”
  • 今日概览：消息总量、活跃会话数、Top 群/私聊、峰值时段
  • 关键词雷达：TF-IDF/关键短语/话题聚类（合并同义词）
  • 任务抽取：含「明天/今晚/帮我/待办/安排」等表述的 To-Do 清单
  • 风险提醒：自定义敏感词/合规词库命中计数与上下文
  • 情绪曲线：按小时的情感分布（积极/中性/消极），标注异常峰值
  • 高价值片段：Top5 信息密度最高的对话摘录（附会话名/时间戳/消息 ID）
  • 附录：原始数据链接（API 可回跳到图片/文件/语音解密流）

四、最佳实践：让日报更聪明

Section titled “四、最佳实践：让日报更聪明”

1）关键词合并表：把常见口头禅对齐成同义词，避免词云涂抹。 2）任务抽取规则：正则 + few-shot，针对「帮我」「记得」「今晚前」等语气词优化。 3）角色画像：按发言密度/影响力为群成员画像，识别“信息源头”。 4）Webhook 增量：配合 chatlog 的 Webhook，在有新消息时推送到你的中间件，做增量更新。 5）CSV/语音：新版本对 CSV 与语音工具链做过修复/增强，关注 Releases Changelog。

五、避坑清单

Section titled “五、避坑清单”
  • 只处理你自己合法拥有的数据。遵守隐私与所在平台条款；项目本身也强调这一点。
  • macOS 获取密钥需临时关闭 SIP，完成后可再开启。
  • 远程 MCP 仍在逐步放开，本地 STDIO 更稳；留意你账号与版本的能力。
  • MCP 连接命令格式 以官方文档为准，claude mcp add --transport http <name> <url>。
  • MCP 概念/认证：用 Streamable HTTP + 标准认证方式，团队部署更安全。

六、进阶：一键换源 & 团队共享

Section titled “六、进阶：一键换源 & 团队共享”
  • 用 Docker 部署 chatlog 到 NAS，预先写死 CHATLOG_WEBHOOK_* 环境变量，方便全家桶联动。
  • 团队用 Claude Connectors/MCP 目录 管理多数据源，非技术同学也能点点就连。

结语

Section titled “结语”

把聊天当数据，用 Claude Code 把它炼成团队可执行的日报。 每天 3 分钟浏览，省下的时间去做真正重要的事。

Codex 接入推荐的 10 个 MCP

2025年9月10日
Vibe Sparking

把 MCP 服务器写进 ~/.codex/config.toml，Codex 就能“长出手脚”，会爬网、会看文档、会点浏览器、会改代码。本文给你可复制配置 + 使用场景 + 避坑提示。

引言

Section titled “引言”

过去我们让模型写代码、查资料、跑任务，总感觉差点“动手能力”。MCP（Model Context Protocol）像 AI 的 USB-C，把外部工具以统一方式接进来，Codex 配好就能随叫随到。OpenAI 文档把 MCP比作“给 AI 的 USB-C 接口”，意思就是接上就能用。vs

微软的 Playwright MCP 也上了车：智能体能真开浏览器、看可达性树、验证变更，不必靠截图。vs

一分钟上手：~/.codex/config.toml

Section titled “一分钟上手：~/.codex/config.toml”

在 Codex CLI 里，MCP 服务器列表写在 mcp_servers 段（TOML）。参考结构：vs

Codex 最近版本支持 per-server 启动超时（startup_timeout_ms），官方发布笔记已合并此能力，社区 issue 也提到默认是 10 秒的超时场景，超时可调更稳。vs

小贴士：很多 MCP 客户端偏好 STDIO 传输；如果遇到只支持 SSE 的远程服务，用 mcp-proxy 做桥接就行（STDIO↔SSE 双向）。vs

抄作业清单：10 个高能 MCP 服务器

Section titled “抄作业清单：10 个高能 MCP 服务器”

每段都给你用途、可复制配置、要点。把块复制到 ~/.codex/config.toml 即可。环境变量记得换成自己的 Key。

1）Context7 —— 一键把最新库文档/示例塞进上下文

Section titled “1）Context7 —— 一键把最新库文档/示例塞进上下文”

用途：按你项目依赖版本，抓最新官方文档与示例，减少“旧接口/幻觉”。vs

要点：支持多客户端；文档里也给了 Cursor/VS Code/Zed/Claude 的配置样例。vs

2）DeepWiki —— 针对任意 GitHub 仓库的问答&结构化知识

Section titled “2）DeepWiki —— 针对任意 GitHub 仓库的问答&结构化知识”

用途：对 DeepWiki 索引过的开源仓库，直接问答、读 wiki 内容/结构。vs

要点：官方宣布免费、免登录；可提交你的仓库去索引。vs

3）Playwright —— 让 Codex 真开浏览器做 E2E 验证

Section titled “3）Playwright —— 让 Codex 真开浏览器做 E2E 验证”

用途：页面交互、可达性树、执行脚本、自动化测试。vs

要点：官方 npm 包与仓库维护活跃；适合回归验证、可视化检查、端到端测试。vs

4）Exa —— 实时 Web 搜索（结构化结果）

Section titled “4）Exa —— 实时 Web 搜索（结构化结果）”

用途：用 Exa Search API 做实时检索，返回结构化标题/URL/摘要，适合研究/资料搜集。vs

要点：官方文档与 npm 包都在更新；支持远程托管版本。vs

5）Spec-Workflow —— 规格驱动开发的整套流水线

Section titled “5）Spec-Workflow —— 规格驱动开发的整套流水线”

远程/SSE 服务器怎么接？（不会就用 mcp-proxy）

Section titled “远程/SSE 服务器怎么接？（不会就用 mcp-proxy）”

很多客户端偏好 STDIO，本地可执行没问题；远程 SSE 就需要桥接。最省事的方法是 mcp-proxy：

  • 模式 A：STDIO → SSE（让只认 STDIO 的客户端连到远程 SSE 服务器）
  • 模式 B：SSE → STDIO（把本地 STDIO 服务器暴露成可远程连的 SSE）vs

一行命令示例 mcp-proxy --port=8080 -- npx @playwright/mcp@latest（把本地 STDIO 的 Playwright 暴露成 SSE）vs

terminalcp 让 Claude Code 像 Playwright 自动管理 Terminal 终端会话

2025年8月13日
Vibe Sparking

一句话：terminalcp = 终端里的 Playwright。 让 AI Agent 能像人类一样启动、操作、读取任何命令行工具（LLDB/GDB、Python REPL、数据库 shell、甚至其他 AI CLI），还能你我随时接管同一会话。

引言

Section titled “引言”

你可能已经把 Playwright/Browser 用在网页自动化上，那**命令行的“自动化控台”**呢？ terminalcp 把“人类在终端里干的事”抽象成 MCP 工具：启动进程 → 输入按键 → 读取输出 → 多会话管理，还支持 Socket 附着，方便你实时旁观或接管 AI 的操作。

为什么它值得上工具链？

Section titled “为什么它值得上工具链？”

1）真·交互式控制 不是简单执行一条命令就退出，而是持续会话：可在 LLDB/GDB 里单步、在 REPL 里试代码、在 vim/htop 这类 TUI 里看“完整屏幕”。

2）两种输出模式，按需选择

  • Terminal 模式（stdout）：拿到渲染后的“屏幕缓冲区”，适合调试器、TUI。
  • Stream 模式：拿到原始文本流（默认剥离 ANSI），还能增量读取 since_last，盯构建/日志更高效。

3）高拟真 PTY 进程运行在伪终端里，保留颜色/光标移动/特殊按键序列，等同人手操作；并且后台管理多个进程，Agent 不被阻塞。

4）Socket 附着：人机协作的关键一环 每个进程自动生成 Unix 域套接字。你可以 attach 到 AI 已开的会话里共同看屏、共同操作，随时救火。

5）比 screen/tmux 更“AI 友好”

  • 结构化响应（JSON），不是只给一堆文本。
  • 干净输出（可剥离 ANSI），更好喂给 LLM。
  • 增量读取与生命周期管理（退出码等）一站式搞定。

和 screen/tmux 的核心差异（实测更适配 AI）

Section titled “和 screen/tmux 的核心差异（实测更适配 AI）”
  • 有结构化响应：JSON 返回，Agent 易解析。
  • 可靠拿输出：stdout 给你“渲染后的屏幕”，stream 支持只取新增。
  • 少污染：默认剥离 ANSI 控制码，避免“转义序列淹没上下文”。
  • 状态感知：能感知进程退出/错误码；
  • 无需猜时序：不用 sleep 等待“也许完成了”。 tmux/screen 想“模拟”这些能力，很费力也不稳。

最佳实践清单

Section titled “最佳实践清单”
  • 用绝对路径启动 CLI（比如 Claude CLI），别名不生效。
  • 交互命令行结合 {"submit":true}，或分两条：文本 + \r。
  • 长任务用 stream+since_last，避免重新拉全量日志。
  • 会话命名 + Socket 附着，便于协作救场。
  • 完成后显式 stop；MCP 退出会自动清理残留进程。
  • 许可证 MIT，可安全融入你的内部工具链。
