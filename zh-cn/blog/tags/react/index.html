Vercel Agent Skills：AI 编程助手的官方技能集

2026年1月28日
Vibe Sparking

AI 编程助手越来越强了，但怎么让它们懂更多的”行规”？Vercel Labs 最近放出了一个技能集合，把多年的工程经验打包成了 AI 能读懂的格式。

vercel-labs/agent-skills 这个仓库包含了五个生产级别的技能：React 优化、Web 设计规范、React Native 模式、React 组合模式，还有一键部署。

什么是 Agent Skills？

Section titled “什么是 Agent Skills？”

Agent Skills 就是给 AI 编程助手的”插件包”。跟普通文档不一样，这些技能是专门为 LLM 设计的：

  • AI 能自己读懂规则
  • 写代码或审查代码时自动应用
  • 识别上下文并激活对应技能
  • 需要的时候还能跑脚本

Vercel 的技能用的是 Agent Skills 格式，所以能在各种 AI 编程平台上用。

有哪些技能？

Section titled “有哪些技能？”

1. React 最佳实践

Section titled “1. React 最佳实践”

react-best-practices 收录了 40 多条规则，分 8 个大类，是 Vercel 工程团队十年来搞 React 和 Next.js 积累的经验。

什么时候用：

  • 写新的 React 组件或 Next.js 页面
  • 实现数据获取（客户端或服务端）
  • Code Review 找性能问题
  • 优化包大小或加载时间

包括：

  • 消除瀑布请求（关键） - 并行数据获取、预取
  • 包体积优化（关键） - Tree shaking、代码分割、动态导入
  • 服务端性能（高） - 服务端组件、流式渲染、缓存
  • 客户端数据获取（中高） - SWR/React Query 模式
  • 重渲染优化（中） - Memoization、useMemo、useCallback
  • 渲染性能（中） - 虚拟列表、CSS containment
  • JavaScript 微优化（低中） - 事件处理、属性访问

2. Web 设计指南

Section titled “2. Web 设计指南”

web-design-guidelines 能审查 UI 代码，100 多条规则覆盖了可访问性、性能和 UX 最佳实践。

什么时候用：

  • “帮我审查 UI”
  • “检查可访问性”
  • “审计设计”
  • “Review UX”
  • “按最佳实践检查我的网站”

包括：

  • 可访问性 - ARIA 标签、语义化 HTML、键盘导航
  • 焦点状态 - 可见的焦点指示器、focus-visible 模式
  • 表单 - 自动完成、验证、错误处理
  • 动画 - prefers-reduced-motion、合成器友好的 transform
  • 排版 - 弯引号、省略号、等宽数字
  • 图片 - 尺寸、懒加载、alt 文本
  • 性能 - 虚拟化、避免布局抖动、preconnect
  • 导航与状态 - URL 反映状态、深度链接
  • 深色模式与主题 - color-scheme、theme-color meta
  • 触摸与交互 - touch-action、点击高亮
  • 国际化 - Intl.DateTimeFormat、Intl.NumberFormat

3. React Native 指南

Section titled “3. React Native 指南”

react-native-guidelines 包含 16 条规则，覆盖移动端性能、架构和平台特定模式。

什么时候用：

  • 开发 React Native 或 Expo 应用
  • 优化移动端性能
  • 实现动画或手势
  • 使用原生模块或平台 API

包括：

  • 性能（关键） - FlashList、memoization、密集计算卸载
  • 布局（高） - Flex 模式、安全区域、键盘处理
  • 动画（高） - Reanimated 库、手势处理
  • 图片（中） - expo-image、缓存、懒加载
  • 状态管理（中） - Zustand 模式、React Compiler
  • 架构（中） - Monorepo 结构、import 优化
  • 平台（中） - iOS/Android 特定模式

4. 组合模式

Section titled “4. 组合模式”

composition-patterns 帮你避免布尔 prop 爆炸，通过复合组件、状态提升和内部组合来构建更灵活的 API。

什么时候用：

  • 重构布尔 prop 太多的组件
  • 构建可复用的组件库
  • 设计灵活的 API
  • Review 组件架构

包括：

  • 提取复合组件
  • 提升状态减少 prop 传递
  • 组合内部实现提升灵活性
  • 构建可组合的 API

5. Vercel Deploy Claimable

Section titled “5. Vercel Deploy Claimable”

vercel-deploy-claimable 让你能直接在 AI 对话中把应用部署到 Vercel。部署是”可认领”的——用户可以把所有权转移到自己的 Vercel 账号。

什么时候用：

  • “帮我部署应用”
  • “部署到生产环境”
  • “上线”
  • “部署并给我链接”

功能：

  • 从 package.json 自动检测 40+ 种框架
  • 返回预览 URL（在线站点）和认领 URL（转移所有权）
  • 自动处理静态 HTML 项目
  • 上传时排除 node_modules 和 .git

什么是 Mitosis？

Section titled “什么是 Mitosis？”

Mitosis 是一个开源编译器，可以将 JSX 组件转换为任何主流 JavaScript 框架的完整功能组件。通过使用框架无关的 JSX 语法编写 UI 组件，你可以将它们部署到任何平台，无需为每个框架重写代码。

最近火出圈的 Lovable，用聊天就能生成网站/应用，收费按credits计价，模式新鲜、速度惊人。与此同时，Mendable 团队放出了开源替代品 open-lovable：把任意网站克隆成现代 React 应用，本地跑、可改、可二开。对很多团队，这是更可控、更经济的一条路。

open-lovable 是什么？

Section titled “open-lovable 是什么？”

一句话：给它一个网址，几秒钟拉下页面结构与资产，生成可运行的 Next.js/React 代码，你在本地直接二次开发。背后两块“发动机”：

  • Firecrawl：把网页内容爬下来，转成 LLM 友好的 Markdown/JSON/截图等结构化数据。
  • E2B Sandboxes：安全隔离的云端沙箱，作为“AI 代码运行时”，让智能体能跑命令、读写文件、调包、联网。

官方 README 开箱就写明了依赖与步骤（支持 Anthropic、OpenAI、Gemini、Groq 任一/多家模型）：E2B_API_KEY + FIRECRAWL_API_KEY 必填，至少配置一个模型提供商的 API Key，npm run dev 即可在 localhost:3000 体验。许可证 MIT。

重点：open-lovable 的定位偏“克隆 + 本地可控”，而 Lovable 更像“托管式 AI App Builder”。

Lovable 做了什么？为什么值得对标

Section titled “Lovable 做了什么？为什么值得对标”

Lovable 是一个聊天式全栈应用构建平台，支持 Agent/Chat/Code/可视化编辑、Figma 导入、集成 GitHub/Supabase/Stripe/Clerk 等，面向非技术用户与小团队非常友好；采用 credit 计费，Pro 起步 $25/月。

  • Agent Mode：会“自己想、自己查、自己改”，能读代码、排错、搜文档、改动代码并调试。
  • Visual Edits：在画面上直接点选元素改文案、样式，不扣 credits。

这家公司增长也很猛，最近融资与营收的新闻不少，说明这条赛道已被验证。

开发者日常的 3 个“真”例子

Section titled “开发者日常的 3 个“真”例子”

例子1：竞品落地页→当天发版的营销站

Section titled “例子1：竞品落地页→当天发版的营销站”

场景：市场要 A/B 测试一版新文案与排版。 做法：

  • 用 open-lovable 克隆竞品页或旧落地页骨架到 React。
  • 在本地把文案、CTA、跟踪脚本接上（GA/分析埋点）。
  • 挂上你自己的样式系统与组件库，接入后端表单/CRM。 好处：几小时完成“迁移+定制+上线”，而不是重切图、重写样式。

技术点：Firecrawl 拿干净的页面内容，减少手抄；Next.js 目录结构清晰，上 CI/CD 很顺。

例子2：把“老项目”现代化

Section titled “例子2：把“老项目”现代化”

场景：公司有一套“能跑但老旧”的官网/Docs。 做法：

  • 直接克隆现站，生成现代化 React 代码。
  • 用 E2B 沙箱驱动脚本做批量替换/重构（比如类名体系、图片优化、i18n 抽离）。
  • 接入 Tailwind/Design System，统一视觉。 好处：不改功能先换壳，提升可维护性，再逐步抽象组件与数据层。

例子3：方案评审当天就能点给领导看

Section titled “例子3：方案评审当天就能点给领导看”

场景：新业务想法很多，领导想“看到东西”。 做法：

  • 克隆一个同类产品公开页面为骨架。
  • 用模型快速替换行业文案、页内交互 Demo。
  • 接入第三方登录/支付/邮件等（后续可迁到真实后端）。 好处：当天就能“有样子”，评审聚焦产品价值而非“页面还没搭”。（真要做产品化，再转 Cursor/本地仓库持续演进。）

如果追求零代码从 0 到 1，Lovable 在 Agent/Visual Edits、集成上更顺滑；开源替代更适合工程化落地与“要代码可控”的团队。

进阶玩法：和日常开发工具链拼起来

Section titled “进阶玩法：和日常开发工具链拼起来”
  • 配合 GitHub Actions：提交 PR 即触发 Lighthouse/Playwright，盯性能与可用性。
  • 接 Supabase/Stripe/Clerk 做 MVP 支付/鉴权/数据。open-lovable 产出是标准 React 代码，直接按常规方式对接即可；Lovable 则内置了对这些服务的教程与模版。
  • 模型“混搭”：场景复杂用 Claude/GPT-5，速度需求用 Groq/Kimi K2；环境变量按需切换。

该怎么选？

Section titled “该怎么选？”
  • 你要掌控代码、节省订阅费、复用现有网页 → open-lovable。
  • 你要0→1 快速出原型/内部工具，更看重托管与协作 → Lovable。
  • 也可以组合拳：先用 open-lovable 把“壳”拉下来，再在本地细改；或先在 Lovable 验证产品/数据流，再“导出/重构”到自托管。

业界讨论普遍把 Lovable 当成“脚手架 + 设计/流程可视化”，严肃项目会回到本地工程继续打磨。open-lovable 恰好补上了“工程化可控”那半步。

结语

Section titled “结语”

AI 辅助开发正在走向两端：一端是托管式、所见即所得的 Lovable；另一端是开源可控、贴近工程实践的 open-lovable。 没有绝对正确，只有你团队的成本/节奏/管控需求最重要。
