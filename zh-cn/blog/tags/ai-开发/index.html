VSDD：AI 写的代码，谁来查作业？

2026年3月3日
Vibe Sparking

没人说的问题

Section titled “没人说的问题”

AI 写了一个函数。生成了测试。测试通过了。代码审查也没问题。

发版？

我反复看到一个现象：半年后用户撞上了一个边界条件。函数处理 null 没问题，但并发访问时挂了。测试覆盖了正常路径和两三个异常路径，偏偏漏掉了输入合法但病态巨大的情况。

AI 没有写错代码。它写了”差不多对”的代码。“差不多对”比”明显错”更难抓，因为它能通过所有为了抓”明显错”而设计的检查。

行业默认的舒适区

Section titled “行业默认的舒适区”

大多数团队用 AI 写代码的流程差不多：

  1. 描述需求
  2. AI 写代码（可能顺便写测试）
  3. 人看一眼，改一改，发版

有些团队加了 TDD。有些让另一个 AI 或者人做 code review。但底层假设没变：测试通过了、reviewer 觉得还行，就够了。

这个假设有盲区。

盲区在哪：AI 太客气了

Section titled “盲区在哪：AI 太客气了”

标准的 LLM 交互天生合作导向。AI 想帮你。想和你达成一致。它审查代码的时候——哪怕审查自己写的代码——起手就带着善意。

“整体看起来不错，以下是一些小建议……”

这不叫审查。这叫夸你之后顺带提两句。

一次长对话里，AI 记得这代码是自己写的，记得你们讨论过的设计决策。它和这个代码库有了”关系”。有关系就有盲区。

什么是 VSDD

Section titled “什么是 VSDD”

VSDD（Verified Spec-Driven Development）来自开源社区，三位贡献者是 dollspace-gay、gitcnd 和 jtremback。他们把三套大多数团队当成互斥选项的方法论叠成了流水线上的三道闸门：

规格驱动开发 (SDD)： 先写契约再写代码。不光定义代码做什么，还要定义什么东西必须是可证明的。

测试驱动开发 (TDD)： 先写测试再写代码。红灯、绿灯、重构。没有失败的测试，就没有新代码。

验证驱动开发 (VDD)： 把活下来的代码扔给一个零容忍的对手反复审查，直到对手被迫编造不存在的缺陷。

三道闸门。规格定义”做什么”。测试执行”怎么做”。对抗性验证确保”没漏掉什么”。

四个角色

Section titled “四个角色”

架构师（人类开发者） — 战略视野、领域专业知识、最终拍板权。你签字确认规格，仲裁 Builder 和 Adversary 的争议，做 AI 做不了的判断。

建造者（Claude 或同类） — 写规格、生成测试、实现代码、做重构。严格 TDD 约束。不许”我先把整个东西写出来再补测试”。

追踪者（Chainlink） — 分层 issue 管理。每个规格条目、测试用例、代码实现都映射到一个可追踪的工作单元。

对手（Sarcasmotron） — 一个不同的 AI 模型，配置为零容忍。审查规格、测试、代码实现。每次审查都用全新的上下文窗口。没有关系漂移，没有累积的人情。

对手是整个方法论里最关键的创新。用不同的模型家族（比如 Builder 用 Claude，Adversary 用 Gemini），引入真正的认知多样性。对手不关心你的感受，只关心你的不变量。

六个阶段

Section titled “六个阶段”

阶段一：把一切写进规格

Section titled “阶段一：把一切写进规格”

写代码之前，Builder 输出两样东西：行为规格和验证架构。

行为规格包含前置条件、后置条件、不变量。边界条件目录列举 null、空、最大值、负数、Unicode、并发等所有边界情况。

验证架构里最关键的一项是”纯度边界图”——把确定性的纯核心和有副作用的外壳（I/O、数据库、网络）明确分开。

为什么这很重要？一个从数据库读数据、做计算、写日志全放在一个函数里的设计，形式化验证工具拿它没办法。但一个接收数据、返回结果、让调用者处理持久化的纯函数——model checker 能推理它。

这个边界必须在规格阶段就画好。因为它决定了模块划分、依赖方向和后面的整个测试策略。

Adversary 在任何测试写出来之前就审查规格。它找歧义、找缺失的边界条件、找没说出来的隐含假设、找自相矛盾的地方。

阶段二：红灯、绿灯、重构

Section titled “阶段二：红灯、绿灯、重构”

规格锁定之后，Builder 写测试。只写测试，不写实现。每个后置条件变成一个断言，每个边界条件变成一个测试。

所有测试必须先失败。如果某个测试没写实现就通过了，这个测试有问题。

然后：拿下一个失败的测试，写刚好让它通过的最少代码，跑全套测试，重复。经典 TDD，由 AI 执行。

阶段三：过火焰山

Section titled “阶段三：过火焰山”

通过测试的代码库连同规格和测试一起交给 Adversary。全新的上下文窗口。

Adversary 查什么：

  • 实现真的满足规格了吗？还是测试把一个误解编码进去了？
  • 有没有同义反复的测试——实现稍微改一下它也照样通过？
  • 隐藏耦合、竞态条件、漏掉的资源清理
  • 输入校验缺口、注入向量
  • 代码里有的行为但规格里没提到的

不许写”整体不错”。每条反馈必须是一个具体缺陷，带具体位置。

阶段四：反馈循环

Section titled “阶段四：反馈循环”

Adversary 的意见按类型回流：

  • 规格级缺陷 → 回到阶段一，改规格，重新审查
  • 测试级缺陷 → 修测试，确认它会失败，再修实现
  • 新发现的边界条件 → 加进规格，写失败测试，写实现

循环到收敛为止。

阶段五：形式化验证

Section titled “阶段五：形式化验证”

因为架构从阶段一开始就是按可验证性设计的：

  • 形式化证明（Kani、Dafny、TLA+）
  • 结构化模糊测试（AFL++、cargo-fuzz）
  • 变异测试（mutmut、Stryker）
  • 纯度边界审计

阶段六：收敛

Section titled “阶段六：收敛”

四个维度同时收敛才算完：

维度收敛信号
规格Adversary 挑的毛病只剩措辞问题
测试Adversary 找不出有意义的未测试场景
实现Adversary 被迫捏造代码中不存在的问题
验证所有形式化证明通过，fuzzer 啥也没找到

这叫”幻觉终止”——对手必须靠幻觉来凑缺陷，因为真的缺陷已经找不到了。达到这个状态的软件被称为 “Zero-Slop”。

对 AI 辅助开发意味着什么

Section titled “对 AI 辅助开发意味着什么”

核心洞察：AI 的礼貌是系统性风险。

用合作型 AI 去审查合作型 AI 的产出，就像让朋友给你批改作业。能找到一些错，但需要对抗性思维才能发现的错——找不到。

VSDD 让 AI 对抗 AI。不同模型，新鲜上下文，零容忍。这不是 code review，这是压力测试。

而人在这个过程中不是被替代了。你是架构师——审批规格、仲裁争议、做 AI 无法替代的判断。你的角色是被抬高了。

什么时候用，什么时候不用

Section titled “什么时候用，什么时候不用”

VSDD 仪式感很重。值得投入的场景：

  • 正确性不能妥协的系统（金融、医疗、基础设施）
  • 需要长期维护的代码库
  • 安全是核心关注点
  • 团队有多个可用的 AI 模型

快速原型或者一次性脚本不需要这套。但即使只加阶段三——拿另一个 AI 做一轮零容忍审查——也能抓到不少隐藏债务。

怎么开始

Section titled “怎么开始”

不用明天就搞全六个阶段。先加一个：

做一次对抗性审查。 下次完成一个功能后，开一个新对话，用不同的 AI 模型。给它零容忍指令。把规格、测试、代码全贴进去。看它找出什么。

大多数试过的开发者会对结果感到不舒服。这种不舒服就对了——说明现有流程有盲区，而你现在知道盲区在哪了。

完整可追溯性

Section titled “完整可追溯性”

VSDD 最强的属性之一：每个产物都能追溯。

问”这行代码为什么存在？“能追溯到一条具体的规格需求。问”这个模块为什么是纯函数？“答案追溯到阶段一的纯度边界图。

没有无缘无故的代码。没有缺少证据的理由。

BMAD 阶段 + Ralph 执行循环：Claude Code 的统一 AI 开发框架

BMAD 阶段 + Ralph 执行循环：Claude Code 的统一 AI 开发框架

Section titled “BMAD 阶段 + Ralph 执行循环：Claude Code 的统一 AI 开发框架”

用 Claude Code 写代码时，你可能遇到过这两种极端：

“随缘编程”模式：描述想要什么，AI 生成代码，复制粘贴。快是快了，但对话一长，AI 就开始”忘记”之前的决定，代码质量参差不齐。

“过度仪式”模式：安装一堆框架，配置无数模板，写大量文档。还没开始写代码，一下午就过去了。

BMAD 方法和 Ralph Wiggum 循环的结合提供了一条中间路线：结构化的开发阶段 + 自主的执行循环。你得到的是可追踪的决策记录和无需持续监督的执行能力。

为什么需要统一框架

Section titled “为什么需要统一框架”

单独使用 BMAD 或 Ralph 都有局限：

框架优势短板
BMAD结构化流程、角色分工、可追溯工件执行阶段需要人工监督
Ralph Wiggum自主执行、迭代优化、无需干预缺乏结构、容易偏离目标

把两者结合起来，BMAD 提供”做什么”和”为什么”，Ralph 提供”怎么做”的自主执行。

统一框架的四个阶段

Section titled “统一框架的四个阶段”

阶段一：发现（Discovery）

Section titled “阶段一：发现（Discovery）”

目标：理解问题空间，收集上下文

BMAD 的 Analyst 代理负责这个阶段。它的工作是：

  • 梳理业务需求和技术约束
  • 识别利益相关者和成功标准
  • 产出：Brief 文档

Ralph 模式：这个阶段不用 Ralph。发现需要人的参与和判断。

阶段二：规划（Planning）

Section titled “阶段二：规划（Planning）”

目标：制定实现方案，拆分任务

Ralph Wiggum vs Ralph Loop：如何选择适合你的 AI 代理模式

Ralph Wiggum vs Ralph Loop：如何选择适合你的 AI 代理模式

Section titled “Ralph Wiggum vs Ralph Loop：如何选择适合你的 AI 代理模式”

如果你在 Claude Code 社区待过一段时间，肯定听人讨论过 “Ralph Wiggum” 和 “Ralph Loop”，好像它们是完全不同的东西。这术语确实容易让人晕头转向。

其实很简单：两者都是 AI 代理的循环模式，但用途不同。一个会一直尝试直到成功。另一个跑完预定的流程就停。选错了会浪费你不少时间。

核心区别

Section titled “核心区别”

Ralph Wiggum 是一个持久的、会自我纠正的循环。它持续运行，检查自己的工作，调整方向，直到满足某个特定条件。你可以把它想象成一个死脑筋、绝不放弃的代理。

Ralph Loop 单次执行任务。你给它派活，它干完，停止。就这么简单。

这不是文字游戏。你的选择会影响怎么写提示词、需要多长时间、能不能得到想要的结果。

Ralph Wiggum：需要坚持的时候

Section titled “Ralph Wiggum：需要坚持的时候”

Ralph Wiggum 擅长那些能从迭代中受益的任务。代码重构、生成测试、合规扫描——任何代理能看自己之前的作品说”我看出刚才哪儿错了”的事情。

最近有个 Reddit 用户分享了用 Wiggum 构建 AI 合规扫描器的经历。扫描器遍历服务条款文档，经过 15 轮循环优化搜索条件，最终达到 92% 的准确率。这就是这种模式发挥作用：每轮都比上一轮更好。

另一个开发者用 Wiggum 现代化了一个老 Node.js 应用。代理用 20 轮迭代处理代码库，减少了 40% 的技术债务。不需要人指导每一步——只需要明确的完成标准和自由迭代的空间。

Wiggum 的特别之处

Section titled “Wiggum 的特别之处”

最核心的是完成承诺（completion promise）。你要明确告诉代理”完成”是什么样子：“100% 测试覆盖率”、“零语法错误”、“所有 PEP8 违规已修复”。没有这个锚点，循环会一直跑下去。

设置起来需要更多前期工作。你得仔细思考退出条件。正如一位开发者在 Reddit 上说的：“大多数人用 Ralph Wiggum 的方式不对——没有设置具体的退出条件，你的循环可能会永远跑下去。”

但配置得当的话，Wiggum 能处理那些需要持续人工监督的复杂任务。它发现过人工测试漏掉的支付网关边界情况。它用三周时间自动处理基础设施配置、成本优化和安全检查，构建了一个无服务器 SaaS。

时间上的取舍

Section titled “时间上的取舍”

Wiggum 不快。每次迭代需要 30-60 分钟。复杂任务可能需要多轮才能完成。但想想替代方案：你花几小时手动调试，或者为每个子任务写详细提示。

有时候慢速自主胜过快速干预。

Ralph Loop：速度和简洁

Section titled “Ralph Loop：速度和简洁”

当你的任务很直接——生成样板代码、更新 Dockerfile、格式化代码——Ralph Loop 5-10 分钟就能搞定。不需要复杂设置，不需要定义完成承诺，指哪打哪。

有用户用 Loop 在五分钟内自动化了整个项目的 Dockerfile 更新。另一个团队把它集成到 CI/CD 流水线生成无服务器函数脚手架，缩短了 60% 的上手时间。

Loop 在从起点到终点的路径很清晰时效果最好。你知道需要发生什么，代理只需要执行。

Loop 的局限

Section titled “Loop 的局限”

缺乏迭代精化限制了 Loop 处理复杂问题的能力。如果代理第一次尝试就出错，那个错误会一直存在。没有自我纠正的机制。

你也错失了 Wiggum 循环中出现的突发问题解决过程。Wiggum 代理可能在工作中途发现更高效的架构。Loop 会按原计划执行，即使有更好的方式存在。

如何选择

Section titled “如何选择”

实际框架是这样的：

选择 Ralph Wiggum 当：

  • 问题有明确成功标准，但路径不清楚
  • 多次尝试能改善结果
  • 你可以离开让它自己工作
  • 边界情况需要迭代发现
  • 任务需要测试驱动开发循环

选择 Ralph Loop 当：

  • 任务简单明确定义
  • 你需要快速结果
  • 手动监督不是问题

Vibe Coding 完全指南：真正有用的 AI 编程方法论

Section titled “Vibe Coding 完全指南：真正有用的 AI 编程方法论”

三个月前，我花了 15 分钟为怀孕的妻子构建了一个”餐厅版 Tinder”应用。

她正处于孕早期的食物厌恶阶段。没什么听起来有食欲的东西。选择晚餐变成了一场 45 分钟的谈判，最终两人都又饿又沮丧。

这个应用向她展示一家餐厅。如果听起来不错，她向右滑动；如果不喜欢，向左滑动。几次滑动后，我们有了一个候选清单。决策完成。

我从事专业编程工作多年。我本可以用”正确”的方式构建——适当的架构、测试、CI/CD 流水线。那会花费两周时间，而且我可能永远不会开始。

相反，我描述了我想要什么，让 AI 生成代码，根据需要调整，然后部署。

总耗时：15 分钟。

我手写的代码行数：0。

这就是 vibe coding。不是放弃你的判断力，而是战略性地选择何时应用它。

思维转变

Section titled “思维转变”

大多数人错误地使用 vibe coding。

他们看着有人用 20 分钟构建”Spotify 克隆版”的演示，感到兴奋，尝试构建”下一个大应用”，在第 15 个组件时遇到障碍，然后得出结论：这整个东西都被过度吹嘘了。

这是本末倒置。

Vibe coding 的核心是匹配投入与风险。

有些代码值得精心设计架构。生产系统。任何有真实用户使用的代码。需要维护多年的代码。

其他代码则不需要。内部工具。一次性脚本。在投入实际工程时间之前测试想法的原型。只有你个人会使用的实用工具。

我们在理智上一直知道这一点。问题在于，当你要手工编写所有内容时，“快速而粗糙”仍然需要数小时。所以我们要么过度设计一次性代码，要么根本不构建它。

Vibe coding 改变了这种经济状况。那些不值得精心设计的代码现在只需要几分钟而不是几小时。你无需投入就能获得工具。

  • 之前： “这个工具值得我花费 8 小时吗？“通常不值得。工具不会被构建。
  • 现在： “这个工具值得我花费 45 分钟吗？“通常值得。

那些明白这一点的人将发布更多、更快地测试想法，并将深度工作留给真正需要它的问题。

Vibe Coding 最适用的场景

Section titled “Vibe Coding 最适用的场景”

Vibe coding 在以下情况中最有价值：

  • 用户是你自己（或你的团队、你的家人）
  • 需求具体（你的确切工作流程，而不是通用最佳实践）
  • “足够好”就足够了（它对你的情况有效，即使有粗糙的边缘）
  • 没有其他人会构建这个（太小众而不成产品，太有价值而不能忽略）

最好的 vibe coded 工具不是你会放到 App Store 上的应用。它们是针对你亲身经历过的问题的解决方案——问题太具体了，其他人都无法解决。

发现值得解决的问题

Section titled “发现值得解决的问题”

在打开任何工具之前，花时间识别符合标准的问题。

在工作中

Section titled “在工作中”

你在哪些重复性任务上浪费时间？手动格式化相同的报告。在系统之间复制粘贴。反复创建类似的文档。反复回答相同的问题。

你在哪些方面缺乏可见性？指标存在于多个地方。你必须深入挖掘的状态更新。应该触手可及但实际不是的信息。

现有工具在哪里让你失望？它们完成了你需要的 80%，但错过了关键的 20%。它们需要太多点击来完成简单任务。它们不是为你的特定工作流程而构建的。

在个人生活中

Section titled “在个人生活中”

你一直在尝试手动解决哪些问题？跨不同应用追踪习惯。做出有太多变量的决策。组织不适合标准工具的信息。

你希望有一个应用存在哪里？针对你的特定爱好或兴趣。针对独特生活情况的工具。你说过”应该有一个应用来做这个”的东西。

花 30 分钟头脑风暴。写下每个问题，无论多小。最好的项目往往来自会累积的”小”烦恼。

三个领域

Section titled “三个领域”

了解你处于哪个领域决定了你应该如何处理构建。

领域 1：工作工具

Section titled “领域 1：工作工具”

内部工具具有完美的 vibe coding 特征：已知用户（你的同事）、清晰需求（你的工作流程）、宽容的标准（它只需要工作）。

示例：演示文稿生成器

在 Tenex，我们为每次客户会议创建自定义幻灯片。相同的结构，不同的内容。数小时的工作感觉就像数据输入。

我构建的工具：输入客户详细信息，选择相关案例研究，它生成一个品牌化的演示文稿，随时可用于会议。

提示词：

工程现实：生成的代码不是我构建这个的方式。状态管理过于简单。没有测试。组件组织……还行。

为什么没关系：三个人使用这个工具。它每周为每个人节省 2 小时。如果它坏了，我们会修复它。如果需要功能，我们会添加它。投资回报率在第一天就是正的。

何时对工作工具使用 vibe-coding：

  • 你已经多次手动完成过任务
  • 步骤足够一致可以系统化
  • 70% 的解决方案仍然能节省大量时间
  • 替代方案是”永远手动继续做下去”

何时正确构建：

  • 工具处理敏感数据
  • 许多人将依赖它
  • 错误会产生真实后果
  • 它需要持续的功能开发

领域 2：产品工作

Section titled “领域 2：产品工作”

这是 vibe coding 为产品团队赢得回报的地方。以前需要冲刺才能测试的想法现在可以在一天内验证。

示例：UX 原型制作

在承诺工程资源之前，你可以 vibe-code 一个工作原型来看看它是否真的有意义。不是 Figma 模型。不是线框图。一个能工作的东西。

提示词：

在 2 小时内，我们有了一些可以放在用户面前的功能。通过观看 5 个人点击它，我们学到的比在 5 次会议上辩论设计学到的更多。

何时对产品使用 vibe-coding：

  • 在承诺工程资源之前需要测试想法
  • 希望利益相关者在方向上达成一致
  • 正在探索 UX 选项
  • 工件将在验证后被丢弃

何时正确构建：

  • “原型”可能会发布
  • 正在构建需要发展的 MVP
  • 技术约束对可行性评估很重要
  • 性能或规模是你正在验证的一部分

Section titled “领域 3：个人工具”

何时干预

Section titled “何时干预”

Vibe coding 并不意味着永远不看代码。它意味着在何时看代码方面具有战略性。

在以下情况干预：

  • AI 重复相同的错误
  • 性能明显很差
  • 安全是一个问题（身份验证、数据处理）
  • 你以后需要维护这段代码
  • 结构阻止你需要的下一个功能

不要在以下情况干预：

  • 代码丑陋但有效
  • 你可以重构它但不需要
  • “正确”的方式需要显著更长的时间
  • 这是一次性代码

判断标准：修复这段代码值得花费时间吗？

常见错误

Section titled “常见错误”

  1. 开始太大： 你想要一个”完整的 CRM”，最终得到一团糟。从一个功能开始。让它工作。扩展。

  2. 不在提示词之间测试： 你堆叠五个提示词，然后发现第二个提示词出了问题。现在你要调试四个提示词的更改。在每个提示词后测试。

  3. 过度设计一次性代码： 你花费 3 小时重构一个 2 个人会使用的工具。发布丑陋版本。根据实际使用改进。

  4. 让原型成为产品： 你的演示很顺利。利益相关者说”就发布这个吧。“拒绝。原型有隐藏的脆弱性。为生产正确重建。

  5. 不使用自己的工具： 构建 → 使用 → 注意摩擦 → 改进。迭代发生在现实生活中，而不是在提示词中。

  6. 忽略移动端： 大多数个人工具在手机上使用。如果你不指定移动优先，你会得到一个在小屏幕上痛苦的桌面应用。

当前局限性

Section titled “当前局限性”

  • 复杂状态管理： 需要复杂状态的应用——实时协作、复杂撤销/重做、离线优先同步——很快变得困难。

  • 规模化性能： Vibe-coded 应用对于小用户群工作正常。在规模上，你会遇到 AI 没有预料到的问题。

  • 安全性： AI 生成的代码可能存在漏洞。对于内部工具还可以；审计任何处理敏感数据的东西。

  • 集成复杂性： 简单的 API 集成有效。具有身份验证、webhook 和边缘情况的复杂集成需要工程关注。

  • “几乎”陷阱： 达到 90% 很快。最后 10%——抛光、边缘情况、移动响应——可能与前 90% 一样长。

总结

Section titled “总结”

Vibe coding 的核心是匹配投入与风险。有些代码值得精心设计架构。有些不值得。AI 让你在几分钟内构建后者。

工作工具： 自动化繁琐的 80%。代码质量标准很低，因为用户群很小，替代方案是”永远不构建它。”

产品工作： 快速测试想法。原型是一次性的——不要让它们成为生产代码。

个人工具： 完全按照你的需要构建。你是唯一的用户，所以抛光不重要。

过程：定义问题。草图流程。首先提示结构。一次添加一个功能。实际使用它。基于实际摩擦迭代。

心态：为自己构建。从小开始。发布丑陋版本。稍后抛光——如果有的话。

vibe coding 通用应用的人会感到沮丧。vibe coding 解决其特定问题的人会获得他们将使用数年的工具。

构建只有你会构建的东西。这就是真正有用的 vibe coding。

Ghostty 创始人的 Vibe Coding：与 Claude 的对话式编程之旅

Section titled “Ghostty 创始人的 Vibe Coding：与 Claude 的对话式编程之旅”

当 Ghostty（现代 GPU 加速终端模拟器）的创始人 Mitchell Hashimoto 想要改进 macOS 更新功能时，他没有直接写代码 —— 他进行了一场对话。在他称之为 “vibe coding”（氛围编程）的精彩会话中，Mitchell 与 Claude 配对，通过对话、探索和共同发现来迭代改进代码库。

这并不是关于 AI 取代开发者。而是关于 AI 作为协作伙伴 —— 转录记录展示了美妙的一面：AI 发现人类容易忽略的问题，人类提供方向和上下文，两者共同构建出任何一方都无法单独完成的东西。

什么是 “Vibe Coding”？

Section titled “什么是 “Vibe Coding”？”

Vibe coding 是一种对话式的软件开发方法，开发者与 AI 通过自然语言对话一起工作。与其将 AI 视为命令执行工具，不如将其视为结对程序员 —— 一起讨论、探索和迭代。

在 Mitchell 的会话中（从 AmpCode 上的真实编码对话捕获），流程是这样的：

  1. 咨询预言家 — 让 Claude 分析代码以寻找改进点
  2. 讨论 — 一起讨论发现，共同做出决策
  3. 实施 — 在 Claude 的协助下编写代码
  4. 迭代 — 从失败中学习，调整方法，再次尝试
  5. 测试 — 验证更改有效，添加测试以建立信心

“预言家”——处于分析模式的 Claude——识别出几类改进：

关键安全问题

Section titled “关键安全问题”
  1. 循环引用 — UpdateDriver.showUpdaterError 中的重试闭包强引用了 self，造成内存泄漏
  2. 线程问题 — UI 面向类型应标记为 @MainActor
  3. 进度值错误 — 进度值需要限制在 [0, 1] 范围内以防止视觉故障
  4. 不可取消的定时器 — “无更新”自动关闭任务可能触发过期更新
  5. 吞没的错误 — try? 隐藏了 startUpdater() 中的潜在失败

需要单元测试的区域

Section titled “需要单元测试的区域”
  1. ReleaseNotes 解析 — 语义版本检测、git 哈希处理、边界情况
  2. UpdateState Equatable — 下载/错误状态的相等性测试
  3. ViewModel 格式化 — 每个状态的文本/图标/颜色映射
  4. 进度数学 — 除零、nil 值、超调场景
  5. 自动关闭计时 — 验证转换和任务取消

UX 改进

Section titled “UX 改进”

  • 限制/缩写过长的错误消息以防止 UI 溢出
