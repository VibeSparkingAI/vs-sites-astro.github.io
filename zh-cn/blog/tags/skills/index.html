PM Skills Marketplace: 65 个技能让 Claude Code 变成产品经理的操作系统

Section titled “PM Skills Marketplace: 65 个技能让 Claude Code 变成产品经理的操作系统”

让 Claude 写个 PRD 试试。两页文字，措辞专业，格式漂亮。但读过 Inspired 的人一眼就能看出问题：功能列表后面没有机会解决方案树，优先级排序没有假设映射支撑，用户故事没用 JTBD 框架。

你不缺知识。Teresa Torres 的持续发现、Marty Cagan 的赋能产品团队、Alberto Savoia 的预型测试、Dan Olsen 的精益产品——这些书你都读过。问题在于：这些框架躺在书架上，而 Claude 正以光速生成没有骨架的文字。

Paweł Huryn 做了一件事：把这些框架变成代码。他是 The Product Compass 的作者，PM Skills Marketplace 就是他的作品。

这是什么

Section titled “这是什么”

65 个技能，36 个链式工作流，8 个插件。全部开源，MIT 协议。

每个技能把一个经过验证的 PM 框架编码进 Claude 的领域知识——不是提示词模板，是让 Claude 在对话中自动调用的结构化认知。聊产品发现时，Claude 会加载 Teresa Torres 的机会解决方案树方法论。排优先级时，它用影响力×风险矩阵并给出实验建议。测试假设时，它用 Alberto Savoia 在 The Right It 里提出的预型测试方法。

GitHub: phuryn/pm-skills 作者: Paweł Huryn (The Product Compass) 协议: MIT 兼容工具: Claude Code, Cowork, Gemini CLI, Cursor, Codex CLI, Kiro

三层架构

Section titled “三层架构”

设计很干净。技能是积木，命令把积木串成工作流，插件按领域分组。

技能 (Skills)

Section titled “技能 (Skills)”

每个技能给 Claude 一项特定 PM 任务的领域知识。有些是独立参考——prioritization-frameworks 涵盖 9 种排序框架（Opportunity Score、ICE、RICE、MoSCoW、Kano 等），Claude 在你讨论优先级时自动调用。有些是工作流组件，多个命令共享。

技能在对话中自动加载，不用手动调用。想强制加载某个技能，用 /plugin-name:skill-name。

命令 (Commands)

Section titled “命令 (Commands)”

命令是亮点。拿 /discover 来说，它串联四个技能：

  1. brainstorm-ideas — 从 PM、设计师、工程师三个视角做头脑风暴
  2. identify-assumptions — 在价值、可用性、可行性、可实现性四个维度识别风险假设
  3. prioritize-assumptions — 用影响力×风险矩阵排序，附带实验建议
  4. brainstorm-experiments — 设计精益实验来验证高优先级假设

一个命令，完整的发现周期。做完之后，它会建议下一步该用哪个命令。

插件 (Plugins)

Section titled “插件 (Plugins)”

8 个插件覆盖 PM 生命周期：

插件技能数命令数覆盖范围

pm-execution1510PRD、OKR、路线图、迭代、回顾、用户故事

pm-toolkit45简历审阅、NDA、隐私政策、校对

安装

Section titled “安装”

在 Claude Code (Cowork) 里：

  1. 点左下角 Customize
  2. 进 Browse plugins → Personal → +
  3. 选 Add marketplace from GitHub
  4. 输入：phuryn/pm-skills

8 个插件自动安装完毕。

实际用法

Section titled “实际用法”

和写提示词的区别在哪？PM Skills 提供的是链式工作流，不是单次对话。

新想法？从 /discover 开始

Section titled “新想法？从 /discover 开始”

走一遍完整的发现周期：三视角创意发散，四维度假设识别，影响力×风险矩阵排序，精益实验设计。

要战略清晰度？用 /strategy

Section titled “要战略清晰度？用 /strategy”

生成 9 个板块的产品战略画布：愿景、目标客户、价值主张、竞争优势、增长策略、收入模型、指标、风险、战略下注。

写 PRD？试 /write-prd

Section titled “写 PRD？试 /write-prd”

准备上线？跑 /plan-launch

Section titled “准备上线？跑 /plan-launch”

定义成功？设 /north-star

Section titled “定义成功？设 /north-star”

定义北极星指标和支撑性输入指标，带业务博弈分类。

知识来源

Section titled “知识来源”

  • Teresa Torres — Continuous Discovery Habits（机会解决方案树、假设映射）
  • Marty Cagan — INSPIRED 和 TRANSFORMED（赋能产品团队、产品战略）

  • Dan Olsen — The Lean Product Playbook（产品市场匹配方法论）
  • Roger L. Martin — Playing to Win（战略选择）

  • Strategyzer — Business Model Generation 和 Value Proposition Design

这不是列名字充门面。每个框架都嵌入了技能的领域知识，直接影响 Claude 的输出结构。

和写提示词有什么区别

Section titled “和写提示词有什么区别”

三点。

结构而非模板。 技能给 Claude 领域知识，不是填空式模板。输出会根据你的具体情况调整，同时保持框架的严谨性。

链式工作流。 命令把技能串联成序列，和 PM 实际的工作流程一致。发现阶段的输出直接喂给战略阶段，战略再喂给执行。你不用手动编排。

自动加载。 技能在对话中按需激活。聊定价时，Claude 会自动调用定价策略、支付意愿分析、竞争定价模型——不用你点名。

适合谁用

Section titled “适合谁用”

已经懂框架、想持续应用的 PM。兼任产品工作的创始人。在没有专职 PM 的情况下做产品决策的技术负责人。读过这些书、想要严谨度但嫌流程重的人。

没读过 Torres 和 Cagan？先去读书。PM Skills 编码的是专业知识，不能替代理解。

粗糙的地方

Section titled “粗糙的地方”

命令（斜杠工作流）只在 Claude Code 和 Cowork 中可用，其他工具只能用技能，不能用链式工作流。CSV 格式的框架定义能用但不够优雅。和所有 AI 工具一样，输出质量取决于输入质量——喂垃圾上下文，出来的是结构化的垃圾。

Vercel Agent Skills：AI 编程助手的官方技能集

AI 编程助手越来越强了，但怎么让它们懂更多的”行规”？Vercel Labs 最近放出了一个技能集合，把多年的工程经验打包成了 AI 能读懂的格式。

vercel-labs/agent-skills 这个仓库包含了五个生产级别的技能：React 优化、Web 设计规范、React Native 模式、React 组合模式，还有一键部署。

什么是 Agent Skills？

Section titled “什么是 Agent Skills？”

Agent Skills 就是给 AI 编程助手的”插件包”。跟普通文档不一样，这些技能是专门为 LLM 设计的：

  • AI 能自己读懂规则
  • 写代码或审查代码时自动应用
  • 识别上下文并激活对应技能
  • 需要的时候还能跑脚本

Vercel 的技能用的是 Agent Skills 格式，所以能在各种 AI 编程平台上用。

有哪些技能？

Section titled “有哪些技能？”

1. React 最佳实践

Section titled “1. React 最佳实践”

react-best-practices 收录了 40 多条规则，分 8 个大类，是 Vercel 工程团队十年来搞 React 和 Next.js 积累的经验。

什么时候用：

  • 写新的 React 组件或 Next.js 页面
  • 实现数据获取（客户端或服务端）
  • Code Review 找性能问题
  • 优化包大小或加载时间

包括：

  • 消除瀑布请求（关键） - 并行数据获取、预取
  • 包体积优化（关键） - Tree shaking、代码分割、动态导入
  • 服务端性能（高） - 服务端组件、流式渲染、缓存
  • 客户端数据获取（中高） - SWR/React Query 模式
  • 重渲染优化（中） - Memoization、useMemo、useCallback
  • 渲染性能（中） - 虚拟列表、CSS containment
  • JavaScript 微优化（低中） - 事件处理、属性访问

2. Web 设计指南

Section titled “2. Web 设计指南”

web-design-guidelines 能审查 UI 代码，100 多条规则覆盖了可访问性、性能和 UX 最佳实践。

什么时候用：

  • “帮我审查 UI”
  • “检查可访问性”
  • “审计设计”
  • “Review UX”
  • “按最佳实践检查我的网站”

包括：

  • 可访问性 - ARIA 标签、语义化 HTML、键盘导航
  • 焦点状态 - 可见的焦点指示器、focus-visible 模式
  • 表单 - 自动完成、验证、错误处理
  • 动画 - prefers-reduced-motion、合成器友好的 transform
  • 排版 - 弯引号、省略号、等宽数字
  • 图片 - 尺寸、懒加载、alt 文本
  • 性能 - 虚拟化、避免布局抖动、preconnect
  • 导航与状态 - URL 反映状态、深度链接
  • 深色模式与主题 - color-scheme、theme-color meta
  • 触摸与交互 - touch-action、点击高亮
  • 国际化 - Intl.DateTimeFormat、Intl.NumberFormat

3. React Native 指南

Section titled “3. React Native 指南”

react-native-guidelines 包含 16 条规则，覆盖移动端性能、架构和平台特定模式。

什么时候用：

  • 开发 React Native 或 Expo 应用
  • 优化移动端性能
  • 实现动画或手势
  • 使用原生模块或平台 API

包括：

  • 性能（关键） - FlashList、memoization、密集计算卸载
  • 布局（高） - Flex 模式、安全区域、键盘处理
  • 动画（高） - Reanimated 库、手势处理
  • 图片（中） - expo-image、缓存、懒加载
  • 状态管理（中） - Zustand 模式、React Compiler
  • 架构（中） - Monorepo 结构、import 优化
