MemU: AI智能体记忆框架

Section titled “MemU: AI智能体记忆框架”

在快速发展的AI智能体领域，一个关键挑战始终存在：如何为AI智能体赋予持久、可解释且真正理解上下文的记忆能力？隆重介绍MemU——由**NevaMind-AI** 开发的革命性智能体记忆框架，在Locomo数据集上实现了92.09%的平均准确率，同时提供独特的双重检索方法，结合了RAG和基于LLM理解的各自优势。

什么是MemU？

Section titled “什么是MemU？”

MemU是专为LLM和AI智能体后端设计的智能体记忆框架。与仅依赖嵌入搜索的传统RAG系统不同，MemU引入了新颖的三层分层架构，将多模态输入转化为结构化、可解释的记忆，同时支持基于嵌入（RAG）和非嵌入（LLM）的检索方法。

该框架围绕一个简单但强大的理念构建：记忆应该是可追溯、可解释且自我进化的。

核心架构：三层记忆系统

Section titled “核心架构：三层记忆系统”

受计算机系统中分层存储架构的启发，MemU将记忆组织为三个不同的层：

层级描述示例
资源层原始多模态数据仓库JSON对话、文本文档、图片、视频、音频文件
记忆项层离散提取的记忆单元个人偏好、技能、观点、习惯、事实
记忆类别层带摘要的聚合文本记忆preferences.md、work_life.md、relationships.md、skills.md

三层架构的关键优势

Section titled “三层架构的关键优势”
  • 完全可追溯性：从原始数据 → 记忆项 → 类别并可反向追踪
  • 渐进式摘要：每一层提供越来越抽象的视图
  • 灵活组织：类别基于内容模式演化
  • 可解释性：记忆以人类可读的markdown文件存储

双重检索方法：两全其美

Section titled “双重检索方法：两全其美”

MemU真正的独特之处在于其双重检索方法：

基于RAG的检索（method="rag"）

Section titled “基于RAG的检索（method="rag"）”
  • 快速的嵌入向量搜索
  • 适合大规模数据
  • 快速响应时间
  • 最适用于：查找查询、事实检索

基于LLM的检索（method="llm"）

Section titled “基于LLM的检索（method="llm"）”
  • 通过直接文件读取实现深度语义理解
  • LLM直接理解自然语言记忆文件
  • 渐进式搜索：类别 → 记忆项 → 资源
  • 最适用于：复杂推理、上下文查询

两种方法都支持：

  • 上下文感知重写：利用对话历史解析代词
  • 渐进式搜索：从摘要导航到细节
  • 下一步建议：迭代式多轮检索
  • 范围过滤：按user_id、agent_id或自定义字段查询

为什么这份白皮书如此重要？

Section titled “为什么这份白皮书如此重要？”

LLM 是预测引擎，而非推理引擎。你提供的每一个字，都在引导模型预测下一个 token。理解这一根本原理，是掌握提示词工程的关键。

技巧会过时，但原理永不过时。

1. 提供示例，别只说要求

Section titled “1. 提供示例，别只说要求”

Few-shot 比 zero-shot 效果好得多。 给 3-5 个例子，模型就能精准理解你的期望，胜过任何冗长的解释。

为什么有效： 示例让模型捕捉你想要的结构、语气和格式。与其解释什么是”专业”，不如直接展示一个专业的回复。

2. 保持简洁

Section titled “2. 保持简洁”

使用动词，避免晦涩表达。 一句清晰的话胜过三段模糊的描述。

LLM 不需要客套话或上下文铺垫，它们需要直接的指令。

推荐动词：

  • 分析（Analyze）
  • 创建（Create）
  • 生成（Generate）
  • 识别（Identify）
  • 总结（Summarize）
  • 比较（Compare）
  • 提取（Extract）

修改前： “我希望你能帮我写一些可能对理解机器学习有用的内容，如果不太麻烦的话。”

修改后： “用 3 句话向非技术人员总结机器学习。“

3. 明确输出格式

Section titled “3. 明确输出格式”

模糊指令 = 随机结果。 要 JSON 就说 JSON，要 500 字就说 500 字。

格式要求越具体，输出越可预测。

弱： “写一篇关于游戏主机的博客。”

强： “写一篇 3 段的游戏主机发展史博客。使用对话风格。每段 4-6 句。每段包含一个具体数据。”

进阶技巧： 请求 JSON 格式可获得结构化、可处理的响应，天然限制幻觉：

4. 用指令，不用约束

Section titled “4. 用指令，不用约束”

告诉模型该做什么，而不是不该做什么。

否定表达对语言模型天生具有歧义性。“不要啰嗦”不如”限制在 100 字内”有效。

避免这样说改成这样说
”不要列出游戏名称""只讨论主机、公司、年份和销量"
"不要啰嗦""每段限制 3 句话"
"不要编造""不确定时回复’我无法核实此信息’”

心理学解释： 正向指令定义了明确的行动空间，而否定约束定义的是无限的排除区域，模型更难理解边界。

5. 控制 Token 长度

Section titled “5. 控制 Token 长度”

成本与 token 线性相关。 生成 1000 字的成本是 100 字的 10 倍。

当效率重要时，始终设置明确的长度约束。

常用技巧：

  • “限制输出在 80 字符内”
  • “用一条推文的长度解释”
  • “精确回复 3 个要点”
  • “最多 50 tokens”

额外收获： Token 限制也会强制简洁，往往提升输出质量。

6. 用变量做模板

Section titled “6. 用变量做模板”

把可变部分抽成变量。 一套精心设计的模板可以复用上百次。

优势：

  • 多次查询保持一致性
  • 便于 A/B 测试
  • 支持程序化生成提示词
  • 版本控制友好

7. 测试不同格式

Section titled “7. 测试不同格式”

问句、陈述句、指令产生的结果各不相同，即使内容相同。

多种方式测试：

  • 问句： “锻炼有什么好处？”
  • 陈述： “锻炼有多种好处。请解释。”
  • 指令： “列出 5 个定期锻炼的好处，附带科学依据。”
  • 角色扮演： “作为健身教练，解释客户为什么应该锻炼。”

9. 适应模型更新

Section titled “9. 适应模型更新”

GPT-4 到 GPT-4.5，Gemini 1.0 到 2.0，提示词都要重新调优。

  • 持续监控输出质量指标
  • 维护提示词变更日志

彩蛋：思维链（CoT）最佳实践

Section titled “彩蛋：思维链（CoT）最佳实践”

对于复杂推理任务，将温度设为 0，适用于需要严格逻辑推理的单一答案问题。

自洽性提示（Self-Consistency Prompting）： 在高温度下多次发送相同的思维链提示，然后对最常见的答案”投票”。这使答案更稳健，减少幻觉。

退后一步提示（Step-Back Prompting）： 不直接解决问题，先让模型思考通用原理或模式，再用该响应引导具体任务。

在快速发展的 AI 开发领域，上下文工程（Context Engineering） 已成为构建有效智能体系统最关键的学科之一。与专注于编写有效指令的提示工程不同，上下文工程解决的是如何全面管理进入模型有限注意力窗口的所有信息。

Agent Skills for Context Engineering 是由 Murat Can Koylan 创建的一个全面、开源的技能集合，旨在教授策划上下文的艺术和科学，以最大化任何平台上的智能体效能。

什么是上下文工程？

Section titled “什么是上下文工程？”

上下文工程是管理语言模型上下文窗口的学科。核心挑战在于上下文窗口不是受原始 token 容量限制，而是受注意力机制限制。

核心问题

Section titled “核心问题”

随着上下文长度增加，模型会表现出可预测的退化模式：

  • 中间丢失现象（Lost-in-the-Middle） - 长上下文中间的信息获得的注意力更少
  • U形注意力曲线 - 模型更关注上下文的开头和结尾
  • 注意力稀缺 - 有限的”注意力预算”必须明智分配

解决方案

Section titled “解决方案”

有效的上下文工程意味着找到最小可能的高信号 token 集合，以最大化预期结果的可能性。这需要理解什么是上下文、它如何退化，以及如何优化它。

关键概念深入解析

1. 上下文基础

Section titled “1. 上下文基础”

理解智能体系统中上下文的结构至关重要：

上下文组成部分：

  • 系统提示词
  • 工具定义
  • 检索到的文档
  • 消息历史
  • 工具输出

每个组成部分都在竞争有限的注意力。有效的智能体会优先处理高信号信息，压缩或消除低信号内容。

2. 上下文退化模式

Section titled “2. 上下文退化模式”

识别失效模式有助于预防它们：

中间丢失（Lost-in-Middle）： 放置在长上下文中间的关键信息获得的模型注意力较少。

上下文污染（Context Poisoning）： 错误或冲突的信息破坏模型的理解。

上下文干扰（Context Distraction）： 无关信息分散了对重要内容的注意力。

上下文冲突（Context Clash）： 矛盾的指令或数据造成混乱。

3. 多智能体模式

Section titled “3. 多智能体模式”

多智能体系统的三种主要架构：

编排器模式（Orchestrator Pattern）： 一个中央智能体协调专门的子智能体，管理任务委派和结果汇总。

点对点模式（Peer-to-Peer Pattern）： 智能体直接通信，无需中央协调，适合协作任务。

分层模式（Hierarchical Pattern）： 分层的智能体结构，高层智能体管理低层智能体组。

Q: 这些技能适用于任何 LLM 吗？ A: 是的，这些原则与 LLM 无关。示例使用 Python 伪代码，可以适应任何环境。

Q: 我怎么知道使用哪个技能？ A: 从基础开始，然后为设计决策使用架构技能，为优化使用运维技能。

Q: 我可以贡献新技能吗？ A: 当然可以！遵循模板结构并提交 pull request。欢迎社区贡献。

Q: 使用这些技能有相关费用吗？ A: 没有，仓库采用 MIT 许可证。但是，你使用这些技能的 LLM 可能有相关费用。

Q: 技能如何处理上下文限制？ A: 渐进式披露意味着只加载相关内容。每个技能设计为保持在 500 行以内。

故障排除

Section titled “故障排除”

技能无法加载：

  • 确保技能文件夹结构正确
  • 检查技能目录中是否存在 SKILL.md
  • 验证文件权限

智能体性能不佳：

  • 查看 context-degradation 技能以获取诊断模式
  • 检查是否超出上下文窗口
  • 评估信息信噪比

多智能体协调问题：

  • 查看 multi-agent-patterns 技能
  • 检查消息传递协议
  • 验证共享记忆访问模式
