SkillsMP：AI 代理技能开放市场

Section titled “SkillsMP：AI 代理技能开放市场”

AI 编程助手已经成为开发者不可或缺的工具，但它们的能力往往局限于内置功能。SkillsMP 改变了这一现状，提供了一个开放的市场平台，开发者可以在多个 AI 平台上发现、安装和共享技能。

什么是 SkillsMP？

Section titled “什么是 SkillsMP？”

SkillsMP 是一个开源市场，托管超过 31,000 个代理技能，可与最流行的 AI 编程助手无缝协作：

  • Claude Code - Anthropic 官方 CLI 工具
  • Codex CLI - OpenAI 命令行界面
  • ChatGPT - 全球最受欢迎的 AI 助手

所有技能都使用统一的 SKILL.md 格式，确保跨平台兼容性，便于在不同 AI 工具之间共享能力。

SKILL.md 标准

Section titled “SKILL.md 标准”

SKILL.md 格式是代理技能生态系统的基础。它通过以下方式定义模块化能力：

  • 指令 - 清晰地指导 AI 技能的作用和使用方法
  • 脚本（可选）- 扩展功能的支持代码
  • 模板（可选）- 常用输出的预构建结构

核心特征

Section titled “核心特征”

技能是模型调用的，而不是用户调用的。这意味着 AI 助手会根据请求的上下文决定何时激活技能，而不需要显式命令。

Codex 也有 Claude Code「斜杠命令」了：把常用提示存成 /命令，一键召唤

2025年9月16日
Vibe Sparking

现在你可以把常用 Prompt 写成 Markdown，放进 ~/.codex/prompts/，在 Codex 里直接用 /命令名 调出。工作流立刻提速。社区已经确认可用，适用于 CLI，安装也很简单（npm i -g @openai/codex 或 brew install codex）。

小提醒：你提到的“暂不支持按目录来区分前缀”，这点在我的实测里也成立。当前是“文件名 = 命令名”，没有目录作用域的前缀隔离。

为什么值得现在用？

Section titled “为什么值得现在用？”

复用 prompt 才是真省时。把“代码审查”“单测生成”“发布清单”这些高频提示固化成 /review、/tests、/ship，每次一敲就来，不用再翻历史记录。Codex 团队最近也在持续升级 CLI/IDE 体验，势头很猛。

更妙的是，这套玩法和 Claude Code 的“自定义斜杠命令”思路一致——用 Markdown 存提示，斜杠唤起，简单高效。迁移成本几乎为零。

三步上手（真的就三步）

Section titled “三步上手（真的就三步）”

① 建目录 确保存在路径：~/.codex/prompts/。没有就创建。

② 写一个 Markdown 例如新建 review.md，内容随你：

“你是代码审查助理。给出结构化建议：安全、性能、复杂度、可读性、测试缺口。分点输出。”

③ 重新打开一次 Codex 会话 进 Codex 输入 /revi… 会自动补全成 /review，回车即用。社区反馈中也有相同使用方式与目录说明。

实操经验：

  • 命令名=文件名（去掉 .md）。
  • 避免与内置命令重名（例如 /init 这类词就别占）。
  • 新文件通常开启新会话后更稳妥地被识别。

现阶段的小坑与规避

Section titled “现阶段的小坑与规避”

  • 不支持目录前缀区分 也就是说 backend/review.md 和 frontend/review.md 暂时做不到分别变成不同 /review。建议在文件名里加后缀：review-fe.md、review-be.md。

  • 参数/占位符能力有限 有用户反馈在自定义命令里使用 @ 一类参数占位并不稳定，遇到这种情况，改为在触发命令后二次补充具体上下文更可靠。

  • 命令命名要“短而记” 这是斜杠菜单的灵魂，和 VS Code/Cursor 的快捷动作类似，别起绕口的长名字。参考 Codex 官方与社区文章的最佳实践。

你可以立刻抄走的 5 个高能命令

Section titled “你可以立刻抄走的 5 个高能命令”

  • /review：代码审查清单（安全/性能/复杂度/可读性/测试）。

  • /tests：为当前文件生成“表驱动”或“参数化”单测，并补齐边界。

  • /ship：预发/发布 Checklist（迁移、回滚预案、监控项、灰度范围）。

  • /perf-hunt：定位热点函数与慢 SQL 的排查步骤。

  • /changelog：把最近提交摘要成可发布的 CHANGELOG。

把上面每条写成一个 .md 放进 ~/.codex/prompts/，重启会话后就能 / 出来了。

和 Claude Code 的搭配姿势

Section titled “和 Claude Code 的搭配姿势”

如果你团队里 Codex + Claude Code 混用，统一把“可复用流程”沉淀成 Markdown Prompt，两边各自的斜杠命令都能吃，减少心智切换。Claude 社区也在推行这种以 markdown 为单位的复用方法。

结语

Section titled “结语”

把常用提示标准化、模块化，是 AI 编程从“灵感”走向“流程”的关键一步。 从今天开始，把你最常用的 3 个 Prompt 丢进 ~/.codex/prompts/，让双手记住 / 的快乐。

想默认高推理？把它写进 ~/.codex/config.toml，或者每次用 --config 覆盖一把。

什么时候用 High reasoning，什么时候用 Minimal/Medium？

Section titled “什么时候用 High reasoning，什么时候用 Minimal/Medium？”

适合 High 的场景

  • 复杂重构 / 设计权衡：例如「把 MVC 项目拆成分层 + 依赖倒置，补齐测试」。需要系统性推理和多步计划。
  • 跨语言迁移：如从 Python → Go 且保留并发/错误语义。需要细致的 API/并发模型映射。
  • 疑难 bug 诊断：复杂堆栈、竞态、边界条件，靠更长的思考链更稳。

适合 Minimal/Medium 的场景

  • 脚手架、批量修 lint、重命名，追求速度且不需要长链路推理。minimal 会大幅减少推理 token、显著提速。

我在本地的推荐工作流（可直接抄）

Section titled “我在本地的推荐工作流（可直接抄）”
  1. 开场就高推理（复杂任务）

然后用 /approvals 收紧自动修改权限，重要变更逐条确认。

  1. 中途降档提速 调试通过后，输入 /model 把推理改成 medium 或 minimal，进入批量小改模式。

  2. 拆解多步任务 把一个大目标拆成多轮：设计 → 接口 → 实现 → 测试。官方也建议**“多回合、一步一回合”**，实际效果更稳。

小贴士：推理力度越高，推理 token越多，延迟可能更高；卡顿时就 /model → minimal/low。

常见问题 & 小坑

Section titled “常见问题 & 小坑”

  • 我用 ChatGPT 登录时，感觉推理被锁在 Medium？ 早期社区确有反馈；但 0.23.0 已在 CLI 内支持运行时切换 /model 与 reasoning effort，实际可在 TUI 里动手改。

  • 命令行切换和配置冲突？ 以 命令行 --config 为准（单次覆盖），长期策略写进 ~/.codex/config.toml。

  • Minimal 到底有什么用？ 官方说明：几乎不产生日志式推理 token，面向对首 token 延迟极其敏感的场景。
