Scrapling：网站改版也不怕的爬虫框架

Section titled “Scrapling：网站改版也不怕的爬虫框架”

你写好一个爬虫，调好 CSS 选择器，数据跑得很顺。两周后目标网站改了 HTML 结构，选择器全挂了，数据管道没输出。你重写一遍。再来一次。

写过爬虫的人都经历过这个循环。Scrapling 就是为了打破它而生的。

Scrapling 跟别的爬虫有什么不同

Section titled “Scrapling 跟别的爬虫有什么不同”

Scrapling 是 Karim Shoair 开发的 Python 爬虫框架，覆盖面很广——HTTP 请求、无头浏览器、反检测抓取、爬虫引擎、代理轮换、命令行工具，甚至还有给 AI 用的 MCP 服务。但真正让它与众不同的是自适应元素追踪。

原理是这样的：抓取页面时，Scrapling 会存储你选中元素的结构指纹。之后，如果网站改了 class 名称、挪动了元素位置、甚至换了整套 CSS 框架，你只要传入 adaptive=True，解析器就能通过结构相似度把原来的元素重新找到。不是猜的，是基于存储的指纹做比对。

传统爬虫把网站变化当成你的问题。Scrapling 把它当成框架该解决的工程问题。

三种抓取模式，对应不同场景

Section titled “三种抓取模式，对应不同场景”

Scrapling 提供三个抓取类，按需选用。

Fetcher —— 纯 HTTP 请求，没有浏览器开销。可以伪造浏览器 TLS 指纹，支持 HTTP/3。

StealthyFetcher —— 带反检测的无头浏览器。不用额外配置就能绕过 Cloudflare Turnstile。

DynamicFetcher —— 基于 Playwright 的完整浏览器自动化，适合 JS 渲染的页面。

和其他工具的对比

Section titled “和其他工具的对比”

对比 Scrapy

Section titled “对比 Scrapy”

Scrapy 是大规模爬虫的老牌选择。Scrapling 的爬虫 API 有意做得跟 Scrapy 很像，但多了自适应元素追踪、内置隐身浏览、更简单的会话管理。Scrapy 的中间件和扩展生态更成熟。新项目如果需要反检测和自适应解析，Scrapling 值得考虑。已有 Scrapy 管线的话，迁移成本不一定划算。

对比 BeautifulSoup

Section titled “对比 BeautifulSoup”

BeautifulSoup 只是解析器，不是框架。Scrapling 完全替代它，速度快 700 倍，还多了元素追踪、抓取和浏览器自动化。如果你现在用 requests + BeautifulSoup，换 Scrapling 是纯升级。

对比 Playwright / Selenium

Section titled “对比 Playwright / Selenium”

Playwright 和 Selenium 是浏览器自动化工具。Scrapling 的 DynamicFetcher 和 StealthyFetcher 底层用的就是 Playwright，但加了反检测、自适应解析和更高层的 API。不用写 Playwright 模板代码就能做浏览器自动化。

使用注意

Section titled “使用注意”
  • 需要 Python 3.10+，不支持更老的版本。
  • 浏览器依赖比较大。 scrapling install 会下载 Chromium 相关二进制文件，预留 ~500MB 磁盘空间。
  • 自适应追踪需要先跑一遍。 必须先用 auto_save=True 抓一次，adaptive=True 才能工作。没有基线就没有自适应。
  • 反检测是场持续的拉锯战。 StealthyFetcher 目前能绕过 Cloudflare Turnstile，但反爬系统在不断进化，保持库更新。
  • 尊重 robots.txt 和速率限制。 工具强大，用得负责。

最后

Section titled “最后”

多数爬虫工具把网站变化当成你的问题。Scrapling 把它当成框架的问题。自适应解析器、内置反检测、爬虫引擎、性能数据——都指向同一个方向：少花时间维护爬虫，多花时间用数据。

找个真实项目试试自适应追踪。抓一次页面，存好指纹，等网站改版，再跑一遍 adaptive=True。当它在改版后依然把元素找齐的那一刻，你就明白这个工具的价值了。

WebReel 是什么

Section titled “WebReel 是什么”

WebReel 是一个开源的命令行工具，通过 JSON 配置文件录制浏览器演示视频。你只需要描述”做什么”——点击哪里、输入什么、滚动到哪——剩下的交给它。

它的工作流程是这样的：启动一个无头 Chrome，打开你指定的 URL，按顺序执行每个步骤，以大约 60fps 截屏，在每帧上面叠加光标动画和按键提示，最后用 ffmpeg 编码输出。Chrome 和 ffmpeg 首次运行时会自动下载，你不用装任何额外的东西。

GitHub: vercel-labs/webreel 文档: webreel.dev 许可证: Apache 2.0 版本: 0.1.4 作者: Chris Tate（Vercel Labs）

常见问题

Section titled “常见问题”

需要自己装 Chrome 或 ffmpeg 吗？ 不用。WebReel 首次运行时会自动下载到 ~/.webreel。如果二进制文件损坏，用 webreel install --force 重新下载。

需要什么版本的 Node.js？ v18 或更高。

Ralph Wiggum vs Ralph Loop：如何选择适合你的 AI 代理模式

Ralph Wiggum vs Ralph Loop：如何选择适合你的 AI 代理模式

Section titled “Ralph Wiggum vs Ralph Loop：如何选择适合你的 AI 代理模式”

如果你在 Claude Code 社区待过一段时间，肯定听人讨论过 “Ralph Wiggum” 和 “Ralph Loop”，好像它们是完全不同的东西。这术语确实容易让人晕头转向。

其实很简单：两者都是 AI 代理的循环模式，但用途不同。一个会一直尝试直到成功。另一个跑完预定的流程就停。选错了会浪费你不少时间。

核心区别

Section titled “核心区别”

Ralph Wiggum 是一个持久的、会自我纠正的循环。它持续运行，检查自己的工作，调整方向，直到满足某个特定条件。你可以把它想象成一个死脑筋、绝不放弃的代理。

Ralph Loop 单次执行任务。你给它派活，它干完，停止。就这么简单。

这不是文字游戏。你的选择会影响怎么写提示词、需要多长时间、能不能得到想要的结果。

Ralph Wiggum：需要坚持的时候

Section titled “Ralph Wiggum：需要坚持的时候”

Ralph Wiggum 擅长那些能从迭代中受益的任务。代码重构、生成测试、合规扫描——任何代理能看自己之前的作品说”我看出刚才哪儿错了”的事情。

最近有个 Reddit 用户分享了用 Wiggum 构建 AI 合规扫描器的经历。扫描器遍历服务条款文档，经过 15 轮循环优化搜索条件，最终达到 92% 的准确率。这就是这种模式发挥作用：每轮都比上一轮更好。

另一个开发者用 Wiggum 现代化了一个老 Node.js 应用。代理用 20 轮迭代处理代码库，减少了 40% 的技术债务。不需要人指导每一步——只需要明确的完成标准和自由迭代的空间。

Wiggum 的特别之处

Section titled “Wiggum 的特别之处”

最核心的是完成承诺（completion promise）。你要明确告诉代理”完成”是什么样子：“100% 测试覆盖率”、“零语法错误”、“所有 PEP8 违规已修复”。没有这个锚点，循环会一直跑下去。

设置起来需要更多前期工作。你得仔细思考退出条件。正如一位开发者在 Reddit 上说的：“大多数人用 Ralph Wiggum 的方式不对——没有设置具体的退出条件，你的循环可能会永远跑下去。”

但配置得当的话，Wiggum 能处理那些需要持续人工监督的复杂任务。它发现过人工测试漏掉的支付网关边界情况。它用三周时间自动处理基础设施配置、成本优化和安全检查，构建了一个无服务器 SaaS。

时间上的取舍

Section titled “时间上的取舍”

Wiggum 不快。每次迭代需要 30-60 分钟。复杂任务可能需要多轮才能完成。但想想替代方案：你花几小时手动调试，或者为每个子任务写详细提示。

有时候慢速自主胜过快速干预。

Ralph Loop：速度和简洁

Section titled “Ralph Loop：速度和简洁”

当你的任务很直接——生成样板代码、更新 Dockerfile、格式化代码——Ralph Loop 5-10 分钟就能搞定。不需要复杂设置，不需要定义完成承诺，指哪打哪。

有用户用 Loop 在五分钟内自动化了整个项目的 Dockerfile 更新。另一个团队把它集成到 CI/CD 流水线生成无服务器函数脚手架，缩短了 60% 的上手时间。

Loop 在从起点到终点的路径很清晰时效果最好。你知道需要发生什么，代理只需要执行。

Loop 的局限

Section titled “Loop 的局限”

缺乏迭代精化限制了 Loop 处理复杂问题的能力。如果代理第一次尝试就出错，那个错误会一直存在。没有自我纠正的机制。

你也错失了 Wiggum 循环中出现的突发问题解决过程。Wiggum 代理可能在工作中途发现更高效的架构。Loop 会按原计划执行，即使有更好的方式存在。

如何选择

Section titled “如何选择”

实际框架是这样的：

选择 Ralph Wiggum 当：

  • 问题有明确成功标准，但路径不清楚
  • 多次尝试能改善结果
  • 你可以离开让它自己工作
  • 边界情况需要迭代发现
  • 任务需要测试驱动开发循环

选择 Ralph Loop 当：

  • 任务简单明确定义
  • 你需要快速结果
  • 手动监督不是问题
  • 工作流可预测且线性
  • 速度比优化重要

真实表现

Section titled “真实表现”

社区数据有些有趣的规律：

  • Ralph Wiggum：有明确完成标准时 85-98% 成功率
  • Ralph Loop：简单任务 95% 成功率

差异在于配置难度。Wiggum 的低成功率下限反映了设置良好退出条件的难度。Loop 的高成功率掩盖了它处理超出原始设计复杂性的无能。

安全措施

Section titled “安全措施”

不管选哪种模式，加上防护措施：

  1. 设置迭代上限 – 限制循环次数在 20-50 轮，防止无限循环
  2. 监控中间输出 – 用日志或仪表板跟踪进度
  3. 结合人工监督 – 让代理处理技术检查，最终评审留给专家

一家金融科技公司的开发者吃过亏——他们的 Wiggum 循环消耗了意料之外的 API 额度。现在他们总会设置最大迭代次数和成本警报。

混合方法

Section titled “混合方法”

很多团队发现两者结合效果最好：用 Loop 做搭建和脚手架，然后切换到 Wiggum 做迭代精化。一家创业公司就这样构建了整个产品——Loop 生成初始代码库，Wiggum 通过测试和优化循环精化它。

接下来是什么

Section titled “接下来是什么”

Claude Code 生态还在演进。开发者在实验跨模型工作流，把 Claude 和 Gemini 以及外部数据源结合。底层的模式不变：有时你需要坚持，有时你需要速度。

选对工具。未来的你会感谢自己。

杰文斯悖论：当 AI 实现完全自主软件开发，需求反而爆炸性增长

Section titled “杰文斯悖论：当 AI 实现完全自主软件开发，需求反而爆炸性增长”

我实现了 AI 100% 自主完成软件开发和交付，而且 AI 的执行效率爆炸性提升后，有意思的事情发生了。

从抱怨开发太慢到被追着要需求

Section titled “从抱怨开发太慢到被追着要需求”

以前，产品经理天天抱怨程序员开发效率太慢。一个功能需求提出去，十天半个月才有动静。催进度、开站会、写 JIRA，整个流程就像挤牙膏。

现在情况完全反转了。

AI 自主开发系统上线后，想法到实现的周期从周级压缩到小时级。产品经理、市场和销售团队突然发现——他们成了瓶颈。程序员开始追着他们要想法、要需求、要反馈。

杰文斯悖论在软件开发中的体现

Section titled “杰文斯悖论在软件开发中的体现”

杰文斯悖论很简单：技术进步往往会增加而不是减少对某种资源的需求，因为新的技术会创造出新的任务和需求，从而抵消了那些被技术替代的任务所节省的时间。

19 世纪，经济学家威廉·斯坦利·杰文斯发现：当煤炭使用效率提高时，煤炭消耗量没有下降，反而大幅增加。因为更高效的煤炭使用降低了成本，催生了更多用煤场景。

把这个逻辑套到 AI 自主软件开发上：

  • AI 让写代码变便宜了 → 更多想法值得实现
  • 开发速度提升了 → 市场验证更快了
  • 实现成本降低了 → 以前不值得做的功能现在值得做了

结果？对软件开发的需求不降反升。

瓶颈的转移：从后端到前端

Section titled “瓶颈的转移：从后端到前端”

AI 压力的转移路径很清晰：

第一阶段：后端程序员 最早受到影响的是写 CRUD、写 API 的后端开发。这部分工作高度模式化，AI 完成度高。

第二阶段：前端开发 UI 组件、页面搭建、状态管理，AI 也能自动生成。前端开发也开始感受到压力。

第三阶段：产品设计和需求定义 当实现不再是问题，问题变成了”要实现什么”。产品经理需要：

  • 更快地提出想法
  • 更快地验证假设
  • 更快地给出反馈

以前可以说”这个需求我们下个版本再看”，现在 AI 几分钟就能出原型，你没法再拖延了。

第四阶段：市场和销售 软件生产速度上来后，需要更多的人来推广、销售、客服。市场需求变成新的限制因素。

新的工作节奏

Section titled “新的工作节奏”

在这种环境下，工作方式被迫改变：

快速迭代成为常态

  • 上午提需求，下午看原型，第二天上线
  • AB 测试可以并行跑几十个版本
  • 用户反馈直接喂给 AI，自动迭代

需求质量更重要

  • 模糊的需求会让 AI 产生大量无效代码
  • 产品经理必须学会更精确地描述需求
  • 需求文档从”给程序员看”变成”给 AI 看”

验证节奏加快

  • 不能等完美了再发布
  • 快速发布、快速收集数据、快速迭代
  • 市场团队需要更高频的运营活动

谁会受到冲击

Section titled “谁会受到冲击”

产品经理 需要转变角色。以前更多是”翻译”业务需求给技术团队，现在需要：

  • 更强的业务洞察
  • 更快的市场判断
  • 更精准的需求表达
  • 数据分析能力（因为跑实验更快了）

市场和销售 软件供应增加后：

  • 需要更多渠道触达用户
  • 需要更精细的用户运营
  • 获客成本可能上升（竞争加剧）

创业者 好消息是想法更容易验证了。坏消息是：

  • 别人也能快速验证想法
  • 竞争加剧，速度优势不再持久
  • 需要在产品定位、品牌、生态上建立护城河

谁不会被替代

Section titled “谁不会被替代”

做决策的人 AI 可以生成选项，但选择还是要人来做。战略决策、权衡取舍、价值判断，这些都需要人。

理解用户的人 AI 可以分析数据，但深度理解用户需要洞察力和同理心。用户访谈、场景分析、痛点挖掘，这些仍需人来做。

定义问题的人 AI 擅长解决问题，但定义问题是什么、为什么要解决这个问题，这些需要产品思维和商业判断。

给从业者的建议

Section titled “给从业者的建议”

给产品经理

  • 学会和 AI 协作，把重复性需求文档工作交给 AI
  • 专注于战略定位、用户洞察、商业模式
  • 提升需求表达的精确度（AI 需要明确指令）
  • 建立快速验证的方法论

给技术人

  • 不要和 AI 比拼写代码速度
  • 转向架构设计、技术决策、系统思考
  • 学习使用 AI 工具，成为 AI 的驾驭者而非竞争者
  • 培养业务理解能力

给创业者和老板

  • 想法更容易实现了，但竞争也更激烈了
  • 速度不再是壁垒，差异化才是
  • 需要建立更深的护城河：品牌、网络效应、数据资产
  • 快速试错，但要有清晰的战略方向

反思：效率提升的代价

Section titled “反思：效率提升的代价”

AI 让软件开发变快了，但这带来了新的问题：

软件过载 当什么都能快速开发时，我们会开发太多东西。注意力成为稀缺资源，用户的时间被大量应用瓜分。

质量参差 快速生成的代码可能有技术债，快速上线的功能可能有体验问题。技术评审、质量把控变得更加重要。

同质化竞争 当大家都能快速实现时，实现本身不再是差异点。战略定位、品牌建设、用户体验这些”软实力”变得更加关键。

总结

Section titled “总结”

杰文斯悖论告诉我们：AI 不会减少对软件开发的需求，反而会激发更多需求。瓶颈从”怎么实现”转移到”实现什么”和”如何推广”。

在这个新时代：

  • 想法容易了，判断更难了
  • 实现快了，决策更重要了
  • 代码多了，价值更稀缺了

AI 自主开发不是职业的终结，而是工作方式的彻底重构。适应的人会发现，自己可以做更多有价值的创造性工作；不适应的人会发现自己被推着跑，永远跟不上节奏。

核心要点：当 AI 把实现成本降到接近零时，限制因素就从”能不能做”变成了”该不该做”和”怎么让别人知道你做了”。产品思维、市场能力、战略判断——这些变得越来越重要。

Section titled “简介”

什么是 Fabric？

Section titled “什么是 Fabric？”

Fabric 是一个开源框架，用于组织和管理 AI 提示词（称为 Patterns/模式），采用模块化、可复用的系统架构。与分散在各种工具和平台上的提示词不同，Fabric 提供了一种集中式、命令行优先的方法来收集、组织和执行 AI 解决方案。

Section titled “Fabric 背后的哲学”

“AI 不是一个事物；它是事物的放大器。而那个事物就是人类的创造力。”

Fabric 的哲学核心是将复杂问题分解为可管理的组件，并系统地将 AI 应用于每个部分。这种方法将 AI 从神秘的黑盒转变为实用的日常工具。
