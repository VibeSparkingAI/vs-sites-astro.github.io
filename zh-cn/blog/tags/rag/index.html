Claude Agent SDK 如何实现与上传文件的对话

2026年1月22日
Vibe Sparking

用户上传了一份员工评估表，然后问：“帮我总结这篇文章的内容”。

AI 回复：“我需要你提供文档才能帮你总结。”

文件明明上传成功了，状态显示 ready，embedding 也生成了。问题出在哪？

问题根源

Section titled “问题根源”

RAG 检索依赖向量相似度，默认阈值是 0.5。

用户说”总结文章”，但没提到”员工”、“评估”、“试用期”这些词。查询和文档之间的语义距离太远，相似度过不了阈值。

这是 RAG 的经典陷阱：用户指着眼前的东西说”这个”，但向量搜索不知道”这个”指的是什么。

解决方案

Section titled “解决方案”

绕过相似度搜索。用户既然明确指定了 attachment_ids，就直接按 ID 取内容。

新增直接检索方法

Section titled “新增直接检索方法”

逻辑很简单：用户指定了哪些文档，就取哪些文档的全部内容。不问相似度，不设阈值。

集成到聊天接口

Section titled “集成到聊天接口”

两种检索方式并行：

  • RAG 搜索：适合开放式问题，从知识库找相关内容
  • 直接检索：适合用户明确指定了文件的情况

Section titled “踩过的坑”

RAG-Anything：全能多模态 RAG 框架深度解析

2025年12月27日
Vibe Sparking

深入探索 HKUDS/RAG-Anything，这是一个基于 LightRAG 构建的综合性多模态文档处理 RAG 系统，能够在统一框架中处理文本、图像、表格和数学公式。

简介

Section titled “简介”

现代文档日益包含多样化的多模态内容——文本、图像、表格、数学公式、图表和多媒体——而传统专注于文本的 RAG（检索增强生成）系统无法有效处理这些内容。由 HKUDS 开发的 RAG-Anything 应运而生，这是一个基于 LightRAG 构建的综合性全能多模态文档处理 RAG 系统。

RAG-Anything 通过在单个集成框架中提供对所有内容模态的无缝处理和查询，消除了对多个专业工具的需求。这种统一的方法使其在学术研究、技术文档、财务报告和企业知识管理等需要统一处理框架的富混合内容场景中特别有价值。

最佳实践

Section titled “最佳实践”
  1. 根据内容类型匹配解析器：复杂布局使用 MinerU，Office 文档使用 Docling
  2. 仅启用所需功能：通过选择性启用图像/表格/公式处理减少开销
  3. 使用适当的查询模式：上下文特定用 local，模式发现用 global，平衡结果用 hybrid
  4. 批量处理相似文档：使用 process_folder_complete 提高效率
  5. 尽可能预解析：对缓存/外部解析结果使用 insert_content_list
