BMAD 最佳实践:AI 驱动的敏捷开发指南

2026年2月13日

BMAD 最佳实践:AI 驱动的敏捷开发指南

Section titled “BMAD 最佳实践:AI 驱动的敏捷开发指南”

简介

Section titled “简介”

欢迎了解 BMAD(Building Methodologies with AI Drivers,AI 驱动的构建方法论)- 一种将 AI 能力集成到开发生命周期每个阶段的 AI 驱动敏捷软件开发方法论。BMAD 提升了不同 AI 模型的开发基准,使团队能够在保持质量的同时更快交付。

BMAD 的独特之处在哪里?

  • AI 优先架构:从头开始为 AI 辅助开发而构建
  • 灵活工作流:从快速迭代到企业级流程
  • 活文档:持续更新,而非静态产物
  • 质量自动化:自动化测试、代码审查和验证
  • 可衡量成果:通过故事点和冲刺状态跟踪进度

Section titled “前置条件”
  • AI 编码助手:Claude Code、Cursor 或兼容的 AI 工具

    • 推荐:Claude Opus 4.5(200K 上下文)
    • 替代方案:Gemini 3 Pro(分析)+ Opus 4.5(实现)
    • 国内:GLM 4.7(200K)或 MiniMax(192K)
  • 项目结构:现有代码库或新项目

Section titled “1. 快速开发(/bmad:bmm:workflows:quick-dev)”

工作流程:

  2. 上下文收集:识别文件、模式、依赖
  3. 执行:持续实现任务
  4. 自检:验证验收标准
  5. 收尾:提交、文档、通知

何时使用:

  • ✅ 专注功能(5-10 个用户故事,15-50 故事点)
  • ✅ 范围明确的 bug 修复
  • ✅ 已知模式的快速迭代

2. 规范驱动开发(/bmad:bmm:workflows:quick-spec)

Section titled “2. 规范驱动开发(/bmad:bmm:workflows:quick-spec)”

Section titled “3. 完整 BMM 流程(企业级)”

**适用于:**大型史诗、平台工作、架构变更

完整生命周期:

  • 启动:/bmad:bmm:workflows:workflow-init
  • 分析:项目简报、需求收集

  • 开发:故事实现与测试
  • 测试:扩展自动化测试覆盖
  • 代码审查:对抗式审查流程
  • 验收:手动抽查 + 自动验证
  • 回顾:史诗后学习捕获

用户故事要求:INVEST 原则

Section titled “用户故事要求:INVEST 原则”

BMAD 中的每个用户故事都必须遵循 INVEST 原则:

原则含义示例
Independent(独立)可以独立开发”添加密码重置”不依赖”添加双因素认证”
Negotiable(可协商)细节可以讨论”快速搜索” → 协商”快速”的定义
Valuable(有价值)提供用户/业务价值”用户可以导出数据” > “重构导出模块”
Estimable(可估算)可以估算复杂度足够清晰以分配故事点
Small(小)适合一次迭代可在 1-3 天内完成
Testable(可测试)明确验收标准”搜索在 10k 记录下 2 秒内返回结果”

故事点量表:

  • 1-3:简单功能,清晰实现
  • 5-8:中等复杂度,部分未知
  • 13+:拆分为更小故事

单角色模型(OPE - 单人实体)

Section titled “单角色模型(OPE - 单人实体)”

  • 效率最大化
  • 无协调开销
  • 决策迅速

1. 保持 UI/UX 一致性

Section titled “1. 保持 UI/UX 一致性”

方法:

方案 A:前期设计

  • 专业设计师创建草稿/原型
  • 首先用模拟数据实现 UI 代码
  • 稍后集成真实数据
  • 专注于业务价值,而非像素完美

方案 B:使用设计系统

  • 提供清晰的 UI/UX 指南和标准
  • 展示好坏设计示例
  • 使用 ui-ux-pro-max 或 frontend-design 等技能

方案 C:迭代改进

  • 快速构建功能版本
  • 基于用户反馈改进
  • ROI 驱动的改进

2. 控制范围和复杂度

Section titled “2. 控制范围和复杂度”

每次迭代限制:

  • 用户故事:5-10 个故事
  • 故事点:总计 15-50 点
  • 持续时间:1-2 周

为什么重要:

  • 适合 AI 上下文窗口
  • 减少认知负荷
  • 实现快速转向
  • 更高完成率

升级信号:

  • 提及多个组件(仪表板 + API + 数据库)
  • 系统级语言(平台、集成、架构)
  • 方法不确定
  • 多层范围(UI + 后端 + 数据)

响应:

  • BMAD 自动检测复杂度
  • 建议适当工作流
  • 执行前提供规划

4. 适应 AI 驱动开发

Section titled “4. 适应 AI 驱动开发”

**关键洞察:**BMAD 无法保证一次迭代内 100% 自动化所有需求。

成功因素:

  • 控制规模:每次迭代 5-10 个故事
  • 管理复杂度:15-50 故事点
  • 提高适应性:学习 AI 的优势/局限
  • 迭代:多个周期胜过完美计划

现实检查:

项目类型预期自动化需要人工决策
CRUD 应用80-90%业务规则、UX 细节
API 服务70-80%架构选择、安全
数据管道60-70%转换逻辑、边缘情况
平台工作40-60%设计模式、权衡

  • 专注于逻辑、安全、可维护性

安装与设置

Section titled “安装与设置”

问:BMAD 应该全局安装还是按项目安装?

答:**推荐按项目安装。**这可以隔离依赖、配置和工作流。全局安装可能导致冲突。

问:我应该使用哪个 AI 模型?

答:**模型能力和上下文长度最重要。**BMAD 提升各模型的基准:

  • 国际:Opus 4.5(200K)或 Gemini 3 Pro + Opus 4.5
  • 国内:GLM 4.7(200K)或 MiniMax(192K)

工作流选择

Section titled “工作流选择”

问:如何在 quick-dev 和完整 BMM 流程之间选择?

答:使用此决策树:

  1. 单文件/组件,方法清晰 → 快速开发(直接)
  2. 多组件,需要规划 → 快速规范
  3. 平台工作,许多未知 → 完整 BMM
  4. “应该如何…”或”最佳方式…” → 从规划开始

问:我可以在项目中途切换工作流吗?

答:可以!BMAD 工作流可组合:

  • 从 quick-dev 开始,需要时升级到 quick-spec
  • 开始完整 BMM,对小的后续任务使用 quick-dev
  • 根据任务复杂度混合工作流

需求与故事

Section titled “需求与故事”

问:我可以修改已完成的史诗或故事吗?

答:**不推荐。**将它们作为历史记录维护:

  • 明确标记过时部分
  • 为变更创建新故事
  • 链接到原始故事以获取上下文
  • 活文档 ≠ 可变历史

问:如果规划后需求发生变化怎么办?

答:使用课程修正工作流:

这有助于在不失去动力的情况下处理重大变更。

团队结构

Section titled “团队结构”

问:BMAD 适合独立开发者吗?

答:适合!明确支持单角色模型:

  • AI 充当队友(分析师、架构师、测试员)
  • 人类专注于决策和创意方向
  • 比传统独立开发更快
  • 风险:易倦怠

问:BMAD 如何帮助团队协作?

答:BMAD 减少协调开销:

  • 明确的角色定义和检查点
  • _bmad-output/ 中的共享产物
  • 阶段间自动化交接
  • 只有人工决策需要会议

范围与局限

Section titled “范围与局限”

问:BMAD 能自动实现所有需求吗?

答:无法保证,但 BMAD 显著加速开发:

  • 成功取决于:规模、复杂度、团队适应性
  • 控制范围:每次迭代 5-10 个故事、15-50 点
  • 迭代改进胜过瀑布式规划
  • AI 处理常规工作;人类处理判断调用

问:BMAD 适合操作系统、嵌入式或固件开发吗?

答:BMAD 针对兼容敏捷的项目进行了优化:

  • Web/移动应用:✅ 非常适合
  • API 和服务:✅ 非常适合
  • 数据管道:✅ 适合
  • 桌面应用:✅ 适合
  • 嵌入式/固件:⚠️ 有限(如果不兼容敏捷)
  • 操作系统内核工作:⚠️ 有限

开发工作流

Section titled “开发工作流”

问:如何在现有项目中开发新功能?

答:三种方法:

  1. 快速功能:/bmad:bmm:workflows:quick-dev "添加功能 X"
  2. 计划功能:/bmad:bmm:workflows:create-story 然后 /bmad:bmm:workflows:dev-story
  3. YOLO 模式:无规划直接执行(实验性)

问:如何确保 UI/UX 一致性?

答:多种策略:

  • 使用 /bmad:bmm:workflows:create-ux-design 进行显式设计阶段
  • 预先提供详细的设计草稿/原型
  • 建立带示例的清晰 UI/UX 指南
  • PO/BA/设计师直接用模拟数据实现 UI
  • 专注于 ROI,而非像素完美细节
