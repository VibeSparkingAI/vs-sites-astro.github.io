Section titled “终端隧道：用 tty-share 随处分享你的终端”

是否曾经需要与同事共享终端会话，进行结对编程、远程调试或协作排查问题？tty-share 是一款极其简洁的工具，让你能够在 Internet 上共享 Linux 或 macOS 终端，而远程参与者无需任何设置。

为什么选择 tty-share？

Section titled “为什么选择 tty-share？”
  • 零远程设置：加入的人无需安装任何工具——可直接从浏览器加入
  • 单一二进制文件：用 Go 编写，无依赖——可在 Linux、macOS 甚至 Raspberry Pi 上运行
  • 完全开源：代码库完全透明，可在 GitHub 上查看
  • 安全连接：机器与中继服务器之间的 TLS 加密通信
  • 灵活共享：可通过公共 Internet 或仅限本地网络共享
  • 只读模式：可选择创建只读会话用于演示

工作原理

Section titled “工作原理”

当你通过 Internet（NAT 外部）共享终端时，tty-share 会连接到代理服务器，该服务器协调参与者之间的通信。官方服务器运行在 tty-share.com，但你也可以托管自己的服务器以获得完全控制权。

随着 AI 代理编写代码的能力日益增强，下一个逻辑步骤就是让它们能够与真实的网页进行交互。Dev Browser 是一个开源的 Claude Code 技能，它正是为此而生——并且在每次调用之间保持浏览器的所有状态。

为什么 Dev Browser 如此重要

Section titled “为什么 Dev Browser 如此重要”

Dev Browser 解决了 AI 代理能力中的一个关键缺口：持久化浏览器自动化。传统的浏览器自动化工具要么一次执行一个操作（慢），要么运行每次都从头开始的完整脚本（脆弱）。Dev Browser 采用了一种不同的方法。

核心特性

Section titled “核心特性”
  • 持久化页面 – 导航到页面一次后将其保存在内存中。无需为每个运行的脚本重新加载 DOM。
  • 灵活执行 – 可以一次性运行完整脚本，或在探索时逐步执行操作。
  • 对 LLM 友好的 DOM 快照 – 结构化、轻量级的页面表示，Claude 可以快速解析。
  • Chrome 扩展 – 控制现有的 Chrome 配置文件，利用已登录的会话、Cookie 和扩展。

这些特性为 Claude Code 代理在测试、数据提取和交互式演示方面提供了实际优势。

  4. 启用”开发者模式”（右上角切换开关）

Section titled “第一部分：三种核心团队管理模式”

1. 主管模式（Supervisor Pattern）- 中心化控制

Section titled “1. 主管模式（Supervisor Pattern）- 中心化控制”

这是最直观、也最常见的模式。想象一个项目经理协调所有工作：一个”主管”代理接收任务，将其分解为多个子任务，分配给不同的”工作”代理，然后监督、验证并最终整合所有结果。

工作流程：用户请求 → Manager分解 → 分配给Sub-Agent → S-Agent执行 → Manager验证 + 综合 → 最终输出

优缺点：优点是控制流程清晰，易于调试和审计。缺点是主管容易成为性能瓶颈，所有协调步骤都是串行的，增加了延迟和Token成本。

2. 流水线模式（Sequential / Pipeline）

Section titled “2. 流水线模式（Sequential / Pipeline）”

这种协作方式的核心是，Sub-Agent按照固定顺序执行，前一个的Output是后一个Sub-Agent的Input。

工作流程：这一编排模式就像人类社会中的”流水线工厂”，后者的输入等于前者的输出。

优缺点：优点是具有极高的稳定性，而且易调试；缺点也同样明显——环节卡死则全线崩溃。

Cursor 团队的 Agent 编程最佳实践：8 大核心原则

2026年1月16日
Vibe Sparking

引言

Section titled “引言”

Cursor 团队最近发表了一篇优秀的博客文章，分享了他们使用 AI Agent 编程的最佳实践。随着 AI 驱动的开发工具成为主流，理解如何有效地与 AI Agent 协作，正成为开发者必备的核心技能。

这些见解不仅适用于 Cursor，对于任何使用 Claude Code、GitHub Copilot 或类似 AI 编程助手的开发者都极具价值。

8 大核心原则

Section titled “8 大核心原则”

1. 先计划再动手

Section titled “1. 先计划再动手”

让 Agent 先列出步骤、提出问题、确认思路，然后再写代码。虽然这看起来会更慢，但实际上更快。

核心洞察： 如果做出来的结果不对，与其反复修补，不如回退代码、修改计划、重新执行一遍。

2. 上下文别给太多

Section titled “2. 上下文别给太多”

Agent 自己会搜索信息，不需要手动塞一堆文件进去。事实上，提供太多上下文反而会干扰 Agent 的判断。

  • 如果知道具体需要哪些文件，直接指定
  • 如果不确定，让 Agent 自己去找

3. 长对话会失焦

Section titled “3. 长对话会失焦”

换任务时开新对话。如果发现 Agent 反复犯同样的错误，也重新开始。

提示： 可以引用历史工作，不需要复制粘贴所有内容。

4. 规则按需加

Section titled “4. 规则按需加”

只在 Agent 反复犯同一个错误时才写规则，不要提前优化。

保持规则精简，引用文件而不是复制内容。

5. 给可验证的目标

Section titled “5. 给可验证的目标”

写测试、配置 linter、使用类型语言。Agent 需要明确的对错信号才能有效迭代。

TDD 流程特别适合 Agent： 先写测试，再让 Agent 写代码直到测试通过。

6. 图片比文字快

Section titled “6. 图片比文字快”

对于 UI 问题或设计规范，截图和设计稿比文字描述高效得多。

7. Prompt 要具体

Section titled “7. Prompt 要具体”

不要说：“给 auth.ts 加测试”

试试：“给 auth.ts 写测试覆盖 logout 边界情况，使用现有测试模式，不要使用 mock”

指令越具体，结果越好。

8. 认真审查

Section titled “8. 认真审查”

AI 生成的代码可能看起来正确但包含微妙错误。Agent 跑得越快，审查就越重要。

实践方法： 结合 Review Agent 使用。可以使用 Cursor 内置的 Review → Find Issues 功能，或集成外部审查工具。

核心理念

Section titled “核心理念”

把 Agent 当作有能力的协作者，而不是代码生成器。 给它计划、目标、反馈，它才能真正发挥作用。

这些技巧不仅适用于 Cursor，也适用于任何 Agent 工具。

推荐工作流程

Section titled “推荐工作流程”
  1. 规划阶段： 让 Agent 列出实施方案
  2. 目标设定： 定义可验证的成功标准（测试、类型）
  3. 迭代开发： Agent 写代码，你审查
  4. 纠偏调整： 需要时回退并修改计划
  5. 最终审查： 仔细检查生成的代码

结语

Section titled “结语”

软件开发的未来是人机协作。通过遵循这些最佳实践，你可以在保持代码质量的同时，大幅提升开发效率。

记住：Agent 是强大的工具，但你仍然是架构师。你的角色从编写每一行代码，转变为提供愿景、反馈和验证。
